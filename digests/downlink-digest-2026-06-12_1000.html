Overview

Tenet Security researchers described Agentjacking as a new class of attack that can trick artificial intelligence coding agents into running arbitrary code on developer machines. The attack can be triggered using a fake Sentry error report, crafted through Sentry, an open-source error-tracking and performance-monitoring platform.

The researchers said the attack exploits a critical architectural flaw at the intersection of Sentry’s event ingestion and the Sentry MCP server. In their description, Sentry’s event ingestion accepts arbitrary payloads from anyone with the DSN, while the Sentry MCP server returns that data to AI agents as trusted system output.

Core Technical Issue

The article describes the problem as rooted in implicit trust when AI agents connect to external services using Model Context Protocol, or MCP. Because an AI agent cannot distinguish between an error event generated by a real application crash and an event injected by an attacker, the agent may treat attacker-controlled content as legitimate diagnostic information.

Tenet Security said the attack uses crafted input in Sentry error events. Those events are then interpreted by coding agents such as Claude Code and Cursor as legitimate diagnostic resolution steps. If the agent follows those steps, it can run attacker-controlled code.

Attack Chain

The attack chain described by Tenet Security includes the following steps:

  • An attacker finds a target’s Sentry Data Source Name, or DSN, which the article describes as a public, write-only credential embedded in websites.
  • The attacker sends a malicious error event to Sentry’s ingest endpoint through a POST request using the DSN.
  • The injected event includes carefully formatted markdown in the message field and context key names.
  • When the Sentry MCP server returns the event to an AI agent, the content is rendered as structured material that appears visually identical to Sentry’s system template.
  • When a developer asks an AI coding agent to fix unresolved Sentry issues, the agent queries Sentry through MCP and receives the malicious event.
  • The agent executes malicious code with the developer’s full privileges.

The researchers emphasized that the attacker does not need to touch the victim’s infrastructure. The malicious instruction arrives as part of an ordinary error event, disguised as a legitimate resolution. When the developer asks the AI agent to fix the Sentry issue, the agent reads the attacker’s command as trusted guidance and runs it on the developer’s machine.

Impact

The article says a successful Agentjacking attack can expose sensitive information. Examples listed include environment variables, Git credentials, private repository URLs, and developer identities.

The article also notes that the attack does not rely on phishing or prior server compromise. Instead, it uses data that organizations publish about themselves, specifically the Sentry DSN, and the trust relationship between MCP-connected external services and AI coding agents.

Affected Tools and Stakeholders

The article specifically names AI coding agents including Claude Code and Cursor as examples of tools that can interpret the injected Sentry events as legitimate diagnostic guidance. Sentry is central to the described attack path because the fake error report is sent through Sentry ingestion and returned through the Sentry MCP server.

Tenet Security is the research organization that described the attack. The article also states that Tenet found at least 2,388 organizations exposed with valid injectable DSNs. Tenet said it tested the attack in a controlled manner against more than 100 organizations.

Reported Testing Results

Tenet Security said it achieved an 85% exploitation success rate against injected errors across some of the most widely used AI coding assistants. The article does not name a threat actor exploiting the technique in the wild. It presents the activity as research conducted by Tenet Security.

Sentry’s Response

Sentry acknowledged the issue, according to the article, but opted not to fix it. The article quotes Sentry as saying the issue is “technically not defensible.”

The article also says Sentry activated a global content filter that blocks a specific payload string. No broader remediation from Sentry is described in the article.

Significance

Tenet Security said Agentjacking stands out because it targets the AI agent that a developer trusts and uses a Sentry DSN as a starting point. The markdown injection is rendered in a way that the agent cannot distinguish from legitimate Sentry guidance.

The article concludes that AI coding agents are becoming part of the attack surface. Tenet said the attack can bypass EDR, WAF, IAM, VPN, Cloudflare, and firewalls because every action in the chain is authorized.

Summary
Researchers found hundreds of real-world RSA and DSA keys with structured zero-bit patterns that made them quickly recoverable, including keys generated by vulnerable versions of CompleteFTP. The flaw came from a big-integer type mismatch in CompleteFTP’s key generation code, and affected users should check and regenerate their keys.

Trail of Bits and Hanno Böck of the badkeys project found hundreds of real-world RSA keys whose bits were heavily biased toward 0 and whose private keys could be quickly recovered. The keys showed structured blocks of zero bits, which the researchers call “short-sleeve” patterns because only part of each big-integer limb contained random-looking data. They recovered private keys for 603 unique RSA public keys and 74 DSA keys generated by vulnerable CompleteFTP versions, plus 26 RSA keys with an unidentified short-sleeve pattern.

The researchers developed a polynomial-based cryptanalytic technique that exploits the repeated zero-bit structure in RSA moduli. By representing the modulus as a polynomial with small coefficients and factoring that polynomial, they could recover the RSA factors in cases where the private factors had the same short-sleeve structure. This approach does not work against correctly generated RSA keys because their prime factors typically do not produce such small-coefficient polynomials.

One identified cause was a type mismatch in old CompleteFTP big-integer key-generation code from EnterpriseDT. The vulnerable function copied 8-bit random values into 32-bit big-integer limbs, leaving large portions of each limb as zeros. EnterpriseDT released CompleteFTP v26.1.0 and a standalone KeyChecker tool to detect vulnerable RSA or DSA keys and alert users to regenerate them.

Trail of Bits, working with Hanno Böck of the badkeys project, identified real-world RSA public keys with unusually sparse moduli and structured blocks of zero bits. These “short-sleeve” keys appeared in two main patterns, both involving regularly spaced zero blocks interleaved with apparently random data. The researchers found that many of these keys could be factored quickly, allowing recovery of corresponding private keys.

The analysis was based on a large corpus of public keys collected by badkeys from sources including Certificate Transparency logs, internet-wide TLS and SSH scans, PGP keys, and other public sources. Pattern 1 appeared in Certificate Transparency logs for certificates issued to organizations including Yahoo and Verizon and on some devices running NetApp software, but its underlying cause remains unexplained. Pattern 2 appeared on SSH hosts running EnterpriseDT’s CompleteFTP software.

The main technical contribution was a polynomial-based factorization method for RSA moduli with short-sleeve structure. The researchers represented integers using base values corresponding to limb sizes, turning them into polynomials whose coefficients were small when the underlying limbs contained large zero regions. Factoring the polynomial could reveal polynomial representations of the RSA prime factors, which could then be evaluated back into integers to recover the private key.

For CompleteFTP, the researchers reverse-engineered the software and found a bug in a big-integer random-bit generation function. The function allocated one byte of random output per 32-bit limb and used Array.Copy in a way that implicitly cast each 8-bit random value into a 32-bit limb, leaving most bits in each limb unset. The article states that RSA keys generated by CompleteFTP versions 10.0.0 through 12.0.0 and DSA keys generated by versions 10.0.0 through 23.0.4 were affected.

The researchers recovered private keys for 603 unique RSA public keys and 74 DSA keys generated by vulnerable CompleteFTP versions, as well as 26 RSA keys with the unidentified short-sleeve pattern. EnterpriseDT responded during disclosure and released CompleteFTP v26.1.0, which checks for vulnerable RSA or DSA keys and alerts users if regeneration is needed. EnterpriseDT also released a standalone KeyChecker tool, and badkeys added support for detecting vulnerable short-sleeve RSA keys.

Overview

Trail of Bits and Hanno Böck of the badkeys project investigated RSA public keys whose moduli showed an unusual bias toward zero bits. The researchers found hundreds of unique keys in the wild that not only had detectable zero-bit structure but could also be quickly factored. They refer to these as “short-sleeve” keys because the zero bits do not fully cover the limbs of the big integers; instead, only a small contiguous part of each limb contains random-looking bits.

The work combined large-scale public-key discovery, cryptanalysis, and reverse engineering. The researchers identified two recurring short-sleeve patterns in real-world RSA moduli. One pattern remains unexplained, while the other was traced to a type mismatch in old CompleteFTP big-integer key-generation code.

Discovery and affected key patterns

The badkeys project collects public keys from sources such as Certificate Transparency logs, internet-wide TLS and SSH scans, PGP keys, and other public datasets. By searching this dataset for unexpectedly sparse RSA moduli, the researchers found keys with regularly spaced blocks of zeros. These blocks appeared between sections of apparently random data.

The article describes two main patterns:

  • Pattern 1 appeared in Certificate Transparency logs for certificates issued to several large organizations, including Yahoo and Verizon, and on some devices running NetApp software.
  • Pattern 2 appeared on SSH hosts running EnterpriseDT’s CompleteFTP software.

The certificates associated with Pattern 1 had already expired, and the researchers shared findings with the companies involved. However, they did not identify the product or implementation responsible for generating Pattern 1 keys. Pattern 2 was traced to CompleteFTP.

Polynomial-based RSA factorization technique

The researchers exploited the mathematical structure created by repeated zero-bit blocks in big-integer limbs. Big integers are commonly represented as arrays of smaller machine-sized values called limbs. For the observed short-sleeve RSA keys, interpreting the modulus using the relevant limb size revealed that each limb contained a small random-looking region and a larger zero-filled region.

The article explains that an integer can be represented as a polynomial by treating the digits in its base-B representation as polynomial coefficients. In this representation, an integer such as a = sum a_i B^i corresponds to a polynomial f_a(x) = sum a_i x^i, and evaluating f_a(B) converts the polynomial back into the integer.

For short-sleeve RSA moduli, choosing a base corresponding to the limb size produces polynomials with unusually small coefficients. If the hidden RSA prime factors p and q also have this short-sleeve structure, their polynomial representations can multiply to the polynomial representation of n. Because factoring polynomials over the integers is easy compared with factoring general RSA moduli, this can reveal polynomial forms of p and q, which can then be evaluated back into integer factors.

The article emphasizes that this does not break properly generated RSA keys. For correctly generated prime factors, the polynomial coefficients are typically full limb-sized values, so the small-coefficient structure needed for the attack is absent.

CompleteFTP root cause

The researchers reverse-engineered CompleteFTP after observing that the second short-sleeve pattern appeared on SSH hosts running that software. Dynamically generated RSA keys in the latest version did not show the vulnerable pattern, so they decompiled the .NET code with ILSpy and investigated older key-generation logic.

They found a vulnerable function called genRandomBits that was intended to fill a big integer with random bits of a requested length. The function calculated the number of 32-bit limbs, allocated a byte array of that length, filled that byte array with nonzero random bytes, and copied the array into the big-integer limb array.

The bug was a mismatch between the size of the random output and the size of the limbs. Each limb required 32 bits of randomness, but the code generated only one 8-bit value per limb. The Array.Copy behavior implicitly cast each 8-bit array element into its own 32-bit limb element, leaving most bits in each limb as zeros.

This behavior matched the observed CompleteFTP short-sleeve key pattern. The researchers also determined that the vulnerable function was compiled into newer versions but was unreachable there, because later versions had refactored key generation to use standard .NET cryptographic APIs.

CompleteFTP RSA and DSA impact

The article states that CompleteFTP RSA key generation was affected in versions 10.0.0 through 12.0.0, covering December 2016 through March 2019. DSA key generation was affected in versions 10.0.0 through 23.0.4, covering December 2016 through December 2023.

The researchers found that DSA key generation also used the vulnerable random-bit function. In that case, the 160-bit DSA private key x was generated with the flawed function, and because the public key and parameters include g and y = g^x, the private key could be recovered using baby-step giant-step. The article notes that Diffie-Hellman key exchange also used the vulnerable function with a 2048-bit exponent, but the researchers believe those DH exchanges are still cryptographically secure.

In total, the researchers recovered:

  • 603 unique RSA private keys generated by vulnerable CompleteFTP versions.
  • 74 DSA private keys generated by vulnerable CompleteFTP versions.
  • 26 RSA private keys with the unidentified short-sleeve Pattern 1.

The article notes that the researchers’ data sources are heavily biased toward RSA SSH keys, so these numbers do not reflect actual prevalence.

Historical trend and containment

The researchers used historical and contemporary SSH scan data from Prof. Nadia Heninger’s collection to examine how the CompleteFTP issue changed over time. The data showed a consistent increase in hosts with vulnerable keys after the RSA vulnerability was introduced in December 2016. After CompleteFTP’s rewritten RSA code was released in March 2019, the trend stopped immediately.

However, the proportion of affected keys later plateaued. The article explains this as consistent with customers who update their software regularly but generate host keys only once. This means that even after vulnerable software versions decline, old vulnerable keys can remain deployed.

EnterpriseDT was described as responsive during disclosure. On May 8, 2026, it released CompleteFTP v26.1.0, which automatically checks whether the system is using a vulnerable RSA or DSA key and alerts the user if regeneration is needed. EnterpriseDT also released a standalone KeyChecker tool, and the badkeys website and standalone tool added detection for vulnerable short-sleeve RSA keys.

Other implementation issues mentioned

The article also mentions a separate CompleteFTP RSA key-generation issue on Linux. In that issue, the private exponent was set to 65537 and the public exponent was large. Trail of Bits disclosed it, and it was fixed in CompleteFTP v26.0.2.

According to the article, EnterpriseDT believes no production users were affected by the Linux issue, based on license data. The researchers’ scans supported that claim because they found no keys in the wild with that property.

Significance

The article’s central finding is that structured randomness failures in cryptographic key generation can produce public keys that are not only detectable at scale but also practically breakable. The short-sleeve structure created by biased big-integer limbs enabled a specialized polynomial factorization technique to recover RSA private keys.

The work also shows that similar failures appeared in independent cryptographic implementations. Pattern 1 remains unexplained, and Pattern 2 was traced to CompleteFTP, indicating that this type of implementation mistake may not be isolated to a single product. The article argues that observing real-world cryptographic failures can inspire more capable cryptanalytic algorithms, which in turn can uncover additional vulnerable systems.

  • The same class of biased-limb key generation failure appeared in independent RSA implementations, suggesting that scanning for structured zero-bit patterns can uncover additional weak keys beyond the known CompleteFTP cases.
  • CompleteFTP’s RSA key generation refactor in March 2019 stopped new vulnerable RSA keys from appearing, but many hosts continued using old vulnerable keys after software updates because keys were generated only once.
  • The public key bit patterns were detectable at internet scale before factoring, enabling projects like badkeys to identify vulnerable short-sleeve RSA keys from public datasets.
  • The CompleteFTP DSA issue persisted much longer than the RSA issue, affecting DSA keys generated through December 2023 even though RSA generation had been fixed years earlier.
  • The vulnerable CompleteFTP code remained compiled into newer versions but was unreachable, showing why reverse engineering older versions was necessary to identify the historical key-generation bug.

01 AI Agents, SaaS Security, and Automated SOC Claims

Tenet Security researchers described “Agentjacking,” a class of attack in which AI coding agents can be induced to execute arbitrary code on developer machines. The reported chain abuses a public Sentry DSN to submit a malicious error event containing crafted markdown; when a developer asks tools such as Claude Code or Cursor to resolve Sentry issues through an MCP server, the agent may interpret attacker-supplied content as trusted diagnostic guidance and run code with the developer’s privileges.

The researchers said they identified at least 2,388 organizations with valid injectable DSNs and observed an 85% exploitation success rate in controlled testing across widely used AI coding assistants. The finding reflects a broader theme in this window: AI-enabled workflows are increasingly operating across sensitive trust boundaries, inheriting user permissions, and acting autonomously on data from external systems.

02 Abuse of Trusted Business Platforms and Sensitive Data Exposure

IRONSCALES reported an invoice phishing case in which an attacker used FedEx Freight’s Salesforce CRM to send a rebill request to a logistics and distribution company. The message had no malicious links or attachments, was sent via Salesforce Quick Action, carried a valid DKIM signature for fedexfreight.com, passed DMARC, and returned compauth=100, while a downstream Securence relay introduced an SPF softfail because the relay IP was not in the original sender’s SPF record.

The payload was purely textual: a request to change the bill-to entity on freight invoice #0584162950. Themis scored the message at 83% confidence as invoice phishing and four mailboxes were quarantined, illustrating how business email compromise-style activity can leverage legitimate CRM infrastructure and authenticated sender identity rather than conventional malware delivery.

Novo Nordisk separately disclosed that attackers accessed internal IT systems and copied non-public clinical trial data. The exposed information included pseudonymized patient IDs, trial participation details, sex, year of birth, biomarkers, health and immunogenicity data, and lifestyle factors such as smoking, alcohol use, and BMI; the company said the data was not directly linked to patient names or other direct identifiers. Information for healthcare professionals was also exposed, including names, registration numbers, email addresses, phone numbers, WhatsApp details, and office locations, while Novo Nordisk said compromised systems were taken offline and core business operations were not impacted.

03 Cryptographic Weaknesses and Windows Update Deployment Issues

Trail of Bits reported a polynomial-based cryptanalytic technique for factoring RSA moduli with structured, regularly spaced zero-bit patterns. Researchers found hundreds of unique RSA public keys in real-world datasets whose corresponding private keys could be recovered quickly, including keys generated by vulnerable CompleteFTP versions affected by a big-integer type mismatch in which 8-bit random values were copied into 32-bit limbs during key generation.

CompleteFTP versions 10.0.0 through 12.0.0 generated vulnerable RSA keys, while versions 10.0.0 through 23.0.4 generated vulnerable DSA keys. The researchers recovered private keys for 603 unique RSA public keys and 74 DSA keys generated by affected versions, and EnterpriseDT released updated detection and checking tools.

Microsoft fixed a known issue affecting Windows updates released since May 2025 when installed using the Windows Update Standalone Installer from a network share. The problem affected Windows 11 24H2/25H2 and Windows Server 2025 enterprise systems, producing ERROR_BAD_PATHNAME in scenarios involving WUSA or double-clicked .msu files from network shares containing multiple .msu files; Microsoft addressed it in June 2026 cumulative updates KB5079391 for Windows 11 and KB5094125 for Windows Server 2025.
