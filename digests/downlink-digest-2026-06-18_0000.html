In recent years, SocGholish has used the fake update flow to trick victims into executing a custom JScript file on their endpoint. Domain Shadowing

The article identifies domain shadowing as a key SocGholish technique for both tier one and tier two hostnames. Domain shadowing occurs when threat actors gain access to the authoritative DNS provider or registrar account for a legitimate domain and create rogue subdomains beneath it.

  • Malicious hostnames inherit the age and reputation of the legitimate parent domain.
  • Defenders may initially assume a hostname under an established business domain is legitimate.
  • New subdomains are often monitored less closely than newly registered domains.
  • Rapid rotation reduces the usefulness of static blocklists.

SocGholish rotates shadowed tier two domains at high frequency, from daily rotation to at least once per week. The article says this shortens the time defenders have to identify and investigate rogue subdomains and lets the actor refresh trusted-looking command-and-control infrastructure before many defenders can classify and respond to it.

Overview

Cloudflare describes how it built a model-agnostic vulnerability research harness for scanning enterprise codebases. The article’s central argument is that the future of agentic security workflows is not in standalone frontier models, prompts, or single-agent sessions, but in durable orchestration layers that can use different models as interchangeable components.

The work is presented as a continuation of Cloudflare’s earlier Project Glasswing research, which examined what happens when frontier security models are pointed at enterprise code. Cloudflare says that market shifts in model availability and capability reinforce the need to decouple the security workflow from any single model provider.

Why Cloudflare says a harness is needed

Cloudflare identifies several limitations of generic coding agents and single-session workflows:

  • Agents hold only one hypothesis at a time.
  • They fill their context window after covering only part of a real repository.
  • They lose information during context compaction.
  • They cannot preserve hundreds of separate investigations across runs without external orchestration.
  • They lack persistence, deduplication, resumability, and fleet-wide dependency tracing.

The article argues that subagents are useful but insufficient on their own. Cloudflare frames the problem as an orchestration challenge rather than a prompting challenge.

From a skill to a pipeline

Cloudflare began with a roughly 450-line security-audit skill that ran against a single repository. The skill performed a seven-phase audit:

  • Three parallel recon agents wrote an architecture.md file.
  • A Hunter agent ran per attack class.
  • Adversarial validators attempted to disprove each finding.
  • Surviving findings were written into a human-readable vulnerability report.
  • Findings were also emitted as findings.json against a schema.
  • A fresh agent independently re-verified every finding.
  • Surviving findings were submitted to an ingest API.

Cloudflare says this skill mapped almost directly onto the later harness. Recon became Recon, Hunters became Hunt, validators became Validate, reports became Report, schema checking became mechanical validation, and fresh verification became independent validation.

The skill exposed three major limitations:

  • Context exhaustion: Long sessions filled the context window and caused the model to lose track of work.
  • Persistence: A crash, rate-limit error, or flaky connection could force work to restart.
  • Cross-repo reasoning: Single-repository sessions could not see relationships between applications and their dependencies.

Cloudflare’s advice is to begin with a skill in the development environment, get prompts working well, and only add architectural stages when their absence becomes the specific bottleneck.

Vulnerability Discovery Harness

Cloudflare’s Vulnerability Discovery Harness, or VDH, is the discovery engine that scans codebases and surfaces potential security issues. It is designed to be model-agnostic and to run across a fleet of repositories without per-language tuning.

The VDH includes the following stages:

  • Recon: Maps target architecture and threat vectors, using three parallel recon sub-agents to write architecture.md.
  • Hunt: Runs per-class attacks, compiles fragments, probes binaries, spawns sibling agents, and writes to the wishlist tool.
  • Validate: Performs mechanical checks and then uses an isolated agent to disprove the finding before filing.
  • Gapfill: Generates new hunt tasks for under-tested area-by-attack-class coverage cells.
  • Dedup: Uses deterministic code and agents to cluster overlapping findings by root cause.
  • Trace: Walks the dependency graph and spawns consumer-repository tasks.
  • Feedback: Uses validation failures, shallow runs, and repeated misses to rewrite queued prompts.
  • Report: Renders human-readable reports without requiring a model.

Stages four through eight run as a continuous producer-consumer loop. This allows late-discovered vulnerabilities to be validated, reported, deduplicated, and checked against other code within the same run.

State, persistence, and context control

Cloudflare emphasizes that the pipeline is split into narrow stages to enforce context controls. The article states that if the model fills the context window, it starts hallucinating, so Cloudflare keeps each agent’s job focused and context usage below 25% of the total window.

Persistence is implemented before parallelism. Every stage writes to one SQLite database keyed by run_id, repo, and stage. Stages can resume, retry, or be pulled into later runs without redoing all work, and findings are streamed and saved as they occur.

Cloudflare also notes a specific operational issue: transient API errors may appear as text in a 200 OK response stream rather than throwing an exception. The orchestrator therefore needs to classify response text instead of relying only on exception types.

Dynamic threat modeling and active testing

During Recon, the agent writes the threat model instead of receiving a fixed one. In addition to about ten built-in attack classes, such as injection, memory corruption, protocol parsing, and timing side channels, Recon can invent repository-specific attack classes with their own methodology.

Hunter agents do more than read source code. They can compile fragments, build small versions, and attack them. Cloudflare says a major quality improvement came from giving Hunters a sandbox built on unshare so they could crash binaries.

The article also notes a nested containerization detail: if the harness runs inside Docker, the sandbox needs seccomp=unconfined and apparmor=unconfined, or it may silently fail to start.

Micro-forks and the wishlist

Cloudflare added two mechanisms to give Hunter agents more autonomy while keeping analysis scoped:

  • Sibling Forking: If a Hunter finds an interesting code path outside its current scope, it can fork a sibling agent with a precise structural seed instead of wandering off task. Cloudflare says this accounts for roughly 9% of fleet-wide tasks, but varies by model from near-zero to about one-fifth.
  • The Wishlist: Agents write to a central wishlist when they need resources they do not have, such as a build environment, VM, production config files, or resources to confirm a proof of concept.

Cloudflare says the wishlist had been written to 25,472 times across 128 repositories since it was added. The article describes it as the main way agents communicate resource needs back to the team.

Fleet-wide cross-repo tracing

The Trace stage checks how software components are connected. It asks whether attacker-controlled input can travel from the outside to a vulnerable part of the system. If so, the Tracer spawns new hunt tasks inside consumer repositories.

Cloudflare says this requires:

  • A unified cross-repo symbol index.
  • An accurate dependency graph.

The article states that cross-repo tracing can uncover systemic flaws that a single-repository scan would miss.

Cloudflare also reports two lessons from fleet-wide scanning:

  • Deduplication becomes its own problem at scale and requires dedicated agents and heuristics.
  • Static analysis should not be wired in too early, because Cloudflare integrated Semgrep but Hunters invoked it zero times in a month of runs, while the wishlist was the most-used tool.

Making findings trustworthy

Cloudflare highlights several failure modes that the harness must defend against. The article says an agent may edit the source code so its exploit works, write a tautological test, or build an exploit that proves nothing because the threat model is invalid.

To reduce these problems, Cloudflare requires a Hunter to state the threat model before filing a finding. The Hunter must define the attacker and the boundary or assumption being violated.

Every confirmed finding must include:

  • A proof-of-concept test that runs against the original, untouched codebase.
  • A proposed patch.
  • Evidence that the test and patch parse correctly.
  • File and path references that deterministic code can mechanically verify.

The Validator cannot file its own findings. Its only role is to aggressively disprove the Hunter’s theory. Cloudflare states that if a Hunter is allowed to grade its own work, it will confidently validate everything it outputs.

Vulnerability Validation System

Findings that leave the harness enter Cloudflare’s Vulnerability Validation System, or VVS. This shared system receives findings from multiple harnesses and currently holds 13,841 findings across 145 repositories, according to the article.

The VVS runs on a different model from the VDH. Cloudflare says this provides an independent check because the validation model evaluates the discovery model’s output with different logical weights and training data.

The VVS has three main jobs:

Deduplication

The deduplication stage determines whether a finding is already in the system or already tracked in an internal Jira ticket. Cloudflare avoids using an LLM to compare every finding with every other finding because that would scale at O(N²).

Instead, deterministic code builds inverted indexes over files, functions, trust boundaries, and rare tokens. The model then reasons over a short candidate list. Stable cross-run keys reopen existing records when bugs are rediscovered.

Contextual judgment

The Judgment agent rechecks the latest information and pulls from available deployment, environment, configuration, wiki, Jira, git, and other context. It evaluates whether the code path is reachable in production and identifies the repository owner.

Cloudflare says this separates findings that are exploitable now from real but latent issues or issues filed against the wrong component.

Automated fixing

The Fixer takes the proposed patch and unit tests, rewrites them to match repository style, applies the diff, and runs targeted tests. A clean fail-to-pass flip is required for the ideal automated cleanup case.

If the post-patch test fails or a broader run detects downstream regressions, the commit is blocked and flagged for human intervention. The Fixer never merges code on its own; a human must review the branch.

Cost and operating model

Cloudflare says running hundreds of agents over many repositories is not cheap, but the cost shape is predictable. Most compute budget goes into the Hunt stage, and Gapfill becomes the lever for trading cost against coverage.

Cloudflare budgets per repository rather than per run because costs vary widely by repository. It enforces a strict task cap per repository and uses worker pools ranging from 50 to 200 workers.

The article says large scans are periodic backlog sweeps rather than per-pull-request checks. A full scan of a complex repository can take hours, and Cloudflare’s worst run took just over 14 hours.

Cloudflare says it does not claim a false-negative rate because there is no labeled set of all real bugs in a codebase. Instead, it tracks whether reruns keep finding bugs and whether coverage grows across runs.

The article describes the system as a filtering funnel. Cloudflare reports the following lifetime numbers at the time of writing:

Cloudflare added automated health signals to detect failures early. If a hunt finishes suspiciously fast and fails to spawn sub-hunts or gap tasks, that may indicate a crashed dependency rather than a clean codebase.

The system flags any Hunter agent that finishes with zero findings as “shallow” and immediately requeues it for a new run.

The bulletin describes two ClickFix-related campaigns.

Fileless macOS ClickFix chain

Netskope Threat Labs reported a Russian-speaking attacker targeting victims mainly in Asia, North America, and Oceania across technology, media, and business services. The campaign used ClickFix pages masquerading as downloads for a malware scanning utility.

The infection chain was described as completely fileless until persistence was established. Victims were socially engineered into running a curl command that fetched a gzip-compressed stager and piped a second-stage AppleScript directly into osascript memory. The second stage, called “Meow (DEBUG),” used a fake system dialog to harvest credentials, browser data, session cookies, and keychain contents, and could trojanize legitimate desktop cryptocurrency wallet applications.

Claude shared chat abuse

Trend Micro reported a separate ClickFix campaign where threat actors abused Anthropic Claude’s shared chat feature. The campaign used Google Ads searches for popular AI developer tools to funnel more than 2,000 victims to malicious download pages before moving the operation onto claude.ai.

The payload was MacSync credential-stealing malware. Asia-Pacific accounted for 67.2% of confirmed victims, with Taiwan representing 30.5% of total traffic. Microsoft 365 device-code phishing

The bulletin describes multiple named malware families and delivery chains.

OnionDrop

Cyderes analyzed OnionDrop, a multi-stage loader used to deliver malware families such as LegionLoader, CGrabber, and Vidar Stealer. The chain begins with a ZIP archive and a legitimate Adobe-signed executable used for DLL side-loading.

The malicious DLL then performs custom byte-pair decoding, Xpress Huffman decompression through RtlDecompressBufferEx, AES-256-CBC decryption with rotating key material, and final shellcode execution through TpPostWork callback abuse inside the Windows Thread Pool. BlackFog described OnyxC2 as a new information stealer marketed on underground forums. The service provides a web panel and payload builder, with refunds for paying users if a build gets caught.

For \(250 per month, operators get a kit that harvests browser credentials, password managers, 2FA data, and crypto wallets across about 210 applications and extensions.
