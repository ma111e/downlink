New CIFSwitch Linux flaw gives root on multiple distributions
bleepingcomputer.com · 30 May 2026 14:16 UTC openai / deepseek-v4-pro

Summary
A new local privilege escalation vulnerability called CIFSwitch in the Linux kernel lets unprivileged users gain root by forging CIFS key requests; affected users should patch their systems or disable the CIFS module and cifs-utils.

A newly disclosed Linux kernel vulnerability called CIFSwitch allows unprivileged local attackers to escalate to root by forging CIFS authentication key descriptions. The flaw stems from the CIFS subsystem’s failure to verify that cifs.spnego key requests originate from the kernel’s own CIFS client, so an attacker can craft a fake request and trigger the root-privileged cifs.upcall helper.

By abusing attacker-controlled fields to force a namespace switch and triggering a Name Service Switch (NSS) lookup before privileges are dropped, a local user can load a malicious NSS module and execute code as root. The vulnerability was introduced in 2007 and impacts multiple distributions with default configurations, including Linux Mint, CentOS Stream, Rocky Linux, AlmaLinux, Kali Linux, and SLES.

A kernel patch adding origin validation has been committed, and mitigations include disabling the CIFS module, removing cifs-utils, or disabling unprivileged user namespaces. A proof-of-concept exploit is available to help organizations verify their patch status.

Security engineer Asim Viladi Oglu Manizada discovered a local privilege escalation vulnerability in the Linux kernel, dubbed CIFSwitch, that allows an unprivileged user to gain root by abusing CIFS authentication key management. The attack targets the kernel’s CIFS client, which uses the cifs.upcall helper (from cifs-utils) to fetch Kerberos credentials for network shares. The core problem is that the kernel does not validate that a cifs.spnego key request genuinely comes from its own CIFS client, enabling an attacker to forge such a request and launch the root‑privileged authentication workflow.

The exploit chain forces the root-running cifs.upcall to trust attacker-controlled fields that it assumes were generated by the kernel. The attacker uses these fields to instigate a namespace switch and then triggers an NSS lookup before privileges are dropped, loading a malicious NSS module to achieve root code execution. The underlying code flaw has existed since 2007 but went unnoticed for nearly two decades.

By default, several distributions are confirmed vulnerable: Linux Mint 21.3⁄22.3, CentOS Stream 9, Rocky Linux 9, AlmaLinux 9, Kali Linux 2021.4–2026.1, and SLES 15 SP7. Other distributions such as Ubuntu, Debian, Pop!_OS, openSUSE, Oracle Linux, and Amazon Linux may be at risk if cifs-utils is installed. Conversely, systems with restrictive SELinux or AppArmor policies (e.g., Ubuntu 26.04, Fedora 40–44, CentOS Stream 10, Rocky Linux 10, SLES 16, AlmaLinux 10, openSUSE Leap 16) are protected by default, and Amazon Linux 2 and older Kali releases are immune because their cifs-utils lack the namespace-switch functionality.

Exploitation requires a vulnerable kernel (6.14 and higher, plus some older versions), a vulnerable cifs-utils version, available user namespaces, and permissive security policies. The vulnerability is therefore not universal, but the conditions are met in many real-world default installations.

Upstream fix commit 3da1fdf adds validation that cifs.spnego requests originate from the kernel CIFS client. Users should update to patched kernels, or apply mitigations such as disabling/blacklisting the CIFS module, removing cifs-utils, or disabling unprivileged user namespaces. A proof-of-concept exploit has been published by the researcher to enable testing.

CIFSwitch is the latest in a string of recently disclosed Linux privilege-escalation bugs, including Copy Fail, Dirty Frag, Fragnesia, DirtyDecrypt, and PinTheft, highlighting ongoing attention to kernel‑level security weaknesses.

Overview

  • A local privilege escalation vulnerability named CIFSwitch was discovered in the Linux kernel by Asim Viladi Oglu Manizada, a SpaceX security engineer.
  • The flaw allows an unprivileged local user to forge CIFS authentication key descriptions, abuse the kernel’s key request mechanism, and gain root privileges.
  • CIFS (Common Internet File System) is used to mount remote network shares; when Kerberos authentication is used, the kernel relies on the root‑privileged cifs.upcall helper from the cifs-utils suite to fetch the necessary credentials.

Technical Details

Normal Authentication Flow

  • When a CIFS share requires Kerberos, the kernel requests a cifs.spnego key.
  • The key request configuration runs cifs.upcall as root to build or retrieve Kerberos/SPNEGO material.

Root Cause

  • The kernel’s CIFS subsystem does not verify that a cifs.spnego key request originates from the kernel’s own CIFS client.
  • An unprivileged user can craft a forged cifs.spnego request and trigger the normal authentication workflow, causing cifs.upcall to trust attacker‑controlled fields that it assumes were generated by the kernel.

Exploitation Steps

  1. The attacker creates a fake cifs.spnego key request.
  2. cifs.upcall processes the request as root and trusts the attacker‑supplied parameters.
  3. The attacker abuses these fields to force a namespace switch.
  4. Before privileges are dropped, a Name Service Switch (NSS) lookup is triggered.
  5. The attacker loads a malicious NSS module, achieving root code execution.

  • Unprivileged user namespaces must be available.
