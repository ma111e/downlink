Vulnerability Overview A critical SQL injection flaw, designated CVE-2026-42208, has been identified in BerriAI’s LiteLLM Python package, which serves as a widely adopted open-source AI gateway. The vulnerability carries a CVSS score of 9.3 and stems from improper input handling during proxy API key validation. Specifically, the database query responsible for checking API keys concatenates caller-supplied values directly into the query string rather than utilizing parameterized inputs. This architectural shortcoming allows unauthenticated threat actors to inject malicious SQL commands by sending specially crafted Authorization headers to standard LLM API endpoints, such as POST /chat/completions. The injected payload is executed when the proxy routes the request through its error-handling path.

Affected Versions & Patch Status The security flaw impacts a specific range of LiteLLM releases.

  • Vulnerable versions: >=1.81.16 and <1.83.7
  • Patched version: 1.83.7-stable, released on April 19, 2026 Project maintainers addressed the issue by correcting the query construction logic to safely handle user-supplied data.

Exploitation Timeline & Attacker Tactics Despite the patch release, active exploitation in the wild was recorded within 36 hours of the vulnerability’s public disclosure. The first exploitation attempt occurred on April 26 at 16:17 UTC, approximately 26 hours and seven minutes after the GitHub advisory was indexed in the global advisory database. Security researchers from Sysdig documented the attack activity, which unfolded in two distinct phases driven by the same operator. The initial phase originated from the IP address 65.111.27[.]132, followed by a secondary phase 20 minutes later using the adjacent IP 65.111.25[.]67. Both phases involved unauthenticated probing of key-management endpoints and deliberate column-count enumeration. Notably, the attacker bypassed standard user and team tables, focusing exclusively on the litellm_credentials.credential_values and litellm_config tables. This highly targeted approach demonstrates that the threat actor already possessed detailed knowledge of LiteLLM’s internal database schema.

Impact & Blast Radius The concentration of high-value credentials within LiteLLM’s architecture significantly amplifies the potential damage of a successful exploit.

  • A single database row often aggregates multiple critical secrets, including OpenAI organization keys with five-figure monthly spend caps.
  • Compromised rows may also contain Anthropic console keys with workspace admin privileges and AWS Bedrock IAM credentials. Because of this data density, extracting a single database row is functionally equivalent to compromising an entire cloud account. This elevates the threat far beyond typical web application SQL injection scenarios.

Mitigation & Recommendations Administrators running vulnerable LiteLLM instances are strongly advised to immediately upgrade to version 1.83.7 or later. For environments where immediate patching is not feasible, maintainers recommend a temporary workaround.

  • Configuration fix: Set disable_error_logs: true under general_settings.
  • Effect: This adjustment neutralizes the vulnerability by removing the error-handling execution path that untrusted input relies on to trigger the malicious query.

Industry Context & Broader Trends The rapid weaponization of CVE-2026-42208 reflects a concerning pattern in AI infrastructure security. The 36-hour exploitation window aligns with data from the Zero Day Clock, which documents the accelerating pace of vulnerability exploitation. Researchers emphasize that modern threat actors no longer wait for public proof-of-concept code to begin attacks. Instead, they leverage official security advisories and open-source database schemas to craft targeted exploits immediately upon disclosure. This incident also follows a previous supply chain attack on LiteLLM orchestrated by the TeamPCP hacking group, underscoring the project’s status as a high-value target for credential theft.
