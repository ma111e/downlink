A flaw in the Linux kernel’s traffic-control subsystem can let a local unprivileged user gain root on affected systems. CVE-2026-46331, nicknamed “pedit COW,” is an out-of-bounds write in the packet-editing action act_pedit that corrupts shared page-cache memory. A public working exploit appeared within a day of the CVE assignment on June 16, and Red Hat rates the flaw as important.

The exploit does not modify the file on disk; it poisons the cached in-memory copy of the setuid root binary /bin/su, injects a small payload, and runs the altered image as root. File-integrity checks can still come back clean while a root shell is already open. The exploit requires act_pedit to be loadable and unprivileged user namespaces to be open, giving the attacker the namespace-local CAP_NET_ADMIN capability needed to trigger the bug.

The PoC author reported unprivileged-to-root exploitation on RHEL 10 and Debian 13, while Ubuntu 24.04 required execution through AppArmor profiles that still permit user namespaces and Ubuntu 26.04 blocks that path by default. The article says administrators should install the patched kernel and reboot, especially on multi-tenant hosts, CI/CD runners, Kubernetes nodes, build workers, and shared lab machines. If patching is not possible, the article recommends blocking act_pedit from loading or disabling unprivileged user namespaces, while noting that dropping the page cache does not undo an already-opened root shell.

  • File-integrity checks may report clean results because the exploit alters only the cached in-memory copy of /bin/su, meaning defenders cannot rely on disk-based verification to rule out compromise.
  • Dropping the page cache can clear the poisoned in-memory binary, but the article warns that this does not undo an already-opened root shell, so affected hosts should be treated as compromised.
  • Ubuntu 26.04 is described as blocking the demonstrated exploit path by default through AppArmor restrictions on unprivileged user namespaces, even though the underlying kernel is still vulnerable.
  • The fix appeared on the public netdev mailing list weeks before CVE assignment as a routine data-corruption patch, showing that exploitable kernel page-cache corruption details may be visible publicly before security tooling or advisories catch up.

In plain words

The article describes DirtyClone, a newly demonstrated Linux flaw that can let someone who already has local access turn that access into full control of the machine by changing a program only in memory, without altering the file on disk. Systems most at risk include shared servers, CI runners, container hosts, and Kubernetes clusters where untrusted users can create isolated work areas. This matters because normal file-checking tools may not see the change, and the same kind of mistake has now appeared in several related Linux kernel flaws, suggesting more similar problems may still exist.

Summary
DirtyClone (CVE-2026-43503) is a Linux kernel flaw that can let local users gain root by corrupting file-backed memory through cloned network packets, especially on systems allowing unprivileged user namespaces. Administrators should install patched kernels or temporarily restrict user namespaces and affected networking modules.

DirtyClone is a Linux kernel privilege escalation flaw in the DirtyFrag family, tracked as CVE-2026-43503 with a CVSS score of 8.8. JFrog Security Research published a working exploit walkthrough on June 25, described as the first public demonstration for this variant. The flaw lets a local user corrupt file-backed memory through a cloned network packet and gain root.

The vulnerability occurs when helper functions that copy network packets drop a safety flag marking packet memory as shared with a file on disk. An attacker can load a privileged binary such as /usr/bin/su into memory, connect those pages to a network packet, force the kernel to clone it, and use an attacker-controlled IPsec tunnel to overwrite the binary’s login checks in memory. The file on disk is not changed, so file-integrity tools may miss the modification, and a reboot restores the original binary.

Exploitation requires CAP_NET_ADMIN to configure the loopback IPsec tunnel, but the article says unprivileged user namespaces on Debian and Fedora can let a local user obtain that capability inside a new namespace. Exposed environments include multi-tenant servers, CI runners, container hosts, and Kubernetes clusters where untrusted users can create namespaces. The upstream fix landed in Linux v7.1-rc5 and has been backported to stable and LTS branches, while temporary mitigations include restricting unprivileged user namespaces or blacklisting esp4, esp6, and rxrpc where possible.

  • The exploit can evade file-integrity checks because it changes only the kernel's in-memory copy of a privileged binary, while the file on disk remains unchanged and is restored to normal after a reboot.
  • Page cache is shared at the host level, so changes made from inside a namespace can affect every process on the machine rather than staying confined to that namespace.
  • The DirtyClone fix addresses a broader contract failure across fragment-transfer helpers, because any path that moves packet fragments without preserving the shared-frag flag can create a similar write primitive.
  • Restricting unprivileged user namespaces is a practical temporary mitigation for affected Debian and Ubuntu systems when immediate patching is not possible.
  • Blacklisting esp4, esp6, and rxrpc can reduce exposure, but it breaks IPsec and AFS and only works if those features are loadable modules rather than compiled into the kernel.
Key Points

  • 02Exploitation requires CAP_NET_ADMIN to configure a loopback IPsec tunnel, and unprivileged user namespaces enabled by default on Debian and Fedora can allow a local user to obtain that capability inside a new namespace.
  • 03Ubuntu 24.04 and later restrict namespace creation via AppArmor, blocking the default exploit path described in the article.
  • 04The affected systems include multi-tenant servers, CI runners, container hosts, and Kubernetes clusters where untrusted users can create namespaces.
  • 05The fix landed upstream in Linux v7.1-rc5 and has been backported to stable and LTS branches, with advisories published by Ubuntu, Debian, and SUSE and a Red Hat Bugzilla tracking entry.
