A critical flaw was found in a widely used piece of software called libssh2 that helps programs like curl, Git, and PHP securely connect to other computers. A rogue server can exploit it to take over any computer that connects to it, with no password or user action needed. Because this software is often hidden inside other products and doesn’t get updated through normal system updates, many organizations may not even realize they are vulnerable. Summary
A public proof-of-concept exploit has been released for a critical memory corruption vulnerability (CVE-2026-55200) in the widely embedded libssh2 SSH client library, allowing malicious servers to potentially execute code on connecting clients; urgent patching or outbound SSH restriction is needed, especially given libssh2's hidden presence in many applications.

A public proof-of-concept has been released for CVE-2026-55200, a critical vulnerability in the libssh2 SSH client library. The flaw, rated CVSS 4.0 score 9.2, allows a malicious or compromised SSH server to trigger memory corruption on any connecting client with no credentials or user interaction. All libssh2 versions up to and including 1.11.1 are affected.

The bug resides in the ssh2_transport_read() function in transport.c, where the incoming packet_length field was not checked for an upper bound. By supplying a length of 0xffffffff, an attacker can cause a 32-bit integer overflow that wraps to a tiny value, leading to an undersized allocation and an out-of-bounds heap write. The fix adds a check to reject packet lengths above LIBSSH2_PACKET_MAXPAYLOAD, and is available via commit 97acf3d, though no tagged release exists yet.

Because libssh2 is embedded in widely used software like curl, Git, and PHP—often as a statically linked copy—many deployments are invisible to standard package updates and remain vulnerable. NHS England Digital has issued an urgent cyber alert, and distributions like Debian are backporting the patch. Until patched, organizations are advised to restrict outbound SSH connections to trusted servers and watch for oversized-packet anomalies or unexpected crashes.

  • The same integer-overflow-to-buffer-overflow bug class in transport.c was fixed in libssh2 version 1.8.1 in 2019 (CVE-2019-3855), yet a near-identical flaw reappeared seven years later, indicating insufficient regression testing or code hardening after the previous critical patch.
  • Although CISA's current exploitation rating for CVE-2026-55200 is 'none' and no wild exploitation is reported, the library's pervasive static linking means many organizations remain unknowingly exposed, creating a high-stakes risk that official metrics do not capture.
  • The public proof-of-concept was posted in an 'exploitarium' repository without prior reporting, and the author admits entries are incomplete and AI-assisted, lowering the bar for attack development despite not being a turnkey remote exploit.
  • No tagged release of libssh2 with the fix yet exists, forcing downstream distributions and projects like Debian to backport the patch themselves, which may lead to inconsistent or delayed patching across the ecosystem.
  • The UK's NHS England Digital issued a specific cyber alert, underscoring that critical healthcare infrastructure relies on applications that embed libssh2, making patient-facing systems a potential attack surface from untrusted SSH endpoints.
