Discovery and disclosure

Rapid7 publicly disclosed the issue after what the article describes as a lack of response to multiple status updates. The Gogs maintainers also requested a CVE ID for the vulnerability. Rapid7 said the fix was implemented through pull request #8301 and recommended that all Gogs users upgrade immediately.

Why default configurations are especially exposed

Although exploitation requires authentication, Rapid7 said default-configured Gogs servers are affected because Gogs ships with open registration enabled by default. The relevant default setting is DISABLE_REGISTRATION = false. Gogs also ships with no limit on repository creation, using MAX_CREATION_LIMIT = -1.

Burgess warned that an unauthenticated attacker can create an account and repository on any default-configured instance. Any registered user who creates a repository automatically becomes its owner. From there, enabling rebase merging is described as a single toggle in settings, and the exploit chain can be operated without interaction from another user.

Technical details and impact

The vulnerability is an argument injection flaw. The article states that attackers can exploit it to compromise the targeted server, read any repository including private repositories, steal credentials, move laterally to other systems on the network, and alter any hosted source code.

Burgess said the flaw is very similar to other argument-injection flaws that the Gogs security team has patched in recent years. The article names CVE-2024-39933, CVE-2024-39932, CVE-2026-26194, and CVE-2024-39930 as examples. However, this issue affects a different code path, Merge(), that had not previously been addressed.

Recommended response

Rapid7 recommends that all Gogs users upgrade immediately to the patched version. For users who cannot patch their Gogs instances immediately, Rapid7 shared several mitigation measures.

The recommended mitigations are:

  • Restrict user registration by setting DISABLE_REGISTRATION = true in app.ini to prevent untrusted users from creating accounts.
  • Restrict repository creation by setting MAX_CREATION_LIMIT = 0 in app.ini, or by configuring the per-user “Max Repo Creation” setting in the admin panel.
  • Audit rebase merge settings, while noting that disabling “Rebase before merging” per repository is not an effective defense against a malicious user who owns or has admin access to a repository because they can re-enable it.

Rapid7 described restricting user registration as the most impactful mitigation because the exploit is self-contained within a single user’s repository. The malicious version, 065ee8eb, performed the same apparent revert but also added _hooks.py, _runtime.bin, and modified __init__.py. By backdating the malicious commit to August 2025, the article says the attacker made it nearly invisible in a casual commit history review.

Injected Malware Components

The attacker placed three malicious components in core/telemetry/:

  • _hooks.py, a 4,022-byte Python loader that downloads Bun v1.3.13 and executes the JavaScript payload.
  • _runtime.bin, a 758,608-byte obfuscated JavaScript credential stealer, despite its .bin extension.
  • A modification to core/telemetry/__init__.py, adding code that spawns a daemon thread on module import to run the loader.

The article emphasizes that the telemetry directory was a deliberate hiding spot because it already contained network-related code and may receive less scrutiny. It also notes that names such as _hooks.py and _runtime.bin were social engineering choices intended to look private, internal, or non-source-like.

Activation Chain

The malware was triggered when a developer ran gpt-pilot and Python imported core/telemetry/__init__.py. The injected code imported threading, defined _setup_reporting(), attempted to import and run core.telemetry._hooks.run, suppressed exceptions, and started the function in a daemon thread.

The Python loader supported Linux, macOS, and Windows across multiple architectures. It used a .loader.lock lock file to prevent duplicate execution and suppressed output to /dev/null. It then downloaded and ran the Bun runtime to execute the JavaScript payload.

JavaScript Payload and Obfuscation

The _runtime.bin payload was a 758KB single-line JavaScript file intended to run under Bun. The article describes five layers of obfuscation:

  • A 1,266-element constant lookup table.
  • A Base91-encoded string array rotated before use.
  • Three custom Base91 alphabets.
  • Control flow flattening using generator functions and while/switch state machines.
  • Obfuscated property access through a lazy string decoder with 3,769 unique references.

The article says the obfuscated code renamed require() and used an @redacted namespace to hide real package names. It gives an example in which obfuscated expressions resolve to module.require and require('crypto').

Credential Targets

The payload targeted credentials across multiple cloud and developer platforms. The article specifically lists AWS keys, npm tokens, GitHub secrets, Kubernetes service accounts, HashiCorp Vault tokens, SSH keys, environment variables, and credential files. It also says the malware queried cloud metadata services such as EC2/ECS IMDS for IAM credentials.

The indicators of compromise include token patterns for npm tokens, GitHub tokens, and AWS access keys. The article also references credential files such as .aws/credentials, .kube/config, and .vault-token in its MITRE ATT&CK mapping.

Command and Control

The malware used GitHub commit messages as a command-and-control channel. It searched the GitHub commits API for the marker string thebeautifulsnadsoftime using a query to https://api.github.com/search/commits?q=thebeautifulsnadsoftime.

The article states that commands were extracted using the regex thebeautifulsnadsoftime ([A-Za-z0-9+/=]{1,30})\.([A-Za-z0-9+/=]{1,700}). The commands were base64-encoded and executed via eval(). The article describes this as a channel that blends into normal developer activity because it uses GitHub commit search API calls.

Exfiltration and Persistence

The malware’s primary exfiltration method was to create GitHub repositories using stolen tokens and commit stolen credentials as files. The article says it impersonated claude@users.noreply.github.com as the commit author and created repositories with discussions, issues, and wiki disabled to reduce visibility.

The secondary exfiltration method used DNS-resolved HTTP endpoints with AES-256-GCM encryption. The payload also planted persistence hooks in developer tools. Specifically, it could modify .claude/settings.json to re-execute on Claude Code session start and .vscode/tasks.json to re-execute when a folder was opened in VS Code.

Anti-Analysis and Sigstore Abuse

The article lists several anti-analysis techniques:

  • A Russian locale exemption that exits if the system locale includes ru.
  • A singleton mutex to prevent multiple instances from running at once.
  • Daemonization using a detached child process with unref().
  • Token monitoring through installTokenMonitor.

The article also describes a capability to abuse Sigstore infrastructure. It says the malware could request signing certificates from Fulcio, create entries in the Rekor transparency log, and generate SLSA provenance attestations. According to the article, this would allow stolen npm tokens to be used to publish malicious packages that appear to have valid SLSA Build Level 3 attestations.

CI Failures and Ruff’s Role

The project’s CI pipeline used ruff check --output-format github and ruff format --check --diff. The first malicious force push failed at 11:02:07 UTC because ruff format --check found a formatting violation in _hooks.py line 59. The article says all six CI jobs failed across three Python versions and two operating system variants.

The attacker tried again at 11:13:07 UTC after fixing the formatting issue. The second CI run failed at 11:13:38 UTC because ruff check reported E402 for a module-level import not at the top of the file and I001 for unsorted or unformatted imports in core/telemetry/__init__.py line 399. Again, all six CI jobs failed, and the article says the attacker gave up.

Connection to Shai-Hulud

The article identifies the incident as a direct instance of the Shai-Hulud malware family and attributes the family to TeamPCP/UNC6780. It lists several matching indicators:

  • Bun v1.3.13 runtime loader pattern.
  • _runtime file naming convention.
  • Matching credential targets and exfiltration methods.
  • The thebeautifulsnadsoftime C2 marker.
  • Russian locale exemption.
  • Sigstore/Fulcio abuse.

The article also notes that attribution is uncertain because TeamPCP publicly released Mini Shai-Hulud source code on May 12, 2026, 27 days before the attack. It says the payload’s sophistication suggests the original actors, but it does not present that as certain.

Disclosure, Community Report, and Deleted Issue

The article states that the compromise was responsibly disclosed to the maintainers via GitHub issue #1182. It also says a community member reported the compromise in GitHub issue #1181 around 11:30 UTC, but that issue was later deleted and returned HTTP 410.

The article says only users with repository admin access can delete GitHub issues. It states that LeonOstrez was the only visible member of the Pythagora-io GitHub organization and that the account was actively force-pushing during the compromise window. Based on those facts, the article says it is overwhelmingly likely that the same compromised account deleted issue #1181, and recommends raising a GitHub support ticket to confirm via internal audit logs and suspend the compromised account.

Affected Users and Recovery Guidance

The article says users may be affected if they cloned or pulled gpt-pilot from GitHub after June 8, 2026 11:01 UTC and before the force push was reverted. It advises checking for the malicious files core/telemetry/_hooks.py and core/telemetry/_runtime.bin, Bun runtime directories in /tmp named rt-*, .loader.lock files, and unexpected .claude/settings.json or .vscode/tasks.json persistence hooks.

The article recommends these recovery steps:

  • Rotate AWS access keys, npm tokens, GitHub PATs, SSH keys, and secrets stored in environment variables or credential files.
  • Audit AWS CloudTrail for unauthorized AssumeRole, GetSecretValue, or ListSecrets calls.
  • Check npm audit logs for unauthorized package publishes or token creation.
  • Inspect GitHub repositories for newly created repositories that the user did not create, especially with discussions, issues, and wiki disabled.
  • Remove unexpected Claude Code or VS Code persistence hooks.
  • Kill suspicious Bun processes running _runtime.bin or processes with the __DAEMONIZED environment variable.

Lessons Highlighted by the Article

The article presents several lessons from the incident. It says CI/CD tooling can serve as an accidental security control when malicious code fails to match a project’s established code style. In this case, ruff blocked both attempted malicious pushes before the code could pass CI.

It also emphasizes that branch protection is not optional. The article recommends required pull request reviews, required status checks, restrictions on force pushes to the default branch, and signed commits. It further states that force pushes to default branches are almost always suspicious in production repositories.

Finally, the article notes that neither the legitimate nor malicious commits were GPG-signed. Because of that, the author identity on the malicious revert commit could not be cryptographically verified, and the article says the claimed author identity was likely spoofed because the commit was pushed via the compromised LeonOstrez account.

  • A code formatter and linter acted as an effective supply-chain defense because the attacker’s injected code did not conform to the project’s existing style and import-ordering rules.
  • Backdating the malicious commit and giving it the same message, author name, and timestamp as a legitimate revert made the compromise harder to notice during casual commit history review.
  • The malware’s use of GitHub commit messages for command-and-control allowed attacker commands to blend into ordinary GitHub API activity rather than relying on a traditional C2 server.
  • The payload’s ability to publish npm packages with valid Sigstore and SLSA Build Level 3 attestations shows that trusted software supply-chain signals can be abused when signing workflows and tokens are compromised.
  • The attacker placed the payload in the telemetry directory and used names like _hooks.py and _runtime.bin, taking advantage of a location and filenames that developers may be less likely to scrutinize.

  • The injected payload included a Python loader, a 758KB obfuscated JavaScript file named _runtime.bin, and a modification to core/telemetry/__init__.py that triggered the malware on module import.

Campaign Evolution

The article emphasizes that this campaign is not repeating a single package compromise pattern. The earlier PyPI wave used executable .pth startup hooks that attempted to locate a bundled JavaScript payload. The newer wave includes additional methods, including native-extension import triggers and a loader that searches across Python’s module search path.

Socket describes at least three relevant PyPI delivery branches in the campaign:

  1. A .pth startup-hook pattern.
  2. A native-extension import-time execution pattern.
  3. A langchain-core-mcp loader variant that searches sys.path for a separate payload.

.pth Startup-Hook Branch

In the first branch, a malicious wheel contains a *-setup.pth file and a bundled _index.js payload. The .pth hook runs during Python startup, downloads Bun if needed, and runs the JavaScript payload. This was the pattern used in the earlier weekend PyPI wave described by Socket.

The article explains that Python supplies the initial execution edge, while Bun supplies the JavaScript runtime for the stealer. The Hades payload itself is a heavily obfuscated JavaScript stealer staged through Bun.

Native-Extension Import-Time Branch

The newer bioinformatics subcluster uses trojanized native .abi3.so extensions. In this branch, the Python source can appear normal because the malicious execution path is inside a compiled native extension. When Python imports the package and loads the extension through dlopen(), the native extension executes _index.js as a side effect of module initialization.

The article states that this is harder to catch with source-only Python review because the malicious trigger is not visible in the package’s .py files. It also notes that native extensions are common and often expected in scientific computing, genomics, and machine learning packages. In this case, that expectation gives cover to a large .abi3.so extension that serves as an import-time launcher for _index.js.

langchain-core-mcp Loader Variant

The langchain-core-mcp variant returns to a .pth startup hook but changes payload discovery. The article says that langchain-core-mcp@1.4.2 installs langchain_core-setup.pth, but the wheel does not ship _index.js. Instead, the .pth file searches every entry in sys.path, first for a direct _index.js, then one directory below each path entry.

The hook also uses a run-once marker in the system temporary directory named .bun_ran. If the marker does not exist, the hook imports modules, searches for the payload, downloads Bun into the temporary directory if absent, executes the discovered JavaScript payload with Bun, and then creates the marker after attempted execution. The article defangs the Bun download URL in its code excerpt.

Socket says this differs from the earlier Hades loader, which attempted to locate _index.js relative to the .pth execution context. The article notes that in standard CPython, that can be unreliable because .pth execution occurs through Python’s site module and __file__ may not point to the .pth file itself. The langchain-core-mcp variant avoids that by scanning sys.path.

The article gives three possible interpretations for the missing bundled _index.js: the actors may have fixed a reliability issue from the earlier loader, the wheel may be a failed or incomplete publish, or the actors may be testing paired-package staging. Microsoft Graph Findings

AzureHound sends GET requests to 17 Microsoft Graph endpoints. SpecterOps mapped those endpoints to validated least-privilege permissions after documentation review and empirical testing.

Microsoft Graph Permission Complexities

The article identifies several complexities in Microsoft Graph permission validation.

First, API-to-permission mapping is one-to-many. For example, /v1.0/groups works with Group.Read.All, GroupMember.Read.All, and Directory.Read.All. Selecting the narrowest permission required knowing which permission worked across the fewest endpoints.

Second, the Microsoft Graph permissions reference could not be treated as complete. SpecterOps found that /v1.0/organization returned HTTP 200 with User.Read.All even though Microsoft did not document User.Read.All as a valid application permission for that endpoint. SpecterOps kept Organization.Read.All in the least-privilege set because production collection should not depend on undocumented behavior.

Third, Microsoft Graph has property-level permission behavior. An endpoint may return HTTP 200 but omit, null, or withhold specific properties unless the token has the right permissions. The article demonstrates that User.ReadBasic.All could call /v1.0/users but returned null for fields such as accountEnabled, createdDateTime, onPremisesSyncEnabled, and userType, while User.Read.All returned the requested values where available. The article also states that signInActivity requires AuditLog.Read.All in addition to User.Read.All.

SpecterOps validated the proposed least-privilege model in several ways.
