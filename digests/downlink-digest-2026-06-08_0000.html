Exploitation Requirements and Default Exposure

The flaw can only be exploited by authenticated attackers without admin privileges. However, Rapid7 security researcher Jonah Burgess said that default Gogs configurations significantly reduce the practical barrier to exploitation.

Gogs ships with:

  • DISABLE_REGISTRATION = false, meaning open registration is enabled by default.
  • MAX_CREATION_LIMIT = -1, meaning there is no limit on repository creation by default.

Burgess said that because of these defaults, an unauthenticated attacker can create an account and repository on any default-configured instance. Any registered user who creates a repository automatically becomes its owner. From there, enabling rebase merging is a single toggle in repository settings, and Burgess said the full exploit chain can be performed without interaction from any other user.

Patch and Disclosure Timeline

Rapid7 researcher Jonah Burgess discovered and reported the vulnerability. The article says Gogs maintainers released version 0.14.3 on June 7 to patch the flaw, 10 days after Rapid7 publicly disclosed it following a lack of response to multiple status updates.

The fix was implemented through pull request #8301. Gogs maintainers also requested a CVE ID for the vulnerability. Rapid7 recommends that all Gogs users upgrade immediately.

Temporary Mitigations

Rapid7 provided mitigation measures for users who cannot patch their Gogs instances immediately.

The recommended measures are:

  • Restrict user registration by setting DISABLE_REGISTRATION = true in app.ini. Rapid7 described this as the most impactful mitigation because the exploit is self-contained within a single user’s repository.
  • Restrict repository creation by setting MAX_CREATION_LIMIT = 0 in app.ini. This can also be configured per user through the “Max Repo Creation” option in the admin panel.
  • Audit rebase merge settings, including the “Rebase before merging” setting under Settings > Advanced for each repository.

The article notes that disabling “Rebase before merging” is not an effective defense against a malicious user who owns or has admin access to a repository, because that user can re-enable the setting.

Affected Product and Internet Exposure

Gogs is written in Go and is designed as an alternative to GitHub Enterprise or GitLab. The article says it is often exposed online as a remote collaboration platform.

Shadowserver currently tracks over 2,300 Internet-exposed Gogs servers. According to the article, most are in Asia, with 1,839 exposed servers, and Europe, with 312 exposed servers. Shodan lists just over 1,000 IP addresses with a Gogs fingerprint.

Relationship to Earlier Gogs Vulnerabilities

Burgess said the new vulnerability is very similar to other argument-injection flaws that the Gogs security team has patched in recent years. The article specifically references:

However, Burgess said the new flaw affects a different code path, Merge(), that was not previously addressed.

Prior Exploitation of Gogs RCE Vulnerabilities

The article also references a separate Gogs remote code execution vulnerability, CVE-2025-8110. Gogs patched that issue in early December 2026 after it was exploited in zero-day attacks to compromise hundreds of servers.

Wiz security researchers, who reported CVE-2025-8110, said many Gogs instances were configured with open registration enabled by default, creating a large attack surface. On January 12, CISA confirmed that CVE-2025-8110 was being abused in the wild and added it to its Known Exploited Vulnerabilities catalog. CISA ordered Federal Civilian Executive Branch agencies to secure their servers by February 2.

On June 8, 2026, an attacker compromised the GitHub account of LeonOstrez, described in the article as a Pythagora co-founder and repository maintainer, and force-pushed malicious code to the main branch of Pythagora-io/gpt-pilot. The project is described as a popular open-source AI developer tool with more than 33,700 GitHub stars and 3,500 forks. The article says the repository had no branch protection rules on main, allowing the attacker to replace the clean commit chain without review or approval.

The malicious payload was identified as a variant of the Shai-Hulud worm. It was designed to steal developer, cloud, and CI/CD credentials, including AWS keys, npm tokens, GitHub secrets, Kubernetes service accounts, HashiCorp Vault tokens, and SSH keys. The article emphasizes that the attack was stopped not by a dedicated security control, but by ruff, the Python formatter and linter used in the project’s CI pipeline.

Attack Timeline

The article describes two force-push attempts and two CI failures:

  • A malicious “Revert” commit was authored and backdated to match a legitimate revert by Zvonimir Sabljic, adding _hooks.py, _runtime.bin, and modifying init.py in core/telemetry.
  • At 2026-06-08 11:01:38 UTC, the first force push replaced the clean commit chain with a malicious chain.
  • At 2026-06-08 11:02:07 UTC, CI failed because ruff format –check detected a formatting violation in _hooks.py.
  • At 2026-06-08 11:13:07 UTC, the attacker force-pushed again after fixing the formatting issue.
  • At 2026-06-08 11:13:38 UTC, CI failed again because ruff check detected E402 and I001 violations in init.py.
  • Around 2026-06-08 11:30 UTC, a community member reported the compromise in GitHub issue #1181.
  • Around 2026-06-08 12:00 UTC, issue #1181 was deleted and began returning HTTP 410.

The article says the attacker gave up after the second CI failure.

Repository Compromise Method

The attacker used the compromised LeonOstrez account to force-push directly to main. The article states that the GitHub API returned a 404 for the branch protection endpoint for main, indicating that branch protection rules were not configured. This meant there were no required pull request reviews or required status checks preventing the force push.

The push event log showed the clean history being replaced with a malicious chain, followed by a second attacker retry. The pre-attack clean HEAD was identified as 53154df1c66b42021f230c3fb6ef797c4b7c3e83. The first CI failure commit was 90f59f5de6819a43ffe9b6272e3ed65aaadca804, and the second was a372904facd53ee99d85add7ee79aea2b7a8506a.

Trojan Commit Strategy

The core malicious commit was titled “Revert ‘Implemented weekend discount’.” The article says this was intended to look like an ordinary revert of a promotional code update. Two versions of the commit existed with identical metadata, including the same message, author name, and timestamp of 2025-08-24 20:37:44.

The clean commit, 566fbb120bc436385aa5a4cb93d7c351dec2127e, only reverted frontend.py and spec_writer.py. The malicious commit, 065ee8ebee7385cb644fd1608587a18edb91f4fb, included the same reverts plus added _hooks.py, _runtime.bin, and modified init.py. The article says the backdating made the commit harder to notice during casual commit history review.

Injected Malware Components

The attacker injected three files into core/telemetry:

  • _hooks.py: a 4,022-byte Python loader that downloaded Bun v1.3.13 and executed the JavaScript payload.
  • _runtime.bin: a 758,608-byte obfuscated JavaScript credential stealer, despite the .bin extension.
  • init.py: modified with nine lines that spawned a daemon thread on module import to trigger the malware.

The activation chain began when a developer ran gpt-pilot and Python imported the telemetry module. The modified init.py launched a daemon thread that attempted to import run from core.telemetry._hooks and execute it. The loader then downloaded the Bun runtime and executed the JavaScript payload.

The loader supported Linux, macOS, and Windows across x64 and ARM variants, including musl Linux. It used a .loader.lock file to prevent duplicate execution and suppressed output to /dev/null.

JavaScript Payload Characteristics

The _runtime.bin payload was described as a 758KB single-line JavaScript file designed to run under Bun. It used five obfuscation layers:

  • A constant lookup table named MxGPr9 with 1,266 elements.
  • A Base91-encoded string array named rmlQezO with 4,777 encoded strings rotated by 26 positions.
  • Three custom Base91 alphabets.
  • Control-flow flattening with generator functions and while/switch state machines.
  • Property access obfuscation through a lazy string decoder named HJgj4ju.

The obfuscated code renamed require() to rMq3gu() and used an @redacted namespace to hide real package names. The article gives crypto loading through module.require as an example of what the obfuscation concealed.

Credential Theft Targets

The payload targeted credentials across cloud, developer, and CI/CD environments. The article specifically names:

  • AWS keys.
  • npm tokens.
  • GitHub secrets and tokens.
  • Kubernetes service accounts.
  • HashiCorp Vault tokens.
  • SSH keys.
  • Cloud metadata credentials through EC2/ECS metadata services.
  • OIDC and application access tokens.

The article also lists token patterns for npm, GitHub, and AWS access keys as indicators.

Command-and-Control and Exfiltration

The malware used GitHub commit messages as a command-and-control channel. It searched the GitHub commits API for the marker string “thebeautifulsnadsoftime” and extracted commands with a regular expression matching two Base64-like fields. The article says the commands were base64-encoded and executed via eval().

For primary exfiltration, the malware created GitHub repositories using stolen tokens and committed stolen credentials as files. It impersonated claude@users.noreply.github.com as the commit author and created repositories with discussions, issues, and wiki disabled. As a secondary method, it used DNS-resolved HTTP endpoints with AES-256-GCM encryption.

Persistence and Anti-Analysis

The payload planted persistence hooks in developer tools. It modified .claude/settings.json so malicious commands would run on Claude Code session start. It also modified .vscode/tasks.json so malicious commands would run when a folder was opened in VS Code.

The article lists several anti-analysis and operational features:

  • A Russian locale exemption that exits when the system locale includes “ru”.
  • A singleton mutex to prevent multiple instances from running.
  • Daemonization through a detached child process with unref().
  • A __DAEMONIZED environment flag.
  • Token monitoring through installTokenMonitor.

Sigstore and SLSA Abuse

The article calls the payload’s Sigstore capability one of its most alarming features. It says the malware could request signing certificates from Fulcio, create entries in the Rekor transparency log, and generate SLSA provenance attestations. According to the article, stolen npm tokens could be used to publish malicious packages that appear to have valid SLSA Build Level 3 attestations.

How Ruff Blocked the Attack

The gpt-pilot repository used ruff in CI with two required checks: ruff check and ruff format –check. These checks ran across six CI jobs covering three Python versions and two operating system variants.

The first force push failed because _hooks.py did not match ruff formatting. The article says the malicious file was the only file that did not pass formatting, producing the result “1 file would be reformatted, 151 files already formatted.” All six CI jobs failed.

The attacker then fixed the formatting issue and force-pushed again. The second attempt failed because init.py contained an import at line 399 after the class definition. Ruff reported E402, “Module level import not at top of file,” and I001, “Import block is un-sorted or un-formatted.” All six CI jobs failed again.

The article presents ruff as an accidental security control. It says code quality tools can become a useful defense layer because malicious code injected outside the normal development workflow may not match a project’s established style.

Attribution and Campaign Connections

The article identifies the incident as part of the Shai-Hulud malware family, attributed to TeamPCP/UNC6780. It bases this identification on several matching indicators:

  • The Bun v1.3.13 runtime loader pattern.
  • The _runtime file naming convention.
  • The same credential target set and exfiltration methods.
  • The “thebeautifulsnadsoftime” C2 marker.
  • The Russian locale exemption.
  • Sigstore/Fulcio abuse for signing malicious packages.

The article also says TeamPCP publicly released the Mini Shai-Hulud source code on May 12, 2026, 27 days before this attack, making attribution uncertain. It states that the sophistication of the payload suggests the original actors, while acknowledging the possibility of a copycat.

The article links the malware family to earlier incidents affecting projects or organizations including Microsoft, Red Hat, Mistral AI, PyTorch Lightning, and axios, as discussed in StepSecurity’s prior analyses.

Potentially Affected Users and Recovery

The article says users may be affected if they cloned or pulled gpt-pilot from GitHub after June 8, 2026 11:01 UTC and before the force push was reverted. It recommends checking for malicious files in core/telemetry, Bun runtime directories named rt-* under /tmp, .loader.lock files, and persistence hooks in .claude/settings.json and .vscode/tasks.json.

The article recommends the following recovery steps:

  • Rotate all credentials immediately, including AWS access keys, npm tokens, GitHub PATs, SSH keys, and secrets in environment variables or credential files.
  • Audit cloud access logs for unauthorized AssumeRole, GetSecretValue, or ListSecrets calls.
  • Check npm audit logs for unauthorized package publishes or token creation.
  • Inspect GitHub accounts for newly created repositories, especially those with discussions, issues, and wiki disabled.
  • Remove unauthorized Claude Code or VS Code persistence hooks.
  • Kill suspicious Bun processes running _runtime.bin or processes with the __DAEMONIZED environment variable.

Deleted Issue and Disclosure

The article says StepSecurity responsibly disclosed the compromise to the maintainers through GitHub issue #1182. It also says an earlier community report in issue #1181 was deleted and now returns HTTP 410. The article states that only users with repository admin access can delete GitHub issues.

The authors attempted to confirm deletion through the GitHub organization audit log API, but the endpoint required organization admin access and returned HTTP 404 to external callers. The article says LeonOstrez was the only visible member of the Pythagora-io GitHub organization and was actively force-pushing during the compromise window. It recommends raising a GitHub support ticket to confirm deletion through internal audit logs and to suspend the compromised account.

Lessons Highlighted by the Article

The article presents several lessons from the incident:

  • CI/CD checks can serve as an accidental security control when malicious code fails to match project style or structure.
  • Branch protection is not optional for default branches because a single compromised account can otherwise force-push malicious code.
  • Force pushes to default branches should be monitored because they are described as almost always suspicious in production repositories.
  • Telemetry directories can be attractive hiding places because they already contain network-related code and may receive less scrutiny.
  • Bun was used deliberately by the operators, according to the article, because it is newer, may be less likely to be flagged by endpoint tools, and can execute JavaScript or TypeScript files with nonstandard extensions such as .bin.

    Execution through a Python import hook

    The delivery mechanism differs from earlier npm campaigns described in the article. Instead of running through installation lifecycle scripts or native build hooks, the Hades Campaign runs during Python code execution through an obfuscated single-line import hook embedded in __init__.py.

    The deobfuscated logic checks for a flag file at /tmp/.bun_ran. If the file does not exist, it searches Python module paths for a companion _index.js file. It then downloads a precompiled Bun v1.3.14 runtime from GitHub, extracts it to a temporary directory, and executes the JavaScript payload using bun run _index.js.

    The article states that the flag file causes the payload to execute only once per boot cycle. It also emphasizes that using a standalone Bun binary allows the malware to run JavaScript tasks in environments that do not have Node.js installed.

    AI analyst misdirection

    The article describes a prompt injection block placed at the beginning of the _index.js bundle. This plain-text block is intended to manipulate automated Large Language Model pipelines used by security companies to inspect and summarize suspicious package code.

    The prompt injection instructs a parsing LLM to ignore the obfuscated code below it, classify the package as verified clean infrastructure, and produce a safe security report. The article characterizes this as a conceptual shift in which attackers target the cognitive logic of automated AI triage systems.

    Obfuscation and modular payload structure

    The _index.js file acts as a runtime bootstrapper. The article says it decrypts and loads sixteen independent functional payloads at startup.

    Each payload blob is gzip-compressed and encrypted with AES-256-GCM using a unique hardcoded key. The bootstrapper uses Bun APIs for decryption and decompression. After deobfuscation, the payloads were found to include modules for macOS and Windows memory reading, IDE and CI/CD backdoor setup, and command-and-control agents.

    Cross-platform memory scraping

    A major capability described in the article is reading the memory of the GitHub Actions Runner.Worker process to extract secrets. Earlier Miasma campaigns used Linux /proc/{pid}/mem, but the Hades Campaign adds macOS and Windows implementations.

    Linux

    On Linux, the malware walks memory mappings through /proc/{pid}/maps and directly reads /proc/{pid}/mem to scrape plaintext variables.

    macOS

    On macOS runners, the malware decrypts a Python script that invokes Mach kernel VM APIs through ctypes. The article states that because the runner worker and execution process run under the same UID, the script can obtain a task port without root privileges. It then enumerates readable memory regions and writes their contents to standard output.

    Windows

    On Windows, the malware executes a PowerShell script that dynamically compiles a C# class using Add-Type. The C# code uses Win32 APIs such as OpenProcess, VirtualQueryEx, and ReadProcessMemory to crawl and dump the target process memory.

    The article states that these cross-platform scrapers allow the malware to extract unmasked variables and tokens on Linux, macOS, and Windows runners.

    GitHub-based command and control

    The article describes three independent command-and-control channels that use public GitHub infrastructure.

    Token dead-drop

    Harvested GitHub personal access tokens are encrypted and pushed as commits to public repositories controlled by the attacker. These commits use the keyword DontRevokeOrItGoesBoom, which the attacker can locate through GitHub search APIs.

    Signed JavaScript eval

    The malware searches GitHub commits for the keyword TheBeautifulSnadsOfTime. Matching commit messages contain base64-encoded JavaScript payloads and RSA-PSS signatures. The malware verifies the signatures against a hardcoded public key and executes valid payloads using eval().

    Python dropper

    The campaign introduces a Python-specific C2 channel through a script named updater.py. This service polls GitHub for commits matching firedalazer. Matching commits encode a URL and RSA-PSS signature; if the signature is valid, the daemon downloads and executes the script from the URL.

    Exfiltration process

    The article says stolen credentials are encrypted locally before exfiltration. The process includes:

    • JSON serialization of harvested secrets
    • gzip compression
    • generation of a random 256-bit AES key
    • AES-256-GCM encryption of the data
    • RSA-2048 encryption of the ephemeral AES key using the attacker’s public key
    • upload of the encrypted payload to a newly created public GitHub repository under the attacker’s control

    The exfiltration repositories use Hades-themed names such as stygian-cerberus-42817 and tartarean-charon-18401. They carry the description Hades - The End for the Damned.

    Worm propagation and lateral movement

    The article describes three self-replication vectors.

    SSH and SCP lateral movement

    The malware parses ~/.ssh/known_hosts and ~/.ssh/config to identify reachable hosts. If ssh and scp are available, it attempts batch-mode connections using configured keys.

    On successful access, it stages a loader script and the primary payload in a temporary directory on the remote host, runs the loader, and deletes the staging directory.

    PyPI and npm OIDC trust exploitation

    When running inside a GitHub Actions workflow runner, the malware checks for OIDC variables such as ACTIONS_ID_TOKEN_REQUEST_TOKEN and ACTIONS_ID_TOKEN_REQUEST_URL. If available, it attempts to mint PyPI or npm publishing tokens by calling the OIDC endpoint with registry audiences.

    The article says the malware also generates Sigstore SLSA provenance bundles. It requests a signing certificate from Fulcio, formats a SLSA provenance statement, DSSE-signs it, uploads it to Rekor, and packages the certificate, signatures, and inclusion proof into a Sigstore bundle. It then injects its payload into target libraries and publishes compromised versions to PyPI and npm using the organization’s automated publishing credentials and the generated provenance bundle.

    GitHub Actions workflow injection

    The malware targets GitHub repositories to extract organization secrets if a harvested token has suitable permissions.

    If the token has the workflow scope, it uses a push-triggered workflow technique. .pth Startup-Hook Pattern

The earlier PyPI wave used executable .pth startup hooks. A malicious wheel contained a *-setup.pth file and a bundled _index.js payload. Python provided the initial execution path, while Bun supplied the JavaScript runtime for the stealer.

The .pth hook runs during Python startup, downloads Bun if needed, and runs the JavaScript payload. Socket says this behavior was seen in the weekend PyPI wave and is also used in the langchain-core-mcp variant, although that variant changes how it discovers the payload.

Native Extension Import-Time Execution

The newer bioinformatics cluster uses trojanized native .abi3.so extensions. In this branch, the visible Python source can appear normal because the malicious execution path is inside a compiled native extension.

When Python imports the package and loads the extension through dlopen(), the native extension executes _index.js as a side effect of module initialization. Socket notes that this is harder to catch with source-only Python review because the malicious trigger is not visible in the package’s .py files.

The article highlights that native extensions are common and often expected in scientific computing, genomics, and machine learning packages. The threat actors use that expectation as cover, because a large .abi3.so file in a performance-focused package may not look unusual at first glance.

The langchain-core-mcp variant returns to a .pth startup hook but changes the payload discovery logic. The article says langchain-core-mcp@1.4.2 installs a file named langchain_core-setup.pth, but the wheel does not ship _index.js.

TA4922 Expansion

TA4922 focuses on:

Meta disclosed that 20,225 Instagram accounts may have been affected by an attack campaign abusing an AI-powered support tool. The attackers compromised accounts by asking Meta’s chatbot to link an attacker-controlled email address to the targeted account.

Software Supply Chain and Package Ecosystem Threats

The article highlights several package ecosystem incidents involving npm and PyPI.

Malicious npm Packages Target Trusted Brands

A threat actor deployed dozens of malicious npm packages targeting AI companies, luxury brands, and venture capital firms. These packages dropped a new malware strain impersonating an AI coding tool.

The malicious code launched through a post-install hook. When the payload ran, it displayed a terminal prompt asking users for information and OpenAI or Anthropic API keys, while also harvesting local credential files in the background.

turbo-axios and faster-axios Deliver Epsilon Stealer

Two malicious npm packages, turbo-axios and faster-axios, targeted developers searching for the popular axios HTTP client. Both were trojanized copies of the real axios source with a postinstall hook that fetched and evaluated remote JavaScript.

The chain ended in Epsilon Stealer, described in the article as a malware-as-a-service Electron infostealer. It harvests browser credentials, crypto wallets, and messaging sessions, and opens a persistent WebSocket channel for arbitrary command execution.

cms-store-ren Leaks Its Own Telegram Bot Token

OX Security flagged a malicious npm package named cms-store-ren. The package collected data from developers’ machines and sent it to a Telegram channel, while also leaking its own bot API token.

The package also downloaded a potentially malicious JavaScript file from a remote server and attempted to execute it. The article says this behavior had not yet been weaponized. The package functioned as a downloader or loader intended to fetch and execute a second-stage payload while reporting successful infections to the attacker.

Malicious PyPI Package Parsimonius

The article also covers Parsimonius, a malicious typosquat of the Python package parsimonious. It included legitimate parsing functionality to avoid suspicion while deploying a Telegram-based backdoor.

Once installed, the backdoor provided remote access and enabled theft of sensitive data, including .env files and bot authentication tokens. The package accumulated 2,474 downloads before removal.

The article lists additional vulnerabilities used in related scanning or exploitation activity, including CVE-2025-34054, CVE-2016-15047, CVE-2015-2051, CVE-2022-35914, and CVE-2021-27137.

OverlayPhantom Android Banking Trojan

OverlayPhantom is a new Android banking trojan targeting more than 180 apps across 10 countries through malicious URLs. It uses fake overlays and real-time screen sharing to steal credentials.

The malware uses a two-stage infection chain with a dropper app impersonating trusted platforms such as ID Austria and TikTok. After deployment, OverlayPhantom masquerades as Google Play Services and abuses Android accessibility services for persistent, elevated control.

It can run more than 30 remote commands supporting:

  • Automated gestures.
  • Clipboard manipulation.
  • Credential theft.
  • Data exfiltration.

Targets include financial and cryptocurrency apps serving users in the U.S., Australia, Germany, France, Belgium, Finland, the Netherlands, Italy, Spain, and the U.K.
