Microsoft Removes 119 Edge Extensions That Hid Malware in Images and Fonts
thehackernews.com · 29 Jun 2026 08:32 UTC openai-codex / gpt-5.5

In plain words

Microsoft removed 119 harmful Edge browser add-ons that looked like normal tools such as ad blockers, VPNs, translators, and video downloaders, but secretly hid dangerous code inside image and font files. These add-ons had been installed by as many as 2.6 million users, though Microsoft says that number is only the maximum possible exposure, not a confirmed victim count. The real-world risk is that some users may have had passwords, Google sign-in codes, WordPress logins, or browser cookies stolen, while the same add-ons also redirected searches and injected ads to make money for the operator.

Summary
Microsoft removed 119 malicious Edge extensions, with up to 2.6 million installs, that hid payloads in images and fonts to evade detection. The StegoAd campaign used the extensions for ad fraud, credential and 2FA theft, cookie exfiltration, and remote code execution, so affected users should remove them and change sensitive passwords.

Microsoft removed 119 malicious extensions from the Edge Add-ons store and suspended more than 90 developer accounts tied to the operation. Microsoft calls the campaign StegoAd and says it was run by a single threat actor active since at least 2021. The extensions posed as common tools such as ad blockers, VPNs, translators, and video downloaders, and had an install base of up to 2.6 million users, though Microsoft says that figure is a ceiling rather than a confirmed victim count.

The campaign hid executable code inside ordinary-looking image and font files, including PNG icons, WebP images, and WOFF2 fonts. Some variants fetched normal-looking images from command-and-control servers, decoded the hidden payload through several transformations, and only received the real file after passing fingerprint and User-Agent checks. The extensions also used delays, server-side validation, execution gating, and DevTools checks to avoid triggering for many installs or when analysts were inspecting them.

Microsoft found that the payloads supported ad fraud, injected ads, hijacked affiliate commissions, redirected searches, ran arbitrary JavaScript from a server, stole Google credentials and second-factor codes, harvested WordPress admin logins, and exfiltrated cookies in bulk. Microsoft advises users to check installed Edge add-ons against the published extension ID list, treat matching or automatically removed extensions as exposure, change sensitive passwords, review sign-in activity, and enable strong two-factor authentication. The article also notes overlap with DarkSpectre, ShadyPanda, and GhostPoster through shared infrastructure, shared hiding methods, and shared extension names, although Microsoft has not named the actor.

  • The extensions often performed their advertised functions and accumulated reviews, which helped them remain in the Edge Add-ons store for years despite carrying hidden malicious code.
  • Some variants used multi-day delays, server-side validation, analyst checks, and a 10% execution gate, meaning many installs may never have triggered the payload and direct probing by researchers could receive only decoy responses.
  • The campaign’s use of steganography evolved as detection improved, moving from JavaScript appended to PNG icons to WebP images and then WOFF2 font files containing hidden code.
  • Microsoft found that seven Google Analytics tracking IDs appeared to give the operator near real-time campaign telemetry through Google’s infrastructure.
  • The article notes overlap with DarkSpectre, ShadyPanda, and GhostPoster through shared infrastructure, shared hiding methods, and shared extension names, although Microsoft has not publicly named the actor.

Public PoC Released for Critical libssh2 CVE-2026-55200 Client-Side SSH Flaw
thehackernews.com · 29 Jun 2026 07:06 UTC openai-codex / gpt-5.5
TAGScisacurlcve-2019-3855cve-2025-15661cve-2026-55199cve-2026-55200debianenglandexploitariumgitlibssh2nhs-england-digitalphptristan-madanivulncheck

In plain words

Someone has published example code for a serious flaw in libssh2, a software library used by many programs to connect to SSH servers. If a program using a vulnerable version connects to a malicious or compromised SSH server, that server may be able to make the program crash or possibly run code on the user’s machine. This matters because libssh2 is built into tools like curl, Git, PHP, backup agents, firmware updaters, and appliances, and some copies are hidden inside products where normal updates may not fix them.

Summary
A public PoC has been released for CVE-2026-55200, a critical libssh2 client-side flaw that could let a malicious SSH server corrupt memory and potentially execute code on connecting clients. Organizations should urgently identify bundled or static libssh2 copies and apply builds containing the upstream patch until an official fixed release is available.

A public proof-of-concept has been released for CVE-2026-55200, a critical flaw in libssh2 that can let a malicious or compromised SSH server trigger memory corruption on a connecting client, with possible code execution. The bug affects every release up to and including 1.11.1 and has a CVSS 4.0 score of 9.2. The article emphasizes that libssh2 is a client-side SSH library embedded in software such as curl, Git, PHP, backup agents, firmware updaters, and appliances.

The flaw is in ssh2_transport_read() in transport.c, where the attacker-controlled packet_length field was rejected only when below 1 and was not checked against an upper bound. A very large length value can wrap around during 32-bit size calculations, causing libssh2 to allocate a tiny buffer before later code writes an oversized packet into it. There is no fixed libssh2 release yet, but the upstream patch adds a check against LIBSSH2_PACKET_MAXPAYLOAD, and the article recommends inventorying bundled or static copies, applying patched builds, and restricting outbound SSH connections to trusted servers until patched.

  • Because libssh2 is often statically linked or bundled inside products, a normal Linux distribution package update may leave vulnerable copies behind and organizations may not know where those copies exist.
  • The public proof-of-concept is not described as a turnkey remote exploit, so immediate risk depends on how libssh2 is embedded in a target application, allocator behavior, and deployed mitigations.
  • The flaw is a repeat of a bug class libssh2 addressed in 2019, with the article noting a near-identical integer-overflow issue in the same transport-read area of the code.
  • Clients that connect to external SSH servers or to hostnames an attacker could redirect should be prioritized, because the vulnerable code is reached when a client connects to an untrusted SSH endpoint.

  • 02The flaw affects every libssh2 release up to and including 1.11.1 and has a CVSS 4.0 score of 9.2.
