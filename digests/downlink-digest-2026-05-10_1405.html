Researchers have disclosed a critical out-of-bounds read vulnerability in Ollama, tracked as CVE-2026-7482, that allows unauthenticated attackers to leak entire process memory. The flaw affects versions prior to 0.17.1 and stems from unsafe memory handling in the GGUF model loader during tensor quantization. Attackers can exploit this by uploading a crafted GGUF file with inflated tensor shapes to trigger the heap read. Once triggered, sensitive data such as API keys, environment variables, and conversation histories can be exfiltrated via the /api/push endpoint.

In addition to this memory leak, two unpatched vulnerabilities in Ollama’s Windows client enable persistent remote code execution. These flaws combine a missing signature verification and a path traversal issue in the auto-update mechanism to drop malicious executables into the Windows Startup folder. Users are strongly advised to apply the latest patches, disable automatic updates on Windows, and restrict network exposure until official fixes are released.

Cybersecurity researchers have identified a critical out-of-bounds read vulnerability in Ollama, designated as CVE-2026-7482 and codenamed Bleeding Llama. This flaw impacts over 300,000 servers globally and carries a CVSS score of 9.1. The vulnerability resides in the GGUF model loader and is triggered when the /api/create endpoint processes a maliciously crafted GGUF file.

During the quantization process, the server reads past the allocated heap buffer due to the framework’s reliance on the Go unsafe package. Successful exploitation allows an attacker to dump the entire process memory, exposing environment variables, API keys, system prompts, and concurrent user conversations. Researchers demonstrated that this leaked data could be exfiltrated by pushing the resulting model artifact to an external registry through the /api/push endpoint.

The impact is further amplified when Ollama is integrated with AI coding assistants like Claude Code, as all tool outputs are automatically flushed into the vulnerable heap. Separately, researchers at Striga detailed two unpatched vulnerabilities in Ollama’s Windows desktop client that chain together to achieve persistent remote code execution. The first flaw, CVE-2026-42248, involves a missing signature verification check that allows unsigned binaries to bypass integrity validation.

The second flaw, CVE-2026-42249, introduces a path traversal vulnerability that stems from unsanitized HTTP response headers used to construct local installer paths. When combined with the client’s default auto-update polling mechanism, an attacker controlling an update server can redirect the installer to the Windows Startup folder. This results in silent, persistent code execution at the user privilege level upon every system login.

Affected Windows versions range from 0.12.10 through 0.17.5, with interim mitigations recommending the disablement of automatic updates and the removal of the Ollama shortcut from the Startup directory.

Vulnerability Overview and Impact Researchers have disclosed a critical out-of-bounds read vulnerability in Ollama, tracked as CVE-2026-7482 and codenamed Bleeding Llama. The flaw affects versions prior to 0.17.1 and likely impacts over 300,000 servers worldwide. It carries a CVSS score of 9.1 and stems from unsafe memory handling within the GGUF model loader.

Exploitation Mechanics The vulnerability is triggered when the /api/create endpoint accepts an attacker-supplied GGUF file with inflated tensor offsets and sizes. During the quantization phase, the server reads past the allocated heap buffer in the fs/ggml/gguf.go and server/quantization.go modules. This behavior is enabled by Ollama’s use of the Go unsafe package, which bypasses standard memory safety guarantees.

Data Exfiltration and Amplified Risks Successful exploitation results in a complete leak of the Ollama process memory. Attackers can extract sensitive information including environment variables, API keys, system prompts, and concurrent user conversation data. The exfiltration chain involves three steps: uploading a crafted GGUF file, triggering the out-of-bounds read via /api/create, and pushing the resulting model artifact to an external registry using /api/push. The risk is significantly heightened when Ollama is connected to AI coding tools like Claude Code, as all tool outputs and conversation data are automatically saved to the vulnerable heap.

Windows Client Remote Code Execution Flaws In a separate disclosure, researchers at Striga outlined two unpatched vulnerabilities in Ollama’s Windows desktop client that chain into persistent remote code execution. The Windows client auto-starts on login, listens on 127.0.0.1:11434, and periodically polls for updates via the /api/update endpoint.

The first flaw, CVE-2026-42248, involves a missing signature verification check that does not validate update binaries prior to installation. The second flaw, CVE-2026-42249, introduces a path traversal vulnerability caused by unsanitized HTTP response headers being used to construct the local staging directory path.

Attack Chain and Persistence When combined, these flaws allow an attacker controlling an update server to execute arbitrary code at the user privilege level. By overriding the OLLAMA_UPDATE_URL to point at a malicious server, an attacker can supply a crafted executable that bypasses signature checks and leverages path traversal to write directly to the Windows Startup folder. This results in persistent, silent code execution on every system login.

Affected Versions and Mitigations The Windows vulnerabilities affect versions 0.12.10 through 0.17.5, with researchers noting that installations up to version 0.22.0 remain vulnerable. Interim mitigations for the Windows client include disabling automatic updates and removing the Ollama shortcut from the %APPDATA%\Microsoft\Windows\Start Menu\Programs\Startup directory.

For all Ollama instances, security guidance emphasizes applying the latest patches, limiting network exposure, and isolating servers behind firewalls. Since the REST API lacks built-in authentication, deploying an authentication proxy or API gateway is strongly recommended to prevent unauthorized access.

EXEC Executive Overview

Covering intelligence reported during the May 10, 2026, 10:00–14:00 UTC window, this digest focuses on a cluster of critical vulnerabilities disclosed within the Ollama local large language model framework. These findings expose significant architectural weaknesses in both the core inference engine and the Windows distribution mechanism, creating unauthenticated attack vectors with direct implications for credential theft and persistent remote code execution.

The primary finding, tracked as CVE-2026-7482, identifies a critical out-of-bounds read condition in Ollama versions prior to 0.17.1. This vulnerability enables unauthenticated attackers to trigger a process memory leak by supplying a maliciously crafted GGUF model file. During the parsing and loading of the GGUF artifact, the out-of-bounds read condition causes the application to expose arbitrary memory regions, which researchers have confirmed can yield sensitive operational data. Exposed contents include stored API keys, authentication tokens, and plaintext conversation histories cached within the process memory space. The vulnerability operates independently of network authentication, leveraging the framework’s default unauthenticated REST API endpoint to initiate the memory disclosure.

Compounding the memory disclosure risk is a pair of unpatched vulnerabilities affecting Ollama’s Windows update subsystem, designated as CVE-2026-42248 and CVE-2026-42249. These flaws target versions 0.12.10 through 0.17.5 and operate in tandem to facilitate persistent remote code execution at the user privilege level. CVE-2026-42248 introduces a path traversal condition within the update installer, while CVE-2026-42249 removes mandatory cryptographic signature verification during the update validation phase. When combined, an attacker can craft a malicious update payload that bypasses integrity checks, traverse directory boundaries, and execute arbitrary code upon installation. This update mechanism flaw establishes a reliable persistence channel that survives system reboots and does not require elevation of privilege beyond the current user context.

The convergence of these vulnerabilities presents a compounding threat profile for environments deploying Ollama for local AI model inference. The memory leak vulnerability provides immediate access to high-value secrets and operational metadata, while the Windows update flaws establish a durable execution pathway. Both attack vectors exploit the framework’s default configuration, which exposes the REST API without built-in authentication controls. The disclosed flaws highlight structural gaps in artifact validation, update integrity verification, and memory boundary enforcement within the local LLM runtime. Current disclosures indicate that the out-of-bounds read condition has been addressed in version 0.17.1, whereas the Windows update mechanism vulnerabilities remain unpatched across the specified version range. The technical details underscore how local AI inference frameworks, when deployed with default network configurations, introduce expanded attack surfaces that integrate traditional software supply chain and memory safety risks into the AI operations layer.
