Technical Mechanism and Classification

The vulnerability originates from the Linux kernel’s algif_aead cryptographic algorithm interface, which was introduced roughly nine years ago. Until official patches are released, organizations must weigh the security benefits against the loss of critical network and file system functionality.

The flaw, tracked as CVE-2025-68670, resides in the xrdp_wm_parse_domain_information function and triggers before client authentication occurs.

When processing a client domain name, the function copies data into a 256-byte stack buffer without proper bounds checking. Attackers can exploit UTF-16 to UTF-8 character expansion to craft a malicious domain name that overflows the buffer and overwrites the return address.

This allows arbitrary code execution within the xrdp server process without requiring valid credentials. The xrdp maintainers quickly patched the issue in version 0.10.5 and backported the fix to earlier releases.

Kaspersky researchers identified a critical remote code execution vulnerability in the xrdp server while conducting a security assessment of the Kaspersky USB Redirector module. The vulnerability, officially designated as CVE-2025-68670, exists within the xrdp_wm_parse_domain_information function.

This function processes the domain name sent by the client during the RDP Secure Settings Exchange phase, which occurs prior to authentication. The client transmits the domain name as a UTF-16 string with a maximum length of 512 bytes.

The xrdp server converts this data to UTF-8 before passing it to the vulnerable function. The function checks if the domain string begins with an underscore and then searches for a double underscore character to isolate a substring.

This substring is copied into a local resultIP buffer allocated on the stack with a fixed size of 256 bytes. The code uses g_strncpy without verifying the actual byte length of the converted UTF-8 string.

Attackers can exploit the character expansion inherent in UTF-16 to UTF-8 conversion to craft a domain name that exceeds the 256-byte limit. When the overflow occurs, the excess data overwrites the thread stack return address.

This allows the attacker to redirect execution flow and achieve arbitrary code execution within the xrdp server process. The xrdp project maintainers responded rapidly to the disclosure and released a patch in version 0.10.5.

They also backported the fix to versions 0.9.27 and 0.10.4.1. Although compiler stack canaries can block simple return address overwrites, the maintainers explicitly warn that relying on them alone is insufficient.

The article concludes by highlighting the importance of responsible disclosure and proactive security audits in strengthening open-source infrastructure.

Proof of Concept and Exploitation Details Kaspersky developed a proof of concept using a custom RDP file containing a malicious domain name. The payload utilized Russian Cyrillic capital K characters to leverage UTF-16 to UTF-8 length expansion.
