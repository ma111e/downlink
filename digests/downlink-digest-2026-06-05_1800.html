What Made the Attack Possible

The article describes modern software supply chains as automated pipelines that pull code, run tests, sign artifacts, and publish releases. These pipelines often hold high-value secrets such as API tokens, service account credentials, signing keys, and publishing tokens. The core risk is that these automated systems trust the code they execute.

A central example is GitHub Actions workflows triggered by pull_request_target. Unlike standard pull_request triggers, pull_request_target runs in the context of the base repository and can have access to secrets and write permissions even when the pull request comes from an external contributor. The article says this behavior is by design and was exactly what TeamPCP exploited.

The article identifies several architectural decisions that enabled the campaign:

  • Git tags are mutable and can be silently rewritten.
  • Workflows using pull_request_target can expose secrets to external contributors if misconfigured.
  • Service accounts can be over-permissioned beyond their intended CI/CD role.
  • Cached npm and PyPI tokens on developer machines can become lateral movement vectors.
  • Malicious code can execute legitimate tools afterward, making CI jobs appear normal.
  • Monitoring systems often lack cross-platform correlation across GitHub, Docker Hub, npm, PyPI, Kubernetes, and other environments.
