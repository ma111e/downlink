Overview

Cybersecurity researchers disclosed details of three now-patched vulnerabilities affecting LangGraph, an open-source framework created by LangChain for building complex, stateful, and multi-agent AI agentic applications. The flaws include a critical vulnerability chain that could result in remote code execution.

Check Point said an SQL injection in a LangGraph function could allow attackers to gain full control via remote code execution of a server by exploiting weaknesses in how the system processes and handles data. Security researcher Yarden Porat is credited with discovering and reporting all three flaws.

Affected Products and Vulnerabilities

The article identifies three vulnerabilities:

  • CVE-2025-67644: A SQL injection vulnerability in LangGraph’s SQLite checkpoint implementation that allows attackers to manipulate SQL queries through metadata filter keys. It affects langgraph-checkpoint-sqlite versions before 3.0.1 and has a CVSS score of 7.3.
  • CVE-2026-28277: An unsafe msgpack deserialization vulnerability in LangGraph that could be used to trigger object reconstruction when a checkpoint is loaded by an attacker who can modify checkpoint data. It affects langgraph versions before 1.0.10 and has a CVSS score of 6.8.
  • CVE-2026-27022: A RediSearch Query Injection in @langchain/langgraph-checkpoint-redis that can be used to bypass access controls. It affects @langchain/langgraph-checkpoint-redis versions before 1.0.1 and has a CVSS score of 6.5.

Exploitation Conditions

Check Point said the vulnerability chain is exploitable in self-hosted deployments using the SQLite or Redis checkpointer with user-controlled filter input. The article states that LangChain’s managed platform, LangSmith Deployment, is not affected.

The remote code execution chain specifically depends on the application exposing the get_state_history() endpoint. That endpoint allows an attacker to retrieve historical checkpoints based on their metadata.

Remote Code Execution Chain

According to Yarden Porat, CVE-2025-67644 and CVE-2026-28277 can be chained to achieve remote code execution.

The article describes the attack sequence as follows:

  1. The attacker prepares a msgpack payload containing instructions to execute arbitrary code.
  2. The attacker sends a malicious filter parameter that exploits the SQL injection vulnerability.
  3. The SQL injection causes a fake checkpoint row to be returned in the database query results.
  4. The fake row’s checkpoint column contains attacker-controlled serialized data.
  5. When the application processes the query results, it deserializes the malicious checkpoint’s BLOB.
  6. The attacker exploits the unsafe deserialization vulnerability to execute the payload, resulting in remote code execution on the server.

LangGraph’s Characterization of CVE-2026-28277

LangGraph described CVE-2026-28277 as a post-exploitation issue. The maintainers said successful exploitation requires the ability to write attacker-controlled checkpoint data and turn that into code execution in the application runtime.

LangGraph also said the issue does not pose risks to existing LangSmith-hosted deployments. In the described threat model, an attacker must tamper with the checkpoint persistence layer used by the deployment, and typical hosted configurations are designed to prevent such access.

The maintainers said escalation from write access to the checkpoint store to code execution may expose runtime secrets or provide access to other systems the runtime can reach.

Significance

Check Point said the findings show how classic vulnerability classes such as SQL injection can become more potent when they appear inside AI agent frameworks. The article notes that these frameworks may carry elevated access and trust, which can open the door to sensitive data exposure.

The issue is significant for self-hosted LangGraph deployments because the vulnerable conditions involve checkpoint storage, user-controlled filter input, and exposed state-history retrieval functionality. The managed LangSmith Deployment platform is explicitly stated to be unaffected.

Recommended Response

Users are advised to apply the latest fixes for the affected packages. The article also recommends implementing authentication for self-hosted LangGraph servers.

Additional recommended measures include:

  • Avoiding long-lived static secrets.
  • Enforcing network segmentation.
  • Treating AI agents as privileged identities.
  • Applying the principle of least privilege to limit an agent’s access footprint.
  • The remote code execution chain depends on the application exposing the get_state_history() endpoint and accepting user-controlled metadata filters, making endpoint exposure and input control central to risk reduction.
  • A flaw described as post-exploitation by LangGraph can become part of a pre-authored remote code execution chain when paired with SQL injection that injects attacker-controlled checkpoint data into query results.
  • The issue highlights that checkpoint stores in AI agent frameworks should be treated as highly sensitive persistence layers because tampering with checkpoint data can expose runtime secrets or provide access to other systems reachable by the runtime.
  • Self-hosted LangGraph deployments face different risk than LangChain's managed LangSmith Deployment platform, so organizations should not assume managed-platform protections apply to their own deployments.
