Summary
A new macOS ClickFix campaign tricks users into executing a Terminal command from fake CAPTCHAs that automatically downloads, mounts, and launches a DMG containing the Atomic macOS Stealer infostealer, evading the need for manual user interaction with the disk image.

A new macOS ClickFix campaign leverages social engineering to infect users with the Atomic macOS Stealer (AMOS) infostealer. It begins with a fake CAPTCHA page instructing victims to open Terminal and paste a command. This single command uses curl to quietly download a malicious DMG file from an attacker-controlled server, then employs hdiutil with the -nobrowse flag to silently mount it, and finally searches for and launches the contained application without any visual indication to the user.

The payload, a self-signed application bundle named “NNApp.app,” belongs to the Atomic Stealer family. Once executed, it displays a fake System Preferences password prompt to steal the user’s login credentials. It targets multiple Chromium-based and Firefox-derived browsers to harvest cookies, login data, autofill information, payment cards, and browser profiles. Additionally, it collects cryptocurrency wallet data from numerous wallets, Telegram and Discord data, Apple Notes databases, Safari cookies, Keychain files, and user documents.

The malware also replaces legitimate Ledger Live and Trezor Suite installations with malicious versions, likely to facilitate further cryptocurrency theft. The harvested data is compressed into a ZIP archive and exfiltrated to attacker-controlled servers, such as svs-verificationdate[.]beer and 196.251.107[.]171. Users are warned never to run Terminal commands prompted by websites claiming to be part of CAPTCHA checks or troubleshooting steps, as fully automated infection chains like this can compromise sensitive financial and personal data with minimal user interaction.

  • The attack fully automates DMG delivery by having the single Terminal command download, silently mount (using hdiutil -nobrowse), and auto-launch the malware, eliminating the need for the user to manually open the disk image, which makes the infection chain harder to notice.
  • The malware specifically replaces legitimate installations of Ledger Live and Trezor Suite with trojanized versions, a technique that can compromise cryptocurrency holdings directly, beyond just stealing wallet files or credentials.
  • The use of the '-nobrowse' flag with hdiutil hides the mounted volume from Finder and the desktop, preventing the user from seeing the malicious disk image after it is mounted, which increases stealth.

Summary
The Miasma campaign exposed a thriving Developer Credential Economy where stolen developer credentials circulate for weeks before being used in automated supply chain attacks, highlighting the need for exposure management over reactive detection.

The Miasma campaign was a self-propagating npm worm that compromised over 89 packages across three waves from June 1–5, affecting repositories under Red Hat, Vapi.ai, and Microsoft Azure. Derived from the open-sourced Mini Shai-Hulud toolchain, the attack exploited a stolen Red Hat developer credential and session cookie that had been sitting in infostealer logs for seven weeks before weaponization. The worm harvested a wide range of secrets from every infected environment, turning each machine that ran npm install into a credential generation node that fed subsequent attack waves.

Technically, the campaign escalated rapidly: Wave 1 delivered malicious packages with valid SLSA Build Level 3 provenance attestations, demonstrating that signed builds do not prove a clean pipeline. Wave 2 introduced the Phantom Gyp technique, abusing a binding.gyp file to execute payloads and bypass the preinstall-script monitoring defenders had just deployed. Wave 3 expanded the attack surface further by dropping persistence files targeting AI coding assistants (Claude Code, Cursor, Gemini CLI, VS Code), ensuring compromise could trigger simply by opening an infected repository in a developer’s IDE.

The campaign is the clearest example to date of the Developer Credential Economy, a three-layer model where credential generation via supply chain compromise feeds underground distribution markets, which in turn enable multiple threat actors to weaponize access for varied objectives. Miasma validated that EDR and reactive detection are insufficient because credential theft occurs in ephemeral CI/CD environments where endpoint tools have no visibility. The public release of the full Mini Shai-Hulud source code in May permanently lowered the barrier for copycat operators, making proactive exposure management and human-gated publishing controls essential to disrupt the attack chain.

  • The Miasma campaign is the first observed supply chain attack that systematically targets AI coding assistants by dropping hidden instruction-layer override files, expanding the attack surface from registries to developer IDEs and AI-assisted workflows.

In plain words

GitHub is changing a built-in automation tool so that when a contributor from outside a project submits code changes, the tool will no longer automatically run that code with the project’s full permissions. Before this fix, attackers could hide malicious code in a contribution and have it executed with access to secrets and write privileges, which let them steal credentials or plant backdoors. The update blocks that common trick by default, closing a door that has been used in recent attacks against several widely used software tools and libraries. This means open-source projects that rely on GitHub’s automation are now safer from that kind of supply chain hijacking.

Summary
GitHub is hardening actions/checkout to refuse fetching fork pull request code by default in pull_request_target and workflow_run workflows, blocking a common attack vector that lets malicious pull requests run with the base repository’s full privileges and steal secrets.

GitHub is updating its official actions/checkout action to block a common attack pattern known as pwn requests. Effective June 18, 2026, version 7 will refuse to fetch code from fork pull requests by default when used inside pull_request_target and workflow_run workflows. The same change will be backported to all currently supported major versions on July 16, 2026. Workflow authors can still opt in to the risky behavior by explicitly setting the allow-unsafe-pr-checkout input to true.

The update addresses the danger of the pull_request_target trigger, which runs automatically with the base repository’s full privileges—including secrets, a read-write GITHUB_TOKEN, and default-branch cache access. If a workflow checks out and executes code from an untrusted fork, an attacker can steal secrets or gain write access. This technique has been exploited in recent supply chain attacks against projects such as Nx (the s1ngularity campaign), PostHog, TanStack, and kubernetes-el.

The new default blocks checkouts where the repository input points to the fork PR repository, the ref input matches refs/pull/*/head or refs/pull/*/merge, or the ref resolves to a fork PR’s head or merge commit SHA. The protection is limited to the actions/checkout action; workflows that use git, the GitHub CLI, or other checkout methods are not covered, and pwn requests triggered through other event types like issue_comment remain out of scope. GitHub and security firm Socket both emphasize that this is a guardrail, not a complete solution, and that any workflow running untrusted code in a privileged context still requires careful review.

  • The protection is confined to the actions/checkout action; workflows that check out fork code using git commands or other actions are not covered, leaving a significant attack surface still open.
  • The change is backported to all currently supported major versions, so existing pipelines relying on the risky pattern will break unless developers actively opt out, potentially causing widespread CI disruption.
  • Even with the new default, any workflow that executes untrusted code in a privileged event context remains a pwn request risk, making the fix a limited guardrail rather than a resolution of the underlying pattern.
  • The article cites four real-world projects (Nx, PostHog, TanStack, kubernetes-el) recently compromised through this exact mechanism, demonstrating that even mature, security-aware teams are susceptible without platform-enforced defaults.
