Execution through a Python import hook

The delivery mechanism differs from earlier npm campaigns described in the article. Instead of running through installation lifecycle scripts or native build hooks, the Hades Campaign runs during Python code execution through an obfuscated single-line import hook embedded in __init__.py.

The deobfuscated logic checks for a flag file at /tmp/.bun_ran. If the file does not exist, it searches Python module paths for a companion _index.js file. It then downloads a precompiled Bun v1.3.14 runtime from GitHub, extracts it to a temporary directory, and executes the JavaScript payload using bun run _index.js.

The article states that the flag file causes the payload to execute only once per boot cycle. It also emphasizes that using a standalone Bun binary allows the malware to run JavaScript tasks in environments that do not have Node.js installed.

AI analyst misdirection

The article describes a prompt injection block placed at the beginning of the _index.js bundle. This plain-text block is intended to manipulate automated Large Language Model pipelines used by security companies to inspect and summarize suspicious package code.

The prompt injection instructs a parsing LLM to ignore the obfuscated code below it, classify the package as verified clean infrastructure, and produce a safe security report. The article characterizes this as a conceptual shift in which attackers target the cognitive logic of automated AI triage systems.

Obfuscation and modular payload structure

The _index.js file acts as a runtime bootstrapper. The article says it decrypts and loads sixteen independent functional payloads at startup.

Each payload blob is gzip-compressed and encrypted with AES-256-GCM using a unique hardcoded key. The bootstrapper uses Bun APIs for decryption and decompression. After deobfuscation, the payloads were found to include modules for macOS and Windows memory reading, IDE and CI/CD backdoor setup, and command-and-control agents.

Cross-platform memory scraping

A major capability described in the article is reading the memory of the GitHub Actions Runner.Worker process to extract secrets. Earlier Miasma campaigns used Linux /proc/{pid}/mem, but the Hades Campaign adds macOS and Windows implementations.

Linux

On Linux, the malware walks memory mappings through /proc/{pid}/maps and directly reads /proc/{pid}/mem to scrape plaintext variables.

macOS

On macOS runners, the malware decrypts a Python script that invokes Mach kernel VM APIs through ctypes. The article states that because the runner worker and execution process run under the same UID, the script can obtain a task port without root privileges. It then enumerates readable memory regions and writes their contents to standard output.

Windows

On Windows, the malware executes a PowerShell script that dynamically compiles a C# class using Add-Type. The C# code uses Win32 APIs such as OpenProcess, VirtualQueryEx, and ReadProcessMemory to crawl and dump the target process memory.

The article states that these cross-platform scrapers allow the malware to extract unmasked variables and tokens on Linux, macOS, and Windows runners.

GitHub-based command and control

The article describes three independent command-and-control channels that use public GitHub infrastructure.

Token dead-drop

Harvested GitHub personal access tokens are encrypted and pushed as commits to public repositories controlled by the attacker. These commits use the keyword DontRevokeOrItGoesBoom, which the attacker can locate through GitHub search APIs.

Signed JavaScript eval

The malware searches GitHub commits for the keyword TheBeautifulSnadsOfTime. Matching commit messages contain base64-encoded JavaScript payloads and RSA-PSS signatures. The malware verifies the signatures against a hardcoded public key and executes valid payloads using eval().

Python dropper

The campaign introduces a Python-specific C2 channel through a script named updater.py. This service polls GitHub for commits matching firedalazer. Matching commits encode a URL and RSA-PSS signature; if the signature is valid, the daemon downloads and executes the script from the URL.

Exfiltration process

The article says stolen credentials are encrypted locally before exfiltration. The process includes:

  • JSON serialization of harvested secrets
  • gzip compression
  • generation of a random 256-bit AES key
  • AES-256-GCM encryption of the data
  • RSA-2048 encryption of the ephemeral AES key using the attacker’s public key
  • upload of the encrypted payload to a newly created public GitHub repository under the attacker’s control

The exfiltration repositories use Hades-themed names such as stygian-cerberus-42817 and tartarean-charon-18401. They carry the description Hades - The End for the Damned.

Worm propagation and lateral movement

The article describes three self-replication vectors.

SSH and SCP lateral movement

The malware parses ~/.ssh/known_hosts and ~/.ssh/config to identify reachable hosts. If ssh and scp are available, it attempts batch-mode connections using configured keys.

On successful access, it stages a loader script and the primary payload in a temporary directory on the remote host, runs the loader, and deletes the staging directory.

PyPI and npm OIDC trust exploitation

When running inside a GitHub Actions workflow runner, the malware checks for OIDC variables such as ACTIONS_ID_TOKEN_REQUEST_TOKEN and ACTIONS_ID_TOKEN_REQUEST_URL. If available, it attempts to mint PyPI or npm publishing tokens by calling the OIDC endpoint with registry audiences.

The article says the malware also generates Sigstore SLSA provenance bundles. It requests a signing certificate from Fulcio, formats a SLSA provenance statement, DSSE-signs it, uploads it to Rekor, and packages the certificate, signatures, and inclusion proof into a Sigstore bundle. It then injects its payload into target libraries and publishes compromised versions to PyPI and npm using the organization’s automated publishing credentials and the generated provenance bundle.

GitHub Actions workflow injection

The malware targets GitHub repositories to extract organization secrets if a harvested token has suitable permissions.

If the token has the workflow scope, it uses a push-triggered workflow technique. It creates a temporary branch, commits a workflow file under .github/workflows/codeql.yml, runs a workflow named Run Copilot, uploads secrets as an artifact named format-results, then deletes the run history and temporary branch.

If the token lacks workflow scope but has repository write access, it uses a deployment-triggered technique. It commits a workflow file, immediately deletes it in a second commit, points a branch to the deletion commit, and triggers a GitHub deployment against the earlier commit that still contains the workflow. The workflow dumps secrets to an artifact, after which the malware deletes the deployment record, workflow run, and branch.

AI coding assistant and IDE rules hijacking

The malware also backdoors local workspace folders. It searches for rule files and configuration directories used by AI agents and IDE systems, including Claude, Codex, Gemini, Copilot, Cline, Aider, Tabby, Amazon Q, Cody, Bolt, and Continue.

Persistence and wiper deterrent

For persistence, the malware installs an update-monitor C2 polling daemon. It also installs a second service named gh-token-monitor.

The gh-token-monitor service polls the GitHub API using the stolen token. If the token is revoked and GitHub returns a 4xx HTTP status, the script triggers a destructive command that removes the user’s home directory and Documents directory. The article states that the service runs for up to 72 hours.

On Linux, the service is registered as a user-level systemd service with user lingering enabled. On macOS, it is installed as a LaunchAgent. The article says the threat actor uses the risk of data destruction to discourage immediate revocation of stolen credentials.
