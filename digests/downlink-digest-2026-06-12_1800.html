Affected Product and Deployment Context

The affected product is Splunk Enterprise, specifically versions 10 and above according to the article’s reading of the advisory. The vulnerable component is the PostgreSQL Sidecar Service Endpoint.

The article describes the following deployment differences:

  • Splunk Enterprise on AWS has the PostgreSQL Sidecar Service installed and enabled by default.
  • One on-premise Windows installation scenario does not install the PostgreSQL Sidecar Service by default.
  • Another on-premise Windows installation scenario installs the PostgreSQL Sidecar Service but does not enable it by default.

The researchers state that the concept of a sidecar appears to have been introduced in Splunk version 10, which aligns with the affected version range from the advisory.

Discovery of the Vulnerable Service

The researchers located sidecar services under /opt/splunk/var/run/supervisor/pkg-run/ and focused on the service with postgres in its name. They confirmed that a splunk-postgres binary was listening locally on ports including 127.0.0.1:5435.

Although the service was only bound to the loopback interface, the researchers suspected that it might still be reachable through another Splunk component. They searched the binary and documentation for PostgreSQL sidecar API routes and identified endpoints under /v1/postgres/, including:

  • /v1/postgres/telemetry
  • /v1/postgres/health
  • /v1/postgres/recovery/backup
  • /v1/postgres/recovery/restore
  • /v1/postgres/recovery/status/{id}
  • /v1/postgres/status

Web Proxy Access to the Local Service

The researchers found that Splunk’s main web application could proxy requests to the local PostgreSQL Sidecar Service API. They used a request to /en-US/splunkd/__raw/v1/postgres/recovery/backup through the main Splunk web interface.

The request included a Basic Authentication header with empty credentials and a JSON body specifying a database and backup file. The endpoint returned a successful response indicating that a backup job had entered a BackupPending state.

The article explains that the vulnerability exists because the PostgreSQL sidecar service endpoint lacks authentication controls, allowing a network-reachable user to invoke file operations without valid credentials.

Initial File Creation and Truncation Primitive

The advisory described arbitrary file creation and truncation, and the researchers reproduced that behavior. By submitting a backupFile value, they caused Splunk to create a file in the PostgreSQL sidecar service directory.

They then used path traversal in the backupFile value to create a file outside the service directory, including under /tmp. At this stage, the researchers summarized their capability as being able to create an empty file at any location and clear the contents of existing files through overwrite or truncation.

The article notes that this primitive alone did not appear to fully explain the 9.8 CVSS score, leading the researchers to investigate further.

Use of pg_dump and User-Controlled Arguments

The researchers analyzed the sidecar service’s backup behavior and identified that the backup endpoint ultimately invoked pg_dump. The command construction included several important arguments:

  • -h localhost
  • -p with the local PostgreSQL port
  • -w to prevent password prompts
  • -U using the username supplied through Basic Authentication
  • -f using the attacker-controlled backupFile
  • the database name using the attacker-controlled database parameter

This explained why arbitrary Basic Authentication usernames were accepted by the Splunk-side endpoint: Splunk forwarded the supplied username to PostgreSQL and relied on PostgreSQL to decide whether authentication should succeed.

Because the researchers could not initially authenticate to the local database, pg_dump did not produce useful dump content and the files remained empty.

PostgreSQL Connection String Behavior

The researchers then relied on documented PostgreSQL behavior where the dbname value can be a connection string. The article quotes PostgreSQL documentation stating that connection string parameters can override conflicting command-line options.

Because the sidecar service passed the user-controlled database value to pg_dump, the researchers supplied connection-string parameters such as hostaddr. This overrode the hardcoded localhost connection and caused Splunk to connect to an attacker-controlled PostgreSQL server.

They then configured an attacker-controlled PostgreSQL database and used the backup endpoint to make Splunk dump that external database to a file on the Splunk filesystem. This gave them the ability to write database dump content, rather than only empty files, to the target system.

Restore Endpoint and Local Database Access

The article then turns to the /v1/postgres/recovery/restore endpoint, which follows a similar structure to the backup endpoint but invokes pg_restore. The restore command also included user-controlled values, including the Basic Authentication username, the database argument, and the backup file path.

The researchers’ plan was to:

  • Force Splunk to dump an attacker-controlled database into an arbitrary file.
  • Force Splunk to restore that dump through pg_restore.
  • Cause SQL contained in the attacker-controlled dump to execute in Splunk’s local PostgreSQL instance.

A key obstacle was authentication to the local PostgreSQL instance. The researchers found that Splunk had a .pgpass file at /opt/splunk/var/packages/data/postgres/.pgpass, which contained credentials for the postgres_admin user. They then used the PostgreSQL passfile connection-string parameter to point pg_restore at that file and authenticate successfully.

Controlled Arbitrary File Write

After gaining the ability to restore attacker-controlled SQL into the local PostgreSQL instance, the researchers created a database dump that would produce a controlled file write. The article shows a template using a PostgreSQL function that calls lo_from_bytea and lo_export to write attacker-controlled content to an attacker-chosen file path.

The function was designed to execute during the restore process by being referenced in a table check constraint and triggered during data insertion. This allowed the researchers to write controlled content to the filesystem as the splunk user.

They demonstrated the result by writing the string pwned to /tmp/pwn.

Remote Code Execution

With controlled arbitrary file write achieved, the researchers moved to remote code execution. They observed that Splunk frequently executed the Python script /opt/splunk/etc/apps/splunk_secure_gateway/bin/ssg_enable_modular_input.py.

They overwrote that script with a Python payload that executed an operating system command to write the output of id into Splunk’s web-accessible directory. After the script was executed, the output file appeared in the Splunk webroot.

The article presents this as achieving remote code execution as the splunk user.

Detection Artifact

The article closes by sharing a Detection Artefact Generator hosted in a GitHub repository. The authors state that the tool is neutered and only determines whether access to /v1/postgres/recovery/backup is blocked.

According to the article, the detector reports vulnerability when the endpoint does not block access while any credentials are provided in the Authorization header, and reports non-vulnerability when the endpoint returns a 401 status code.

  • The PostgreSQL Sidecar Service listened only on 127.0.0.1, but Splunk’s externally reachable web application exposed it through the /splunkd/__raw/ proxy path, making the localhost-only binding ineffective.
  • The sidecar API accepted arbitrary Basic Authentication usernames because Splunk passed the supplied username directly to pg_dump or pg_restore and relied on PostgreSQL to enforce authentication.
  • PostgreSQL’s documented behavior allowing a dbname value to be a connection string let the researchers override Splunk’s hardcoded localhost database target and force Splunk to connect to an attacker-controlled PostgreSQL server.
  • Splunk’s local .pgpass file contained usable PostgreSQL credentials and could be referenced through the passfile connection-string parameter, enabling authenticated restore operations against the local PostgreSQL instance.
  • The researchers converted an initial empty-file creation and truncation primitive into controlled arbitrary file write by restoring a malicious PostgreSQL dump that executed lo_export during the restore process.

Second Wave

The article also describes a second wave that used:

  • bun install js-digest

This wave was pushed from a separate set of accounts that community trackers linked to the same npm publisher as atomic-lockfile. Its payload was a different ELF binary by hash, and the community also flagged it as malicious.

Malware Capabilities

Independent researcher Whanos reverse-engineered the deps payload and described it as a Rust credential stealer aimed at developer workstations and build systems.

The malware collects:

  • Cookies, tokens, and local storage from Chromium-based browsers such as Chrome, Edge, and Brave.
  • Session data from Electron apps including Slack, Discord, and Microsoft Teams.
  • GitHub, npm, HashiCorp Vault, and OpenAI/ChatGPT bearer material and account metadata.
  • SSH keys, known_hosts, and shell histories.
  • Docker and Podman credentials.
  • VPN profiles.

The article says stolen files are sent over HTTP to temp.sh. Command and control runs through a Tor onion service via a local loopback proxy.

Persistence

The malware installs persistence using systemd with Restart=always.

When it runs with root privileges, it:

  • Copies itself under /var/lib/.
  • Writes a unit under /etc/systemd/system/.

When it runs as a normal user, it:

  • Uses the home directory.
  • Creates a per-user unit under ~/.config/systemd/user/.

The article stresses that removing the AUR package is not enough once the payload has executed, because the package manager can only remove files it knows about.

eBPF Rootkit Behavior

The article says early write-ups overstated the eBPF rootkit. The eBPF component is optional and only loads when the binary already has root and the required capability.

It is not used to gain privileges. When it activates, it hides the malware’s own:

  • Processes.
  • Process names.
  • Socket inodes.

It uses pinned BPF maps named:

  • hidden_pids
  • hidden_names
  • hidden_inodes

The malware also kills attempts to attach a debugger. Because the eBPF rootkit can hide evidence after root execution, the article advises reinstalling from trusted media if the payload ran as root.

Sonatype’s first write-up counted more than 20 hijacked packages. Within a day, community trackers and the Arch aur-general thread had cataloged over 400 packages.

One master list compiled by grepping the AUR git mirror put the number at around 408, while consolidated lists were climbing higher. The article says the affected package list is large, still growing, and not complete.

The atomic-lockfile npm package itself showed only 134 weekly downloads on Socket before it was removed from the registry. The article therefore frames the main exposure as the AUR build path rather than direct npm installs.

ThreatLabz assesses with high confidence that earlier Shai-Hulud waves are linked to TeamPCP, which Mandiant tracks as UNC6780. Major Developments

The article identifies several key developments after Shai-Hulud V2:

Campaign Evolution Across the Trust Stack

The article emphasizes that Shai-Hulud is notable for moving systematically through multiple layers of the software supply chain trust stack. Earlier waves focused on maintainer authentication and package install-time execution. Later waves moved into security tooling, provenance systems, OIDC-based publishing, developer tooling, and AI-assisted analysis.

The campaign evolved from npm postinstall and preinstall hooks into Python .pth files, which execute whenever the Python interpreter starts. It also evolved from GitHub dead drops to more takedown-resistant command-and-control approaches, including Internet Computer Protocol canisters and Session Protocol.

March 2026: Miasma and PyPI Expansion

The Miasma wave marked the campaign’s first major expansion beyond npm into PyPI. Instead of directly targeting package maintainers, TeamPCP compromised Aqua Security’s Trivy vulnerability scanner through GitHub Actions cache poisoning.

Trivy is widely used in build pipelines for container and dependency scanning. The attackers exploited the absence of version pinning in downstream consumers, meaning that pipelines installing Trivy without a pinned version downloaded and executed the attacker’s binary.

LiteLLM, a Python library for calling LLM providers, was among the first major victims. LiteLLM version 1.82.8 was published with a 34KB malicious file named litellm_init.pth placed in Python’s site-packages directory.

.pth File Persistence

The article highlights Python .pth files as a significant persistence advancement over npm install-time hooks. Python processes .pth files in site-packages during interpreter startup, so any process invoking Python can trigger the payload.

The article gives examples such as web server startup, test suite execution, Jupyter notebook sessions, and scheduled cron jobs. Unlike npm preinstall or postinstall hooks, .pth persistence can survive package reinstallation and may persist across virtual environment recreation if the base site-packages directory is affected.

May 2026: Hades and SLSA Build Level 3 Bypass

In May 2026, TeamPCP exploited a pull_request_target misconfiguration in the TanStack open-source monorepo. The article explains that pull_request_target executes in the context of the base repository rather than a contributor’s fork, which can grant access to repository secrets.

The reported attack chain involved a malicious contribution that triggered the privileged workflow. Malicious code executed inside the runner context and scraped an OIDC token from the Runner.Worker process memory. That token was then presented to GitHub’s OIDC federation endpoint to generate a valid npm publish token.

The attacker used the token to publish malicious packages from TanStack’s legitimate environment. The article stresses that no maintainer credentials needed to be stolen because the trusted environment itself became the access path.

Valid Provenance, Malicious Output

A key finding is that malicious artifacts can carry valid provenance when the trusted build process is abused. Within a six-minute window, 84 malicious artifacts were published across 42 @tanstack/ packages.

Those artifacts carried valid Sigstore provenance attestations through fulcio.sigstore.dev and were recorded in the Rekor transparency log. The attestations were cryptographically valid because the malicious build ran from inside the trusted system.

The article also notes that days later, on May 19, the campaign mass-republished the @antv data visualization namespace. Snyk reported roughly 314 versions published within a single six-second window. Public reporting also linked the broader wave to compromises affecting AI-related infrastructure packages associated with Mistral AI, Guardrails AI, UiPath, and OpenSearch.

Open-Sourcing of the Toolkit

On May 12, 2026, TeamPCP published the complete worm source code to GitHub under an MIT license with the commit message, “Open Sourcing The Carnage.” The release reportedly included full propagation code, an OIDC token scraping module, operational documentation for customizing encryption keys and C2 infrastructure, and a $1,000 Monero prize announcement on BreachForums for the largest supply chain attack built from the codebase.

The article states that this changed the threat landscape and made attribution harder. Before publication, multiple waves could be linked more straightforwardly to the same operator. After publication, copycat actors could reuse the same code and tradecraft, making malware overlap or similar operations insufficient for confident attribution.

June 1–2, 2026: Red Hat OIDC Trusted Publishing Abuse

The June Red Hat wave showed a different path into trusted publishing. Public reporting cited in the article indicates that a Red Hat engineer’s GitHub account was compromised, though the initial takeover method has not been publicly disclosed.

With access to the account, the attacker reportedly pushed orphan commits to internal Red Hat repositories, injected GitHub Actions workflows through those orphan commits, allowed the workflows to request OIDC tokens through normal trusted publishing flows, and published malicious packages with valid provenance generated by authorized infrastructure.

The result was 32 packages and 96 versions published within hours. As with the TanStack wave, the trust chain remained cryptographically valid while the identity and execution context at its root had been compromised.

C2 Camouflage in the Red Hat Wave

Samples from the June Red Hat wave introduced a network-layer evasion technique. They staged a C2 channel to route traffic to api.anthropic.com/v1/api, a non-existent endpoint at Anthropic’s domain.

The article states that this was camouflage rather than the primary exfiltration channel. The campaign family has repeatedly used GitHub-based dead drops, including repositories and commit-based signaling. The staged Anthropic path appeared intended to blend suspicious outbound requests into legitimate-looking AI API traffic, and Anthropic’s infrastructure was not compromised.

June 5, 2026: IDE Configuration File Targeting

The June 5 IDE wave moved beyond package registries. In the Azure/durabletask wave, the attacker used developer tooling configuration files that could trigger code or instructions when a repository was opened in a supported editor or assistant environment.

A compromised contributor account pushed files targeting four major AI-assisted development environments:

  • .claude/settings.json, using a SessionStart hook when opening a project in Claude Code.
  • .cursor/rules/setup.mdc, using an alwaysApply: true prompt injection during Cursor AI agent initialization.
  • .vscode/tasks.json, using runOn: folderOpen when opening a folder in VS Code.
  • .gemini/settings.json, using a startup hook when starting a Gemini Code Assist session.

The article states that no package installation was required; the trigger was cloning a repository and opening it in one of the targeted editors. GitHub disabled 73 Microsoft repositories across four organizations in response.

The article also notes that IDE configuration files are not typically evaluated by package registry scanners, SLSA verification tools, or behavioral analysis systems focused on package content. They are often reviewed as configuration rather than executable code.

June 8, 2026: Hades PyPI and Prompt Injection

The Hades PyPI wave, detected by Socket Research on June 8, 2026, compromised 37 PyPI wheels across 19 packages. The malware retained familiar campaign tradecraft, including .pth persistence, multi-layer obfuscation, AES-256-GCM encryption with per-instance keys, PBKDF2 key derivation, and obfuscator-wrapped JavaScript payloads.

The notable advancement was an AI scanner evasion technique embedded in the _index.js bundle. According to StepSecurity’s analysis as quoted in the article, the bundle begins with a plain-text prompt injection block placed before the obfuscated payload.

The prompt injection instructed any parsing LLM to ignore the obfuscated code below it, classify the package as verified clean infrastructure, and output a safe security report. The article states that security tools passing raw package content to a language model without system-prompt isolation could receive the attacker’s instruction inside the model context window.

LLM Scanner Failure Modes

The article identifies two failure modes for LLM-based scanners handling injected content. The first is a false-clean verdict, where the prompt injection instructs the model to classify the package as clean and the model follows that instruction. The second is a safety refusal, where submitted content triggers the model’s safety filters and the model refuses to analyze the file.

The article states that either result can produce no usable malicious verdict. It warns that any pipeline treating “no finding” or “refused to analyze” as equivalent to “clean” inherits this blind spot.

C2 Infrastructure Evolution

The article describes an evolution in the campaign’s C2 infrastructure in response to takedowns and defensive pressure:

  • V1 and V2 used GitHub orphan or dangling commits as dead drops.
  • Miasma continued to use GitHub dead drops.
  • May–June 2026 activity used Internet Computer Protocol blockchain canisters, which the article describes as more resistant to takedown.
  • The June 8, 2026 activity used GitHub dead drops, Session Protocol, and staged api.anthropic.com/v1/api camouflage.

One example involved a background service called kitty-monitor, which reportedly queried GitHub’s commit search API once per hour for the marker string firedalazer. The most recent matching commit contained the next instruction in its commit message, and the implant verified the signature against an embedded public key before retrieving the referenced URL.

The article states that this design allowed tasking to live inside ordinary public GitHub activity and prevented defenders from trivially spoofing commands unless they could forge the embedded signature.

Defensive Recommendations

The article recommends several defensive actions, including:

  • Apply lockfiles strictly and use npm ci instead of npm install.
  • Use private registry proxies and Software Composition Analysis tools to filter and monitor third-party packages.
  • Restrict open-source package consumption on corporate devices and CI systems to enterprise-controlled package managers.
  • Audit and remove unused packages to reduce dependency surface.
  • Apply least privilege with scoped, short-lived keys and tokens.
  • Enable phishing-resistant MFA such as FIDO2 and WebAuthn on npm, PyPI, GitHub, and cloud platforms.
  • Revoke and rotate npm tokens, GitHub PATs, cloud keys, and CI/CD secrets on suspected exposure.
  • Restrict build environments to internal mirrors and limit outbound network access.
  • Pin CI/CD tool versions, including scanners, formatters, and runtimes.
  • Audit pull_request_target usage in GitHub Actions workflows.
  • Monitor repositories with publish permissions for orphan commits and unexpected workflow files.
  • Monitor Python site-packages for unexpected .pth files, especially unusual names such as leading-hyphen or non-package names.
  • Treat IDE and AI-agent configuration files as executable code.
  • Do not treat SLSA or Sigstore provenance as proof of safety.
  • Enforce system-prompt isolation in LLM-based scanning pipelines.

  • Enforce a release cooldown period to prevent users from immediately checking out newly released packages.

Overview

Maine has taken its public data breach reporting portal offline after fraudulent breach disclosures were submitted and published on the state’s website. The fake disclosures impersonated Discord and VRChat. The Maine Attorney General’s Office described the submissions as apparent abuse of the state’s data breach reporting system.

The office said that after conversations with VRChat, one of the two affected companies, it became clear that the reported data breaches were hoaxes. According to the statement, the hoaxes were submitted by an unknown entity unrelated to either company. The false reports were removed from the database.

Affected parties

The fraudulent disclosures named two companies:

  • VRChat, described in the article as a multiplayer social virtual reality platform.
  • Discord.

VRChat told BleepingComputer that the filing was fraudulent and had been submitted using the name of a fictitious employee. After BleepingComputer contacted VRChat about the filing, the company confirmed the disclosure was fake and said it had not submitted the notice to Maine authorities.

BleepingComputer also contacted Discord about the fraudulent notice submitted to the site, but the article says Discord did not respond.

What the fake filing claimed

The fraudulent VRChat filing claimed that the company had suffered a data breach affecting more than 2.4 million people. The disclosure also included a fabricated employee contact name. VRChat disputed the filing and confirmed it was not submitted by the company.

The Maine Attorney General’s Office said it had no knowledge of any recent legitimate data breach reports from either VRChat or Discord.

How the portal was abused

Before the shutdown, submitted breach notices were automatically published to Maine’s public database. The Attorney General’s Office told BleepingComputer that it did not have independent knowledge of the breaches submitted through the system. It said the submitting entity filled out the information, and the information went directly onto the site.

This process allowed the fraudulent disclosures to appear in an official public database. The article states that the incident demonstrates how automatically published breach disclosures can be abused to spread misinformation and damage a company’s reputation.

Maine’s response

The Maine Attorney General’s Office temporarily disabled public access to the breach notification database. The office said it is reviewing reporting procedures to reduce similar abuse in the future. It also removed the false reports from the database.

Companies can continue to submit breach notifications through the reporting service. However, members of the public seeking copies of disclosures must now contact the Attorney General’s Office directly.

Significance of the portal

The article says Maine’s data breach portal is commonly used by journalists, researchers, and threat intelligence firms. These groups use the portal to monitor newly disclosed security incidents. They also use it to determine whether organizations are reporting cyberattacks or data breaches affecting consumers.

Because the portal was public and notices were automatically published, fraudulent submissions could be seen by people who rely on the database for breach monitoring. The article says it is unclear how many additional fraudulent breach notices may have been submitted through the portal before public access was suspended.

Known actor information

The article does not identify the entity that submitted the fake notices. The Maine Attorney General’s Office described the submitter as an unknown entity unrelated to VRChat or Discord. No named threat actor or group is attributed in the article.
