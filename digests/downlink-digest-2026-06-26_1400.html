In plain words

Splunk disclosed a critical flaw in Splunk Enterprise that can let someone with no login reach a hidden database recovery feature through Splunk’s website and make the server create or change files, eventually running their own code. The affected versions are Splunk Enterprise 10.2.x before 10.2.4 and 10.0.x before 10.0.7, while Splunk Enterprise 9.4 and earlier and Splunk Cloud Platform are not affected. This matters because many organizations use Splunk to collect and review security and operations records, so a compromise could let attackers tamper with logs, hide alerts, steal sensitive data, and use the Splunk server to move deeper into a network.

Summary
Splunk Enterprise has a critical unauthenticated RCE flaw, CVE-2026-20253, that is being actively exploited and can let attackers abuse exposed PostgreSQL recovery endpoints through Splunk Web to write files and run code as the Splunk service account. Organizations should urgently upgrade affected 10.2.x and 10.0.x versions, restrict Splunk Web exposure, or temporarily disable the PostgreSQL sidecar service where safe.

Splunk disclosed CVE-2026-20253 on June 10, 2026, a critical unauthenticated remote code execution vulnerability in Splunk Enterprise with a CVSS score of 9.8. The flaw stems from missing authentication on a PostgreSQL sidecar service recovery endpoint that can be reached through the Splunk Web interface. Splunk updated its advisory on June 18, 2026 to warn of active exploitation, and CISA added the vulnerability to its Known Exploited Vulnerabilities catalog the same day.

The affected versions are Splunk Enterprise 10.2.x prior to 10.2.4 and 10.0.x prior to 10.0.7. Splunk Enterprise 9.4 and earlier and Splunk Cloud Platform are not affected. The article recommends identifying all Splunk Enterprise instances, upgrading to a fixed release, restricting network access to the Splunk Web management interface, and temporarily disabling the PostgreSQL sidecar service if an immediate upgrade is not possible and the instance does not rely on Edge Processor, OpAmp, or SPL2 data pipelines.

Technically, the issue allows an unauthenticated attacker to send crafted requests to Splunk Web on port 8000, which proxies them to internal PostgreSQL recovery endpoints. The attack chain can progress from creating or truncating arbitrary files to injecting PostgreSQL connection string parameters, reusing plaintext PostgreSQL credentials stored in a .pgpass file, and overwriting a Python script that Splunk executes on a schedule. Successful exploitation runs code under the Splunk service account and can enable log tampering, alert suppression, sensitive data harvesting, persistence, and movement to other internal systems.

  • The vulnerable PostgreSQL sidecar listens only on 127.0.0.1:5435, but Splunk Web can expose it remotely through a proxy path, so removing direct internet access to Splunk Web on port 8000 is a practical way to reduce exploit reachability.
  • The recovery endpoints accept any Basic Authorization header value, including empty credentials, which means exploit attempts do not require valid Splunk credentials at any step.
  • The temporary workaround of disabling the PostgreSQL sidecar service can break Edge Processor, OpAmp, and SPL2 data pipelines, so organizations must check feature usage before applying it.
  • Attackers can reuse Splunk’s plaintext PostgreSQL credentials stored in /opt/splunk/var/packages/data/postgres/.pgpass by injecting a passfile parameter, turning a file-write path into authenticated database abuse without knowing the password.
  • Compromise of a Splunk Enterprise instance is especially damaging because the platform stores and analyzes security telemetry, allowing attackers to tamper with logs, suppress alerts, and harvest sensitive data from indexed events.
Key Points

  • 02The vulnerability affects Splunk Enterprise 10.2.x prior to 10.2.4 and 10.0.x prior to 10.0.7, while Splunk Enterprise 9.4 and earlier and Splunk Cloud Platform are not affected.

The group combines web shells and open-source tools like SoftEther VPN, Mimikatz, and VNT with TinyRCT to conduct reconnaissance, lateral movement, command execution, file theft, and screenshot capture.

A Chinese-speaking advanced persistent threat actor has been linked to a new custom backdoor called TinyRCT in attacks targeting government entities and critical infrastructure in Southeast Asia. Palo Alto Networks Unit 42 tracks the activity as CL-STA-1062 and said it overlaps with UAT-7237, a group previously associated with attacks on web infrastructure entities in Taiwan. The activity has focused particularly on state-owned enterprises in the energy and government sectors. In one September 2025 campaign, the actor infiltrated a Southeast Asian government entity, deployed a web shell to exfiltrate data from an MS SQL server, and conducted network reconnaissance against another government entity in the same country.   • CL-STA-1062 disguises tools as VMware executables or an XDR agent using names such as "XDRAgent.exe," "vmtools.exe," and "vmwared.exe," making ordinary-looking administrative or security-related filenames part of the observed activity.
  • TinyRCT delivery used a malicious "chrome_setup.zip" archive that contained a legitimate executable, a configuration file, and a rogue DLL to trigger AppDomainManager injection and download the backdoor.
  • TinyRCT communicates with its remote server over HTTP but encrypts exchanged data with AES-128 in CBC mode, using GET requests to receive instructions and POST requests to send stolen data.
  • Unit 42 observed the actor staging and exfiltrating an entire directory of web server source code from a government entity, showing that the campaign included theft of application source code as well as database-related data.
  • The same September 2025 activity included compromise of one government entity and reconnaissance against a separate government entity in the same country, which Unit 42 said suggested an effort to find lateral movement opportunities and broaden access.

In plain words

Several software packages from Immobiliare Labs used with Backstage, an internal developer portal, were secretly changed on npm so that installing them could run hidden code. The affected users are organizations and platform engineering teams using the listed GitLab and LDAP authentication plugins, especially in self-hosted Backstage setups. This matters because the hidden code tries to steal many kinds of passwords and access tokens from developer tools, cloud services, package registries, and GitHub Actions, and it may also try to spread by publishing altered software packages using stolen access.

Summary
Multiple Immobiliare Labs Backstage npm plugins were compromised with malicious patch releases that execute during npm install via a binding.gyp node-gyp hook. The payload steals a wide range of developer, cloud, CI/CD, and registry credentials and attempts persistence through AI coding assistant configurations.

On June 26, 2026, multiple versions of several npm packages maintained by Immobiliare Labs were found to contain a malicious payload that runs during npm install. The affected packages are @immobiliarelabs/backstage-plugin-gitlab, @immobiliarelabs/backstage-plugin-gitlab-backend, @immobiliarelabs/backstage-plugin-ldap-auth, and @immobiliarelabs/backstage-plugin-ldap-auth-backend. All compromised versions were published within a 30-second window as new patch releases across supported major release series.

The compromised packages include a 5 MB index.js file and a new binding.gyp file that were absent from prior releases. The binding.gyp file uses node-gyp shell expansion to execute node index.js, which bypasses tools that only monitor scripts.postinstall in package.json. The payload uses multiple layers of obfuscation, downloads the Bun JavaScript runtime from GitHub releases, writes the decrypted payload to a temporary file, executes it with Bun, and deletes the file.

Static analysis found that the payload targets credentials from GitHub Actions, cloud providers, Kubernetes, Vault, package registries, password managers, and SSH keys. It also includes an infectHost function that modifies configuration files for AI coding assistants including Claude Code, GitHub Copilot, Cursor, VS Code, Aider, Amazon Kiro, Sourcegraph Cody, Google Gemini, and OpenAI Codex. The payload contains functions suggesting the ability to publish modified packages to npm and PyPI using stolen registry credentials, consistent with self-propagating supply chain worm behavior described in the article.

  • Tools that only monitor package.json postinstall scripts may miss this attack because the payload is launched through binding.gyp and node-gyp shell expansion instead.
  • The attackers inserted malicious patch releases into every supported major release series at nearly the same time, increasing the chance that normal dependency updates would select a compromised version.
  • The payload uses Bun rather than Node.js because Bun does not support the --require hook interception used by many Node.js security monitoring tools.
  • The credential theft logic includes an unusual GitHub Actions technique: reading /proc/<pid>/mem from the Runner.Worker process to access masked secrets.
  • A package cooldown policy would have been directly relevant here because all compromised versions were newly published within a 30-second window on the day they were detected.

Threat actors are creating OpenAI tenants that impersonate legitimate companies and inviting employees to join them. Rolldown Pulls Rust React Compiler Integration After Binary Size Increase

In plain words

Vite and Rolldown removed a planned React Compiler feature written in Rust because it made the tool’s download about 5MB larger for everyone, even people who do not use React. This affects Vite and Rolldown users, React developers who wanted faster builds, and other framework communities because Vite is meant to serve many kinds of projects rather than favor one. The bigger issue is how web development tools can add faster native features without making every install heavier and slower to download.

Summary
Rolldown and Vite removed a Rust-based React Compiler integration because it increased the default binary size by about 5MB, which maintainers said was too much for a framework-agnostic tool used by many non-React projects. The team is exploring smaller and more opt-in approaches that could still bring React users native compiler speedups without making all users carry the cost.

Rolldown and Vite withdrew a Rust-based React Compiler integration after maintainers decided the binary-size cost was too high for a feature that would ship to all users by default. Boshen said the integration increased the binary size from 28.7MB to 33.8MB, a 17% increase. The work had been discussed for over a year and was intended to help React apps drop Babel pipelines when they only needed Babel for React Compiler.

The decision prompted debate because Vite is framework-agnostic and used across React, Vue, Svelte, Solid, Astro, and other projects. Critics argued that React-specific compiler support should not add weight to every install and should instead live in an opt-in plugin or package. Evan You said Vite cannot go down a path of adding several megabytes for one framework and then face the same question for others, especially with Vite downloaded 12 million times a week.

The article also describes the technical tradeoff behind the decision: a JavaScript plugin can preserve opt-in behavior, but a direct Rust integration can be faster by avoiding data passing and AST cloning overhead. The Oxc work showed that part of the size and speed cost came from returning a JSON-serialized Babel AST and parsing it back into Rust types, and removing that round trip reduced transform time and addon size. Boshen is now experimenting with a smaller implementation that he said shows up to a 2x performance improvement with only a 1.4MB binary size increase.

  • A roughly 5MB native-code increase was treated as significant because Vite is downloaded 12 million times per week and the cost is paid by every uncached install, including projects that do not use React.
  • The article shows a tension between opt-in plugins and direct native integration: a plugin can avoid shipping React-specific code to everyone, but the native Oxc path can be up to 50% faster because it avoids data passing and AST cloning overhead.
  • Part of the original binary and performance cost came from preserving the JavaScript plugin interface, including a JSON-serialized Babel AST round trip inside the same process, and removing that round trip reduced both transform time and addon size.
  • Native addon compression is being explored as a practical distribution fix, with one napi-rs proposal showing a Vite 8.1.0 native addon footprint dropping from 25.74MB to 8.15MB on darwin-arm64 while keeping runtime speed unchanged after decompression and caching.
Key Points
  • 01Rolldown and Vite withdrew a Rust-based React Compiler integration after maintainers decided the increase from 28.7MB to 33.8MB, or 17%, was too costly for a feature shipped to all users by default.
  • 02The integration originated from Oxc work intended to let React apps avoid Babel pipelines when they only needed Babel for React Compiler, and it was later wired into Rolldown behind a new transform.reactCompiler option before the Rolldown PR was closed on June 9.
  • 03Critics argued that React-specific compiler support should not add weight to Vite or Rolldown core because Vite is framework-agnostic and used across many frameworks.
  • 04Evan You said Vite cannot give special treatment to React without raising the same question for Vue, Svelte, Solid, Astro, and other frameworks, especially given Vite’s 12 million weekly downloads.
  • 05Boshen is experimenting with a smaller implementation that he said shows up to a 2x performance improvement with only a 1.4MB binary size increase, but the code still needs to be made maintainable and upstreamed.
