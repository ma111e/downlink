Check Point Research published a technical analysis of vulnerabilities in LangGraph’s checkpointer persistence layer. LangGraph is described as an open-source framework for stateful, multi-agent AI systems with built-in persistence and more than 50 million monthly downloads according to PyPI stats. The research focuses on how checkpointers store execution state and what can happen when their persistence layer is exposed through unsafe filtering and deserialization paths.

The first vulnerability, CVE-2025-67644, affects the SQLite checkpointer. The issue is in _metadata_predicate, where a user-controlled query_key from the filter dictionary is inserted into a SQL expression for json_extract(CAST(metadata AS TEXT), '$.{query_key}'). Because the key is not safely bound, an attacker can include a quote character and inject arbitrary SQL into the generated query.

The article explains that this SQL injection can be used to add attacker-controlled rows to the query results with a UNION SELECT. The checkpointer’s list() function selects checkpoint fields including the serialized checkpoint BLOB and then passes the result to self.serde.loads_typed((type, checkpoint)). By injecting a fake checkpoint row with a controlled serialization type and data, an attacker can force LangGraph to deserialize malicious content.

The second vulnerability, CVE-2026-28277, involves unsafe msgpack deserialization. LangGraph uses ormsgpack.unpackb() with a custom extension hook, and the article shows that a msgpack extension using EXT_CONSTRUCTOR_SINGLE_ARG can cause the code to import a named module, retrieve a named attribute, and call it with attacker-controlled input. The example given is importing os, retrieving system, and calling it with a command such as echo PWN > /tmp/pwned.txt.

The attack chain requires an entry point where a developer exposes get_state_history() with a user-controlled filter, because that method calls the checkpointer’s list() method. The article states that teams self-hosting LangGraph with SQLite or Redis checkpointers are at risk under those conditions, while LangSmith Deployment is not vulnerable because it runs PostgreSQL. A third issue, CVE-2026-27022, affects the Redis checkpointer with the same class of filter-key injection and similar preconditions.

LangChain patched all three issues. The article says users should update to langgraph-checkpoint-sqlite 3.0.1+, langgraph 1.0.10+, and langgraph-checkpoint-redis 1.0.2+. The disclosure timeline says the vulnerabilities were disclosed to LangChain on 2025-11-19, with fixes released for SQLite on 2025-12-10, Redis on 2026-02-20, and msgpack deserialization on 2026-03-05.

  • langgraph 1.0.10+
  • langgraph-checkpoint-redis 1.0.2+

LangGraph Checkpointers

LangGraph uses checkpointers as a persistence layer to store execution state at each step. The article explains that LangGraph supports SQLite and PostgreSQL checkpointer implementations, and it also discusses a Redis checkpointer package affected by a related issue.

The SQLite checkpointer uses an internal checkpoints table. The table includes fields such as thread_id, checkpoint_ns, checkpoint_id, parent_checkpoint_id, type, checkpoint, and metadata. The metadata column stores contextual information in JSON format, such as a user ID, step number, or source.

CVE-2025-67644: SQLite SQL Injection

The SQLite SQL injection is located in the filtering logic used by the list() function. The function accepts a filter parameter that is used to query checkpoints based on metadata.

The internal _metadata_predicate function builds SQL predicates by iterating over the filter dictionary. The vulnerable line constructs a JSON metadata query using the filter key directly inside a SQL string:

json_extract(CAST(metadata AS TEXT), '$.{query_key}') {operator}

Because query_key is attacker-controlled when user input reaches the filter dictionary, a key containing a quote character can escape the JSON path string and inject SQL.

SQL Injection to Arbitrary Deserialization

The article explains that the list() function queries checkpoint data from the database, including the checkpoint BLOB. The query selects fields including thread_id, checkpoint_ns, checkpoint_id, parent_checkpoint_id, type, checkpoint, and metadata.

The returned rows are then processed, and the serialized checkpoint data is deserialized with:

self.serde.loads_typed((type, checkpoint))

The SQL injection can be used to inject a UNION SELECT statement that returns a fake checkpoint row. In that injected row, the attacker controls the type and checkpoint values. The article describes this as allowing an attacker to place attacker-controlled serialized data into the result set, which is then deserialized by the application.

CVE-2026-28277: Unsafe MsgPack Deserialization

The second vulnerability involves LangGraph’s handling of msgpack data during deserialization. The loads_typed() function supports several serialization types, including null, bytes, bytearray, json, msgpack, and optionally pickle when pickle fallback is enabled.

The article notes that pickle is disabled by default. It also states that the JSON path using json.loads() with an object hook was discussed in prior LangGrinch research, but does not lead to code execution in this case. The msgpack path is the relevant path for this exploit chain.

When the type is msgpack, LangGraph calls ormsgpack.unpackb() with a custom extension hook:

ext_hook=self._unpack_ext_hook

The article shows a custom msgpack extension hook that handles EXT_CONSTRUCTOR_SINGLE_ARG. That hook unpacks data into a tuple and then executes:

getattr(importlib.import_module(tup[0]), tup[1])(tup[2])

According to the article, an attacker who controls the serialized msgpack data can provide a tuple such as:

  • os
  • system
  • A command such as echo PWN > /tmp/pwned.txt

This causes the code to import the os module, retrieve the system function, and call it with the attacker-controlled command.

Combined Attack Chain

The article describes the entry point as an application exposing get_state_history() with a user-controlled filter. The get_state_history() method internally calls the checkpointer’s list() method with the supplied filter.

The attack flow described in the article is:

  1. The attacker crafts a malicious msgpack payload that contains instructions to execute code.
  2. The attacker sends a malicious filter parameter that exploits the SQL injection.
  3. The SQL injection adds a fake checkpoint row to the query results, with the malicious msgpack payload in the checkpoint column.
  4. The application processes the query results and deserializes the injected checkpoint data.
  5. The unsafe msgpack extension hook executes the attacker-controlled payload, resulting in remote code execution.

CVE-2026-27022: Redis Checkpointer Injection

The article also describes a third vulnerability, CVE-2026-27022, in langgraph-checkpoint-redis. It says the same injection class affects the Redis checkpointer because user-controlled keys in the filter dictionary are interpolated directly into the query instead of being bound as parameters.

The stated preconditions match CVE-2025-67644: the application must expose get_state_history() with a user-controlled filter and use the Redis checkpointer. The article says this issue was patched in langgraph-checkpoint-redis 1.0.2.

Additional SQL Injection Findings

Beyond the primary filter-based SQL injection, the researchers identified additional defense-in-depth SQL injection issues in both the SQLite and PostgreSQL checkpointers. These involved direct concatenation of integer values, including LIMIT and ttl parameters, into SQL queries instead of using parameterized bindings.

The article notes that Python does not enforce type hints at runtime, so these parameters could still accept malicious string input. Check Point Research says it worked with the LangChain team during disclosure to remediate these issues using parameterized queries.

Disclosure and Vendor Response

The article provides the following disclosure timeline:

  • 2025-11-19: CVE-2025-67644, CVE-2026-28227, and CVE-2026-27022 were disclosed to the LangChain team.
  • 2025-12-10: CVE-2025-67644 was fixed and publicly released in langgraph-checkpoint-sqlite 3.0.1.
  • 2026-02-20: CVE-2026-27022 was fixed and publicly released in langgraph-checkpoint-redis 1.0.2.
  • 2026-03-05: CVE-2026-28277 was fixed and publicly released in langgraph-checkpoint 4.0.1.

The article says the LangChain team responded quickly to fix the critical SQL injection vulnerability, which effectively breaks the described attack chain. It also says the team continued work on additional remediation efforts, including the msgpack deserialization issue.

SafeDep reported that the Miasma credential-stealing framework was briefly made available for free on GitHub after multiple repositories named “Miasma-Open-Source-Release” appeared beginning June 8, 2026. The article also references EDRStartupHinder, a prior technique that prevents EDR programs from starting. EDRStartupHinder uses Windows Bindlink to redirect a DLL from System32 and takes advantage of behavior involving DLLs signed by programs protected with Protected Process Light.

After reboot, services depending on those libraries fail to start, leaving the endpoint without defenses.

When Exchange Online or hybrid on-premises Exchange is combined with an external MX record such as a third-party email server or spam protection system, attackers may spoof emails from any sender to any recipient in a target tenant. The article says the spoofed emails can bypass SPF, DKIM, and DMARC policies and that Outlook may resolve the profile picture for internal spoofed senders.
