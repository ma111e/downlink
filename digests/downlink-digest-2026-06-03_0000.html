The article also notes that Redis said it had no evidence of exploitation in its own or customer environments. As of publication, no public in-the-wild reports had surfaced. However, the full technical chain is now public, increasing the risk of follow-on exploitation.

Technical root cause

The flaw is located in unblockClientOnKey() in src/blocked.c. This function runs when a key event wakes a blocked command. It dispatches the queued command through processCommandAndResetClient() and then continues using the same client pointer.

The problem is that processCommandAndResetClient() can free the client as a side effect, and the article states that its own header comment says so. The caller ignores the return value and continues reading the freed structure, creating a use-after-free condition categorized as CWE-416.

According to Wiz’s analysis cited in the article, the bug took two commits to create:

  • A January 2023 refactor, PR #11012, added the unchecked call.
  • A March 2023 change, PR #11568, added more client access after the call.
  • Neither commit was dangerous alone, but together they reached general availability in Redis 7.2.0.
  • The combined issue survived multiple rounds of security review.

Published exploit chain

The article describes the published exploit as a three-stage chain.

First, the attacker leaks a heap pointer using a one-line Lua script: EVAL "return tostring(redis.call)" 0. This provides the heap address needed for later stages.

Second, the attacker grooms client memory limits, parks a bloated client on a stream, then drops the limits and wakes the blocked client. Redis frees the blocked client mid-call, and a pipelined SET immediately reclaims the freed slot with a fake client structure.

Third, Redis’s normal memory accounting in updateClientMemoryUsage() performs an out-of-bounds decrement using attacker-controlled fields. The article says this write is aimed at the Global Offset Table to repoint strcasecmp() at system(). The next command Redis parses then runs as a shell command.

Conditions required for exploitation

The full chain requires an authenticated session with access to several Redis capabilities. The article lists the needed categories and commands as:

  • CONFIG SET
  • EVAL
  • Stream commands such as XREAD and XADD
  • Basic SET and GET
  • ACL categories @admin, @scripting, @stream, and @read/@write

The default user has all of these privileges. The article also says that in many deployments, these privileges are grouped into a single shared application or operator role. Denying CONFIG breaks the specific published chain, although it does not remove the underlying use-after-free.

Docker image detail

The official Redis Docker image makes the final stage easier, according to the article. It ships with only partial RELRO, which leaves the Global Offset Table writable at runtime. ASLR and PIE do not help in this case because the write is relative to a global whose offset is fixed at build time.

Affected and fixed versions

Redis released fixed minor versions for affected branches on May 5. The article says minor upgrades within a series are meant to be drop-in.

Affected branches and fixed versions are:

  • Redis 7.2.x: 7.2.0 to 7.2.13 are affected; fixed in 7.2.14.
  • Redis 7.4.x: 7.4.0 to 7.4.8 are affected; fixed in 7.4.9.
  • Redis 8.2.x: 8.2.0 to 8.2.5 are affected; fixed in 8.2.6.
  • Redis 8.4.x: 8.4.0 to 8.4.2 are affected; fixed in 8.4.3.
  • Redis 8.6.x: 8.6.0 to 8.6.2 are affected; fixed in 8.6.3.

The article says managed Redis services patch on their own schedules, and Redis said Redis Cloud is already patched.

Mitigation guidance in the article

The article recommends upgrading to the patched minor release for the relevant Redis series. If patching is not immediately possible, it recommends several risk-reduction steps:

  • Keep Redis off the public internet.
  • Place Redis behind TLS.
  • Tighten ACLs so no single role holds @admin, CONFIG, and @scripting together.
  • Deny @scripting if Lua is not used, which stops the Stage 1 heap leak.
  • Prioritize internet-exposed instances, shared application credentials, and roles combining CONFIG, scripting, and stream access.
  • Rotate broadly shared Redis credentials.

Broader disclosure context

CVE-2026-23479 was one of five RCE-class Redis flaws disclosed the previous month, according to the article. The article also notes that it follows Redis’s 2025 RediShell flaw, another authenticated use-after-free involving Lua scripting. Its most unusual element is that it was found by an AI tool after two commits introduced the issue and the bug remained hidden for more than two years.

Overview

Check Point Research investigated a large-scale malware distribution ecosystem that impersonates open-source and freeware project websites to capture search traffic. The fake sites include lookalikes for widely trusted tools, including security and researcher tooling such as Ghidra, dnSpy, ILSpy, and SpiderFoot, as well as developer and utility projects such as grpcurl, MQTT Explorer, MFCMAPI, WinSetupFromUSB, CrystalDiskMark, and GUIFormat.

The article emphasizes that the sites are not obviously suspicious at first glance. They are professionally designed, may appear high in Google search results, and often preserve visible links to legitimate upstream resources such as GitHub releases. The core deception is not only in the page design, but in what happens after the user clicks a download button or link.

Relationship to prior reporting

The article states that the broader phenomenon of websites impersonating popular open-source and freeware projects had already been documented by late 2025. It specifically references a November 2025 Fullstory report that identified a large cluster of fraudulent domains and did not find direct abuse in examined samples at that time, while warning about downstream phishing or watering-hole-style risks.

Check Point Research states that the ecosystem evolved by at least December 2025, when sites in the cluster had embedded Traffic Distribution System scripts. From early January 2026 onward, Check Point Research recorded active malware distribution through the same infrastructure.

Click interception and TDS routing

The fake sites load CloudFront-hosted JavaScript staging scripts from Amazon CloudFront. The scripts follow consistent URL patterns using CloudFront domains and campaign-style identifiers, and Check Point Research identified more than 100 active websites embedding them.

A key tactic is that the visible download link can remain legitimate-looking. For example, hovering over a download button may show a real GitHub repository URL, because the original href remains intact. However, browser-side JavaScript can intercept the first eligible user interaction and send the browser to a TDS-controlled destination instead.

The article describes several technical behaviors used by the script:

  • It can use different primary events depending on browser type, such as mousedown on desktop Chrome and click on desktop Firefox.
  • It can use capture-phase event handlers to intercept the interaction before normal navigation proceeds.
  • It records the user’s intended destination and forwards it downstream.
  • It can invoke different serving methods, including cached window.open, temporary blank tabs, or dynamically created anchor elements.
  • It calls preventDefault() and stopImmediatePropagation() to cancel the visible link navigation and block other handlers.

The routing logic also uses browser-side state and frequency caps, including localStorage. This creates a reproducibility issue because the first eligible click may trigger the TDS path, while later refreshes, repeated clicks, or return visits may fall back to the original visible link target.

TDS gating and branching

After the click handoff, sessions enter a sequence of redirects. Check Point Research observed all tested sessions first redirecting to oundhertobeconsist[.]org/<token>, but notes that this domain was not hardcoded in the page or scripts and was supplied dynamically through decoded CloudFront stage configuration.

The TDS applies multiple forms of gating and routing. The article describes controls and inputs including:

  • first-visit state,
  • mandatory click confirmation,
  • anti-bot and anti-analysis logic,
  • VPN and datacenter filtering,
  • geography,
  • browser type,
  • browser fingerprint,
  • click context,
  • original entry domain,
  • frequency caps.

Different branches produced different outcomes. Some sessions were routed to an offer wall or content locker, affiliate-tagged downloads of legitimate software, potentially unwanted applications, or benign software such as Opera. Other branches led to malware delivery infrastructure.

Malware families observed

Check Point Research observed several malware families downstream of the TDS infrastructure:

  • SessionGate, a previously unknown multi-stage loader with heavy obfuscation and extensive anti-analysis measures, observed delivering potentially unwanted applications.
  • RemusStealer, a newly emerged infostealer targeting browser data, cryptocurrency wallets, two-factor authentication tools, password managers, files, registry values, clipboard contents, and screenshots.
  • AnimateClipper, a cryptocurrency clipper capable of hijacking transactions across more than 20 blockchain ecosystems.

The article states that VirusTotal telemetry showed more than 5,000 total submissions across relevant samples, indicating substantial reach within the subset visible through public sharing.

SessionGate delivery chain

SessionGate drew attention because of its multi-stage delivery chain, validation logic, and anti-analysis approach. The article states that VirusTotal telemetry for individual SessionGate-associated samples reached approximately 2,000 to 3,500 submissions, with notable visibility in Turkey, Poland, Brazil, Germany, France, Russia, and the United Kingdom.

Check Point Research identified two landing pages initiating downloads of SessionGate samples:

  • originaldownloads[.]info
  • getfluxfile[.]com

Each landing page generates a short-lived, unique payload download URL per client session, bound to the client’s browser and IP address. The HTML page contains obfuscated JavaScript that performs server-side validation through javascriptapiusa[.]com/lic? before allowing access to the executable payload.

SessionGate decoy and obfuscation

The SessionGate loader contains an embedded 7-Zip archive and can fall back to a benign installer experience when the gated delivery path does not proceed. The article notes that the downloaded archive is about 20 MB, while the visible contained file is about 15 MB, with the remaining approximately 5 MB consisting of heavily obfuscated loader code.

The loader uses several obfuscation techniques, including:

  • injected junk code,
  • opaque predicates,
  • string encryption,
  • large bloated functions that can exceed 500 KB,
  • encrypted string blobs placed inside function bodies after conditional branches.

These techniques can disrupt static analysis tools, including IDA’s decompiler and graph mode, and can cause disassemblers to misinterpret string data as executable code.

SessionGate environment checks and C2 gating

The SessionGate loader performs environment checks that influence whether it proceeds with delivery or falls back to decoy behavior. It checks for certain services using Adler-32 hashes rather than plaintext service names, including indicators such as eelam, ehdrv, eamonm, epfwwfp, epfw, ekbdflt, edevmon, npf, npcap, and sysmondrv.

It also enumerates running processes using Toolhelp-based scanning and compares indicators through hash-based logic. Additional checks include Windows Defender PUA/PUS-related registry settings and Windows Enterprise edition detection.

The loader communicates with C2 domains such as:

  • appfreshstart[.]com
  • appgetonline[.]com
  • webinnosetup[.]com
  • appmakingcenter[.]com

The C2 requests use a distinctive URL structure, include stage-specific encrypted tokens, and use the User-Agent NSIS_InetLoad (Mozilla). The loader calculates a signature using SHA1 over a base path, timestamp, and salt. For check-in responses, the loader sums decimal digits in a returned hex string and aborts execution if the resulting value is even.

SessionGate stage 2 and DLL keying

The second-stage SessionGate binary is another large Windows GUI executable that impersonates a 7-Zip SFX installer. It contains a PDB path, D:\code\cpp-downloader-scb-reg-other\Plugins\7ZipDownloader\Output\SFXWin.pdb, which Check Point Research used to find more than 200 similar samples on VirusTotal, with the earliest appearing in late August 2025.

The second stage validates its command line, requiring a numeric token and a base64-looking string. If the checks fail, it falls back to benign 7-Zip SFX behavior. If the gate passes, it reads its own image, extracts two embedded DLL payloads, decrypts them using AES-CBC, and loads them in memory via manual PE mapping.

DLL #1 acts as a key broker. It validates the command line, decrypts the base64 argument using AES-256-CBC with a hardcoded key and zero IV, checks a filename suffix pattern, performs anti-analysis checks, and contacts CRC C2 domains such as:

  • yourfastcrc[.]com
  • mobileversioncrc[.]com
  • webcrcprove[.]com
  • integritycrc[.]com

The CRC request includes the uppercase SHA-256 hash of the stage-2 loader. The server returns token material that DLL #1 transforms into a 32-character base62 token, which is then used to derive AES key material for DLL #2. Check Point Research observed that the CRC endpoint can return different values for the same loader hash, and assesses that the key release may be tied to one-time semantics and victim context such as IP address or session state.

SessionGate final module behavior

After successfully decrypting DLL #2, Check Point Research found a network-controlled installer or bundler framework. The article states that this module did not show stealer or RAT behavior in the analyzed sample. Instead, it implements configurable delivery and silent installation of additional software.

The module’s workflow includes:

  • building an encrypted request,
  • retrieving encrypted configuration from a C2 server such as appmakingcenter[.]com,
  • decoding configuration into a key/value table,
  • retrieving a product download URL,
  • downloading a payload,
  • executing it silently via cmd.exe,
  • sending telemetry and tracking events.

The module contains product-facing strings and expected install paths for products including PDF Spark, PDF Proton, PDF Ignite, PDF Skill, Document Sparkle, NibblrAI, and PCPooch. The actual download URL is supplied dynamically by backend configuration, with a hardcoded 7-Zip installer URL used as a fallback if the configuration lacks the relevant key.

RemusStealer branch

In the RemusStealer case, the TDS chain ends at a landing page that provides a link to a password-protected ZIP archive and the password needed to open it. The extracted executable is about 850 MB on disk, but is artificially inflated with zero-filled padding, with roughly 32 MB of non-zero content after padding is removed.

The executable is a first-stage loader written in Go. It contains an embedded malicious payload in .rdata, decodes it at runtime, and executes it through manual PE mapping.

The embedded payload is RemusStealer, described as a C2-controlled infostealer and Malware-as-a-Service product. The article states that the first public listing observed for “Remus” was posted on a Russian-language underground forum by a user named RemusStealer on February 12, 2026, with two advertised subscription tiers.

RemusStealer capabilities

RemusStealer uses encrypted C2-driven tasking. The server defines what to collect by sending encrypted JSON tasks, and the agent executes tasks sequentially until the server sends a stop command.

The article lists RemusStealer capabilities including:

  • Chromium browser data theft, including History, Login Data, Login Data For Account, Network\Cookies, and Web Data.
  • Firefox/NSS profile collection, including key4.db, cert9.db, cookies.sqlite, logins.json, formhistory.sqlite, places.sqlite, prefs.js, and extensions.webextensions.uuids.
  • Chromium master key extraction from Local State using DPAPI and upload as a /Key artifact.
  • Extension-driven theft using server-provided extension target lists.
  • Server-controlled file system search and exfiltration with environment variable expansion.
  • Server-controlled registry reconnaissance.
  • Clipboard theft.
  • Screenshot capture when enabled by an internal flag.

The captured C2 traffic included 332 browser extension identifiers. The targeting was heavily skewed toward cryptocurrency wallets, password managers, and 2FA extensions, including examples such as 1Password, Bitwarden, LastPass, Dashlane, Keeper, RoboForm, NordPass, Proton Pass, KeePassXC, Zoho Vault, MetaMask, Rabby Wallet, Coinbase Wallet, Trust Wallet, OKX Wallet, Binance Wallet, Bitget Wallet, Phantom, Solflare Wallet, Keplr, Cosmostation, SubWallet, TronLink, Exodus, Ronin Wallet, Tonkeeper, MyTonWallet, Yoroi, UniSat Wallet, Suiet, Pontem, Authy, and 2FAS.

AnimateClipper branch

The AnimateClipper branch led to a ClickFix-style phishing page at processing-in-progress-x4.t3.storage[.]dev. The page imitated a Cloudflare verification screen and instructed the user to run mshta.exe with a remote URL pointing to https://185.0xA1.0xFB[.]58/navy.7z.

The fetched object was not a normal 7-Zip archive. Its beginning contained an HTA page with obfuscated VBScript, while the appended archive content was benign decoy data. The VBScript retrieved a resource from http://194.150.220[.]218/4SLEYpfAk57hGubo/fo0suc2ki2.rtf, which was actually a heavily obfuscated PowerShell script.

The PowerShell chain reconstructed an additional PowerShell stage in memory, used an RC4-based routine to decrypt the next payload, and then downloaded https://cdn-1415.brightcanvas[.]digital/fo0suc2ki2.rtf. That file was a ZIP archive containing a bundled Python environment, third-party libraries, Node.js modules, and a large obfuscated Python script stored as node_modules.asar.

The Python loader embedded shellcode, copied it into executable memory, and transferred execution to it through ntdll!LdrCallEnclave. The shellcode then decrypted and decompressed an embedded payload container and manually mapped the resulting PE payload in the same process.

AnimateClipper final payload

The final payload is a cryptocurrency clipper. It monitors the clipboard for cryptocurrency wallet strings, identifies wallet formats locally, replaces copied addresses with attacker-controlled wallet addresses embedded in the sample, and writes the modified value back to the clipboard.

AnimateClipper resolves its C2 by querying a smart contract over the public BNB Smart Chain Testnet JSON-RPC endpoint. At the time of analysis, the contract response resolved to the C2 domain kr.hugo-lapp[.]co. The malware uses HTTPS to communicate with the resolved C2 and includes logic for periodic refresh check-ins and reporting address-replacement activity.

Check Point Research also reviewed incoming transactions to wallet addresses embedded in the analyzed sample. In the dataset examined, the earliest inbound payments were recorded in July 2025, with the first observed transaction dated July 12, 2025. The article states that this indicates the operation has likely been active for a prolonged period and that the TDS-driven infection chain may be one of several distribution paths used to deploy the malware.

Significance and impact

The article’s central conclusion is that a polished website and a legitimate-looking visible link are not reliable security signals. The impersonation sites can appear official, rank well in search results, and show real upstream URLs on hover, yet still redirect the first download interaction into a gated TDS stack.

The campaign’s significance lies in how the TDS acts as a broker between traffic acquisition infrastructure and downstream delivery operators. The same search-driven traffic pipeline can route different users to different outcomes, including benign offers, potentially unwanted applications, or malware. The article states that attribution and accountability are made harder because the impersonation operator does not need to be the malware author to enable malware delivery at scale.

Protections and indicators

The article states that Check Point Threat Emulation and Harmony Endpoint provide coverage for the attacks and threats described. It also provides indicators of compromise, including SHA-256 hashes for SessionGate, AnimateClipper, and RemusStealer samples, as well as domains and URLs associated with SessionGate C2s, RemusStealer C2s, and AnimateClipper infrastructure.

  • The fake download pages can appear trustworthy even under hover inspection because they preserve legitimate upstream href values, while JavaScript intercepts the actual click event and redirects the user elsewhere.
  • The campaign creates a reproducibility trap for analysts because the first eligible click may trigger the malicious TDS path, while refreshes, repeated clicks, or return visits can fall back to the visible legitimate link.
  • SessionGate’s decoy 7-Zip SFX behavior can cause sandboxes and analysts to observe only a normal installer experience when the gated delivery path does not proceed.
  • SessionGate’s DLL #2 may be undecryptable without the correct one-time server-provided key material tied to the matching session, IP context, and generated build.
  • AnimateClipper resolves its C2 through a BNB Smart Chain Testnet smart contract, while its replacement cryptocurrency wallet addresses are embedded directly in the binary.

    Cybersecurity researchers have discovered a remote denial-of-service exploit called HTTP/2 Bomb. The article says it affects major web servers, including NGINX, Apache HTTPD, Microsoft IIS, Envoy, and Cloudflare Pingora. Calif said the vulnerable behavior exists in each server’s default HTTP/2 configuration.

    According to the article, HTTP/2 Bomb was discovered by OpenAI Codex. Calif said the discovery came from chaining together two known techniques: a compression bomb and a Slowloris-style hold. The result is a denial-of-service condition that can consume and retain server memory.

    Technical mechanism

    HTTP/2 Bomb targets HPACK, the header compression scheme used by HTTP/2. HPACK is described in the article as a dedicated header compression algorithm for HTTP/2 that compresses request and response metadata using Huffman encoding. The article says HPACK produces an average reduction of 30% in header size and was designed to be resilient to attacks such as CRIME.

    Calif said the HTTP/2 Bomb behavior is based on a small amount of data on the wire causing disproportionate server-side allocation. In the company’s description, one byte on the wire becomes one full header allocation on the server, repeated thousands of times per request. A zero-byte flow-control window then keeps the server from freeing the allocated memory.

    The article also describes the second component as a Slowloris-style hold. Slowloris is described as a denial-of-service technique in which an attacker opens and maintains many simultaneous HTTP connections with a target server. The article characterizes Slowloris as an application-layer attack.

    Relationship to earlier techniques and vulnerabilities

    The article says HTTP/2 Bomb is inspired by several known approaches and vulnerabilities. These include HPACK Bomb, also known as CVE-2016-6581, which was first disclosed in 2016. The article also references CVE-2025-53020, a memory exhaustion vulnerability in Apache httpd’s HTTP/2 implementation.

    Two additional Apache HTTP Server denial-of-service flaws are also cited. One is CVE-2016-8740, involving crafted CONTINUATION frames. The other is CVE-2016-1546, involving worker-thread starvation in an HTTP/2 connection.

    Calif said the new aspect of HTTP/2 Bomb is the source of amplification. In the classic bomb, a large value is placed into a table and referenced repeatedly, leading servers to cap total decoded header size. In HTTP/2 Bomb, the header is nearly empty, so the decoded-size limit does not fire because there is almost nothing to decode.

    Why the attack is effective

    Calif said the amplification in HTTP/2 Bomb comes from per-entry bookkeeping allocated by the server. This differs from a model where the primary risk comes from a large decoded header. The article states that the server allocates memory around very small header entries, and the client can hold that memory in place.

    The article says the hold is enabled by a zero-byte flow-control window. This prevents the server from freeing the allocated memory. Calif characterized the deeper issue as a specification-level framing of memory risk that focuses on amplification ratio while missing the effect of retained allocations.

    Calif said a 70:1 amplifier is harmless if the memory is freed when the request completes. It becomes an attack because HTTP/2 lets the client hold the connection open almost for free. This can pin every allocated byte for as long as the client chooses, according to the article.

    Reported impact

    The article describes the impact as remote denial of service against affected web servers. In a hypothetical attack scenario, it says a home computer on a 100Mbps connection could render a vulnerable server inaccessible within seconds. The article also says a single client can consume and hold 32GB of server memory against Apache HTTPD and Envoy in about 20 seconds.

    The affected products named in the article are:

    • NGINX
    • Apache HTTPD
    • Microsoft IIS
    • Envoy
    • Cloudflare Pingora

    The article states that the vulnerable behavior exists in each server’s default HTTP/2 configuration.

    Mitigations and response

    The article lists mitigations for NGINX and Apache HTTPD and notes the patch status for other affected products. For NGINX, it recommends upgrading to version 1.29.8 or later. The article says this version adds the max_headers directive with a default of 1000.

    If upgrading NGINX is not an option, the article recommends disabling HTTP/2 with http2 off;. For Apache HTTPD, the article says the issue is fixed in mod_http2 v2.0.41. If upgrading Apache HTTPD is not an option, it recommends setting Protocols http/1.1 to disable HTTP/2.

    For Microsoft IIS, Envoy, and Cloudflare Pingora, the article says no patch was available at the time of writing.

    Key takeaway

    HTTP/2 Bomb is a remote denial-of-service technique that abuses HPACK header handling and HTTP/2 flow control to cause and retain server-side memory allocations.
