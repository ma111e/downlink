Overview

watchTowr Labs analyzed CVE-2026-50751, a Check Point Remote Access VPN IKEv1 authentication bypass with a CVSS score of 9.3. The vulnerability affects Check Point Mobile Access/SSL VPN, Remote Access VPN, and Spark Firewall products, specifically in deprecated IKEv1 VPN code. Check Point released hotfixes on June 8, 2026 through sk185033.

The article states that the vulnerability had reportedly been exploited in the wild since May 7, 2026, roughly a month before patches were available. It also says Check Point reported “a few dozen targeted organizations,” and that at least one incident had been linked to a Qilin ransomware affiliate. The vulnerability was described as a logic flaw in how certificates are validated during the IKEv1 key exchange.

Affected Products and Preconditions

The article lists the affected Gaia versions as:

  • R80.20.X
  • R80.40
  • R81
  • R81.10
  • R81.10.X
  • R81.20
  • R82
  • R82.00.X
  • R82.10

watchTowr says exploitation depends on several conditions:

  • The target accepts legacy Remote Access clients.
  • IKEv1 is permitted rather than IKEv2-only.
  • Machine certificate authentication is not mandatory.

The article notes that these prerequisites may still leave many potential targets because legacy client support is common in enterprise environments.

Research Method

watchTowr compared R82.10 Jumbo Hotfix Take 19, which they describe as vulnerable, against the same build with the sk185033 hotfix applied. The researchers first examined vpnd, but found that vpnd did not bind UDP 500 or UDP 4500. They then determined that on R81.10 and later, Check Point redirects IKE negotiation to a newer daemon, iked, which listens internally on UDP 30500.

After diffing the correct binary, the changed code appeared in:

  • ikeMainMode.cc
  • ikeAggressive.cc
  • machine_cert_utils.cc

A key change was the removal of a trailing integer argument from process_cert_payloads. In the vulnerable version, that argument controlled whether machine certificate payloads were processed. In the patched version, the function instead consults the appliance’s own machine certificate policy from the negotiation state.

Root Cause

The vulnerable code allowed the decision to process certificate material and validate authentication to depend on flags that could be influenced by the client. In process_cert_payloads, the vulnerable implementation used a caller-supplied process_machine_certs flag to decide whether to process Check Point machine certificate payloads of type 0xF6. The patched version removed that argument and used a field populated from the gateway’s configured machine certificate policy.

The more important issue appeared in the caller process_auth_pl and in verify_peer_auth. In process_auth_pl, the code read a flags dword from state offset 0x4bc4 and tested bit 0x2. If this bit was not set, the code did not call seekPayload for the 0xF9 signature payload and did not call processSigPayload.

In verify_peer_auth, the same flag word controlled whether verifyMessagePhase1 ran. If bit 0x4 was set, an OR condition short-circuited and the function returned authentication success without calling verifyMessagePhase1. The article identifies verifyMessagePhase1 as the phase-1 signature check.

Client-Controlled Authentication Flags

watchTowr searched for writes to state offset 0x4bc4 and found two. One was initialization in FwIkeNewPhase1State, which zeroed the field for a new negotiation. The other was in processVendorIDPayload.

The Vendor ID involved was the 16-byte value:

  • 3c f1 87 b2 47 40 29 ea 46 ac 7f d0 ea f2 89 f5

The gateway logged this as VPNExtFeatures. According to the article, processVendorIDPayload required at least four bytes after the 16-byte magic value, read those four attacker-supplied bytes, byte-swapped them, and stored them directly into state + 0x4bc4. This meant the client could provide the flags used by the gateway to decide whether to perform authentication checks.

The article highlights an additional inconsistency: a peer that sent only the VPNExtFeatures Vendor ID was logged by the gateway as “not a Check Point peer,” while the gateway still honored the client-supplied flag that disabled authentication verification.

Proof of Concept Behavior

To prove the issue, watchTowr wrote a minimal IKEv1 Main Mode client using AES-256, SHA1, Diffie-Hellman group 2, and Check Point’s 0xFADD hybrid authentication method. They sent the VPNExtFeatures Vendor ID with the four trailing bytes set to 0x00000004, setting bit 0x4. They then completed Diffie-Hellman and sent a completely random HASH_I in encrypted message 5.

They tested identical packets against two gateways:

  • A vulnerable R82.10 Take 19 appliance.
  • A patched Take 19 appliance with sk185033.

The vulnerable gateway logged that the peer was not a Check Point peer, proceeded through MMCreate6, and saved the user in IkeSAFromState. The patched gateway logged an authentication failure with hybrid and sent an authentication failed notification. The article says the vulnerable gateway created a phase-1 SA based on a signature generated with os.urandom.

watchTowr also performed negative controls:

  • VPNExtFeatures with capability 0x00000004 skipped verifyMessagePhase1 and produced the bypass.
  • VPNExtFeatures with capability 0x00000000 caused verifyMessagePhase1 to run and resulted in rejection.
  • No Vendor ID caused verifyMessagePhase1 to run and resulted in rejection.

Authentication as a Provisioned User

The researchers then moved from proving acceptance of invalid phase-1 data to showing authentication as a provisioned Remote Access user. They switched from the hybrid method to RSA signature authentication, IKE authentication method 3, using the certificate path.

The credentials they presented were:

  • An ID payload containing a distinguished name in the form CN=,OU=users,O=.
  • A self-signed X.509 certificate minted by the researchers.
  • A signature payload containing 256 bytes of os.urandom.
  • The VPNExtFeatures Vendor ID with bit 0x4 set.

Because verifyMessagePhase1 was skipped, the gateway did not verify the signature and did not validate the certificate trust chain. The article states that the gateway only checked whether the subject distinguished name resolved to a provisioned Remote Access user.

According to watchTowr, an attacker needed two pieces of information:

  • A valid username.
  • The ICA organization string for the distinguished name.

The article says the ICA organization string is present in the gateway’s public TLS certificate and can be read from there. It also says a wrong username fails at user lookup while a correct username succeeds, making the same probe a username oracle.

In their demonstration, watchTowr’s tool authenticated as the user “watchtowr” with a forged certificate and invalid signature, decrypted the gateway’s final phase-1 message, and retrieved the gateway internal IP and VPN certificate. The gateway also logged that the user CN=watchtowr,OU=users,O=watchtowr was saved.

Exploitation Over UDP and TCP 443

The article states that the vulnerability was exploitable over UDP 500. It also says watchTowr was able to exploit it over TCP 443.

When UDP 500 and UDP 4500 are filtered, Check Point clients can fall back to Visitor Mode. Visitor Mode tunnels the same IKE exchange over a raw TCP stream on port 443 using a framing protocol called TCPT. watchTowr says they reversed the protocol enough to demonstrate that the same bypass worked over this path.

Authentication Modes

watchTowr reports that the vulnerability appeared exploitable across several Check Point certificate modes for SSL Network Extender user authentication. Their table states:

  • Certificate mode was bypassed over UDP 500 and TCP 443.
  • Certificate with enrollment mode was bypassed over UDP 500 and TCP 443.
  • Mixed mode was bypassed over UDP 500 and TCP 443.
  • Legacy username/password mode was not bypassable.

The article explains that Legacy username/password mode still required a real password through XAUTH, so the bypass did not remove that gate.

Response and Detection Artifact

Check Point released hotfixes through sk185033 on June 8, 2026. The patched code removed the client-controlled process_machine_certs function argument and instead based machine certificate processing on the gateway’s own configured policy. The patched gateway in watchTowr’s test rejected the malicious exchange with an authentication failure.

watchTowr also released a Detection Artefact Generator at the GitHub repository watchtowrlabs/watchTowr-vs-Check-Point-CVE-2026-50751. The article says the decision to release it was based on the attackers having already had the vulnerability for a month.

  • The vulnerable gateway could log a peer as “not a Check Point peer” while still honoring that peer’s client-supplied flag to skip authentication checks.
  • The bypass required only a valid username and the ICA organization string, and the article states the ICA organization value could be obtained from the gateway’s public TLS certificate.
  • Wrong username guesses failed at user lookup while valid usernames succeeded, making the same authentication probe usable as a username oracle.
  • Filtering UDP 500 or UDP 4500 was not sufficient mitigation in the researchers’ testing because the same IKEv1 exchange could be tunneled over TCP 443 using Check Point Visitor Mode.
  • The plain Legacy username/password mode was described as not bypassable because XAUTH still required a real password, while the certificate-based modes were bypassed.

Major campaign developments

The article identifies several major developments across 2026:

  • In March 2026, the Miasma wave expanded into PyPI through a compromised vulnerability scanner and introduced .pth-based persistence.
  • In May 2026, the Hades/TanStack wave abused a GitHub Actions CI misconfiguration to scrape OIDC tokens from runner memory and publish malicious packages with valid SLSA provenance.
  • On May 12, 2026, the worm source code was open-sourced under an MIT license.

  • On June 5, 2026, the IDE wave moved execution into IDE configuration files and contributed to GitHub disabling 73 Microsoft repositories.

March 2026: Miasma and PyPI expansion

The Miasma wave marked the campaign’s first major expansion beyond npm. Instead of targeting package maintainers directly, TeamPCP compromised Aqua Security’s Trivy vulnerability scanner through GitHub Actions cache poisoning. Trivy is widely used in build pipelines for container and dependency scanning.

The attack relied on the absence of version pinning in downstream consumers. When Trivy’s repository cache was poisoned, pipelines that installed Trivy without a pinned version downloaded and executed the attacker’s binary. LiteLLM, a Python library for calling LLM providers, was among the first major victims, and LiteLLM version 1.82.8 was published with a malicious 34KB litellm_init.pth file in site-packages.

The use of .pth files represented a persistence advancement over the npm preinstall and postinstall hooks used in earlier waves. Python processes .pth files in site-packages during interpreter startup, so any invocation of Python in an impacted environment can execute the payload. The article notes that this persistence can survive package reinstallation and can persist across virtual environment recreation if the base site-packages directory is affected.

May 2026: Hades, TanStack, and valid malicious provenance

On May 11, 2026, TeamPCP exploited a pull_request_target misconfiguration in the TanStack open-source monorepo. The pull_request_target trigger runs in the context of the target repository rather than the contributor’s fork, which can grant access to repository secrets when misconfigured. The attackers submitted a malicious contribution that triggered the workflow in TanStack’s privileged CI environment.

The malicious code executed inside the privileged runner context and scraped an OIDC token from the Runner.Worker process memory. The token was presented to GitHub’s OIDC federation endpoint to generate a valid npm publish token. That token was then used to publish malicious packages from TanStack’s legitimate environment.

The article highlights that no maintainer credentials needed to be stolen in this wave. The trusted environment itself became the access path. Within a six-minute window, 84 malicious artifacts were published across 42 @tanstack/ packages.

Those artifacts carried valid Sigstore provenance attestations signed through the legitimate CI path and recorded in the Rekor transparency log. Cryptographically, the attestations were valid because the malicious build process ran from inside the trusted system. The article states that this demonstrates why SLSA/Sigstore provenance should not be treated as proof of safety, because provenance validates the build process rather than the integrity of the CI system or the identity at the root of the workflow.

Days later, the campaign mass-republished the @antv data visualization namespace. Snyk reported roughly 314 versions published within a six-second window. Public reporting also linked the broader wave to compromises affecting additional AI-related infrastructure packages associated with Mistral AI, Guardrails AI, UiPath, and OpenSearch.

May 12, 2026: Worm source code release

On May 12, 2026, TeamPCP published the complete worm source code to GitHub under an MIT license with the commit message, “Open Sourcing The Carnage.” The release reportedly included full propagation code, an OIDC token scraping module, operational documentation for customizing encryption keys and C2 infrastructure, and a $1,000 Monero prize announcement on BreachForums for the largest supply chain attack built from the codebase.

The article states that this release changed the threat landscape and made attribution harder. Before publication, it was more straightforward to link multiple waves to the same operator. After the release, copycat actors could reuse the same code and tradecraft, so malware overlap or similar operations alone were no longer enough to confidently attribute later activity to TeamPCP.

June 1–2, 2026: Red Hat OIDC trusted publishing abuse

The June Red Hat wave used a different path into the same trusted publishing model. Public reporting cited in the article indicates that a Red Hat engineer’s GitHub account was compromised, although the initial takeover method was not publicly disclosed. With access to the account, the attacker pushed orphan commits to internal Red Hat repositories, injected GitHub Actions workflows through those orphan commits, allowed the workflows to request OIDC tokens through normal trusted publishing flows, and published malicious packages with valid provenance.

The result was 32 packages and 96 versions published within hours. From the perspective of signing and provenance checks, the packages appeared legitimate. As with the TanStack wave, the trust chain remained cryptographically valid while the identity and execution context at its root had been compromised.

Samples from the June wave also introduced C2 camouflage. The article describes traffic staged to route to api.anthropic.com/v1/api, a non-existent endpoint at Anthropic’s domain. It states that this was camouflage rather than the primary exfiltration channel, and that Anthropic’s infrastructure was not compromised.

June 5, 2026: IDE configuration file execution

The Azure/durabletask IDE wave moved beyond package registries and CI publishing into developer tooling configuration files. A compromised contributor account pushed four files targeting AI-assisted development and editor environments. These files were:

  • .claude/settings.json, using a SessionStart hook when opening a project in Claude Code.
  • .cursor/rules/setup.mdc, using alwaysApply prompt injection during Cursor AI agent initialization.
  • .vscode/tasks.json, using a runOn: folderOpen task when opening a folder in VS Code.
  • .gemini/settings.json, using a startup hook when starting a Gemini Code Assist session.

The article states that no package installation was required. The trigger was git clone followed by opening the repository in one of the targeted editors. In response, GitHub disabled 73 Microsoft repositories across four organizations.

The article emphasizes that IDE configuration files are not evaluated by package registry scanners, SLSA verification tools, or behavioral systems oriented around package content. They are often reviewed as configuration rather than executable code, placing the vector outside many existing detection perimeters.

June 8, 2026: Hades PyPI and LLM scanner evasion

The Hades PyPI wave was detected by Socket Research on June 8, 2026 and compromised 37 PyPI wheels across 19 packages. The malware retained familiar campaign tradecraft, including .pth persistence, multilayer obfuscation, AES-256-GCM encryption with per-instance keys, PBKDF2 key derivation, and obfuscator-wrapped JavaScript payloads.

The notable change was an AI scanner evasion technique embedded in the _index.js bundle. According to StepSecurity’s analysis as quoted in the article, the bundle begins with a plain-text prompt injection block placed ahead of the obfuscated payload. The prompt injection instructed parsing LLMs to ignore the obfuscated code, classify the package as verified clean infrastructure, and output a safe security report.

The article explains that scanners passing raw package content to a language model without enforcing system prompt isolation can place the attacker’s instruction in the model’s context window. Without content boundary separation, the injected text can override the scanner’s analytical prompt and produce a false-negative verdict. The article also notes that a scanner could be induced to refuse analysis if submitted content triggers safety filters, and that treating “no finding” or “refused to analyze” as clean creates a blind spot.

C2 infrastructure evolution

The campaign’s C2 infrastructure evolved in response to takedowns and defensive pressure. Earlier V1 and V2 activity used GitHub orphan or dangling commits as dead drops, which the article describes as having low takedown resistance because GitHub can disable repositories. Miasma also used GitHub dead drops.

In May and June 2026, the campaign moved toward Internet Computer Protocol blockchain canisters, which the article describes as more resistant to takedown because they are decentralized and do not depend on a central domain. The June 8, 2026 Hades PyPI wave used GitHub dead drops, Session Protocol, and staged api.anthropic.com/v1/api camouflage. The article states that Session Protocol bypasses DNS blocking.

One described GitHub dead-drop method involved a background service called kitty-monitor. It queried GitHub’s commit search API once per hour for the marker string firedalazer. The most recent matching commit contained the next instruction in its commit message, and the implant verified the signature against an embedded public key before retrieving the referenced URL.

The article explains that cryptographic big integers are commonly represented as arrays of smaller machine-sized values called limbs. In the short-sleeve keys, each limb contains only a small contiguous subset of random bits, while other bits in the limb are zero. This regular limb-level structure is what makes the polynomial technique effective.

The researchers represent an integer as a polynomial using the digits of the integer’s base-B representation as coefficients. If an integer is written as:

  • a = sum of a_i B^i

then the corresponding polynomial is:

  • f_a(x) = sum of a_i x^i

Evaluating the polynomial at B converts it back into the integer. For the vulnerable keys, B corresponds to the limb size, such as 2^32 for 32-bit limbs.

This representation is useful because multiplication is preserved by evaluation: multiplying two integer evaluations corresponds to evaluating the product of the two polynomials. If the RSA modulus n is the product of p and q, and the short-sleeve structure makes the corresponding polynomial coefficients small, then factoring the polynomial can reveal polynomial factors corresponding to p and q. Evaluating those polynomial factors at the limb base recovers the RSA private factors.

The article emphasizes that this does not break ordinary RSA. Correctly generated RSA prime factors typically produce polynomial coefficients with full limb-sized randomness, so the necessary small-coefficient structure is absent. The short-sleeve property is essential to making the polynomial factorization approach practical.

The vulnerable genRandomBits function was still compiled into the latest version examined by the researchers, but it was unreachable. According to the article, CompleteFTP has generated RSA keys using .NET’s RSACryptoServiceProvider since version 12.1.0. It has generated DSA keys using the DSA.Create API since version 23.1.0.

The researchers found that CompleteFTP’s DSA key generation also called the vulnerable genRandomBits function in older versions. The 160-bit DSA private key x was generated using the flawed function, while the public key and parameters included a generator g and target y = g^x.

The article states that the private DSA key is easily recoverable using baby-step giant-step. After identifying the pattern, the researchers found vulnerable DSA keys in the wild as well.

The article also notes that Diffie-Hellman key exchange used the vulnerable function with a 2048-bit exponent, but the researchers state this is not vulnerable and that they believe those DH key exchanges remain cryptographically secure.

Historical spread and containment

The researchers used historical and contemporary SSH scan data from Prof. Nadia Heninger’s collection to study how the CompleteFTP issue changed over time. The scans typically included hundreds of CompleteFTP hosts per IPv4-wide scan.

The historical trend aligned with CompleteFTP’s release history. After the RSA vulnerability was introduced in December 2016, the number of hosts with vulnerable keys consistently increased. When rewritten RSA code was released in March 2019, that trend immediately stopped.

However, the article notes that vulnerable keys persisted even as fewer hosts ran affected versions of the software. The researchers interpret the plateau in the proportion of affected keys as consistent with customers updating software while generating host keys only once.

EnterpriseDT was described as responsive throughout disclosure. The company released CompleteFTP v26.1.0 on May 8, 2026. This update automatically checks whether the system is using a vulnerable RSA or DSA key and alerts the user if regeneration is needed.

EnterpriseDT also released a standalone KeyChecker tool for the same purpose. The article states that users who generated CompleteFTP host keys during the affected periods should use the tool or updated software to determine whether their keys need to be regenerated.

The badkeys website and standalone badkeys tool were also updated to support detection of vulnerable short-sleeve RSA keys.

The researchers report recovering private keys for:

  • 603 unique RSA public keys generated by vulnerable CompleteFTP versions
  • 74 DSA keys generated by vulnerable CompleteFTP versions
  • 26 RSA keys with the unidentified short-sleeve pattern

They caution that their data sources are heavily biased toward RSA SSH keys, so these counts do not reflect actual prevalence.

The article includes a footnote describing a separate CompleteFTP RSA key-generation issue on Linux. In that issue, the private exponent was set to 65537 and the public exponent was large. The researchers disclosed the issue, and it was fixed in CompleteFTP v26.0.2.

The article states that the Linux version has different features and is less popular than the Windows version. According to EnterpriseDT license data, the company believes no production users were affected, and the researchers’ scans found no keys in the wild with that property.

Significance

The article’s central significance is that structured implementation mistakes in big-integer key generation can leave public RSA moduli with detectable patterns and factorable private keys. The researchers found similar short-sleeve failures in independent RSA implementations, although one pattern remains unexplained. They argue that the regular spacing of the leakage creates useful structure for cryptanalysis and may be worth looking for in other implementations.

The research also shows how practical cryptanalysis can move from detecting unusual public-key patterns, to developing a specialized factoring method, to reverse-engineering software and identifying the underlying bug. In this case, the result was both recovery of real private keys and vendor-released tooling to identify affected CompleteFTP RSA and DSA keys.

  • The vulnerable CompleteFTP code remained compiled into newer versions but became unreachable after RSA key generation was refactored to standard .NET cryptographic APIs, which is why dynamic testing of the latest version did not reproduce broken keys.
  • Historical SSH scan data showed that new vulnerable CompleteFTP RSA keys stopped appearing immediately after the March 2019 rewrite, but many affected keys persisted because customers updated software without regenerating host keys.
  • The same short-sleeve failure mode appeared in at least two independent RSA implementations, indicating that biased limb-level big-integer generation is a practical class of implementation failure worth detecting beyond the confirmed CompleteFTP cases.
  • The polynomial factorization attack depends on the regular spacing of zero-bit blocks across big-integer limbs, turning an otherwise hard integer factorization problem into an easy polynomial factorization problem for this special key structure.
