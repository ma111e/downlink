The article describes this as a scope takeover in which publish access to the trusted @mastra scope allowed the attacker to fan out one malicious dependency across many packages.

Mastra is described as a widely adopted TypeScript framework for building AI agents and applications. Its scope includes the mastra and create-mastra entry points, @mastra/core, database and vector connectors, auth integrations, deployers, and provider SDKs. The article emphasizes that developers using this framework often hold high-value secrets, including cloud keys, LLM provider keys, database credentials, and source-control tokens.

Why the Incident Matters

The affected packages are described as heavily used. Mend records over 20 million downloads for @mastra/core and over 19 million downloads for @mastra/schema-compat. The article states that any project that auto-updates these packages is in scope.

The attack did not require malicious application code to be inserted into each @mastra package. Instead, each compromised release differed from its clean predecessor by a single functional change: an added dependency on easy-day-js. This design meant the malicious code lived in the dependency, while the carrier packages could appear clean to per-package scanning.

Initial Attack Vector

The payload reached victim machines through normal npm dependency resolution. The attacker added the following dependency pattern to compromised @mastra packages:

  • "easy-day-js": "^1.11.21"

The article identifies easy-day-js as a typosquat of the dayjs date library. Version 1.11.21 of easy-day-js was a clean prior release with no install hook. Version 1.11.22 added the malicious obfuscated postinstall dropper.

The caret range was central to the attack. Because npm resolves ^1.11.21 to the newest matching version at install time, a fresh install of any affected @mastra package pulled easy-day-js@1.11.22 even though the declared version range referenced the clean 1.11.21 line.

Stage 1: Postinstall Execution

The malicious easy-day-js@1.11.22 package added a lifecycle script to its package.json:

  • postinstall: node setup.cjs --no-warnings

This script ran automatically when the package was installed. The --no-warnings flag suppressed Node runtime warnings so that unusual behavior would not appear in install output. The executed file, setup.cjs, acted as the dropper.

Stage 2: Dropper Behavior

The setup.cjs dropper was obfuscated with javascript-obfuscator. The article describes several obfuscation details, including a rotated 40-entry string array, a custom-alphabet base64 decoder, and an XOR-encoded byte array that reconstructed the marker string easy-day-js.

After deobfuscation, the dropper performed a short sequence of actions:

  • It disabled TLS certificate verification by setting process.env.NODE_TLS_REJECT_UNAUTHORIZED = '0'.
  • It wrote two marker files into the OS temp directory.
  • It fetched a second-stage payload from hxxps://23[.]254[.]164[.]92:8000/update/49890878.
  • It wrote the downloaded payload to a randomly named 24-hex-character .js file in the OS temp directory.
  • It launched the payload with the installer’s Node binary as a detached, hidden background process.
  • It passed 23[.]254[.]164[.]123:443 as the second-stage C2 argument.
  • It deleted setup.cjs from disk after spawning the second stage.

The dropper also wrote <tmpdir>/.pkg_history, containing the package install path, and <tmpdir>/.pkg_logs, containing easy-day-js stored as an XOR-0x80 byte sequence.

Stage 3: Second-Stage Payload

The dropper downloaded the second stage at install time rather than embedding it in the npm package. The article states that this allowed the operator to update or replace the second stage remotely. The second stage was a Node.js script staged in the OS temp directory and launched using the installer’s own Node binary.

The second stage ran as a detached and hidden process, allowing the npm install process to finish while the malware continued running. It inherited the disabled TLS setting and communicated with 23[.]254[.]164[.]123:443 without certificate validation.

Functionally, the article describes the second stage as a remote-access tool. It harvested sensitive credentials and cryptocurrency wallet data from the host, exfiltrated them to the C2, and could receive follow-up commands.
