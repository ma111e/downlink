Unlike CVE-2021-22204, which targeted eval, this vulnerability targets system() as the dangerous sink. The article frames the research as a taint-analysis exercise, tracing user-controlled metadata values to command execution functionality.

Exploitation prerequisites

The article explicitly states that exploitation requires the -n flag, also known as -printConv. This flag outputs machine-readable data without additional processing.

The exploit path also relies on ExifTool’s -tagsFromFile feature. The vulnerable SetMacOSTags function is triggered when metadata is copied into FileCreateDate, not when FileCreateDate is written directly.

Vulnerable code path

The vulnerable sink is located in the SetMacOSTags function. The system() call receives a command assembled from three inputs:

  • $file, which the article says is properly sanitized.
  • $setTags, which is processed iteratively.
  • $val, which is user-controlled and left unsanitized in the vulnerable branch.

The vulnerability is tied to macOS file creation date handling. ExifTool uses the Spotlight system attribute MDItemFSCreationDate to handle file creation dates, and this attribute maps to the internal alias FileCreateDate.

When ExifTool parses an image, it iterates through discovered tags. The tag name is assigned to \(tag, and the tag content is assigned to \)val. The vulnerable path is triggered when $tag matches MDItemFSCreationDate or FileCreateDate.

At that point, the tag content flows into $val and is passed to SetMacOSTags. The article states that the filename parameter is escaped, but the date value is not, allowing an attacker to inject quotes into the metadata field and break the intended command structure.

Payload delivery method

The article explains that direct assignment to FileCreateDate is heavily validated. Initial attempts to inject malformed dates failed because ExifTool rejected the input as an invalid date/time value.

The researcher instead used a source date-time tag such as DateTimeOriginal. The article notes that CreateDate may also be usable, but the analysis specifically used DateTimeOriginal.

The attack stores an invalid or crafted raw value in the source tag and then copies that source tag into FileCreateDate using -tagsFromFile. This copy operation invokes SetMacOSTags and places the attacker-controlled value into the vulnerable $val parameter.

Role of the -n flag

The -n flag is central to the bypass described in the article. ExifTool’s normal handling applies the PrintConvInv filter, which rejects invalid date formats during direct writes.

Using -n skips the PrintConvInv conversion step. The article identifies this conversion step as where input sanitization occurs for the date-time value.

By skipping that step, the researcher could store raw data in a date-time metadata tag, including a single quote needed for command injection. That value could then be copied into FileCreateDate to trigger the vulnerable path.

Patch analysis

The article shows the vulnerable pattern as:

The patched form uses list-form system execution:

This change removes the need for manual escaping because arguments are passed directly rather than interpreted as a shell command string. The article also describes a new System() wrapper that redirects STDOUT and STDERR to /dev/null and restores them after the call, preserving previous behavior.
