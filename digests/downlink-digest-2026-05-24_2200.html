Overview

Threat actors are exploiting CVE-2026-26980, a critical SQL injection vulnerability in Ghost CMS, to compromise websites and turn them into delivery points for ClickFix attacks. According to QiAnXin XLab, the attackers are using the flaw to obtain Ghost admin API keys without authorization and then using those keys to modify published articles through the Ghost Admin API. The result is the insertion of malicious JavaScript into otherwise legitimate sites.

Ghost addressed the flaw in February 2026 with version 6.19.1. The article states that the vulnerability has a CVSS score of 9.4 and affects Ghost’s Content API, allowing an unauthenticated attacker to read arbitrary data from the database. It also notes that the vulnerability was discovered by Anthropic using Claude.

Why the Vulnerability Matters

The article emphasizes that the flaw is particularly severe not only because it enables arbitrary database reads, but because it can expose a site’s admin API key. Once the attackers have that key, they can invoke the Ghost Admin API and directly alter published content.

This capability lets the attackers poison websites at scale by embedding malicious code into articles. In the observed campaign, the malicious JavaScript was inserted at the bottom of pages to support fake CAPTCHA attacks associated with ClickFix activity.

Campaign Scope and Attribution

QiAnXin XLab described the operation as a “large-scale poisoning” campaign weaponizing CVE-2026-26980. The article says at least two different threat clusters are believed to be behind the activity. In some cases, certain websites were implanted with malicious code within a single day.

The activity was first detected on May 7, 2026. According to the article, more than 700 websites have been compromised.

Technical Infection Flow

Initial Site Poisoning

The attackers exploit the Ghost CMS flaw to obtain the target site’s admin API key. They then use the Ghost Admin API to bulk modify articles and insert a malicious JavaScript loader into the page content.

JavaScript Loader Architecture

The injected code functions as a two-stage loader. Its purpose is to retrieve the main payload at runtime from the external domain clo4shara[.]xyz/11z77u3.php.

The article says this design gives the attackers flexibility, because they can change the delivered payloads while keeping the same loader in place across many compromised sites.

Traffic Distribution and Cloaking

XLab said that directly accessing clo4shara[.]xyz/11z77u3.php reveals a traffic distribution script. According to the article, the script collects browser fingerprint information from users and uploads it to the server, then performs actions such as redirection, popups, and downloads based on returned instructions.

The article states that the PHP script is powered by Adspect, a commercial cloaking service. Its use allows the attackers to serve benign content to security scanners and crawlers while presenting the real payload only to intended victims.

The script is also said to support 19 different commands for running arbitrary JavaScript code and facilitating remote control of the victim’s browser.

ClickFix Delivery Chain

Victims who pass the attackers’ selection logic are shown a fake CAPTCHA verification page inside an iframe. The page tells them to prove they are human by copying and pasting a Base64-encoded command into the Windows Run dialog.

That command acts as a dropper. It downloads a ZIP archive, extracts a Windows batch script, and runs it.

The batch script then executes a PowerShell command to download a DLL file from a remote domain, launch it via rundll32.exe, and open a fake web page as a distraction for the user.

Later Payload Variants

The article says later iterations of the malware replaced the DLL with a JavaScript payload. Despite this change, the goal remained the same: to drop a Windows executable onto the victim’s system.

Two payload paths were described:

  • In the DLL-based path, the dropped executable was a PuTTY client signed with a valid code-signing certificate.
  • In the JavaScript-based path, the binary was an Inno Setup installer for an Electron application.

Final Malware Stage

The Electron application is described as a modified version of the open-source Grape desktop client. According to the article, it is designed to establish persistence and poll the remote server web-telegram[.]ug every 30 seconds.

The application can then process attacker-issued instructions, including commands to run JavaScript code or executable files.

Response and Mitigation

The article advises Ghost CMS users to take several actions:

  • Upgrade Ghost instances to the latest version
  • Rotate all credentials
  • Clean up compromised sites
  • Audit access logs for suspicious activity
  • Notify users who may have visited affected sites during the contamination period

These steps are recommended because the campaign involved site tampering and may have exposed visitors to malware through fake CAPTCHA lures.
