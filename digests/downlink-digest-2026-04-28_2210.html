A critical SQL injection flaw in the popular LiteLLM open-source LLM gateway is currently under active exploitation by threat actors. Tracked as CVE-2026-42208, the vulnerability resides in the proxy API key verification step and requires no authentication to exploit.

Attackers leverage specially crafted Authorization headers sent to any LLM API route to execute arbitrary SQL commands against the backend database. This access allows them to read and modify sensitive stored data, including API keys, virtual and master keys, and environment configuration secrets.

The LiteLLM project, which boasts 45,000 GitHub stars, serves as a critical middleware layer for developers managing multiple AI models. Following public disclosure on April 24, researchers at Sysdig detected deliberate exploitation attempts approximately 36 hours later.

These attacks targeted the /chat/completions endpoint with malicious Bearer tokens designed to extract specific credential tables. The attackers skipped benign probing and went straight for secret storage locations, suggesting advanced reconnaissance or insider knowledge.

During the second phase of the campaign, the threat actor switched IP addresses to evade tracking while refining their payloads for higher precision. LiteLLM maintainers addressed the issue in version 1.83.7 by implementing parameterized queries to eliminate the string concatenation flaw.

For environments that cannot immediately patch, the maintainers recommend setting disable_error_logs to true under general_settings to block the exploitation path. Security analysts warn that any internet-exposed LiteLLM instance running a vulnerable version should be presumed compromised.

Immediate credential rotation for all virtual keys, master keys, and provider credentials is mandatory. This incident follows a recent supply-chain attack against LiteLLM’s PyPI package, underscoring sustained adversarial focus on the platform.

The flaw enables an authenticated user with repository push access to achieve remote code execution on the backend infrastructure using a single git push command. Google-owned cloud security firm Wiz discovered and reported the issue on March 4, 2026, prompting GitHub to validate and deploy a fix to GitHub.com within two hours. Researchers have identified a severe defect in VECT 2.0 ransomware that causes it to act as a data wiper for files larger than 128 KB. The malware attempts to speed up encryption by processing large files in chunks, but it incorrectly reuses the same memory buffer for generating encryption nonces. As each chunk is processed, the newly generated nonce overwrites the previous one in memory. Once encryption finishes, only the last generated nonce is written to disk, meaning only the final 25% of the file can be decrypted. The nonces for the first three quarters of the file are lost forever and are never sent to the attackers, making recovery impossible even after ransom payment. This flaw is present across all VECT 2.0 variants, including those designed for Windows, Linux, and ESXi environments.
