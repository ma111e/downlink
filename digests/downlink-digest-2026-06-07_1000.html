Overview

CVE-2026-44825 is a hardcoded credentials vulnerability in Apache Solr’s Basic Authentication setup tool, bin/solr auth enable. The article states that the tool can silently install undocumented template accounts with publicly known default credentials. This can give a remote attacker full administrative access to a SolrCloud cluster.

The vulnerability has a CVSS v3.1 base score of 8.1, rated High. Exploitation requires no prior authentication. The central risk is that accounts may exist in the deployment even though administrators did not intentionally create them.

Technical Details

The vulnerability is tied to the process of enabling Basic Authentication using bin/solr auth enable. When this tool is used, it writes a security.json file. The article says that this file may include additional template user accounts.

The accounts specifically identified are:

  • superadmin
  • admin
  • search
  • index

Each of these accounts is configured with a password that matches the username. Because the credentials are described as identical to documented defaults, an attacker aware of the issue can attempt authentication without additional reconnaissance.

Impact

The article describes the practical impact as full administrative control of the cluster. An attacker who authenticates as superadmin can read, modify, or delete any index. The attacker can also alter cluster configuration.

The article further states that an attacker could pivot to any system the Solr cluster is authorized to reach. This makes the issue significant not only for Solr data and configuration, but also for connected systems accessible from the Solr cluster.

Affected Assets

Apache Solr versions 9.4.0 through 9.10.1 and version 10.0.0 are affected. The vulnerability is specific to SolrCloud deployments where administrators enabled Basic Authentication using bin/solr auth enable.

The article explicitly states that deployments are not affected by this specific issue if they configured BasicAuth through other means. Deployments that never enabled BasicAuth at all are also not affected by this specific issue.

Exposure Context

The article notes that Apache Solr is used for enterprise search, e-commerce product catalogs, log analytics pipelines, and document management platforms. It says SolrCloud clusters are commonly internet-facing or reachable from cloud environments. It also notes that administrative interfaces are often left accessible beyond the intended perimeter during initial setup or after infrastructure changes.

CyCognito identified externally reachable Apache Solr assets that may be exposed across a range of industries. Because no asset file was provided for the advisory, the article says per-sector percentages are not available. The article describes exposure as likely broadly distributed rather than concentrated in a single sector.

Detection and Visibility Challenges

The article emphasizes that the accounts may be silently installed. Administrators who verify their own account credentials after setup may have no obvious reason to check for additional accounts they did not create. As a result, the template users may persist for extended periods without detection.

The article also notes that organizations may inherit Solr through platform bundles or third-party applications. This can delay visibility into version currency and authentication state. It also highlights the risk of older staging or development instances remaining accessible after production systems have been upgraded.

Fixes and Mitigation

Apache has released Solr 9.11.0 and 10.1.0 to address CVE-2026-44825. Organizations running affected versions should upgrade to the corresponding fixed release.

For deployments where immediate upgrading is not possible, Apache recommends deleting the template accounts created by bin/solr auth enable. The accounts to audit and remove are superadmin, admin, search, and index. These accounts are defined in security.json and can be removed or have their credentials rotated through the Solr Security API.

The article recommends that defenders verify directly with Apache’s security advisory and inspect their deployment’s security.json contents. It warns that the absence of weak credentials in self-created accounts does not rule out the presence of silently installed template accounts.

Recommended Actions

The article recommends the following actions until patching is confirmed:

  • Audit security.json on all SolrCloud nodes for undocumented template accounts.
  • Remove or rotate credentials for superadmin, admin, search, and index if present.
  • Restrict network access to the Solr admin interface at the firewall or WAF layer.
  • Monitor Solr authentication logs for logins using default credential patterns.
  • Inventory all Solr deployments, including staging and development instances, to identify versions in the affected range.
  • Verify that BasicAuth-enabled clusters were not configured using bin/solr auth enable on any affected version.

CyCognito published an Emerging Threat Advisory for CVE-2026-44825 in the CyCognito platform. The company says it is actively researching enhanced detection capabilities for this vulnerability. The article also directs readers to contact CyCognito to request a demo.

02 Apache Solr CVE-2026-44825 Hardcoded Credential Exposure

CyCognito detailed CVE-2026-44825, a hardcoded credentials vulnerability affecting Apache Solr’s Basic Authentication setup tool, bin/solr auth enable. The flaw has a CVSS v3.1 base score of 8.1 and affects Apache Solr versions 9.4.0 through 9.10.1 and version 10.0.0 in SolrCloud deployments where Basic Authentication was enabled using the affected tool.

The issue arises when the setup process writes a security.json file containing template accounts named superadmin, admin, search, and index, each with a password matching the username. In affected deployments, authentication as superadmin can provide full administrative control over the SolrCloud cluster, including the ability to read, modify, or delete indexes and alter cluster configuration.

Apache released Solr 9.11.0 and 10.1.0 to address CVE-2026-44825. The reporting also notes that organizations unable to immediately upgrade can identify exposure by examining security.json for the template accounts, making the vulnerability particularly relevant to internet-exposed or broadly accessible SolrCloud environments where administrative authentication is reachable.
