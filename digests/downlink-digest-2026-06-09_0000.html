Background and Event

The article opens with the Miasma worm, which had recently compromised 32 official Red Hat packages on npm. The worm added a malicious preinstall script to each compromised package so that node index.js ran automatically the moment the dependency was installed, harvesting cloud credentials, CI tokens, SSH keys, and more before any of the victim’s own code ran. In the days that followed, Miasma spread beyond its initial targets across npm, PyPI, and GitHub, hitting packages including @vapi-ai/server-sdk (71k weekly downloads) and ai-sdk-ollama (31k weekly downloads).

The new wave introduces a new trick: instead of using a preinstall or postinstall hook in package.json, it moved the trigger into binding.gyp, a file npm will execute at install time. As a result, auditing package.json and seeing no lifecycle hooks no longer proves a package is safe.

What node-gyp and binding.gyp Are

Many npm packages ship native C/C++ add-ons that must be compiled before Node can load them. node-gyp is the cross-platform build tool npm bundles and invokes for this. It wraps GYP (Generate Your Projects), a build system Google created for Chromium and later abandoned; node-gyp now uses a Node.js-maintained fork. node-gyp reads binding.gyp from the package root to know what to build. The file is JSON-like but technically a Python literal. Crucially, when npm installs a package and notices a binding.gyp in its root, it automatically runs node-gyp rebuild without any script registered in package.json. The mere presence of the file is enough for code to run during installation.

How Miasma Exploited It

The worm dropped a binding.gyp with a target named Setup whose sources array contained the string <!(node index.js > /dev/null 2>&1 && echo stub.c). The <!(...) syntax is a GYP command expansion: GYP runs the enclosed shell command and substitutes its output. So node-gyp executed node index.js (the Miasma credential-stealer/worm payload), discarded all output with > /dev/null 2>&1 to keep logs clean, and && echo stub.c printed a harmless filename so the build continued normally. No preinstall hook was necessary.

Expansion Syntax

GYP offers several command-expansion flavors, all executing at parse time before compilation:

  • <!(command) / >!(command) / ^!(command) — run the command and substitute raw output as a single string.
  • <!@(command) / >!@(command) / ^!@(command) — run the command and split output into a list.
  • <!pymod_do_main(module args) — import a Python module and call its DoMain() function.
  • <|(name item1 item2 ...) — create a file named name at parse time.

GYP does not scope expansion to documented fields. It recursively walks the entire parsed structure and expands tokens inside any string value under any key at any depth, even an invented key like some_random_key. This makes review painful because payloads can hide anywhere.

The Sandbox Escape

binding.gyp is a Python dictionary handed to Python’s eval. GYP strips builtins by calling eval(file_contents, {"__builtins__": {}}, None), intending to deny access to dangerous functions like import and open. However, the sandbox can be escaped. A single Python expression beginning from an empty tuple () traverses object relationships to catch_warnings, recovers builtins and import, loads os, and runs a shell command (e.g., node evil.js). The article shows the entire malicious binding.gyp can be just this one expression, with no JSON or targets/sources fields.

The expression can also be tucked into a normal-looking build file, for example hidden in an openssl_fips variable, requiring no <!(...) expansion. Conditions are treated the same way: GYP compiles and evaluates condition strings like "OS=='win'" via eval with the same stripped builtins, so a condition can hold the same sandbox-escape expression. The author stresses this is more dangerous than documented features because nothing about binding.gyp was ever intended to host pure Python code.

Hiding Code in Included and Auto-Included Files

binding.gyp supports an includes key to factor out shared settings. Included files are processed exactly like the main file, so all expansion and sandbox tricks apply. The included file can be named anything (no .gyp/.gypi extension required, e.g. config or LICENSE) as long as it holds valid JSON-formatted data, and includes are transitive, so a payload can be several files away.

Beyond explicit includes, node-gyp auto-includes config.gypi and common.gypi from the package root if present, processing them like any gyp file even when binding.gyp is just { }. Worse, node-gyp also auto-includes ~/.gyp/include.gypi from the user’s home directory into every gyp build that user runs. A payload dropped there once persists across every future native npm install with a binding.gyp.

Dependencies

Separate from includes, GYP targets can declare dependencies on targets in other .gyp files (e.g. "dependencies": ["dep.gyp:dep_target"]). The referenced file is parsed and expanded like any other, giving a second independent way to reach code in another file. Referenced file names are irrelevant as long as they hold valid JSON, and dependencies can also be transitive.

Compiler Hijacking

binding.gyp controls how native code is built. GYP exposes compiler settings via per-target keys (cflags, defines, include_dirs) and make_global_settings (CC, CXX, LINK, AR, CFLAGS, LDFLAGS). Because compilation happens at install time, an attacker can:

  • Replace the compiler, e.g. setting CC to <(module_root_dir)/cc-evil.sh, which runs evil.js and then execs the real compiler so the build succeeds.
  • Use the documented *_wrapper convention (meant for launchers like ccache), e.g. CC_wrapper pointing at cc-evil-wrapper.sh, which receives the real compiler as an argument.
  • Inject malicious flags. On make-based builds, flag values become make variables, and make evaluates $(shell ...) inside them, so flags like cflags or global CFLAGS can carry $(shell node <(module_root_dir)/evil.js). The command’s output is passed to the compiler as a harmless flag.

The exact mechanism varies by build tool and OS, but compiler and linker settings should be treated as code.

Executing Code Through Actions, Rules, and Postbuilds

GYP has documented features that run commands by design:

  • actions — a build step in a target’s actions array that runs an arbitrary command (e.g. "action": ["node", "evil.js"]), needing no expansion syntax.
  • rules — like actions but fire once per input file matching a given extension; an attacker ships a throwaway file (e.g. trigger.poc) so the rule’s command runs against it.
  • postbuilds — a command that runs after a target is built, carrying the same action array.

Key Takeaway and Detection

The central message is that a binding.gyp file runs code at install time exactly like a preinstall or postinstall hook in package.json and deserves the same suspicion. A clean package.json is no longer evidence that nothing runs. Supply-chain attackers like those behind Miasma are seeking new install-time execution methods, and binding.gyp is easy to miss, especially undocumented behavior like the sandbox escapes.

  • The standard manual review practice of checking package.json for preinstall/postinstall hooks is now insufficient as a safety signal, because a binding.gyp file alone causes code execution at install time regardless of what package.json declares.
  • The sandbox escape is qualitatively more dangerous than documented command expansions or postinstall hooks because nothing about binding.gyp was ever intended to host pure Python code, so existing reviewer intuitions and scanner rules built around expected dangerous fields fail to anticipate it.
  • Included files can be named anything (such as config or even LICENSE) and need no .gyp/.gypi extension as long as they hold valid JSON, and includes/dependencies can be transitive, so the actual payload may sit several files away from the binding.gyp under review.

Obfuscation and analysis resistance

MLTBackdoor uses several methods to hinder analysis. The article says it uses indirect system calls, API hashing, and obfuscation methods applied at compilation time with an LLVM-based obfuscator.

Mixed Boolean-Arithmetic

MLTBackdoor uses Mixed Boolean-Arithmetic obfuscation to rewrite simple arithmetic expressions into mathematically equivalent but much more complex expressions. ThreatLabz provides examples showing that noisy multi-line expressions collapse into simple operations after replacing bitwise-not expressions such as ~x with -x - 1.

The article states that MLTBackdoor uses MBA so extensively that around 95% of its code consists of extra, unnecessary calculations. This makes the malware’s logic harder to follow during reverse engineering.

Control Flow Flattening

MLTBackdoor also uses Control Flow Flattening. This technique replaces normal conditional logic with a large loop and switch-based state machine, which obscures the real execution order of code blocks.

ThreatLabz notes two additional complications in MLTBackdoor’s CFF implementation:

  • The state value is stored at a stack offset and XORed before each comparison.
  • The calculation of the next state is wrapped in MBA.

Stack-built strings

The malware does not generally encrypt or encode strings. Instead, it constructs strings at runtime byte-by-byte on the stack.

The article highlights that this becomes more difficult to analyze when combined with MBA and CFF. For example, a C2 domain such as cwrtwright[.]com may be constructed through multiple functions and stitched together, while the flattened state machine fragments the string-building logic. ThreatLabz states that this defeats tools such as FLOSS that look for consecutive characters in memory.

API resolution and indirect system calls

MLTBackdoor resolves Win32 APIs, system calls, and Beacon Object File symbols at runtime using DJB2 hashing. ThreatLabz observed three hashing patterns:

  • Normal WinAPI lookups, such as djb2("WinHttpConnect").
  • Lowercase API name lookups, such as djb2("enumwindows").
  • Beacon-prefixed lookups, such as djb2("BeaconNtCreateFile").

The malware also uses a Hell’s Gate-style indirect system call technique. At startup, it walks ntdll exports, matches them against a list of 31 Nt-function hashes, and builds a runtime table containing hashes, system service numbers, and syscall gadget addresses. When it needs to call a Windows API function, it calls its own wrapper, looks up the hash, retrieves the system service number and gadget address, and jumps to the corresponding ntdll system call address.

The article lists the Nt APIs covered by the malware’s resolved hashes, including NtAllocateVirtualMemory, NtProtectVirtualMemory, NtFreeVirtualMemory, NtWriteVirtualMemory, NtCreateThreadEx, NtQueryInformationProcess, NtQuerySystemInformation, NtOpenProcess, NtClose, NtOpenFile, NtCreateFile, NtReadFile, NtWriteFile, and several registry-related calls.

Anti-analysis behavior

MLTBackdoor includes multiple anti-analysis techniques for detecting debuggers and sandboxed environments. The article explicitly notes that detection does not halt execution. Instead, the malware aggregates the results of 10 checks into a bitmask and sends that bitmask as part of its initial request.

The documented checks include:

  • Hypervisor checks that look for known virtualization vendor IDs such as VMwareVMware, VBoxVBoxVBox, XenVMMXenVMM, and KVMKVMKVM.
  • An additional hypervisor-related check that treats some other hypervisor vendor IDs differently and accounts for Windows 10⁄11 hosts with Virtualization-Based Security enabled.
  • A timing check using repeated RDTSC, CPUID, and RDTSC loops.
  • A debugger check using NtQueryInformationProcess with the ProcessDebugPort information class.
  • A process-name check that hashes running process names with SHA256 and compares them to a hardcoded list.
  • A window-title check using strings such as x64dbg, x32dbg, ollydbg, windbg, idapro, process monitor, process explorer, wireshark, fiddler, dnspy, and cff explorer.
  • A sandbox driver check for names such as vbox, vmci, vmhgfs, virtio, vioscsi, and xenbus.
  • A RAM check for systems below 2 GB.
  • A CPU-count check for systems with one processor.
  • An uptime check for systems running less than five minutes.

Built-in capabilities

MLTBackdoor includes a small set of built-in commands focused on filesystem operations. The commands described in the article are:

  • download, which retrieves a file from the victim’s machine.
  • upload, which drops a file on the victim’s machine.
  • ls, which lists files in a directory.
  • delete, which deletes a file or folder.
  • rename, which renames or moves a file or folder.
  • mkdir, which creates a new folder.

The article states that the BOF loader is what significantly increases MLTBackdoor’s capabilities beyond this built-in command set.

Beacon Object File loader

MLTBackdoor can load Beacon Object Files, which the article describes as Microsoft Common Object File Format compiled files containing sections, a symbol table, and relocations. The malware can map and execute BOFs inside its own process and then free them.

ThreatLabz describes the BOF dispatcher as performing the following steps:

  • Creating one memory block per section.
  • Walking the COFF symbol table.
  • Applying relocations per section.
  • Changing section permissions to read and execute only.
  • Handling crashes and returning them as results.
  • Locating the entry point in the symbol table.
  • Removing and freeing allocated memory.

The BOF loader is compatible with Cobalt Strike beacons that rely on a small subset of DJB2-hashed imports. The article lists supported imports such as BeaconDataParse, BeaconDataInt, BeaconDataShort, BeaconDataLength, BeaconDataExtract, BeaconPrintf, and BeaconOutput.

A notable difference described by ThreatLabz is that the loader also includes 19 additional Beacon-prefixed cases that route calls to MLTBackdoor’s own indirect system call wrappers. These include wrappers for calls such as BeaconNtAllocateVirtualMemory, BeaconNtProtectVirtualMemory, BeaconNtFreeVirtualMemory, BeaconNtClose, BeaconNtCreateFile, BeaconNtOpenFile, BeaconNtReadFile, BeaconNtWriteFile, BeaconNtQueryInformationFile, BeaconNtSetInformationFile, BeaconNtQueryDirectoryFile, BeaconNtQuerySystemInformation, BeaconNtOpenProcessToken, BeaconNtQueryInformationToken, BeaconNtCreateKey, BeaconNtOpenKey, BeaconNtSetValueKey, BeaconNtDeleteValueKey, and BeaconNtDeleteKey.

Network communication

MLTBackdoor uses a custom encrypted binary protocol over TLS on port 443. The fixed HTTP path is /api/v1/telemetry, and the User-Agent is Microsoft-Delivery-Optimization/10.1, which the article says is used to masquerade as legitimate traffic.

The malware performs an Elliptic-Curve Diffie-Hellman key exchange using NIST curve P-256. It generates a new key pair per session and performs ECDH with a P-256 public key shared by the C2 server. The resulting shared secret is concatenated with both the session public key and the C2 server public key, then hashed with SHA256 to derive a shared secret used as an AES-256-GCM session key.

After the key exchange, subsequent messages are encrypted with the AES session key and a random 12-byte nonce. The article describes the packet structure as containing a header, encrypted payload, and a 16-byte AES-GCM tag.

DGA and command-and-control resilience

Some MLTBackdoor samples use hardcoded stack-built C2 domains together with a DGA. Other samples rely only on hardcoded domains or only on the DGA.

The article describes the DGA as a deterministic date-based algorithm that generates a new domain per day. ThreatLabz states that the DGA is designed to maintain control of infected systems if hardcoded C2 domains are unreachable.

ThreatLabz notes that the domain generated for April 29, 2026, hrs2y15sungu[.]com, was used not only for C2 communication but also for the distribution campaign described in the infection-chain section.

Protocol naming and message types

ThreatLabz named the malware MLTBackdoor based on the first four magic bytes of the network communication protocol’s header. The magic value is 0x014D4C54, represented as \x01MLT.

The packet header includes:

  • A 32-bit magic value.
  • A 32-bit session ID.
  • A 32-bit message type.
  • A 32-bit payload length.
  • A 12-byte nonce.
  • A 4-byte unknown field.

The session ID consists of four random bytes generated through BCryptGenRandom and is regenerated on each handshake.

The article lists supported message types for C2 communication, including check-in messages, BOF tasks, sleep commands, process exit, command execution results, ECDH key exchange, file download and upload operations, directory listing, delete, rename, mkdir, and BOF stdout.

Kerberos Primer

The article establishes the foundational Kerberos exchange:

  • The client sends an AS-REQ to the KDC containing a timestamp encrypted with a key derived from the client’s password; the username is sent in plaintext so the KDC can look up the matching key in AD.
  • On success, the KDC builds a TGT with the user’s metadata in the PAC, encrypts it with the KRBTGT key, generates a session key (placed inside the TGT and in the EncASRepPart of the AS-REP), and the client caches the TGT and session key in LSASS.
  • To access a service, the client sends a TGS-REQ with the TGT, target SPN, and an authenticator encrypted with the session key. The authenticator proves possession of the session key. The KDC issues a service ticket (ST) encrypted with the service’s long-term key, containing a fresh session key and the copied PAC.
  • The client presents the ST to the service in an AP-REQ with a new authenticator; the service decrypts the ticket with its long-term key, recovers the session key, validates the authenticator, uses the PAC for authorization, and replies with an AP-REP.

The Problem U2U Solves

Windows services typically run as built-in accounts (e.g., NT AUTHORITY\SYSTEM) or dedicated service accounts that have an SPN. Regular AD user accounts do not have an SPN. In peer-to-peer (P2P) scenarios where a user authenticates to a service hosted by another user, the KDC cannot locate a long-term key and returns KDC_ERR_S_PRINCIPAL_UNKNOWN. U2U solves this by encrypting the service ticket with the hosting user’s TGT session key instead of a service account’s long-term key.

How U2U Works Mechanically

  • The client sends a KERB-TGT-REQUEST directly to the target service.
  • The service responds with a KERB-TGT-REPLY containing its own TGT.
  • The client submits this TGT in the additional-tickets field of a modified TGS-REQ with the ENC-TKT-IN-SKEY flag set.
  • The KDC decrypts the additional TGT with the KRBTGT key, extracts the session key, and uses it to encrypt the new U2U service ticket.

Why Handing Out TGTs Is Safe

The TGT is encrypted with the KRBTGT account’s long-term key, so the session key inside is inaccessible without that key. AD randomly generates the KRBTGT password, making it computationally infeasible to crack (cracking it would enable the Golden Ticket attack, a threat independent of U2U). A TGT alone is unusable without its associated session key, which is why Rubeus output is a KRB-CRED structure bundling the ticket with the session key, flags, expiration, and metadata.

Why Session Keys Beat Long-Term Keys

The KDC randomly generates session keys, delivers them in the AS-REP’s EncASRepPart, and does not store them. Their validity is tied to the TGT lifetime (default 10 hours). Because they are not password-derived, there is no dictionary or brute-force path, so a captured U2U ticket would offer at most a ten-hour window against a random key — not a realistic attack surface.

Where the Draft Falls Short of Reality

The draft-swift-win2k-krb-user2user specification defines three U2U-initiation scenarios:

  1. Client knows U2U upfront — goes directly to the service with KERB-TGT-REQUEST, then performs U2U. Implemented in Windows (RDP with NLA).
  2. KDC enforces U2U via account policy — draft says implementations “MAY” apply a per-account policy, with the KDC returning KDC_ERR_MUST_USE_USER2USER (0x1B).
  3. Server enforces U2U after the TGS exchange — server returns KRB_AP_ERR_USER_TO_USER_REQUIRED (0x45) to an AP-REQ.

The authors found significant divergence from reality:

  • There is no per-account policy, userAccountControl flag, or Group Policy setting in Windows to force U2U.
  • KDC_ERR_MUST_USE_USER2USER does not exist in kerberos.dll; attempts to trigger it (via UPN in the SPN field or an empty SPN) only returned KDC_ERR_S_PRINCIPAL_UNKNOWN.
  • Even if it existed, the error provides no path to the needed TGT, since the KDC does not track TGTs after issuance.
  • The third scenario is not entirely fictional but only applies when a service is hardcoded to require U2U and explicitly returns the error — application-level logic, not a missing SPN or KDC policy.

Microsoft chose to enforce U2U earlier in the flow rather than at the AP layer, so only the first scenario is consistently observed in Windows.

The most common legitimate U2U observation is RDP when the target server has NLA enabled. NLA requires authentication before the server renders a login screen and relies on CredSSP to forward credentials in encrypted form. Through lab traffic captured in Wireshark, the authors traced the exchange:

  • The RDP client makes a standard TGS-REQ for TERMSRV/hostname.domain.local and receives a valid service ticket that is then never used or cached.

The article connects U2U primitives to the UnPAC-the-Hash attack:

  • In PKINIT authentication, the user authenticates with a certificate rather than a password, so the client has no NT hash to cache for the msv1_0 package.   • PAC_CREDENTIAL_INFO is encrypted with the AS reply key, derived from the Diffie-Hellman shared secret and two nonces (server and client). This key encrypts the EncASRepPart in PKINIT and is only temporarily stored in LSASS.
  • Natively, the LSA decrypts EncASRepPart to get the TGT session key, then requests a service ticket for itself (HOST/FQDN) to obtain a decryptable copy of the PAC, decrypts PAC_CREDENTIAL_INFO with the AS reply key, and caches the NT hash for NTLM.

The attack reuses these primitives:

  • The attacker first obtains a target’s certificate and private key (via ADCS ESC or Shadow Credentials attacks, or by dumping a PKINIT-obtained TGT).
  • They initiate their own PKINIT exchange with an attacker-controlled ephemeral DH key pair, receive an AS-REP, and derive the AS reply key, yielding a valid TGT and session key.
  • They request a self-targeted U2U service ticket, which the KDC encrypts with the held TGT session key, allowing the attacker to decrypt it entirely on their own.
  • They access the embedded PAC and decrypt PAC_CREDENTIAL_INFO with the derived AS reply key to retrieve the user’s NT hash.

The authors close by teasing additional U2U content in part 2 and thank Duane Michael and Elad Shamir.

  • The U2U specification draft diverges significantly from Windows reality: the authors manually attempted to trigger KDC_ERR_MUST_USE_USER2USER and confirmed it does not exist in kerberos.dll, and even an error like this would be of limited use since the KDC does not track TGTs after issuance and cannot direct the client to the material it needs.
  • In RDP with NLA, the client first obtains a perfectly valid standard service ticket for TERMSRV/hostname but then never uses or caches it, instead deliberately switching to U2U solely to obtain a fresh, non-cacheable, short-lived session key that protects the plaintext credential delivery in CredSSP from interception and key extraction.
  • The KDC ignores the sname/SPN field entirely once ENC-TKT-IN-SKEY is set and simply reflects whatever target value was supplied back into the issued ticket's sname, contradicting the draft's specification that sname should reflect the target service's UPN.
  • Microsoft's inclusion of the NT hash in the PAC during PKINIT authentication, intended only to support legacy NTLM services when no password is available, is precisely the design decision that makes UnPAC-the-Hash possible by allowing an attacker who controls the DH key pair to derive the AS reply key and decrypt PAC_CREDENTIAL_INFO.
Key Points
  • Kerberos user-to-user (U2U) authentication solves peer-to-peer scenarios where a service runs under an account that has no SPN, by encrypting the service ticket with the hosting user's TGT session key rather than a service account's long-term key, using the ENC-TKT-IN-SKEY flag in a modified TGS-REQ that includes the target's TGT in the additional-tickets field.
  • Handing out TGTs is safe because they are encrypted with the randomly-generated KRBTGT long-term key, and a TGT is unusable without its associated session key, which is why Rubeus exports tickets as KRB-CRED structures containing both the ticket and the session key.
  • Although the draft RFC defines three U2U scenarios, the authors found that Windows only consistently implements the first (client-initiated) scenario, as the KDC_ERR_MUST_USE_USER2USER error does not exist in kerberos.dll and there is no per-account policy attribute to force U2U.
  • RDP with Network Level Authentication (NLA) is the most common legitimate use of U2U, where CredSSP calls InitializeSecurityContext with the ISC_REQ_USE_SESSION_KEY flag to obtain a fresh, non-cacheable session key used to securely deliver the client's plaintext credentials.
  • The UnPAC-the-Hash attack reuses U2U primitives: after obtaining a user's certificate, an attacker performs a PKINIT exchange to derive the AS reply key and TGT session key, requests a U2U service ticket for themselves, and decrypts the PAC's PAC_CREDENTIAL_INFO structure to recover the user's NT hash for Pass-the-Hash.
