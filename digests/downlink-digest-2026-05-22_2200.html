Summary
An attacker rewrote every git tag across four Laravel-Lang Composer packages to inject a malicious autoloaded payload that silently steals CI secrets and environment variables, making all version tags unsafe regardless of pinning unless a pre-attack commit hash is used.

On May 22, 2026, a threat actor compromised four Composer packages maintained by the Laravel-Lang organization by rewriting every existing git tag to point at malicious commits. Each commit added src/helpers.php to the Composer autoload.files map, causing the payload to execute immediately whenever any dependent application booted, without requiring any class instantiation or function call. The payload contacted flipboxstudio.info, dropped hidden PHP and ELF artifacts in /tmp, exfiltrated CI runner environment data, and then deleted its own files to hinder forensic analysis.

The attack made every tag across laravel-lang/lang, laravel-lang/http-statuses, laravel-lang/actions, and laravel-lang/attributes unsafe, regardless of version pinning, because the tags themselves were moved. Only pre-attack commit hashes verified against a trusted source remain safe, and any project that ran a Composer install or update after the attack window must rotate all secrets that were accessible to the affected environment.

A single threat actor executed a broad supply chain attack against the Laravel-Lang package ecosystem on May 22, 2026, rewriting every git tag in four repositories to point at malicious commits. The attacker did not publish a new version; instead, they force-pushed new commits under the tags, so any project using version constraints (e.g., ^3.4, ~2.0) and running composer update would resolve to a poisoned release. The malicious commits all used the fake author identity “Your Name” with email you@example.com and modified only composer.json and src/helpers.php, adding the latter to Composer’s eager autoload.files map.

The payload in src/helpers.php fetched a second stage from the typosquat domain flipboxstudio.info, saved it under a hidden filename in the system cache directory, and executed it in a backgrounded process. The stage then dropped a hidden PHP loader and an ELF binary into /tmp, exfiltrated sensitive environment data (likely including GITHUB_TOKEN and other secrets), and removed the dropped artifacts within seconds to avoid detection. The orphaned processes, now reparented to init, continued running, making the network calls to the C2 the most reliable detection indicator.

Analysis using Harden-Runner in audit mode confirmed end-to-end exploitation: an autoload step triggered a GET request to download the payload, followed by a POST to exfiltrate data, all while the implant processes survived beyond the workflow step. The four affected packages — laravel-lang/lang (the core translations package with 502 tags), laravel-lang/http-statuses, laravel-lang/actions, and laravel-lang/attributes — share the same malicious commit structure and timeline, indicating a single compromised credential with organization-wide push access.

The impact is severe because no tagged version can be trusted; every tag from the earliest to the latest was moved. Recovery requires inspecting composer.lock for the listed imposter commit SHAs, rotating all secrets exposed to any runner or developer machine where the package was installed after the attack window, and auditing for orphaned processes or network calls to flipboxstudio.info. The author reported the compromise to Laravel-Lang maintainers and recommended revoking all credentials, notifying Packagist, and restoring tags from reputable clones or the dist mirror.

Overview

On May 22, 2026, a single threat actor compromised four popular Composer packages maintained by the Laravel-Lang organization. Rather than introducing a new malicious version, the attacker rewrote every existing git tag in each repository to point at a new malicious commit. The affected packages are laravel-lang/lang (the flagship Laravel translations package with 502 tags), laravel-lang/http-statuses, laravel-lang/actions, and laravel-lang/attributes. All tag rewrites occurred between 22:32 UTC on May 22 and 00:00 UTC on May 23, and each malicious commit is signed with the fake author name “Your Name” and email you@example.com, strongly indicating a single compromised credential with organization-wide push access.

Attack Mechanism

The attacker manipulated Composer’s autoloading behavior. Each malicious commit added src/helpers.php to the autoload.files map in composer.json, a mode that eagerly requires the listed file the moment vendor/autoload.php is loaded. Since every Laravel and Symfony application (and most PHP frameworks) include this autoloader at startup, the payload executes immediately without any explicit invocation.

The contents of src/helpers.php act as a dropper: they construct a URL (https://flipboxstudio.info/payload), fetch a second stage, write it to a hidden temporary file with a random hex name, and execute it in a backgrounded PHP process using @exec("php \"$f\" > /dev/null 2>&1 &"). The second stage then drops a hidden PHP loader and an ELF binary into /tmp, both under random hexadecimal names, and causes them to exfiltrate environment data via a POST to https://flipboxstudio.info/exfil before deleting the dropper files from disk. The entire compromise, from initial autoload to self-deletion, completes in roughly 3 seconds.

Technical Evidence and Detection

The article details a live detonation of laravel-lang/http-statuses v3.4.5 inside an isolated GitHub Actions runner monitored by Harden-Runner. Key observations from that run include:

  • A PHP process (the autoload step) spawning a background sh wrapper that launched the PHP loader and then exited, causing the loader and the subsequent ELF binary to be reparented to init (PID 1).
  • Two network calls: a GET to /payload (fetching the second stage) and a POST to /exfil (exfiltrating data).
  • The dropper files (/tmp/.laravel_locale/<random>.php and /tmp/.<random> ELF) were deleted within 2 seconds of creation, leaving the orphaned processes as the only runtime evidence.

Indicators of compromise identified in the article:

  • Network: DNS and HTTPS traffic to flipboxstudio.info (a typosquat of flipboxstudio.com).
  • Filesystem: /tmp/.laravel_locale/<12 hex chars>.php and /tmp/.<8 hex chars> (ELF binary), though these self-delete quickly.
  • Process: Orphaned php and unnamed ELF processes with parent PID 1.
  • Git: Commits with author “Your Name” / you@example.com, timestamps within the attack window, and diffs touching only composer.json and src/helpers.php.

Because the tags themselves were moved, every version — including historical releases — is poisoned. Only projects whose composer.lock pins a pre-attack commit SHA (and that SHA is not one of the listed imposter commits) are safe as long as they only run composer install with that lockfile.

Impact and Affected Versions

All tags across the four repositories were rewritten. The article lists specific compromised commit SHAs for example versions:

  • laravel-lang/lang: 15.29.5, 15.29.1, 2.0.4, 1.0.2, and every other of its 502 tags.
  • laravel-lang/http-statuses: v3.4.5, v3.4.0, v3.0.0, v2.0.0, v1.0.0, and all intermediate tags.
  • laravel-lang/actions: 1.12.2, 1.10.0, 1.0.0, and all 46 tags.
  • laravel-lang/attributes: v2.4.1, 2.6.0, 2.5.0, and all 86 tags.

Any project that ran composer update or composer install with these packages on or after 2026-05-22 22:32 UTC may be compromised. The payload exfiltrates environment variables, meaning CI tokens (GITHUB_TOKEN, cloud credentials, deploy keys, database credentials) must be rotated immediately.

Recovery and Remediation

The article prescribes the following recovery steps:

  • Halt all composer update and composer install operations for affected packages.
  • Inspect composer.lock files; if any locked commit matches an imposter SHA, treat the project as compromised.
  • Rotate all secrets accessible to any environment that installed the package after the attack window.
  • Check for orphaned php and unnamed ELF processes with ppid=1, and look for lingering artifacts in /tmp.
  • If using Harden-Runner, search for detections of flipboxstudio.info across the tenant to identify compromised runs.
  • Block flipboxstudio.info at the egress, DNS, or firewall level.

For maintainers, the recommendations include:

  • Force-update all tags to their original commits using the Packagist dist mirror or pre-attack clones.
  • Audit and rotate every credential with push access to the Laravel-Lang organization; re-enforce 2FA.
  • Notify Packagist and audit the rest of the organization for similar rewrites.

Conclusions

This incident demonstrates how tag rewriting can bypass version-pinning defenses and how a small change to Composer’s autoload configuration can achieve immediate and stealthy code execution. The self-deleting, reparented payload leaves minimal forensic evidence, making network egress monitoring the most reliable detection method. The uniform attack pattern across four repositories underscores the danger of broad-scope credentials in open-source organizations.

  • The attacker rewrote every git tag, not just the latest, meaning even projects pinning to specific versions like `v3.4.5` were compromised; only commit hash pinning from before the attack is safe.
  • The payload, its second stage, and the dropped ELF binary all self-delete within 3 seconds and then reparent to init, leaving almost no forensic trace on disk aside from network traffic to the typosquat C2 domain.
  • By adding a single file to Composer's `autoload.files` array, the attacker ensured the payload runs immediately on every application boot that loads the autoloader, without any function call or class instantiation required.
  • The attack was carried out across four major packages within 88 minutes, suggesting the actor used a single compromised credential with org-wide push access and demonstrated how rapidly a whole package ecosystem can be poisoned.

Summary
CISA is launching a new form for outside security researchers and vendors to directly nominate actively exploited vulnerabilities to its Known Exploited Vulnerabilities catalog, aiming to speed up threat detection and defensive patching.

The Cybersecurity and Infrastructure Security Agency (CISA) launched a new nomination form and email pathway that enables external researchers, vendors, and industry partners to directly submit actively exploited vulnerabilities to its Known Exploited Vulnerabilities (KEV) catalog. The KEV catalog is an authoritative list used to mandate federal patching timelines, and it has become a key resource for defenders both inside and outside government. The new submission process aims to speed up the identification, validation, and sharing of critical threat information by crowdsourcing exploitation intelligence.

Organizations remediate KEV-listed vulnerabilities 3.5 times faster than non-KEV bugs, and the additional submissions are expected to accelerate KEV additions and subsequent defensive actions. However, the growing volume of AI-discovered vulnerabilities, and the speed at which AI can weaponize flaws, is putting pressure on the timeline. CISA is now considering shortening the standard three-week remediation deadline to just three days, and the new structured form, which requires evidence of exploitation, is intended to help defenders prioritize real-world risk over theoretical severity.

CISA’s Known Exploited Vulnerabilities catalog has served since 2021 as an internal mandate for federal agencies: vulnerabilities added to the KEV must be patched within a specific timeframe, typically three weeks. Over time the list has become an industry-wide reference, and research shows that organizations fix KEV bugs 3.5 times faster than non-KEV ones. Its influence stems from its focus on flaws that are being actively exploited, rather than on those that merely carry high severity scores.

On Thursday, CISA announced a new nomination form—and a continued email option—that allows researchers, vendors, and industry partners outside the U.S. government to report vulnerabilities they believe should be added to the KEV. Chris Butera, CISA’s Acting Executive Assistant Director for Cybersecurity, said this reporting capability enhances the agency’s ability to identify, validate, and quickly share threat information. Former CIO Robert Costello described the form as a practical way to operationalize the partnership with the security community, telling the publication that crowdsourced exploitation intelligence through a standardized process would lead to faster KEV additions and faster defensive action across the ecosystem.

The submission process is more structured than the previous email-only approach. Mayuresh Dani of Qualys noted that the old inbox had no transparency around how many submissions were accepted; the new form demands detailed information and evidence of exploitation. Dani emphasized that visibility into the post-submission verification process, along with guardrails against false or incorrect reporting, will determine the initiative’s real effectiveness. He also pointed out that commercial alternatives to the KEV already exist, and some now view the official catalog as a trailing indicator of exploitation.

The article highlights that AI is compressing the window between vulnerability disclosure and exploitation, prompting CISA leadership to consider a dramatic policy change. While most initial KEV entries carried a three-week fix deadline, recent months have seen the agency assign three-day and even 24-hour deadlines to particularly dangerous bugs. According to a Reuters report cited in the article, CISA Acting Director Nick Anderson and U.S. National Cyber Director Sean Cairncross have floated the possibility of cutting the KEV deadline for all new vulnerabilities to just three days, directly citing concerns over AI-enabled exploit development.

Chris Doyle of JupiterOne said improvements like the new nomination form strengthen the signal quality and timeliness of the KEV, which helps defenders prioritize real-world risk. The overall aim is to accelerate defense efforts, vulnerability disclosure, and exploitation tracking. By making the intake process more rigorous and open, CISA hopes to keep the KEV catalog current with the fast-evolving threat landscape.

In sum, the new reporting pathway represents a convergence of several pressures: a widely adopted yet potentially lagging KEV list, a flood of AI-discovered vulnerabilities that may or may not be weaponized, and the accelerating speed of exploitation. The move is broadly welcomed as a necessary step to surface actionable intelligence faster and keep patching cadences ahead of attackers.

CISA Opens KEV Nomination to External Researchers: A Comprehensive Analysis

Background: The Known Exploited Vulnerabilities Catalog

  • The KEV catalog is an authoritative, mandatory list maintained by CISA that tells federal agencies which software and hardware vulnerabilities must be patched.
  • Since its debut in 2021, the catalog has grown substantially and is now used by private-sector organizations as a de facto prioritization reference.
  • Research cited in the article indicates that organizations remediate KEV-listed vulnerabilities 3.5 times faster than non-KEV bugs, demonstrating its practical impact.
  • The catalog’s core value proposition is its focus on vulnerabilities that are being actively exploited by hackers and nation-state actors, rather than on those with high theoretical severity scores.

The New Nomination Mechanism

  • CISA announced a new nomination form (hosted on a Qualtrics survey) alongside an existing email channel to allow “researchers, vendors, and industry partners” to report bugs for KEV inclusion.
  • Acting Executive Assistant Director for Cybersecurity Chris Butera said the capability enhances CISA’s ability to “identify, validate, and quickly share critical threat information.”
  • Submitters must provide detailed information about the vulnerability and, critically, evidence of its active exploitation. This is a departure from the previous informal email intake, which lacked standardization.
  • Former CISA CIO Robert Costello framed the move as operationalizing the agency’s partnership with the security research community. In his view, crowdsourcing exploitation intelligence through a standardized process will lead to faster KEV additions and, ultimately, faster defensive action across the federal, private, and critical infrastructure networks.

Why This Move Is Happening Now: The AI Acceleration Factor

  • The article repeatedly underscores the role of AI in both discovering and exploiting vulnerabilities at an unprecedented pace.
  • Costello explicitly stated that AI is accelerating discovery and exploitation, making “early, coordinated disclosure more critical than ever.”
  • There is a growing deluge of AI-discovered vulnerabilities, many of which are insignificant and unlikely to be exploited, amplifying the need for a signal that separates real-world risk from noise.
  • The pressure from AI is also forcing a re-examination of remediation timelines, which is discussed below.

Verification Challenges and the “Trailing Indicator” Concern

  • Mayuresh Dani of Qualys noted that while CISA previously accepted KEV submissions via email, there were no external reports on how many vulnerabilities were added as a result—the process was a black box.
  • The new form mandates critical details and evidence, creating an opportunity for transparency into what happens after submission. Dani said, “Hopefully, this functionality will now provide visibility into what exactly happens post submission.”
  • He stressed that CISA must establish robust verification procedures and guardrails against incorrect or false reporting so that only validated exploitation observations make it onto the KEV list.
  • Dani also highlighted that commercial alternatives to the KEV exist, and some now consider the official catalog a trailing indicator of vulnerability exploitation. This implies that CISA’s initiative may be, in part, an effort to catch up with commercial threat intelligence offerings and restore the KEV’s leading role.

Evolution of Remediation Deadlines

  • When the KEV launched, nearly all entries were assigned a three-week remediation deadline for federal agencies.
  • The article reports that in the past year, a growing number of vulnerabilities have been given three-day or even 24-hour patch deadlines, signaling the shrinking window between disclosure and active exploitation.
  • A Reuters report (cited in the article) revealed that CISA Acting Director Nick Anderson and U.S. National Cyber Director Sean Cairncross are considering cutting the KEV deadline for all new bugs to just three days. Their concern is that hackers are using powerful, emerging AI systems to develop exploits much faster than before.
  • This consideration directly links the new nomination effort to a broader policy shift: if the official KEV is to reflect near-real-time threats, it must be populated more rapidly, which is exactly what external nominations aim to facilitate.

Expert Reactions and Expected Benefits

  • JupiterOne’s Chris Doyle said improvements like the new nomination form “can help strengthen the signal quality and timeliness of KEV, which ultimately benefits defenders trying to prioritize real-world risk over theoretical severity.”
  • The overarching expectation is that the initiative will speed up defense efforts, vulnerability disclosure processes, and exploitation tracking across the entire cybersecurity ecosystem.
  • By combining internal validation with external intelligence, CISA hopes to make the KEV a more dynamic and current resource that keeps pace with attacker innovation.

Conclusion

CISA’s creation of a formal nomination pathway for the KEV catalog is a direct response to the dual pressures of AI-accelerated exploitation and the growing availability of commercial alternatives that threaten to make the official list less relevant. The new form demands evidence of exploitation, promises better transparency into the inclusion process, and is designed to shorten the time between vulnerability discovery, KEV listing, and mandated patching. If successful, it will close the feedback loop between the global research community and federal defense mandates, reinforcing the KEV’s role as the authoritative source for prioritization at a moment when every hour counts.

  • The new nomination form may be a reactive move, as commercial alternatives already offer exploitation intelligence and some view the KEV as a trailing indicator, suggesting CISA is trying to catch up.
  • The consideration of cutting the KEV remediation deadline from three weeks to just three days reveals that AI-powered exploit development is compressing the window between vulnerability disclosure and exploitation to a point where near-immediate patching is becoming necessary.
  • The structured form replaces an opaque email submission process, introducing mandatory detailed evidence and the potential for much greater transparency into CISA’s validation workflow, which was previously a black box for outside researchers.
