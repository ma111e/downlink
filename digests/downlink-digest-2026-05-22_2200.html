Each commit added src/helpers.php to the Composer autoload.files map, causing the payload to execute immediately whenever any dependent application booted, without requiring any class instantiation or function call. Only pre-attack commit hashes verified against a trusted source remain safe, and any project that ran a Composer install or update after the attack window must rotate all secrets that were accessible to the affected environment.

A single threat actor executed a broad supply chain attack against the Laravel-Lang package ecosystem on May 22, 2026, rewriting every git tag in four repositories to point at malicious commits. The attacker did not publish a new version; instead, they force-pushed new commits under the tags, so any project using version constraints (e.g., ^3.4, ~2.0) and running composer update would resolve to a poisoned release. The malicious commits all used the fake author identity “Your Name” with email you@example.com and modified only composer.json and src/helpers.php, adding the latter to Composer’s eager autoload.files map.

The payload in src/helpers.php fetched a second stage from the typosquat domain flipboxstudio.info, saved it under a hidden filename in the system cache directory, and executed it in a backgrounded process. The stage then dropped a hidden PHP loader and an ELF binary into /tmp, exfiltrated sensitive environment data (likely including GITHUB_TOKEN and other secrets), and removed the dropped artifacts within seconds to avoid detection. The orphaned processes, now reparented to init, continued running, making the network calls to the C2 the most reliable detection indicator.

Analysis using Harden-Runner in audit mode confirmed end-to-end exploitation: an autoload step triggered a GET request to download the payload, followed by a POST to exfiltrate data, all while the implant processes survived beyond the workflow step. The four affected packages — laravel-lang/lang (the core translations package with 502 tags), laravel-lang/http-statuses, laravel-lang/actions, and laravel-lang/attributes — share the same malicious commit structure and timeline, indicating a single compromised credential with organization-wide push access.

The impact is severe because no tagged version can be trusted; every tag from the earliest to the latest was moved. Recovery requires inspecting composer.lock for the listed imposter commit SHAs, rotating all secrets exposed to any runner or developer machine where the package was installed after the attack window, and auditing for orphaned processes or network calls to flipboxstudio.info. The author reported the compromise to Laravel-Lang maintainers and recommended revoking all credentials, notifying Packagist, and restoring tags from reputable clones or the dist mirror.

Overview

On May 22, 2026, a single threat actor compromised four popular Composer packages maintained by the Laravel-Lang organization. Rather than introducing a new malicious version, the attacker rewrote every existing git tag in each repository to point at a new malicious commit. The affected packages are laravel-lang/lang (the flagship Laravel translations package with 502 tags), laravel-lang/http-statuses, laravel-lang/actions, and laravel-lang/attributes. All tag rewrites occurred between 22:32 UTC on May 22 and 00:00 UTC on May 23, and each malicious commit is signed with the fake author name “Your Name” and email you@example.com, strongly indicating a single compromised credential with organization-wide push access.

Attack Mechanism

The attacker manipulated Composer’s autoloading behavior. Each malicious commit added src/helpers.php to the autoload.files map in composer.json, a mode that eagerly requires the listed file the moment vendor/autoload.php is loaded. Since every Laravel and Symfony application (and most PHP frameworks) include this autoloader at startup, the payload executes immediately without any explicit invocation.

The contents of src/helpers.php act as a dropper: they construct a URL (https://flipboxstudio.info/payload), fetch a second stage, write it to a hidden temporary file with a random hex name, and execute it in a backgrounded PHP process using @exec("php \"$f\" > /dev/null 2>&1 &"). The second stage then drops a hidden PHP loader and an ELF binary into /tmp, both under random hexadecimal names, and causes them to exfiltrate environment data via a POST to https://flipboxstudio.info/exfil before deleting the dropper files from disk. The entire compromise, from initial autoload to self-deletion, completes in roughly 3 seconds.

Technical Evidence and Detection

The article details a live detonation of laravel-lang/http-statuses v3.4.5 inside an isolated GitHub Actions runner monitored by Harden-Runner. Key observations from that run include:

  • A PHP process (the autoload step) spawning a background sh wrapper that launched the PHP loader and then exited, causing the loader and the subsequent ELF binary to be reparented to init (PID 1).
  • Two network calls: a GET to /payload (fetching the second stage) and a POST to /exfil (exfiltrating data).
  • The dropper files (/tmp/.laravel_locale/<random>.php and /tmp/.<random> ELF) were deleted within 2 seconds of creation, leaving the orphaned processes as the only runtime evidence.

Indicators of compromise identified in the article:

  • Network: DNS and HTTPS traffic to flipboxstudio.info (a typosquat of flipboxstudio.com).
  • Filesystem: /tmp/.laravel_locale/<12 hex chars>.php and /tmp/.<8 hex chars> (ELF binary), though these self-delete quickly.
  • Process: Orphaned php and unnamed ELF processes with parent PID 1.
  • Git: Commits with author “Your Name” / you@example.com, timestamps within the attack window, and diffs touching only composer.json and src/helpers.php.

Because the tags themselves were moved, every version — including historical releases — is poisoned. Only projects whose composer.lock pins a pre-attack commit SHA (and that SHA is not one of the listed imposter commits) are safe as long as they only run composer install with that lockfile.

Impact and Affected Versions

All tags across the four repositories were rewritten. The article lists specific compromised commit SHAs for example versions:

  • laravel-lang/lang: 15.29.5, 15.29.1, 2.0.4, 1.0.2, and every other of its 502 tags.
  • laravel-lang/http-statuses: v3.4.5, v3.4.0, v3.0.0, v2.0.0, v1.0.0, and all intermediate tags.
  • laravel-lang/actions: 1.12.2, 1.10.0, 1.0.0, and all 46 tags.
  • laravel-lang/attributes: v2.4.1, 2.6.0, 2.5.0, and all 86 tags.

Any project that ran composer update or composer install with these packages on or after 2026-05-22 22:32 UTC may be compromised. The payload exfiltrates environment variables, meaning CI tokens (GITHUB_TOKEN, cloud credentials, deploy keys, database credentials) must be rotated immediately.

Recovery and Remediation

The article prescribes the following recovery steps:

  • Halt all composer update and composer install operations for affected packages.
  • Inspect composer.lock files; if any locked commit matches an imposter SHA, treat the project as compromised.
  • Rotate all secrets accessible to any environment that installed the package after the attack window.
  • Check for orphaned php and unnamed ELF processes with ppid=1, and look for lingering artifacts in /tmp.
  • If using Harden-Runner, search for detections of flipboxstudio.info across the tenant to identify compromised runs.
  • Block flipboxstudio.info at the egress, DNS, or firewall level.

For maintainers, the recommendations include:

  • Force-update all tags to their original commits using the Packagist dist mirror or pre-attack clones.
  • Audit and rotate every credential with push access to the Laravel-Lang organization; re-enforce 2FA.
  • Notify Packagist and audit the rest of the organization for similar rewrites.

This incident demonstrates how tag rewriting can bypass version-pinning defenses and how a small change to Composer’s autoload configuration can achieve immediate and stealthy code execution. The self-deleting, reparented payload leaves minimal forensic evidence, making network egress monitoring the most reliable detection method. The uniform attack pattern across four repositories underscores the danger of broad-scope credentials in open-source organizations.

  • The payload fetches a second stage from flipboxstudio.info, drops hidden PHP and ELF artifacts in /tmp, exfiltrates CI runner environment data, and then deletes its own files from disk.
  • Projects that ran composer update or composer install after 2026-05-22 22:32 UTC may be compromised, and all secrets accessible to those environments should be rotated immediately.

The npm ecosystem is facing a severe wave of software supply chain attacks, with the TeamPCP group systematically poisoning popular packages in a self-perpetuating cycle. Against this backdrop, GitHub has delivered two significant security enhancements aimed at cutting off common attack paths that rely on automated or covert package distribution and installation.

The centerpiece is staged publishing, now generally available on npm. Instead of an immediate publish that exposes a new package version to consumers, the process uploads a prebuilt tarball to a staging queue where a human maintainer must complete a two-factor authentication challenge to approve the release. This introduces a mandatory proof of presence for every publish, including those originating from non-interactive CI/CD workflows and OpenID Connect trusted publishing, effectively breaking the automated token abuse observed in recent compromises.

To combat installation from untrusted sources, npm also introduced three explicit-allowlist flags alongside the existing —allow-git flag. The —allow-file flag controls installs from local file paths and local tarballs, —allow-remote governs installs from remote URLs including HTTPS tarballs, and —allow-directory controls installs from local directories. Together, they allow developers to shut down non-registry installation vectors that have been used to smuggle malicious code.

Usage of staged publishing requires maintainers to have publish access, the target package must already exist on the npm registry (meaning brand new packages cannot be staged), and two-factor authentication must be enabled on the account. Developers need npm CLI version 11.15.0 or newer and invoke the command ‘npm stage publish’ from the root directory of the package. GitHub recommends pairing staged publishing with trusted publishing via OIDC for optimal protection.

The twin measures represent a direct response to a threat landscape where massive-scale package poisoning attacks have surged. By gating the final publish step behind a human-verified authentication step and restricting install sources to an explicit allowlist, npm aims to raise the bar for attackers and restore trust in the software supply chain.

Staged Publishing Feature

How It Works

  • Instead of a direct publish that immediately makes a new package version available, the maintainer now uploads a prebuilt tarball to a staging queue.
  • A human maintainer must explicitly approve the release by passing a two-factor authentication (2FA) challenge before the package becomes installable from the npm registry.
  • This creates a mandatory “proof of presence” for every publish, ensuring that even automated builds from non-interactive CI/CD workflows or those using trusted publishing with OpenID Connect (OIDC) cannot bypass final human verification.

Requirements for Maintainers

  • The maintainer must have publish access to the package.
  • The package must already exist on the npm registry; brand new packages cannot use staged publishing at this time.
  • Two-factor authentication must be enabled on the maintainer’s account.
  • The maintainer must use npm CLI version 11.15.0 or newer and run ‘npm stage publish’ from the package’s root directory to submit a release to the staging area.

Best Practice Recommendation

GitHub advises pairing staged publishing with trusted publishing via OIDC for optimal security, combining automated integrity of the build pipeline with the human approval gate.

Install Source Controls

New Explicit-Allowlist Flags

Alongside the existing —allow-git flag, npm has introduced three new install source flags that let developers apply a white-list approach to all non-registry install sources:

  • –allow-file: Controls installations from local file paths and local tarballs.
  • –allow-remote: Controls installations from remote URLs, including HTTPS tarballs.
  • –allow-directory: Controls installations from local directories.

These flags shift the default behavior to a deny-by-default posture for any installation source other than the main registry, closing off covert vectors often abused by attackers to inject malicious code during package installs.

Impact and Significance

The staged publishing feature ensures that even if attackers compromise an automated CI/CD pipeline or abuse OIDC-based trusted publishing tokens, they still cannot push a tarball to public consumption without a maintainer’s explicit 2FA-approved step. This breaks a key link in recent supply chain attack chains. The install source controls add a parallel layer of defense by limiting what can be installed outside the vetted registry, reducing the attack surface for sideloading malware. Together, these changes represent a hardening strategy against the type of large-scale package poisoning campaigns currently targeting the npm ecosystem.

Security teams are confronting a wave of actively exploited vulnerabilities and sophisticated supply chain compromises across multiple ecosystems. Two high-severity flaws — CVE-2026-48172 in the LiteSpeed cPanel plugin and CVE-2026-9082 in Drupal Core — are under active attack, while coordinated supply chain intrusions have hit Packagist and Laravel-Lang packages, exfiltrating secrets and deploying hidden malware. 01 Supply Chain Attacks and Registry Security Measures

Amid a surge in software supply chain attacks, GitHub introduced staged publishing and install source controls for npm. Staged publishing now requires a maintainer to approve a package release using a two-factor authentication challenge before the package becomes publicly installable, adding proof of presence even for CI/CD and trusted publishing workflows. New install flags (--allow-file, --allow-remote, --allow-directory) give developers explicit control over non-registry install sources. The updates follow widespread poisoning campaigns by groups such as TeamPCP.

A supply chain attack targeting the PHP ecosystem infected eight Packagist packages by inserting malicious code into package.json lifecycle hooks. The injected postinstall script downloads a Linux binary from a GitHub Releases URL, saves it as /tmp/.sshd, makes it executable, and runs it in the background. The binary’s filename (gvfsd-network) mimics the legitimate GNOME Virtual File System daemon, and the GitHub account hosting the second-stage payload is no longer available. Even without the binary, the installer enables remote code execution and attempts to evade detection by disabling TLS verification and suppressing errors.

In a more invasive attack, a threat actor rewrote every existing git tag across four Laravel-Lang Composer packages to point at new malicious commits. Each added src/helpers.php to the Composer autoload.files map, causing automatic execution as soon as any dependent application loads its autoloader. The payload fetches a second stage from flipboxstudio.info, drops hidden PHP and ELF artifacts in /tmp, exfiltrates CI runner environment data, and then deletes itself. Projects that ran composer update or composer install after 2026-05-22 22:32 UTC may be compromised; all accessible secrets should be rotated. The attacker used the fake identity “Your Name” you@example.com consistently across all four repositories.

02 Actively Exploited Vulnerabilities

CVE-2026-48172, a maximum-severity flaw in the LiteSpeed User-End cPanel Plugin (versions 2.3–2.4.4), allows any cPanel user to execute arbitrary scripts with root privileges. The vulnerability is being actively exploited in the wild, and LiteSpeed released a patch in version 2.4.5 along with a specific grep-based indicator of compromise to help defenders identify potential exploitation. Additional attack vectors were later addressed in cPanel plugin version 2.4.7, bundled with WHM plugin version 5.3.1.0.

Drupal Core’s SQL injection vulnerability CVE-2026-9082 was added to CISA’s Known Exploited Vulnerabilities catalog because of active exploitation. The flaw enables privilege escalation and remote code execution through specially crafted database abstraction API requests. Imperva observed over 15,000 attack attempts targeting nearly 6,000 sites across 65 countries, with most probes focused on reconnaissance and validation. Drupal has released patches for supported versions, while Drupal 9.5 and 8.9 require manual patching. Federal civilian executive branch agencies are recommended to apply fixes by May 27, 2026.

03 AI-Assisted Vulnerability Discovery and CISA KEV Evolution

Anthropic’s Project Glasswing, using the Claude Mythos Preview model, uncovered more than 10,000 high- or critical-severity vulnerabilities across systemically important open-source software. From over 6,200 flaws impacting more than 1,000 projects, 1,726 were validated as true positives and 1,094 assessed as high- or critical-severity. One critical finding was a flaw in WolfSSL (CVE-2026-5194, CVSS 9.1) that could allow attackers to forge certificates and impersonate legitimate services. The effort has led to 97 upstream patches and 88 security advisories, prompting Anthropic to urge developers to shorten patch testing and deployment cycles in response to the accelerating pace of AI-assisted discovery.

CISA launched a new nomination form enabling outside researchers, vendors, and partners to report actively exploited vulnerabilities directly to the KEV catalog. The catalog has become a critical resource that drives organizations to remediate listed vulnerabilities 3.5 times faster. CISA officials are also considering shortening the remediation deadline for all new KEV entries to just three days, driven by concerns over AI-powered exploit development that could drastically compress the window for patching.

04 Piracy Operation Dismantled

Italian authorities dismantled the CINEMAGOAL piracy app, which provided unauthorized access to streaming platforms including Netflix, Disney+, and Spotify. The operation used virtual machines in Italy to harvest valid decryption keys from subscriptions opened under false identities every three minutes, masking end users’ IP addresses. Over 70 resellers sold annual subscriptions for €40–€130, causing an estimated €300 million in damages. Law enforcement conducted 100 searches nationwide and seized servers in France and Germany containing the app’s source code and decoding functions.
