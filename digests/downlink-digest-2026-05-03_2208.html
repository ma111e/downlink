Incident Overview

Microsoft Defender recently generated widespread false positive alerts by incorrectly flagging legitimate DigiCert root certificates as Trojan:Win32/Cerdigent.A!dha. This detection error originated from a Defender signature update released on April 30th. The false positives caused the antivirus software to automatically remove these certificates from the Windows AuthRoot trust store. Affected systems experienced disruptions, leading some users to mistakenly believe their devices were infected and prompting others to reinstall Windows.

Affected Certificates and Remediation

Cybersecurity experts identified the specific certificate hashes triggering the false positives. The flagged certificates were automatically quarantined under the HKLM\SOFTWARE\Microsoft\SystemCertificates\AuthRoot\Certificates\ registry path. Microsoft resolved the issue by releasing Security Intelligence update version 1.449.430.0, followed by version 1.449.431.0. These updates automatically install and restore the removed certificates to the trust store. Administrators can also manually trigger the update through Windows Security by navigating to Virus and threat protection, selecting Protection updates, and clicking Check for Updates.

DigiCert Security Incident Context

The Defender false positives occurred shortly after DigiCert disclosed a security breach involving compromised support staff. Attackers targeted a customer support team member by sending a malicious ZIP file disguised as a screenshot. After bypassing initial defenses, the threat actor accessed a second system and exploited a feature in DigiCert’s internal support portal. This access allowed the attacker to view customer accounts and obtain initialization codes for approved but undelivered EV code-signing certificate orders. DigiCert revoked 60 code-signing certificates as a precautionary measure, including certificates linked to the Zhong Stealer malware campaign.

Threat Actor Attribution and Malware Distribution

Security researchers, including independent analysts and threat intelligence teams, reported that the compromised certificates were used to sign malware. The certificates were attributed to the #GoldenEyeDog threat group, also known as APT-Q-27. The malware, named Zhong Stealer, functions similarly to a remote access trojan rather than a traditional infostealer. Distribution methods involved phishing emails delivering decoy images, first-stage executables displaying fake screenshots, and second-stage payloads hosted on cloud storage services like AWS. The campaign utilized signed binaries and loaders tied to legitimate vendors to maintain persistence and evade detection.

  • Phishing emails deliver a fake image or screenshot to the victim.
  • A first-stage executable runs and displays a decoy image to the user.
  • Retrieval of a second-stage payload occurs from cloud storage such as AWS.
  • Use of signed binaries and loaders includes components tied to legitimate vendors.

Microsoft’s Stance and Technical Distinction

Microsoft has not confirmed a direct connection between the Defender detections and the DigiCert breach. The timing and focus on DigiCert-related certificates suggest a possible correlation, but the technical details differ significantly. The certificates flagged by Microsoft Defender are root certificates residing in the Windows trust store. In contrast, the revoked DigiCert certificates were EV code-signing certificates used to sign malicious payloads. This distinction indicates that the Defender rule likely triggered on unrelated certificate hashes rather than the specific compromised assets.

EXEC Executive Overview

Microsoft Defender Antivirus experienced a significant false positive event involving the incorrect identification and removal of legitimate DigiCert root certificates from Windows trust stores. This widespread disruption was triggered by a signature update deployed on April 30th, which erroneously classified these certificates as Trojan:Win32/Cerdigent.A!dha. The false positives resulted in the automatic removal of valid cryptographic assets, potentially impacting system integrity verification processes relying on these roots. The issue was resolved following the release of Security Intelligence update version 1.449.430.0, which automatically restored the removed certificates to the Windows trust store, restoring the expected trust chain state for affected endpoints. The April 30th signature update introduced patterns that matched the hash or structural properties of the DigiCert root certificates, leading to the classification. The resolution update reversed these changes, ensuring that trust store integrity was preserved without requiring manual intervention from system administrators or security operations teams.

The false positive event occurred in close temporal proximity to a public disclosure regarding a security breach at DigiCert. According to security research, threat actors successfully compromised initialization codes associated with code-signing certificates utilized within the Zhong Stealer malware campaign. Attribution analysis has linked this compromise to the #GoldenEyeDog threat group. Researchers note that the breached certificates were actively used to sign malware payloads distributed through phishing campaigns, demonstrating the exploitation of trusted signing infrastructure to bypass detection mechanisms and facilitate malware deployment. The Zhong Stealer campaign, associated with the #GoldenEyeDog threat group, represents a sustained effort to deploy credential-harvesting tools. The compromise of initialization codes allowed actors to generate valid signatures, complicating the distinction between legitimate and malicious code-signing artifacts.

Despite the overlapping timeline, Microsoft has not confirmed a direct causal link between the Defender detections and the DigiCert incident. Technical analysis reveals that the certificates flagged by Defender are root certificates, whereas the revoked code-signing certificates associated with the Zhong Stealer campaign represent a different class of cryptographic assets. Security researchers have identified connections between the compromised certificates and the #GoldenEyeDog threat group, emphasizing the continued sophistication of threat actors in leveraging compromised code-signing capabilities. The incident highlights the sensitivity of signature updates to cryptographic metadata and the potential for collateral false positives when threat intelligence regarding compromised certificates intersects with routine detection engine updates. This overlap underscores the operational challenges posed by the intersection of high-value certificate compromises and automated security product behaviors.
