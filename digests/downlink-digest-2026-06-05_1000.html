Overview

Cyera Research reported six vulnerabilities in protobuf.js, the JavaScript implementation of Protocol Buffers. The article says protobuf.js has over 48 million weekly npm downloads and is widely used to load schemas, build reflection objects, and generate specialized JavaScript encoder and decoder functions. The central finding is that protobuf.js treated schema-related metadata as trusted data even when that metadata could be influenced by attackers and later converted into executable code or unsafe runtime behavior.

The vulnerabilities can lead to remote code execution or denial of service depending on the input surface and application behavior. The article emphasizes that not every protobuf.js deployment is equally exposed: trusted schemas compiled at build time and flat, authenticated message flows are much lower risk than systems dynamically loading schemas, decoding untrusted recursive protobuf messages, or running pbjs static generation on contributed schemas.

CVEs and fixed versions

The article lists six assigned CVEs:

  • CVE-2026-44289: denial of service through unbounded protobuf recursion.
  • CVE-2026-44290: process-wide denial of service when loading schemas with unsafe option paths.
  • CVE-2026-44291: code generation gadget after prototype pollution.
  • CVE-2026-44292: prototype injection in generated message constructors.
  • CVE-2026-44294: denial of service from crafted field names in generated code.
  • CVE-2026-44295: code injection in pbjs static output from crafted schema names.

The article states that all issues were fixed in protobuf.js 7.5.6 and 8.0.2, and in protobufjs-cli 1.2.1 and 2.0.2. It also says the mitigation work is tracked in PR #2163.

Core technical pattern

protobuf.js reads .proto files or JSON descriptors, converts them into JavaScript objects representing message types and fields, and then generates specialized JavaScript functions for encoding and decoding. This design is used for performance because it avoids interpreting the schema repeatedly for every message. The risk described by Cyera is that schema values can cross a boundary from metadata into code.

The article repeatedly identifies the same class of mistake in different forms:

  • Type names reached generated JavaScript functions.
  • Namespace names reached static JavaScript files emitted by pbjs.
  • Option paths walked through internal object structures and overwrote built-in functions.
  • Field names broke generated code by injecting raw control characters.
  • JSON object keys bypassed prototype safety checks in generated constructors.
  • Nested binary messages triggered unbounded recursion and crashed the process.

Runtime RCE chain after prototype pollution

The runtime RCE chain requires a separate prototype pollution primitive in the Node.js application or one of its dependencies. The article explicitly states that protobuf.js did not provide the standalone prototype pollution bug in this chain; instead, it provided the code generation gadget that could be reached after pollution had already occurred.

protobuf.js built primitive type lookup tables using plain objects. Because those objects inherited from Object.prototype, a polluted Object.prototype entry could be returned during a lookup for a field type. Cyera gives the example of adding a property to Object.prototype with a crafted type-like string, causing types.basic[payload] to return a value instead of undefined.

Once the polluted lookup made the attacker-controlled string appear to be a valid primitive type, protobuf.js used that same string in generated encoder or decoder code. The value was inserted through a %s placeholder without validation or escaping and then compiled with Function(). At that point, the article says the type name stopped being a type name and ran as JavaScript.

The same sink could also be reached through descriptors. The article says ext/descriptor copied descriptor.typeName into field.type, so an attacker-controlled FileDescriptorProto could carry the payload in typeName. With the same prototype pollution prerequisite, that descriptor string could reach the same type lookup and code-generation sink.

pbjs static code generation RCE path

The article describes a separate RCE path in pbjs, the protobuf.js CLI tool that generates static JavaScript files from schemas. Unlike the runtime chain, this issue does not require prototype pollution. The attacker-controlled input is the schema, and the output is a JavaScript file that executes when imported.

The vulnerable logic was in cli/targets/static.js. The escapeName() helper only handled JavaScript reserved words and did not actually escape schema names containing newlines, semicolons, quotes, or other syntax-significant characters. Names for namespaces, enums, services, and methods could pass through to emitted JavaScript through direct string concatenation.

The article describes a CI/CD supply-chain scenario in which an attacker submits a pull request containing a crafted .proto file. When CI runs pbjs to generate JavaScript and the generated file is imported by a build, test suite, or downstream consumer, the injected code runs. The article says this could expose build secrets, signing credentials, cloud tokens, package registry credentials, and internal repositories.

Other denial-of-service and prototype issues

CVE-2026-44294 involves field names with raw control characters. util.safeProp() escaped backslashes and double quotes but missed characters such as newlines, carriage returns, and null bytes. A crafted field name could make generated JavaScript syntactically invalid, causing Function() to throw a SyntaxError whenever encode, decode, or verify was called for that message type.

CVE-2026-44290 involves unsafe option paths. util.setProperty() blocked proto and prototype path segments but did not block constructor. The article says an option path such as constructor.keys could walk from an options object to the global Object function and set Object.keys to a non-function value, causing Object.keys() calls to throw process-wide.

CVE-2026-44292 involves generated message constructors. The base Message constructor already skipped proto when copying properties from input objects, but generated constructors did not. If input came from JSON.parse(), proto could appear as an own enumerable key and be assigned to this[”proto”], changing the prototype of that specific message instance.

CVE-2026-44289 involves unbounded recursion in generated decoders for nested message fields. Self-referencing protobuf message types are valid, but the generated decoder called nested decode routines without a depth parameter, counter, or maximum. A deeply nested binary payload could recurse until V8 threw RangeError: Maximum call stack size exceeded.

Real-world impact demonstrated by Cyera

Cyera demonstrated the recursion denial-of-service against Baileys, described in the article as the most popular open-source WhatsApp bot library with over 2 million weekly npm downloads. Baileys uses protobuf.js to decode binary messages from WhatsApp servers, and the WhatsApp protocol schema includes naturally recursive types. Cyera crafted a roughly 34KB binary payload with about 10,000 nesting levels and sent it as a WhatsApp message to a bot.

When Baileys called Type.decode() on the incoming binary, protobuf.js recursed through the nested structure until V8 ran out of stack space and the process crashed. The article says the persistence makes the impact worse because the malicious message remains in chat history. When the bot restarts and catches up on missed messages, it decodes the same payload and crashes again, causing a restart loop until the message is manually cleared.

Cyera also confirmed CVE-2026-44292 against Baileys. By injecting proto keys carrying properties such as isAdmin: true and canDelete: true into a message object, the generated constructor copied those keys without filtering. The result was a single message object with properties its schema never defined.

The article also says Cyera reproduced the recursion crash against a Google Cloud Functions pattern recommended in Google’s Firestore trigger documentation. The same 34KB payload crashed a Cloud Function decoding Pub/Sub events. Because the event is redelivered until acknowledged, the function crashes again on each retry until the message reaches the dead-letter queue.

Exposure conditions

The article identifies several exposure scenarios:

  • Applications loading .proto files or JSON descriptors from users, plugins, tenants, or shared repositories are exposed to schema-driven issues such as crafted field names, unsafe option paths, and static generation injection.
  • Applications decoding protobuf binary data from untrusted senders are exposed to the stack overflow denial-of-service issue when recursive message types are present.
  • Node.js processes with a separate prototype pollution bug can expose the protobuf.js code-generation gadget that turns polluted type lookup into JavaScript execution.
  • Projects running pbjs in CI on contributed schemas can expose build pipelines to static code-generation injection.

The article does not attribute exploitation to any named threat actor. Its focus is vulnerability research, proof-of-concept exploit chains, and affected usage patterns.

Response and mitigations

The article says the maintainers handled the reports promptly through GitHub Security Advisories. Reports were opened in April 2026, maintainers accepted them and linked mitigation work to PR #2163, fixed protobuf.js versions 7.5.6 and 8.0.2 were released, fixed protobufjs-cli versions 1.2.1 and 2.0.2 were released, and GitHub assigned the CVEs in May 2026.

The fixes described in the article include:

  • Building type lookup tables with Object.create(null) to eliminate inherited property access.
  • Validating typeName before interpolation.
  • Sanitizing all reflection names in static code generation, including fields, namespaces, and services.
  • Replacing unsafe string concatenation with safe templating.
  • Blocking constructor alongside proto and prototype in option-path traversal.
  • Making generated constructors skip proto keys during property copying.
  • Enforcing a configurable maximum nesting depth during recursive decoding.

Main takeaway

The article’s central conclusion is that protobuf.js’s trust boundary failed where metadata became structure and structure became code. Type names, namespace names, option paths, field names, object keys, and nested binary messages were all treated as passive metadata, but in affected versions they could influence generated JavaScript, process-wide built-ins, object prototypes, or recursive control flow. Cyera’s recommended response is to upgrade to the fixed protobuf.js and protobufjs-cli versions, especially for applications that process untrusted schemas, descriptors, or protobuf messages.

  • The most severe runtime code execution path is not a standalone protobuf.js prototype pollution bug; protobuf.js acts as a code-generation gadget that can turn prototype pollution elsewhere in the same Node.js process into arbitrary JavaScript execution.
  • The pbjs static generation issue creates a supply-chain risk because malicious schema names in pull requests or shared schema sources can become executable JavaScript during CI builds or tests when the generated file is imported.
  • The recursion denial-of-service can become persistent in message-driven systems because malicious payloads may remain in WhatsApp chat history or be redelivered by Pub/Sub until acknowledged or moved to a dead-letter queue.
  • A schema option path such as constructor.keys could corrupt Object.keys process-wide, showing that schema metadata could damage unrelated application code outside protobuf.js itself.
  • Applications using only trusted schemas compiled at build time and flat, non-recursive messages from authenticated sources had much lower exposure than applications dynamically loading schemas or decoding untrusted recursive protobuf data.

The article says SOCs reporting value use AI systems that capture and persist institutional knowledge. Examples include the assets that matter, the analysts whose judgment shaped past incidents, sanctioned actions, escalation criteria, tickets that turned out to be nothing, and tickets that turned out to be significant. Without that grounding, the article says AI produces “the average of the internet,” which it describes as the wrong answer in most environments.

Governance Requirements

The article identifies governance as a major requirement for effective AI in the SOC. It notes that the SOC-CMM 2026 report identifies effective SOC governance as the single most challenging area of SOC improvement, with 39% of respondents naming it. The article says AI governance and SOC governance overlap.

According to the article, an agentic SOC must operate inside customer-defined guardrails. It must expose a defensible reasoning trace for every action. It must also earn autonomy in stages instead of asking for it upfront.

The article says AI in the SOC cannot be a black box. It argues that analysts must trust the system enough to give it standing authority, and that this trust is what produces productivity gains. Without that trust, the system stalls.

The Architecture Problem

The article describes the main problem as point AI operating inside fragmented architecture. If detection engineering and investigation teams work in different tools, AI in either tool may accelerate that team’s work but not improve the handoff between them. If threat hunters cannot test hypotheses across the same telemetry that investigations use, AI in either workflow only advances that workflow.

The article gives a similar example for remediation. If remediation playbooks live in a SOAR tool that does not see what an investigation agent concluded, AI remediation may execute against stale context. The article says the fix is to connect the stages rather than add more AI into the same fragmented structure.

The article defines “second wave” AI as this connective fabric. The first wave delivered AI per stage, while the second wave must deliver AI across stages.

What the Second Wave Must Deliver

The article says the five stages of the SOC must operate as one agentic fabric grounded in the customer’s environment. It repeats that every closed investigation should calibrate the next detection, every threat hunt result should update the next intelligence cycle, and every remediation should feed back into the next playbook. The intended result is a compounding SOC workflow.

In practice, the article says a platform built this way sits on top of the SIEM, EDR, identity, cloud, ticketing, and threat intelligence stack an organization already owns. It does not require replacing the existing stack. Instead, the connective layer allows each stage to feed the next.

The article says that when this architecture is in place, SOCs report sharper investigations completed faster, detections that are surfaced and tuned instead of left silent or noisy, threat hunts that run continuously rather than episodically, and remediation that operates inside defined guardrails with full reasoning traces and audit-grade decision records.

01 Enterprise AI Agents, SOC Adoption, and Healthcare Risk

Cyera reported that its analysis of 7,246 publicly reported AI incident records from September 2023 through May 2026 identified 344 verified enterprise-relevant cases of agent-inflicted damage. Of those, 188 involved autonomous AI systems causing direct organizational harm without an external attacker, with incidents grouped into poor access control and privilege escalation, data and secrets exposure, and real-world damage such as financial loss, code destruction, service disruption, and silent integrity failure.

The research said incident volume rose sharply beginning in December 2025, coinciding with broader enterprise adoption of AI coding agents and autonomous development tools including Claude Code, Cursor agent mode, Devin, and OpenClaw. Health-ISAC separately warned that agentic AI in healthcare can intensify risks around over-permissioned accounts, weak governance, and credential misuse as AI-enabled workflows take on delegated operational decisions.

In security operations, the SOC-CMM 2026 Maturity Report found that only about 10% of surveyed SOCs reported excellent value from AI, while roughly 71% reported only some value or no value. The reporting attributed weaker outcomes to first-wave AI features embedded in isolated SIEM, EDR, SOAR, and ticketing workflows, while describing stronger-performing SOCs as those applying AI across threat intelligence, hunting, detection engineering, investigation, and remediation with shared context and governance.

02 OP-512 IIS Espionage Activity and protobuf.js Proto6 Vulnerabilities

ReliaQuest reported a previously unreported threat cluster tracked as OP-512 targeting Microsoft IIS servers with a custom web shell framework. The activity was assessed with moderate to high confidence as China-linked and espionage-focused, and the observed target was a legacy IIS server running Windows Server 2016 with end-of-life .NET Framework 4.0.

The OP-512 framework reportedly consists of three web shells enabling remote access, file management, authenticated command execution through two access paths, and automated compromise reporting. The web shells used timestomping by calculating the median last-modified timestamp of nearby files and folders, then modifying their own timestamps to blend into the file system; the actor also attempted privilege escalation to SYSTEM using the Potato Suite.

Cyera also disclosed six vulnerabilities in protobuf.js under the name Proto6, affecting scenarios where schema-derived data is poisoned or crafted. The findings included a runtime remote code execution chain requiring a separate prototype pollution primitive, a protobufjs-cli static generation flaw that can emit crafted schema names into generated JavaScript that executes on import, and recursion-based denial-of-service cases reproduced against Baileys WhatsApp bots and a Google Cloud Functions Pub/Sub decoding pattern; fixes were released in protobuf.js 7.5.6 and 8.0.2, and protobufjs-cli 1.2.1 and 2.0.2.

03 Executive Impersonation, Typosquatting, and Authenticated Fraud Emails

Cyble reported that executive impersonation has become a highly targeted and financially damaging attack vector for organizations in the UAE, Saudi Arabia, and Qatar in 2026. Gulf executives were described as attractive targets because of their roles in large international transactions, visibility on LinkedIn, and frequent connections to government-linked, energy, or sovereign wealth entities.

The reported methods include LinkedIn impersonation, WhatsApp CEO fraud, lookalike domains, and AI-generated voice or video deepfake fraud. The article also noted that Saudi Arabia’s SAMA Cybersecurity Framework requires financial-sector organizations to address social engineering and targeted attacks against key personnel, including monitoring unauthorized executive identity use and maintaining response procedures.

IRONSCALES described a funding-fraud email sent from a typosquat domain with one extra letter compared with a legitimate commercial lender. The domain was configured as a Salesforce Marketing Cloud sending identity, allowing the message to pass SPF, DKIM, DMARC, and Microsoft CompAuth checks, while Pardot tracking links redirected to the legitimate lender’s site and the reply-to address pointed back to the typosquat domain; Microsoft still scored it as high-confidence spam with SCL=9 and SFTY:9.25.

04 EU Technology Sovereignty and Open-Source Infrastructure

The European Commission proposed a technology sovereignty package covering Chips Act 2.0, the Cloud and AI Development Act, an Open Source Strategy, and a roadmap for digitalizing the energy system. The Commission said the EU depends on foreign countries for more than 80% of key digital products, services, infrastructure, and intellectual property.

The Open Source Strategy would fund long-term maintenance and security for Europe’s critical open-source infrastructure and steer public administrations toward open-source tools through procurement guidance. The Cloud and AI Development Act defines four assurance levels for public bodies, with higher tiers requiring EU ownership, EU control, and protection from third-country interference, while Chips Act 2.0 would accelerate approval timelines for new fabrication plants and extend state aid for first-of-a-kind facilities not already present in the EU.
