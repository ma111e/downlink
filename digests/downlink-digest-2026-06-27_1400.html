Ukraine Says Russian Intelligence Used Fake Support Texts to Steal Messaging Credentials
thehackernews.com · 27 Jun 2026 17:27 UTC openai-codex / gpt-5.5
TAGSbelaruscredential-thefteuropeoysterbluesqr-code-phishingrussiarussian-intelligence-servicessecurity-service-of-ukrainesms-phishingstar-blizzardukraineunc1151unc4221unc5792united-states

In plain words

Ukraine’s security service, working with the FBI, says Russian intelligence sent fake text messages pretending to be from messaging app support in order to trick people into giving away access to their accounts. The people affected include Ukrainian citizens as well as government officials, military personnel, politicians, and activists in Ukraine, Europe, and the United States. This matters because the stolen accounts could expose private military, political, economic, and personal information.

Summary
Ukraine’s SSU says Russian intelligence ran a long-running phishing campaign using fake messaging support texts to steal credentials from officials, military personnel, politicians, activists, and other users in Ukraine, Europe, and the U.S. Users are urged to review active sessions, enable two-factor authentication, and never share codes, passwords, recovery keys, or scan suspicious QR codes.

The Security Service of Ukraine said it worked with the U.S. Federal Bureau of Investigation to uncover a long-running campaign by Russian intelligence services targeting messaging accounts. The campaign aimed at government officials, military personnel, politicians, activists, and other users in Ukraine, Europe, and the United States. The SSU said the purpose was to gain access to sensitive military, political, and economic information, as well as victims’ personal data.

According to the SSU, the attackers sent SMS messages that pretended to come from a messaging platform’s support bot. These messages urged users to disclose their account credentials, allowing the attackers to break into messaging accounts. The SSU said the targeting included not only organizations, officials, and public figures, but also personal accounts belonging to Ukrainian nationals.

The SSU did not name a specific hacking group behind the campaign. The article notes that similar attacks against Signal and WhatsApp users have been attributed to Star Blizzard, UNC5792, and UNC4221. Recommended defenses include reviewing active messaging app sessions, logging out of unknown connections, enabling two-factor authentication, avoiding QR codes from unknown users, and not sharing confirmation codes, PINs, passwords, or recovery keys.

  • The campaign did not only target high-profile people or organizations; the SSU said personal accounts belonging to Ukrainian nationals were also included.
  • The phishing approach relied on impersonating a messaging platform’s support bot, showing that ordinary support-style messages can be used to pressure users into giving away account access.
  • The article links this activity to a broader pattern of attacks against commercial messaging apps, including prior waves aimed at Signal and WhatsApp users.
  • The FBI’s related warning highlights backup recovery keys as a specific target, meaning account recovery information can be as sensitive as passwords or confirmation codes.

Clean GitHub repo tricks AI coding agents into running malware
bleepingcomputer.com · 27 Jun 2026 14:22 UTC openai-codex / gpt-5.5
TAGSanthropicaxiomclaude-codedns-txt-recordfake-job-postingsgithubmozilla-0dinpythonreverse-shell
82
In plain words

Mozilla researchers showed that a harmless-looking GitHub project could trick an AI coding tool like Claude Code into running hidden instructions while trying to set up the project. If this worked, an attacker could get control of a shell on the developer’s computer with the same access as that developer. This matters because developers using AI tools to install and run projects could have their secret keys, settings, and local files exposed without seeing any obviously dangerous code in the project itself.

Summary
Mozilla 0DIN researchers showed that an apparently clean GitHub repository can trick an AI coding agent like Claude Code into running setup commands that fetch and execute an attacker-controlled payload from a DNS TXT record. The key risk is that agents may automatically “fix” normal-looking setup errors and execute hidden command chains that give attackers a shell with the developer’s privileges.

Mozilla’s Zero Day Investigative Network (0DIN) researchers demonstrated how an agentic coding tool could be tricked into running malware from a seemingly benign GitHub repository. The example used Claude Code to clone and set up a project that contained no malicious code in the repository itself. 0DIN said the compromise can happen with no exploit code, no warning, and no suspicious command that anyone had to approve.

The method relies on a clean-looking repository, a Python package that refuses to run until initialized, and an initialization command that calls a shell script. That script retrieves a configuration value from an attacker-controlled DNS TXT record and executes it as a command. Claude Code treats the package error as a normal setup issue and automatically runs the suggested command while trying to recover.

If successful, the attacker would obtain an interactive shell running with the developer’s privileges. The article says this could expose environment variables, API keys, local configuration files, and allow the attacker to establish persistence. 0DIN warned that threat actors could distribute such repositories through fake job postings, tutorials, blog posts, or direct messages, and suggested that AI agents should disclose the full execution chain of setup commands, including dynamically fetched scripts and code.

  • The repository itself can remain clean because the malicious command is stored outside the codebase in an attacker-controlled DNS TXT record, making it harder for scanners, AI agents, and human reviewers to detect before execution.
  • The attack abuses an AI agent’s routine error-recovery behavior rather than requiring the agent to approve an obviously suspicious command.
  • The malicious outcome is separated from the agent’s visible decision by multiple steps: a trusted error message, a setup script, and a dynamically fetched DNS value.
  • 0DIN’s recommended mitigation is for AI agents to disclose the full execution chain of setup commands, including scripts and code fetched dynamically at runtime.

  • 030DIN said Claude Code treats the package error as a normal setup issue and automatically runs the suggested initialization command while attempting to recover.
  • 04If successful, the attacker would obtain an interactive shell running with the developer’s privileges, potentially exposing environment variables, API keys, local configuration files, and enabling persistence.
  • 050DIN warned that threat actors could distribute such repositories through fake job postings, tutorials, blog posts, or direct messages.
