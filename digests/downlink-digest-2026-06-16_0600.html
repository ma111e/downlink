The vulnerability centers on how n8n parses webhook request bodies. A middleware function, parseRequestBody(), chooses between a multipart/form-data parser and a regular body parser based on the Content-Type header. Multipart requests are handled through Formidable and populate req.body.files with file upload metadata, while other content types are decoded into req.body by the regular body parser. The vulnerable path is in the Form Webhook node. The article says the Form webhook calls prepareFormReturnItem without first verifying that the Content-Type is multipart/form-data. Because prepareFormReturnItem processes files from req.body.files and calls copyBinaryFile using the filepath value stored there, an attacker who sends a non-multipart body can control req.body.files and point filepath to local files on the n8n instance. This enables arbitrary file read by causing local files to be copied into workflow data.

Core Technical Cause

The vulnerability is presented as a Content-Type confusion issue in n8n’s webhook request handling. The shared webhook execution flow calls a middleware function named parseRequestBody(), which decides how to parse the request body based on the Content-Type header.

According to the article:

  • Requests with Content-Type multipart/form-data are handled by parseFormData(), described as the file upload parser.
  • Other content types are handled by parseBody(), described as the regular body parser.
  • The regular body parser decodes the HTTP request body and stores the result in the global req.body variable.
  • The file upload parser wraps Formidable’s parse() function and populates req.body.files with uploaded file metadata.

The article explains that Formidable saves uploaded files to randomly generated paths in a temporary directory, which normally prevents users from controlling upload file paths. The security issue arises not from Formidable itself, but from n8n code that trusts req.body.files in a path where the request was not necessarily parsed as multipart/form-data.

Normal File Upload Handling

The article describes the expected file-upload pattern in n8n as fetching uploaded files from req.body.files. It uses the ChatTrigger webhook as an example of a safer pattern because it first verifies that the Content-Type header is multipart/form-data before calling a file handler.

This check matters because req.body.files is only populated by the file upload parser when multipart/form-data is used. If a request uses another content type, the regular body parser handles the body and places decoded input directly into req.body. That means an attacker can supply a body containing a files property and thereby control req.body.files.

Vulnerable Form Webhook Path

The vulnerable path identified in the article is the Form Webhook node. The Form node is described as a common external interface for workflows that need user input, including workflows where users upload CVs, screenshots, error logs, or other files.

The article says the function that handles Form submissions is formWebhook. It calls prepareFormReturnItem without verifying that the Content-Type is multipart/form-data. prepareFormReturnItem is described as a file-handling function that calls copyBinaryFile for each file in req.body.files.

The key problem is that copyBinaryFile copies a file from the filepath value stored in req.body.files[id].filepath to persistent storage, either on disk or S3 object storage depending on configuration. Because the Form webhook path does not first ensure that req.body.files came from a real multipart upload, an attacker can send a request with another content type, such as application/json, and define req.body.files directly. This gives the attacker control over filepath.

Arbitrary File Read Primitive

The article demonstrates that controlling req.body.files[number].filepath allows an attacker to make n8n copy local files instead of uploaded files. The example given is /etc/passwd, which is loaded into an organizational knowledge-base workflow.

The exploit flow shown in the article is:

  1. Intercept the HTTP request sent when uploading a file through a Form.
  2. Change the Content-Type from multipart/form-data to something else, such as application/json.
  3. Craft the request body so it overrides req.body.files.
  4. Set req.body.files[number].filepath to a local file path on the n8n instance.
  5. Let the Form workflow process that local file as if it were an uploaded file.

In the article’s example, the copied local file becomes available through a RAG-based organizational knowledge base. The attacker can then ask the chat interface about the loaded file and retrieve its contents. The article shows this with /etc/passwd.

Escalation to Authentication Bypass

The article then explains how arbitrary file read can lead to an authentication bypass in local n8n deployments. It describes n8n authentication sessions as using a cookie named n8n-auth. After login, n8n creates a payload dictionary containing key user details, including the user ID and the first 10 characters of a SHA256 hash derived from a string combining the user’s email and password. This payload is then signed with a secret key and stored in the n8n-auth cookie.

The article states that the pieces needed to forge a valid session are available in local files on n8n instances:

  • User records are stored in the n8n database.
  • In Docker deployments, the local database is located at /home/node/.n8n/database.sqlite.
  • The encryption secret key is stored in a configuration file.
  • The configuration file location is /home/node/.n8n/config.

The demonstrated authentication bypass involves first using the vulnerability to load the n8n database into the knowledge base, then extracting the admin user ID, email, and hashed password through the chat interface. The attacker then loads the config file using the same technique and extracts the encryption key. With those values, the attacker creates a signed session cookie and sets it as the n8n-auth cookie, resulting in admin access.

Remote Code Execution

After administrator access is obtained, the article says code execution is achieved by creating a new workflow containing n8n’s built-in Execute Command node. The screenshot described in the article shows the command id being executed and returning output indicating the process runs as the node user.

This final step does not require exploiting another bug according to the article; it uses n8n functionality available after gaining admin access. The article presents this as the final link in the chain from unauthenticated request handling to full instance takeover.
