The SDK checked whether that bucket existed, but it did not verify whether the victim owned it. Because Cloud Storage bucket names are globally unique, an attacker could create the expected bucket first in their own Google Cloud project.

Once the attacker created the expected bucket, the victim’s SDK would upload model files to the attacker-controlled bucket. The attacker could then replace the uploaded model with a malicious one.

Required attacker knowledge and conditions

The attacker needed only:

  • A Google Cloud project of their own.
  • The victim’s project ID, which the article says is often public.

The article states that the attacker did not need credentials, phishing, or a foothold in the target.

The attack worked only under specific conditions:

  • The victim’s default staging bucket did not already exist in that region.
  • The victim left the staging_bucket parameter unset.

The article notes that the first condition is common for a new project in Vertex AI in a region, while the second depends on the developer relying on the SDK’s default instead of naming their own bucket.

Model replacement and code execution

Many Python machine learning models are saved with pickle or joblib. The article notes that these formats can run code when a file is loaded.

The attack depended on speed. Unit 42 measured about 2.5 seconds between the victim’s upload and Vertex AI reading the file.

The payload in Unit 42’s proof of concept stole an OAuth token from the serving container’s metadata server and sent it to the attacker. In Unit 42’s test environment, the token was not limited to the compromised deployment.

  • GKE cluster names.
  • Internal container image paths.
