Overview

A critical security vulnerability affecting the Funnel Builder plugin for WordPress is being actively exploited in the wild. The attacks target WooCommerce checkout pages by injecting malicious JavaScript intended to steal payment information from customers.

The vulnerability affects all versions of Funnel Builder before 3.15.0.3. The issue currently does not have an official CVE identifier. Funnel Builder is used in more than 40,000 WooCommerce stores.

FunnelKit, which maintains Funnel Builder, has released a patch for the vulnerability in version 3.15.0.3.

Reported Exploitation

Sansec published details of the activity and said attackers are using the flaw to inject arbitrary JavaScript into every checkout page on affected stores. The malicious code is being placed in the plugin’s External Scripts setting.

According to Sansec, attackers are planting fake Google Tag Manager scripts in that setting. The injected scripts are designed to appear like ordinary analytics code alongside a store’s legitimate tracking tags.

The apparent purpose of this disguise is to make the malicious code look familiar and less suspicious during review. Sansec noted that presenting skimmers as Google Analytics or Google Tag Manager code is a recurring Magecart pattern.

Technical Details

The flaw involves a publicly exposed checkout endpoint in Funnel Builder. That endpoint allows an incoming request to choose the type of internal method to run.

In older plugin versions, the endpoint did not check the caller’s permissions. It also did not limit which internal methods could be invoked.

As described in the article, a bad actor could exploit this by sending an unauthenticated request that reaches an internal method capable of writing attacker-controlled data into the plugin’s global settings. Once the malicious snippet is added there, it is injected into every Funnel Builder checkout page.

This means an attacker can plant a malicious <script> tag that runs during every checkout transaction on a vulnerable WordPress site.

Observed Payload Behavior

Sansec observed at least one payload that masqueraded as a Google Tag Manager loader. The payload launched JavaScript hosted on a remote domain.

The JavaScript then opened a WebSocket connection to the attacker’s command-and-control server at wss://protect-wss[.]com/ws. Through that connection, it retrieved a payment skimmer tailored to the victim’s storefront.

The skimmer’s goal is to siphon data entered by site visitors during checkout. The article specifically names credit card numbers, CVVs, billing addresses, and other personal information as targeted data.

Affected Parties and Impact

Affected parties are WooCommerce stores using vulnerable versions of the Funnel Builder plugin. The plugin is installed on more than 40,000 WooCommerce stores, according to the article.

The disclosed impact is the injection of malicious JavaScript into checkout pages and theft of payment-related information. Because the malicious code can be written into global plugin settings, the injected script can appear across every Funnel Builder checkout page on a compromised store.

The article does not identify a specific threat actor by name. It refers to attackers and a bad actor exploiting the vulnerability.

Response and Mitigation

FunnelKit has released a fix in Funnel Builder version 3.15.0.3. Site owners are advised to update the plugin to the latest version.

The article also recommends that site owners review the plugin’s External Scripts area. Specifically, they should check Settings > Checkout > External Scripts for unfamiliar entries and remove anything suspicious.

Related Disclosure Mentioned in the Article

The article also notes a separate campaign detailed by Sucuri involving Joomla websites. In that campaign, Joomla sites were backdoored with heavily obfuscated PHP code that contacted attacker-controlled command-and-control servers.

The PHP script acted as a remote loader, sending information about the infected website and waiting for instructions. According to Sucuri’s description, the attacker could dynamically inject spam product links, redirect visitors, or display malicious pages without modifying local files again.

That Joomla campaign is described as having the ultimate aim of abusing the compromised sites’ reputation to inject spam.

  • The exploit abuses a publicly exposed checkout endpoint that allowed unauthenticated requests to invoke internal plugin methods without permission checks or an allowlist of permitted methods.
  • The malicious code is inserted into Funnel Builder’s global settings, which causes the skimmer to appear on every Funnel Builder checkout page rather than requiring attackers to modify individual pages.
  • At least one observed payload disguised itself as a Google Tag Manager loader and then used a WebSocket connection to wss://protect-wss[.]com/ws to retrieve a storefront-specific skimmer.
  • A practical detection step for store owners is to inspect the plugin’s Settings > Checkout > External Scripts area for unfamiliar entries, because attackers are planting the skimmer through that setting.
