Microsoft links Mastra AI supply chain attack to North Korean hackers

Summary
Microsoft says North Korea’s Sapphire Sleet hackers hijacked an npm maintainer account to push malicious updates to more than 140 Mastra AI packages. The attack installed a cross-platform infostealer aimed at stealing developer credentials, API keys, tokens, and cryptocurrency wallets.

Microsoft attributed a recent Mastra AI supply chain attack to Sapphire Sleet, a North Korean state actor also known as BlueNoroff. The attack compromised more than 140 npm packages after threat actors hijacked the npm maintainer account “ehindero,” which had publishing privileges across the Mastra package environment. Microsoft said the group primarily targets the financial sector and has previously been linked to cryptocurrency theft campaigns, malicious browser extensions, fake job offers, and software supply chain compromises.

The attackers used the compromised account to publish malicious updates for packages in the @mastra scope that added a dependency named “easy-day-js,” a typosquat of the legitimate dayjs JavaScript library. When installed, the dependency ran a post-install hook that executed an obfuscated dropper, disabled TLS certificate verification, contacted attacker-controlled command-and-control infrastructure, downloaded a second-stage payload, and ran it as a detached hidden process. The second-stage payload was a cross-platform information stealer targeting Windows, Linux, and macOS systems.

The malware collected host information, browser histories, installed applications, and running processes, and checked for 166 cryptocurrency wallet browser extensions including MetaMask, Phantom, Coinbase Wallet, Binance Wallet, and TronLink. It used operating-system-specific persistence methods such as Windows Registry Run keys, macOS LaunchAgents, and Linux systemd services. Microsoft said systems that contacted the attackers’ command-and-control servers showed follow-on activity associated with Sapphire Sleet, including a PowerShell backdoor, Microsoft Defender exclusions, and a malicious Windows service granting SYSTEM privileges.

  • A single compromised npm maintainer account with broad publishing privileges was enough to affect more than 140 packages across the Mastra package environment.
  • The malicious dependency name “easy-day-js” closely imitated the legitimate dayjs library, showing the attackers relied on a package-name lookalike inside a trusted update chain.
  • The malware was built to persist differently on Windows, macOS, and Linux, indicating the campaign was designed to compromise developers across multiple operating systems.
  • The implant checked for 166 cryptocurrency wallet browser extensions, making cryptocurrency theft a central and highly targeted objective of the payload.
  • Microsoft linked the campaign to Sapphire Sleet not only through initial malware behavior but also through follow-on activity, including a PowerShell backdoor and C2 infrastructure used in prior campaigns.
Key Points
  • 01Microsoft attributed the Mastra AI supply chain attack that compromised more than 140 npm packages to the North Korean hacking group Sapphire Sleet, also known as BlueNoroff.
  • 02The attackers compromised the npm maintainer account “ehindero” and used it to publish malicious updates for packages in the @mastra scope.
  • 03The malicious updates injected a dependency named “easy-day-js,” a typosquat of the legitimate dayjs JavaScript library, which executed a post-install hook on installation.
  • 04The second-stage payload was a cross-platform information stealer targeting Windows, Linux, and macOS systems, with checks for 166 cryptocurrency wallet browser extensions.
  • 05Microsoft said systems that communicated with the attackers’ command-and-control servers showed follow-on activity using tactics previously associated with Sapphire Sleet, including a PowerShell backdoor and additional persistence mechanisms.

Hackers Exploit Gravity SMTP WordPress Plugin Bug to Expose API Keys

Summary
Attackers are actively exploiting CVE-2026-4020 in the Gravity SMTP WordPress plugin to expose system details and email service API keys from vulnerable sites. Site owners should update to version 2.1.5 or later and rotate any configured third-party email credentials.

Threat actors are exploiting a recently patched flaw in Gravity SMTP, a WordPress plugin installed on about 100,000 sites. The vulnerability is tracked as CVE-2026-4020 and has a CVSS score of 5.3. It can allow unauthenticated attackers to extract sensitive data, including configuration data, API keys, secrets, and OAuth tokens used by the plugin’s email integrations.

Wordfence said the issue is caused by a REST API endpoint at /wp-json/gravitysmtp/v1/tests/mock-data with a permission callback that always returns true. When the ?page=gravitysmtp-settings query parameter is appended, the endpoint can return about 365 KB of JSON containing the full System Report. The exposed data can include PHP and web server versions, WordPress and plugin details, database information, and API keys or tokens for services such as Amazon SES, Google, Mailjet, Resend, and Zoho.

A patch has been released in Gravity SMTP version 2.1.5. Wordfence said it has blocked more than 17 million exploit attempts targeting CVE-2026-4020, with activity starting in early May 2026 and spiking around June 6, 2026. Site owners running vulnerable versions with third-party email integrations are advised to assume compromise, update the plugin, rotate credentials, and review server logs for suspicious requests to the affected API endpoint.

  • Site owners using vulnerable Gravity SMTP versions with third-party email integrations are advised to assume compromise and rotate credentials after updating, because live API credentials may already have been exposed.
  • The exploit requires only an unauthenticated HTTP GET request with the ?page=gravitysmtp-settings query parameter, making log review for that specific endpoint and parameter combination especially actionable.
  • The exposed data is not limited to email credentials; the returned system report can include software versions, active plugins, theme details, database table names, and configuration information that can help attackers plan follow-on attacks.
  • Wordfence observed exploitation beginning at the start of May 2026 before a sharp spike around June 6, 2026, so investigations should review logs from at least early May onward.
Key Points
  • 01Threat actors are exploiting CVE-2026-4020, a medium-severity information disclosure flaw in the Gravity SMTP WordPress plugin installed on about 100,000 sites.
  • 02The flaw involves a REST API endpoint at /wp-json/gravitysmtp/v1/tests/mock-data whose permission callback allows unauthenticated access.
  • 03Attackers can use the vulnerable endpoint with the ?page=gravitysmtp-settings query parameter to retrieve sensitive data such as system details, plugin versions, database information, and API keys or tokens for email integrations.
  • 04Wordfence said it has blocked more than 17 million exploit attempts targeting CVE-2026-4020, with activity spiking around June 6, 2026.
  • 05The vulnerability has been patched in Gravity SMTP version 2.1.5, and affected site owners are advised to update and rotate third-party email credentials.

New Prinz Eugen ransomware prioritizes recent files for encryption

Summary
Prinz Eugen is a new hands-on ransomware operation that likely uses stolen RDP credentials and legitimate RMM tools, then encrypts the most recently modified files first to maximize disruption. The malware leaves no ransom note, self-deletes, and moves extortion communications out-of-band to reduce forensic traces.

A new ransomware operation named Prinz Eugen prioritizes recently modified files for encryption and leaves no ransom note on infected systems. Threatdown, Malwarebytes’ enterprise cybersecurity arm, found that the attackers use a hands-on-keyboard style and prefer legitimate remote monitoring and management software and living-off-the-land tools. Researchers said initial access is likely achieved through stolen RDP credentials, followed by manual download and execution of the main payload, servertool.exe.

In an investigated incident, the attackers used the RemotePC RMM tool and a backdoor administrator account for persistence. Prinz Eugen is not currently operating as ransomware-as-a-service, and its developers are not currently recruiting affiliates. Its data leak site lists three victims, but Threatdown identified at least five victims and said the Standard Bank attacker demanded 1 BTC and was refused.

The Go-based encryptor recursively scans directories with no depth limit or exclusions and encrypts virtually every file except those with the .prinzeugen extension. It uses ChaCha20-Poly1305 encryption, a 32-byte master key, random initialization vectors for each file, and key derivation based on Argon2id, SHA-256, and HKDF-SHA256. Threatdown said the lack of a ransom note can reduce forensic artifacts and move ransom communications to email, phone contact, or dark-web victim portals.

  • Prinz Eugen’s choice to encrypt the most recently modified files first is an impact-focused strategy because those files are more likely to be actively used and business-critical.
  • The lack of a ransom note is intentional rather than an omission, as Threatdown says it reduces forensic artifacts and pushes ransom communication to email, phone contact, or dark-web victim portals.
  • The encryptor checks that a file can be decrypted before deleting the original when the --delete flag is used, showing the malware is designed to avoid corrupting files in a way that would prevent recovery after payment.
  • Prinz Eugen attempts to prevent key recovery by overwriting the encryption key with zeroes, forcing garbage collection, and self-deleting from disk.
Key Points

  • 05Threatdown identified at least five Prinz Eugen victims and said that in the Standard Bank breach the attacker demanded 1 BTC and was refused.

In plain words

A mid-size equipment finance company received a fake document-sharing email that pretended to come from “Jennifer Johnson” and led through several trusted-looking web services before ending at a Google Docs presentation used for stealing login details. The email should have looked suspicious because its sender checks failed, but it still reached the inbox after other security services treated it as safe. This matters because attackers can hide harmful links behind familiar, trusted services, making scam emails look normal to both people and some protective tools.

Summary
A phishing email impersonating a document-sharing notification hid its credential-harvesting link behind EdgePilot, Barracuda LinkProtect, a throwaway redirect domain, and Google Docs, allowing trusted infrastructure to camouflage the attack. Despite failing SPF, DKIM, and DMARC, it reached the inbox because downstream trust in an intermediary gateway overrode the raw authentication failures.

IRONSCALES detected a credential-harvesting campaign in May 2026 targeting a mid-size equipment finance company. The email impersonated a document-sharing notification from “Jennifer Johnson” and referenced a company-specific aging report. The actual sender address was davidbr@bellnet[.]ca, a Canadian consumer ISP account with no connection to the target organization or document-sharing platform.

The attack routed its payload link through three layers before reaching a Google Docs presentation. The chain used EdgePilot, Barracuda LinkProtect, the throwaway domain smstkart[.]com/bee, and then docs[.]google[.]com/presentation/d/e/.... The article states that legitimate link-protection services and Google Docs provided trusted-domain camouflage for the phishing payload.

The message failed SPF, DKIM, and DMARC at the recipient’s Microsoft 365 tenant, but Microsoft still delivered it to the inbox with SCL=-1. AppRiver’s SecureTide gateway had logged the links as wrapped and returned a clean verdict before forwarding the message. Themis flagged the message at 84% confidence and auto-resolved it as phishing based on behavioral signals including a first-time sender, a consumer ISP address, a display-name mismatch, and an inconsistent communication pattern.

  • Legitimate link-protection services can become camouflage when attackers place their own redirect infrastructure behind trusted wrapper domains.
  • A clean verdict from an upstream email gateway can create downstream trust even when Microsoft 365 later records SPF, DKIM, and DMARC failures.
  • URL reputation alone was insufficient in this case because the visible and terminal domains included trusted services such as EdgePilot, Barracuda LinkProtect, and Google Docs.
  • Behavioral signals such as first-time sender status, consumer ISP origin, display-name mismatch, and unusual communication patterns identified the phishing attempt when authentication and link checks did not.
