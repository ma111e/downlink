Vulnerability Overview

Researchers from Xint.io and Theori have disclosed CVE-2026-31431, a high-severity Linux local privilege escalation (LPE) flaw codenamed Copy Fail. The vulnerability carries a CVSS score of 7.8 and fundamentally alters how unprivileged users interact with the kernel’s page cache. By exploiting a logic flaw in the kernel’s cryptographic subsystem, attackers can write four controlled bytes into the page cache of any readable file on the system. This primitive ultimately grants the attacker full root access without requiring remote network access.

Technical Mechanism and Exploitation

The core issue originates from a logic flaw within the algif_aead module, introduced via a source code commit in August 2017. Exploitation is achieved through a compact 732-byte Python script that follows a precise four-step sequence. The script first opens an AF_ALG socket and binds it to authencesn(hmac(sha256),cbc(aes)). It then constructs a shellcode payload within the script environment. Next, the script triggers a write operation that targets the kernel’s cached copy of a setuid binary, specifically /usr/bin/su. Finally, the script executes execve("/usr/bin/su"), which loads the injected shellcode and runs it with root privileges.

Impact and Cross-Container Scope

While the vulnerability is strictly local and cannot be exploited remotely, its impact extends far beyond a single user session. An unprivileged user can reliably corrupt the page cache of a setuid binary to achieve immediate root access. Because the Linux page cache is shared across all processes on the host, the vulnerability carries significant cross-container implications. This shared memory architecture allows a single local compromise to cascade across isolated containers, effectively neutralizing container sandboxing boundaries.

Vendor Response and Historical Context

Major Linux distributions have already responded to the disclosure by publishing official security advisories. Affected vendors include Amazon Linux, Debian, Red Hat Enterprise Linux, SUSE, and Ubuntu. Security analysts note that Copy Fail shares architectural similarities with the Dirty Pipe vulnerability (CVE-2022-0847), as both rely on page cache corruption primitives. However, Copy Fail operates within a different subsystem and leverages a distinct exploitation path. Bugcrowd researcher David Brumley clarified that the 2017 in-place optimization in algif_aead allows a page-cache page to incorrectly end up in the kernel’s writable destination scatterlist. This misconfiguration enables an unprivileged process to drive splice() into the socket and complete small, targeted writes into the page cache of files it does not own.

Key Differentiators and Reliability

The Copy Fail vulnerability stands out due to its consistent reliability and lack of complex exploitation requirements. Unlike many kernel flaws, it does not depend on race conditions or kernel offset detection to function. The researchers describe the vulnerability as uniquely portable, tiny, stealthy, and cross-container. These combined properties allow any user account, regardless of privilege level, to escalate to full administrative access. Furthermore, the exploit bypasses standard sandboxing mechanisms and remains effective across all Linux versions and distributions shipped since 2017.

The flaw primarily impacted headless CI environments where the tool automatically trusted workspace folders without explicit user consent. Google addressed this by mandating explicit folder trust through the GEMINI_TRUST_WORKSPACE environment variable and hardening tool allowlisting in –yolo mode. Google recently addressed a maximum severity CVSS 10.0 security flaw in the Gemini CLI npm package and its associated GitHub Actions workflow. This vulnerability permitted unprivileged external attackers to inject malicious configuration files that triggered direct command execution on host systems. The flaw circumvented the agent’s sandbox initialization and lacked a CVE identifier. It impacted Gemini CLI versions below 0.39.1 and 0.40.0-preview.3, as well as the GitHub Actions workflow below version 0.1.22. Google clarified that the impact is restricted to headless CI workflows that previously auto-trusted workspace folders for configuration loading. To remediate the issue, Google requires administrators to explicitly trust directories using the GEMINI_TRUST_WORKSPACE environment variable. The update also enforces tool allowlist evaluation in –yolo mode, which may cause silent failures in pipelines relying on unrestricted command execution. In parallel, Novee Security disclosed a CVSS 8.1 vulnerability in Cursor IDE versions prior to 2.5. This flaw enables arbitrary code execution via prompt injection by exploiting autonomous Git operations within nested bare repositories. The attack chain begins when a user opens a repository containing a malicious post-checkout hook and requests a codebase explanation. Cursor’s agent then parses AGENTS.md, triggers a git checkout, and silently executes the hook outside both the agent’s reasoning chain and user visibility. Researchers attribute this to a Git feature interaction that becomes dangerous when AI agents autonomously manage uncontrolled repositories. Furthermore, LayerX Security identified an unpatched CVSS 8.2 access control vulnerability dubbed CursorJacking. This flaw allows any locally installed extension to extract sensitive API keys and session tokens from a local SQLite database. Cursor maintains that exploitation requires prior user installation and permission grants for the extension. Organizations are urged to update Gemini CLI immediately, configure explicit folder trust, and restrict Cursor extensions to only trusted sources.

Google has patched a CVSS 10.0 remote code execution flaw in the @google/gemini-cli npm package and its associated GitHub Actions workflow. Exploitation could lead to session token exposure, unauthorized backend access, and financial loss from unauthorized API usage. Cursor states that access is limited to the local machine where the extension was installed and granted permissions. The Silver Fox threat group is deploying a multi-stage attack disguised as tax correspondence that uses a modified Rust loader and ValleyRAT backdoor to deliver a new Python-based backdoor called ABCDoor, targeting organizations across Russia, India, and other regions.

The Silver Fox threat group launched a phishing campaign in late 2025 and early 2026 disguised as official tax correspondence targeting organizations in India and Russia. Attackers delivered malicious archives containing a modified Rust-based loader called RustSL, which was customized to bypass antivirus detection and enforce geographic restrictions. Once executed, the loader deployed the ValleyRAT backdoor, which subsequently installed a previously undocumented Python-based backdoor named ABCDoor. ABCDoor establishes persistence through registry entries and scheduled tasks, communicates over HTTPS, and utilizes ffmpeg to broadcast victim screens across multiple monitors. The campaign impacted various sectors including industrial, consulting, retail, and transportation, with over 1,600 malicious emails recorded.

In December 2025 and January 2026, the Silver Fox threat group executed a coordinated phishing campaign targeting organizations across India, Russia, Indonesia, South Africa, Cambodia, and Japan. The attackers sent emails styled as official tax notices, using PDF attachments containing download links to bypass email security gateways. Inside the downloaded archives, victims encountered a modified Rust-based loader known as RustSL. This customized loader featured a steganography module for payload unpacking and a guard module that restricted execution to specific target countries using multiple public IP services. The loader also implemented a Phantom Persistence technique that intercepts system shutdown signals to force malware execution upon reboot. Upon activation, the RustSL loader decrypted and executed shellcode that downloaded the ValleyRAT backdoor. ValleyRAT managed command-and-control communications and deployed custom modules responsible for fetching a new Python-based backdoor called ABCDoor. ABCDoor was distributed as a large archive containing a Python environment and the ffmpeg tool for screen capture. Once running, ABCDoor maintained persistence through Windows registry modifications and scheduled tasks. The backdoor communicated with its command-and-control servers over HTTPS and utilized the Desktop Duplication API to stream up to four monitors simultaneously. Attackers also employed JavaScript loaders and self-extracting archives in later stages of the campaign to distribute the payload. ABCDoor evolved rapidly from late 2024, adding features like clipboard exfiltration, file encryption, and self-update capabilities. The threat group consistently used command-and-control domains sharing the abc third-level domain. Over 1,600 malicious emails were recorded during the campaign, affecting industrial, consulting, retail, and transportation sectors.

Campaign Overview

The Silver Fox threat group conducted a multi-stage cyberattack campaign beginning in December 2025 and expanding in January 2026. The operation targeted organizations in the industrial, consulting, retail, and transportation sectors across India, Russia, Indonesia, South Africa, Cambodia, and Japan. Attackers recorded over 1,600 malicious emails throughout the campaign period.

Initial Access and Delivery

The campaign relied heavily on phishing emails disguised as official tax correspondence. In the January wave targeting Russia, emails contained PDF attachments with clickable links to malicious archives hosted on attacker-controlled websites. The December wave targeting India delivered archives directly as email attachments or embedded malicious code within files. Attackers specifically used PDF attachments with external download links to evade email security gateways, as the attached documents only contained benign-looking links requiring further analysis.

Initial Payload: Modified RustSL Loader

The primary malware delivered was a customized version of the publicly available RustSL loader, dubbed Silver Fox RustSL. The attackers modified the open-source framework to include extensive antivirus bypass capabilities, such as multiple encryption methods, sandbox detection techniques, and payload execution strategies. Two key modules were added to the loader:

  • The steganography.rs module handled encrypted payload unpacking using a custom XOR-based decryption algorithm.
  • The guard.rs module enforced strict geofencing by querying five public IP services to verify the victim’s location before execution. The loader also incorporated a Phantom Persistence technique that intercepts system shutdown signals, aborts the shutdown process, and triggers a reboot to ensure the malware executes upon startup.

Secondary Payload: ValleyRAT Backdoor

Once the RustSL loader executed, it decrypted a shellcode payload that downloaded the ValleyRAT backdoor, also known as Winos 4.0. The infection chain progressed through an Online module and a Login module, which handled command-and-control communications and module deployment. ValleyRAT stored its configurations and modules in specific Windows registry keys. During this campaign, the backdoor loaded two previously unseen modules responsible for downloading and executing the ABCDoor backdoor.

Tertiary Payload: ABCDoor Python Backdoor

The custom ValleyRAT modules downloaded a 52.5 MB archive containing a Python environment and the ffmpeg utility. This archive was extracted and executed via a batch script that copied files to a directory mimicking the legitimate Tailscale VPN service. The resulting Python-based backdoor, named ABCDoor, operated within a legitimate pythonw.exe process to maintain stealth. ABCDoor established persistence through Windows registry entries and scheduled tasks that ran every minute. The backdoor communicated with command-and-control servers over HTTPS using asynchronous event handlers. Its capabilities included:

  • Screen broadcasting across up to four monitors using the Desktop Duplication API.
  • File management, clipboard exfiltration, and process manipulation.
  • Self-update mechanisms and self-deletion capabilities.
  • Cryptographic functions limited to DPAPI encryption. ABCDoor utilized command-and-control domains that all shared the abc third-level domain.

Evolution and Distribution Methods

ABCDoor first appeared in late 2024 and evolved rapidly throughout 2025. Early distribution relied on C++ and Go stagers that downloaded the payload from attacker-controlled servers. Between May and August 2025, attackers used TinyURL redirects and channel-named URLs to deliver the malware. Starting in November 2025, the group shifted to JavaScript loaders distributed via self-extracting archives. These archives contained PowerShell scripts that installed NodeJS if missing, executed obfuscated JavaScript, and downloaded the final ABCDoor archive. The JavaScript loader logged operations in Chinese and handled all subsequent payload extraction and execution.

Targeting and Impact

The RustSL loader and ABCDoor modules enforced strict geographic restrictions, allowing execution only in target countries. While the original public RustSL code only supported China, the Silver Fox variant expanded to include India, Indonesia, South Africa, Cambodia, and Japan. Telemetry showed the highest concentration of attacks in India, Russia, and Indonesia. The campaign successfully impacted organizations across multiple critical sectors, demonstrating the group’s expanding geographic focus and sophisticated multi-stage delivery architecture.
