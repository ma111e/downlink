Mandiant revealed new details on how hackers exploited Cisco Catalyst SD-WAN vulnerability CVE-2026-20245 in zero-day attacks to create rogue root accounts on targeted devices. The flaw affects Cisco Catalyst SD-WAN Manager, Controller, and Validator and allows authenticated attackers with local access to execute arbitrary commands as root by uploading a crafted file. Cisco had previously warned that the vulnerability was exploited in a limited number of attacks and released security updates, stating that no workarounds were available.

According to Mandiant, the intrusion began with unauthorized SD-WAN peering connections observed on a service provider’s infrastructure. Beginning in March 2026, the threat actor established rogue peer connections and authenticated to affected SD-WAN Manager devices using the vmanage-admin account. The attackers changed the default admin account password, logged in to the SD-WAN Manager web interface, extracted configuration information, and later restored the original password, likely to reduce detection.

Mandiant said the attackers exploited CVE-2026-20245 through a tenant-upload feature in the SD-WAN command-line interface by uploading a malicious CSV file named evil_tenant.csv. The payload backed up system configuration files such as /etc/passwd and /etc/shadow before creating a new root-level account named troot, which the attackers used to gain full control of the device. The attackers also deleted the malicious file, removed temporary files, erased evidence of the rogue root account, and ran a validation script to confirm traces of compromise had been removed.

  • CVE-2026-20245 was used after the attackers had already gained access to SD-WAN devices, so patching this flaw alone may not address the initial access path in affected environments.
  • The attackers restored the default admin password after using it, making password state alone an unreliable indicator of whether the account had been abused.
  • Mandiant observed rogue peering activity on systems that were not vulnerable to previously disclosed authentication-bypass flaws, and Cisco said stolen certificates from a prior compromise may have been used to regain access.
  • The attackers ran a validation script to confirm that compromise traces had been removed, so organizations may need to rely on diagnostic data, unauthorized peering checks, and published indicators rather than obvious on-device artifacts.
Key Points

  • CVE-2026-20245 is a high-severity command injection flaw in Cisco Catalyst SD-WAN Manager, Controller, and Validator that allows authenticated attackers with local access to execute arbitrary commands as root by uploading a crafted file.
  • Mandiant said the attackers used a tenant-upload feature in the SD-WAN command-line interface to upload a malicious CSV file named "evil_tenant.csv" that created a new root-level account named "troot."
  • The attackers changed the default admin account password, extracted configuration information for edge devices, controllers, and SD-WAN templates, and later restored the original password to reduce detection.

In plain words

A popular GitHub add-on used to automate software releases, codfish/semantic-release-action, was secretly changed so that many projects using its version labels would run an attacker’s code the next time their release process ran. The projects most affected are those that used the listed version labels, especially because release setups often contain secret keys that can publish software or access GitHub. This matters because the attacker could steal those secrets and potentially use them to take over or tamper with software releases, spreading the problem beyond one project.

Summary
The codfish/semantic-release-action GitHub Action was compromised after attackers force-pushed malicious commits and repointed multiple version tags, causing affected CI workflows to run obfuscated code that could steal release secrets such as GITHUB_TOKEN and NPM_TOKEN. Users should stop using the affected tags, check for the listed malicious commits and payload hash, and rotate any exposed credentials.

On Jun 24, 2026, the codfish/semantic-release-action GitHub Action was compromised through an imposter commit attack. An attacker force-pushed two malicious commits into the repository and repointed sixteen tags to them, including v2, v3, v4, and v5. Any workflow referencing one of those tags will pull and run the attacker’s code on its next CI run.

The action has been a standard way to wire semantic-release into GitHub Actions since 2019 and has over 100 GitHub stars. Workflows using it for automated releases almost always hold a GITHUB_TOKEN and frequently an NPM_TOKEN with publish access. The article says this is exactly the kind of access an attacker wants to land inside.

The malicious commits replaced action.yml with a composite action that first called a clean pinned version of the real action, then installed Bun through oven-sh/setup-bun, and finally executed an obfuscated index.js payload. The payload contained the string thebeautifulsnadsoftime, which the article links to the Miasma credential stealing toolkit and its GitHub public commit search dead drop mechanism. The article lists malicious commits, the index.js SHA-256 hash, affected tags, and confirmed clean versions as indicators of compromise.

  • The compromise was designed to preserve normal release behavior by first running a clean pinned version of the real action, making successful workflow runs potentially misleading.
  • The old Dockerfile and entrypoint files were left in the repository even though they were no longer used, which made a casual file listing less likely to reveal the switch to a composite action.
  • The attacker exploited the fact that GitHub tags are not protected by default and are resolved when a workflow runs, so previously written workflows could be changed retroactively without modifying the workflow files.
  • The use of GitHub’s public commit search API as a dead drop channel means the malware can receive commands without relying on traditional attacker-owned infrastructure.
  • The article identifies specific clean versions, showing that not all releases of codfish/semantic-release-action were affected and that version selection matters for remediation.
Key Points
  • On Jun 24, 2026, the codfish/semantic-release-action GitHub Action was compromised when an attacker force-pushed two malicious commits and repointed sixteen tags, including v2, v3, v4, and v5.
  • Any workflow referencing the affected tags will pull and run the attacker’s code on its next CI run, and these workflows often contain GITHUB_TOKEN and sometimes NPM_TOKEN credentials with publish access.
  • The malicious commits changed action.yml from a Docker-based action into a composite action that first ran a clean pinned version of the original action and then installed Bun and executed an obfuscated index.js payload.
  • The injected index.js payload was 781,580 bytes, had SHA-256 hash 9f93d77d32833a515bc406c46da477142bb1ac2babeecb6aa42f98669a6db015, and contained the dead drop marker string thebeautifulsnadsoftime.

Researchers describe a macOS privilege-escalation technique that lets a standard user impersonate trusted app components through CDHash cache and NIB payload abuse to call privileged XPC services, potentially disabling EDR and MDM tools without admin rights. Vendors should harden XPC client validation using active code-signing checks rather than relying on cached trust.

A novel macOS privilege escalation technique allows standard user accounts to silently disable leading enterprise security products, including EDR and MDM solutions, without administrator credentials, kernel exploits, or security alerts. The article says the technique exploits how trusted application components are identified, how trust relationships are inherited, and how privileged operations are authorized. It was validated against CrowdStrike Falcon Sensor and Kandji MDM Agent, with CrowdStrike adding detections and preventions and Kandji fixing the issue assigned CVE-2026-39118.

The attack chains CDHash kernel cache exploitation with NIB payload injection to impersonate trusted application components. After a legitimate signed application binary is launched to populate the kernel cache, the application bundle can be manipulated so injected code runs in the context of the trusted component. From that position, the payload can communicate with privileged background daemons and invoke exposed XPC methods such as runProcessWithCommand:, terminateAppsAndAgents:, and ceaseSystemExtensionWithReply:.

The article argues that relying only on passive CDHash validation in kernel caches is insufficient for securing privileged XPC service boundaries. It recommends that developers explicitly validate client identity inside listener protocol logic, including by using xpc_connection_set_peer_code_signing_requirement() or by revalidating the on-disk signature state with SecCodeCheckValidity() and SecCodeCopyGuestWithAttributes() using the incoming connection’s audit token. The author also built XPC Hunter, an open-source automated framework intended to discover exploitable XPC privilege escalation surfaces across installed macOS applications at scale.

  • The article says the core security assumption that only a vendor’s own signed software can communicate with privileged background processes is unreliable when trust is based on cached CDHash validation alone.
  • The described technique is especially difficult for defenders because it abuses legitimate operating system behavior rather than memory corruption or a conventional exploit path, leaving few obvious forensic traces.
  • The exposed XPC methods found during testing were not obscure primitives but built-in management functions such as running commands, terminating agents, and stopping system extensions.
  • The article identifies peer validation at the XPC listener handshake, using active code-signing checks tied to the live connection, as the practical point where developers can break the impersonation chain before privileged methods are exposed.

In plain words

The article is a checklist for companies choosing tools that spot and respond to suspicious activity across cloud services, business apps, user accounts, and newer AI systems. It says many organizations have security tools but still cannot see enough of how these systems really behave, especially when attackers use valid logins, permissions, or connected apps. This matters because choosing the wrong tool can leave blind spots, create too many useless alerts, or cause slow and risky responses when a real break-in happens.

Summary
The article provides a checklist for evaluating Cloud Detection and Response platforms, emphasizing that buyers should prioritize broad visibility across cloud, SaaS, identity, workloads, and AI systems over feature counts. The key test is whether a CDR platform can measurably reduce detection time, response time, and blast radius in real-world cloud attacks.

The article presents a checklist for evaluating Cloud Detection and Response platforms and vendors based on real operational needs rather than feature lists. It says many enterprises have security tools but still lack visibility that matches how cloud, SaaS, identity, and AI systems behave. The article warns that some CDR platforms are posture tools with alerts, while others are legacy detections wrapped in cloud branding.

The checklist emphasizes visibility across existing EDR telemetry, cloud control planes, SaaS, identity providers, serverless environments, workloads, storage, databases, APIs, networking logs, and configuration changes. It says attackers move through APIs, tokens, roles, and permissions, and that a platform that cannot correlate identity, SaaS, and cloud control planes is only watching fragments rather than full attacks. It also highlights AI-driven triage, cross-domain correlation, automatic attack path reconstruction, and identity-aware detection as important evaluation areas.

The article says CDR platforms should support response and containment without breaking the business, including compromised identity response, SOAR and SOC integrations, context-preserving investigation, and business-impact-aware actions. It also recommends assessing risk-aware prioritization, forensic readiness, long-term data retention, threat hunting, enterprise scalability, predictable telemetry costs, and vendor expertise in cloud investigations. The final decision question is whether the platform measurably reduces time-to-detect, time-to-respond, and blast radius across critical cloud, SaaS, and identity paths.

  • The article warns that some products marketed as CDR may actually be posture tools with alerts or legacy detection products rebranded for cloud, so buyers should pressure-test vendor claims against operational outcomes.
  • The checklist treats agentless visibility as essential because important cloud, SaaS, serverless, and identity activity often occurs where traditional security agents cannot run.
  • The article frames response quality as a balance between speed and business impact, noting that fast containment without context can cause outages while slow containment can lead to breaches.
  • The article says CDR should function as an intelligence layer that strengthens existing SIEM, EDR/XDR, and SOAR tools rather than becoming another dashboard analysts ignore.
  • The article highlights forensic readiness as a buying criterion, including long-term data retention independent of native cloud limits and unified audit trails across cloud, identity, and workloads.
Key Points
  • The article says Cloud Detection and Response platforms should be evaluated by real operational needs rather than by feature lists alone.
  • A CDR platform should provide visibility across cloud control planes, SaaS, identity providers, workloads, storage, databases, APIs, networking logs, and configuration changes.
  • The article argues that identity telemetry must be central because attackers in cloud environments can use valid credentials and chain access across SaaS, cloud, and workloads.
  • The checklist says CDR platforms should use AI-driven triage, cross-domain correlation, automatic attack path reconstruction, and machine-speed detection to reduce analyst noise.
  • The article says the final decision should be based on whether a CDR platform measurably reduces time-to-detect, time-to-respond, and blast radius across critical cloud, SaaS, and identity paths.
