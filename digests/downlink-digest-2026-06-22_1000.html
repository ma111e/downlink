In plain words

JFrog researchers found a serious flaw in FFmpeg, a widely used tool that many apps and websites rely on to handle video files. A specially made video file can make those apps crash, and in demonstrations it let the researchers run commands on Jellyfin and Nextcloud servers when the file was uploaded or automatically scanned. This matters because FFmpeg is built into many desktop video players, file preview tools, media servers, cloud services, and even some AI systems, so one bug can spread silently through many products. The article says users should update FFmpeg or disable the affected MagicYUV video decoder if they do not need it.

Summary
JFrog found PixelSmash, a critical FFmpeg MagicYUV decoder flaw that lets a crafted media file crash or potentially execute code in apps and services that automatically process video, including Jellyfin and Nextcloud. Users and vendors should upgrade FFmpeg immediately or disable the MagicYUV decoder if it is not needed.

JFrog Security Research discovered and disclosed PixelSmash, CVE-2026-8461, a heap out-of-bounds write in FFmpeg’s MagicYUV decoder with a CVSS score of 8.8 High. The vulnerability can be triggered when an application using FFmpeg’s libavcodec processes a crafted AVI, MKV, or MOV media file. JFrog says the issue can crash applications that use FFmpeg and demonstrated remote code execution against Jellyfin and Nextcloud by uploading a crafted 50 KB AVI file.

The article emphasizes that FFmpeg is bundled or linked by many media-processing applications, including desktop players, file-manager thumbnail generators, media servers, cloud transcoding pipelines, and AI/ML video-processing tools. JFrog confirmed crashes against products including Kodi, mpv, ffmpegthumbnailer, Jellyfin, Emby, Nextcloud, Immich, PhotoPrism, OBS Studio, and vLLM. The researchers describe PixelSmash as a software supply chain vulnerability because downstream projects inherit the vulnerable MagicYUV decoder through FFmpeg rather than introducing the bug themselves.

Technically, the flaw is caused by a rounding mismatch in how the MagicYUV decoder handles slice heights for chroma planes, allowing attacker-controlled bytes to be written past the end of a heap buffer. JFrog escalated the bug to code execution by overwriting FFmpeg’s AVBuffer structure so that normal cleanup could call system() with an attacker-controlled command, though the demonstrated Jellyfin exploit required ASLR to be disabled. The article recommends upgrading to the fixed FFmpeg version as soon as possible or disabling the MagicYUV decoder at build time if it is not needed.

  • The exploit can be triggered without playing the file because thumbnail generation, preview generation, and media library scanning can automatically process a malicious video as soon as it appears in a folder or upload location.
  • The article highlights that Plex’s approach of building FFmpeg with disabled decoders and a minimal allow-list was the only effective defense the researchers observed, while most tested projects inherited the default MagicYUV decoder exposure.
  • Jellyfin and Emby sometimes processed the malicious file with exit code 0 and no error output, even though ASAN-instrumented builds confirmed the same out-of-bounds write occurred.
  • The demonstrated Jellyfin remote code execution required ASLR to be disabled because the exploit used hardcoded addresses, while the article states the MagicYUV flaw alone is sufficient for reliable denial of service against every tested target.
  • The heap layout needed for the Jellyfin exploit changed based on ffprobe command-line arguments and input file path length, and filenames of equal length produced identical layouts in the researchers’ testing.
Key Points
  • JFrog Security Research discovered and disclosed PixelSmash, CVE-2026-8461, a heap out-of-bounds write in FFmpeg’s MagicYUV decoder with a CVSS score of 8.8 High.
  • The vulnerability can be triggered by processing a crafted AVI, MKV, or MOV media file in any application that decodes video using FFmpeg’s libavcodec.
  • JFrog demonstrated remote code execution against Jellyfin via automatic library scanning and against Nextcloud via the video preview provider by uploading a crafted 50 KB AVI file.
  • The article says the MagicYUV decoder is enabled by default in upstream FFmpeg builds and tested distribution packages before version 9.0, affecting many downstream applications that bundle or link FFmpeg.
  • Users are advised to upgrade to a fixed FFmpeg version as soon as possible or disable the MagicYUV decoder at build time if it is not needed.

If opened on WhatsApp Desktop or Web, the scripts install a preconfigured ManageEngine Endpoint Central agent that gives attackers remote access to the victim’s Windows system.

In June 2026, Kaspersky observed an active malware campaign distributing malicious VBScript files through direct messages in WhatsApp. The campaign affected users in multiple countries and territories, including Malaysia, Brazil, India, Mexico, Singapore, the UK, Spain, Taiwan, Australia, Russia, and Vietnam, with the highest number of victims in Malaysia. The final stage silently installs the ManageEngine Endpoint Central agent using bundled configuration files, certificates, and an MSI installer, enabling remote access to the victim’s system through attacker-controlled management servers. The article notes that some infrastructure overlaps with IP addresses previously associated with ValleyRAT and Gh0st RAT activity, but says the evidence is insufficient to attribute the campaign to a known threat actor.   • The attackers abuse legitimate ManageEngine Endpoint Central software rather than deploying only custom malware, allowing the final stage to provide remote access through a real enterprise management tool.
  • One Stage 2 script repeatedly requests elevated permission to change User Account Control behavior, indicating the attack chain still depends on the victim granting administrative privileges for that change to succeed.

Summary
Starting September 30, 2026, certified Android devices in Brazil, Indonesia, Singapore, and Thailand will block normal installs of apps from developers who have not verified their identity with Google, including apps distributed outside Google Play. The move is framed as anti-malware protection but is drawing strong opposition from open-source and independent app distributors like F-Droid.

Google has set September 30, 2026, as the start date for enforcing Android developer verification in Brazil, Indonesia, Singapore, and Thailand. On certified Android phones in those countries, normal installs of apps from developers who have not registered an identity with Google will be blocked. The rule applies whether apps come from Google Play or stores run by Samsung, Xiaomi, OPPO, vivo, Honor, and Transsion.

Google says the program is meant to reduce malicious apps and scams, especially those involving victims being convinced to install harmful APKs.   • Google’s limited-distribution option does not resolve the concerns of larger independent repositories, because it is capped at 20 devices and is aimed at students and hobbyists rather than broad public distribution.
