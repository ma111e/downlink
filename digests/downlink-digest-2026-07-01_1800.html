Researchers say a large campaign stole login details from many Fortinet firewall devices and is connected to the INC and Lynx ransomware groups. The campaign reportedly targeted more than 430,000 FortiGate firewalls worldwide and placed tools on about 19,000 devices to capture sign-in information, with around 11,000 still compromised after notifications. This matters because the stolen access could help criminals break into organizations later, and some victim information appears to overlap with organizations later named on a ransomware leak site.

Summary
Researchers linked the FortiBleed credential-theft campaign, which stole credentials from tens of thousands of Fortinet devices, to the INC and Lynx ransomware operations. The findings suggest the stolen FortiGate access was being used to support future ransomware intrusions.

The FortiBleed credential-theft campaign has been linked to the INC and Lynx ransomware operations, suggesting stolen Fortinet credentials were intended to support future network intrusions. The campaign came to light after a server exposed on the internet was found containing credentials stolen from more than 73,000 Fortinet devices. Researchers found downloaded FortiGate configuration files, harvested credentials, and infrastructure used to crack password hashes and perform credential-stuffing attacks.

SOCRadar said its Threat Research Unit tied the operation to INC and Lynx after identifying a Windows server used in the FortiBleed infrastructure. The company said artifacts on that server showed the threat actor had accessed ransomware negotiation panels for both Lynx and INC, including dashboards with victim chats used during negotiations. SOCRadar also reported overlaps between victim information harvested during FortiBleed and organizations later listed on the INC ransomware leak site.

The campaign used a custom tool called “FortiGate Sniffer” on compromised FortiGate firewalls to intercept VPN credentials and other authentication data from network traffic. SOCRadar said the operation targeted more than 430,000 FortiGate firewalls worldwide, deployed traffic sniffers on about 19,000 devices, and still had around 11,000 compromised devices after impacted organizations were notified. The researchers also said they identified roughly 500 servers used by the operation, persistent backdoor accounts named “adminin,” and evidence suggesting the attackers exploited a previously undisclosed Nextcloud zero-day vulnerability.

  • The link to INC and Lynx was based on browser sessions from FortiBleed infrastructure accessing both ransomware groups’ negotiation panels, which ties the credential-theft operation to ransomware activity beyond just possession of stolen Fortinet data.
  • SOCRadar said the number of compromised devices dropped from about 19,000 to around 11,000 after impacted organizations were notified, indicating that notification and remediation efforts had a measurable effect.
  • The reported overlap between FortiBleed-harvested victim information and organizations later listed on the INC ransomware leak site suggests the stolen Fortinet credentials were not only collected at scale but also connected to later extortion activity.
  • The discovery of persistent backdoor accounts using the username “adminin” gives defenders a concrete artifact to check for on compromised systems mentioned in the investigation.
  • SOCRadar’s finding of more than 200 additional operational servers and roughly 500 servers overall shows the campaign’s infrastructure was far larger than initially understood.

Unpatched Argo CD Repo-Server Flaw Could Let Attackers Take Over Kubernetes Clusters
thehackernews.com · 01 Jul 2026 19:40 UTC openai-codex / gpt-5.5
TAGSargo-cdargo-cd-repo-serverargo-cdownargoprojcve-2024-31989cve-2025-55190cve-2026-42880cycodegenerate-manifesthelmkuberneteskustomizeredisredis-cache-poisoningsynacktiv

In plain words

A serious unfixed weakness was found in Argo CD, a tool many organizations use to roll out software onto Kubernetes systems. If an attacker can reach one of Argo CD’s internal parts, they may be able to make it run their own commands and even cause the system to deploy attacker-controlled software. This matters because Argo CD often has powerful access to both software code and the systems that run it, so a single exposed internal service could put an entire Kubernetes setup at risk.

Summary
Argo CD has an unpatched repo-server flaw that can let an unauthenticated attacker with internal network access execute code and potentially take over a Kubernetes cluster. Until a fix is released, users should strictly isolate repo-server and Redis with Kubernetes network policies, especially Helm installs where those policies are off by default.

Argo CD has an unpatched flaw in its repo-server component that can let an unauthenticated attacker run code if they can reach the component’s internal network port. Synacktiv, which found the bug, says it can lead to a full Kubernetes cluster takeover. The firm says it reported the flaw to Argo CD maintainers in January 2025, but roughly eighteen months later there is still no fix and no CVE.

The flaw affects repo-server, the Argo CD component that reads Git repositories and builds Kubernetes manifests. Synacktiv found that an unauthenticated request to the repo-server’s GenerateManifest service can abuse kustomize’s --helm-command option by pointing it to a script from an attacker-controlled Git repository. When kustomize runs, it executes that script instead of the helm binary.

The article says Argo CD ships Kubernetes network policies that isolate repo-server, but the Helm chart leaves those policies disabled by default with networkPolicy.create set to false. Synacktiv showed that code execution on repo-server could be used to read the Redis password, poison Argo CD’s deployment cache, and cause Argo CD to deploy an attacker-supplied workload on the next automatic sync. Until a patch is available, the article says the defense is to enable network policies so only Argo CD’s own components can reach the repo-server and Redis ports.

  • The main risk is not only the unpatched code execution flaw, but also the default Helm chart behavior: network policies that would isolate Argo CD components are disabled by default.
  • The article highlights that Argo CD’s Redis password fix for CVE-2024-31989 can be bypassed in practice if an attacker first gains repo-server code execution and reads the password from an environment variable.
  • Administrators can quickly check whether they are exposed by running kubectl get networkpolicy -A and confirming that policies exist for repo-server and Redis.
  • Synacktiv is delaying publication of its automated attack tool, argo-cdown, specifically to give defenders time to enable network isolation before the attack becomes easier to reproduce.
  • The article points to a recurring pattern in which Argo CD internal services and low-privilege access paths have exposed high-value cluster or repository secrets across multiple recent flaws.
Key Points

  • 02Synacktiv reported the flaw to Argo CD maintainers in January 2025 and published details after roughly eighteen months because there is still no fix or CVE.
  • 03The attack abuses kustomize’s --helm-command option through the repo-server’s GenerateManifest service to execute a script from an attacker-controlled Git repository instead of the helm binary.
  • 04Synacktiv showed that code execution on the repo-server could be used to read the Redis password, poison Argo CD’s deployment cache, and cause Argo CD to deploy an attacker-supplied workload on the next automatic sync.
  • 05The article says the current defense is to enable Kubernetes network policies so only Argo CD components can reach the repo-server and Redis ports, because Helm installs leave those policies disabled by default.

In plain words

Security researchers found that several GitHub projects claiming to contain example exploit code were secretly installing a spying program called ChocoPoC through hidden Python add-ons. The people most affected are cybersecurity researchers, vulnerability testers, and penetration testers who download and run this kind of code while studying security flaws. Once installed, the program can run commands, steal browser data and files, collect system details, and send information out through online services. This matters because even code that appears useful for security work can be turned into a trap, so untrusted projects should only be tested in isolated environments.

Summary
Attackers are targeting cybersecurity researchers with GitHub proof-of-concept exploits that look legitimate but install malicious PyPI dependencies delivering the ChocoPoC RAT. Researchers should treat unverified PoCs as hostile and run them only in isolated environments.

Multiple weaponized proof-of-concept exploits on GitHub were found delivering a Python-based remote access trojan named ChocoPoC. The campaign is believed to target cybersecurity researchers who download and test PoC exploits. Sekoia identified at least seven GitHub repositories distributing ChocoPoC through exploits for vulnerabilities affecting FortiWeb, React2Shell, MongoBleed, PAN-OS, Ivanti Sentry, Check Point VPN, and Joomla SP Page Builder.

ChocoPoC stands out because the malware is not embedded directly in the exploit file. Instead, malicious Python packages are added to the PoC dependency list, with a trojanized package named frint pulling a malicious dependency named skytext from PyPI. When the PoC runs, a compiled native Python extension decrypts embedded Python code, triggers a downloader, and retrieves the final ChocoPoC payload from a Mapbox dataset.

The RAT can execute shell commands and Python code, upload files and directories, collect browser passwords, cookies, autofill data, and browsing history, search for documents and database files, gather shell history and network configuration, and enumerate running processes. Sekoia found that skytext was downloaded 2,400 times, mostly on Linux-based systems, with download surges after disclosure of a popular vulnerability. The researchers assessed with high confidence that the attacker primarily used compromised accounts to publish malicious PyPI packages and PoC repositories, and they recommend running unverified code only in isolated environments.

  • The campaign’s delivery method is harder to spot because the exploit code can remain intact while the malicious behavior is shifted into seemingly harmless dependency packages.
  • The attackers used PyPI as part of the infection chain, meaning reviewing only the GitHub repository code may not reveal the malware risk.
  • Mapbox datasets were used both to retrieve the final ChocoPoC payload and to exfiltrate data, showing that legitimate online services can be abused in multiple stages of the same campaign.
  • Download spikes after disclosure of a popular vulnerability show that newly publicized flaws can quickly become lures for researchers seeking proof-of-concept code.
  • Sekoia’s findings indicate that compromised accounts were likely used to publish the malicious packages and repositories, reducing the reliability of judging a project by account identity alone.
Key Points

  • 02ChocoPoC is delivered through malicious Python packages added to the proof-of-concept dependency list, rather than by embedding the malware directly in the exploit file.
  • 03Sekoia identified at least seven GitHub proof-of-concept repositories distributing ChocoPoC for vulnerabilities affecting FortiWeb, React2Shell, MongoBleed, PAN-OS, Ivanti Sentry, Check Point VPN, and Joomla SP Page Builder.
