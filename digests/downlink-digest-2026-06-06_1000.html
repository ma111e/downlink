Overview

The article reports that the Miasma worm campaign reached Microsoft’s Azure GitHub organizations on June 5, 2026. A malicious commit was pushed to the Azure/durabletask repository using a previously compromised contributor account. GitHub later disabled 73 repositories across four Microsoft GitHub organizations in a 105-second automated sweep.

The core significance of the incident is the attack’s move from package registry poisoning to developer-tool and AI-coding-agent hijacking. The article describes this as a shift from “execute on package install” to “execute on folder open.” In this case, cloning the repository was described as safe, while opening it in targeted tools was not.

Background: May 19 durabletask PyPI compromise

The article links the June 5 incident to an earlier compromise on May 19, 2026. In that earlier incident, three malicious versions of Microsoft’s durabletask PyPI package were uploaded during a 35-minute window. Those packages planted a credential-harvesting payload that stole secrets from AWS, Azure, GCP, Kubernetes, and more than 90 developer tool configurations.

According to the article, the attacker bypassed the repository’s CI/CD pipeline and uploaded directly to PyPI using a compromised publishing token. The June 5 incident reused the same compromised contributor account, but used GitHub source repository injection rather than package registry poisoning.

The malicious Azure/durabletask commit

The malicious commit was identified as 5f456b8 and was pushed to Azure/durabletask. The article highlights several suspicious properties in the commit metadata:

  • The commit message was “Switched DataConverter to OrchestrationContext [skip ci].”
  • The commit added five files and modified no source code files.
  • The commit timestamp was backdated to 2020-03-09T15:59:47Z.
  • The “[skip ci]” flag suppressed CI pipeline execution.

The article notes that the commit message claimed a code change, but no source code was modified. All added files were either tool configuration files or the malicious payload itself.

Planted files and targeted tools

The commit added five files intended to trigger execution across four developer tools or AI coding environments.

Claude Code

The .claude/settings.json file defined a SessionStart hook. That hook matched all sessions and executed the command:

node .github/setup.js

The article states that this caused the payload to execute automatically whenever a Claude Code session started in the repository.

Gemini CLI

The .gemini/settings.json file used the same structure as the Claude Code hook. It also triggered on Gemini CLI session start and ran the same payload.

Cursor

The .cursor/rules/setup.mdc file was described as a prompt injection. It contained an alwaysApply rule instructing the Cursor AI agent to run:

node .github/setup.js

The article says the instruction framed execution as required for project setup and IDE integration. The alwaysApply flag made the rule active regardless of which file the developer was working on.

VS Code

The .vscode/tasks.json file defined a shell task named “Setup.” Its command was:

node .github/setup.js

The task used runOptions with runOn set to folderOpen, which the article says caused it to run automatically when a developer opened the folder in VS Code.

Payload file

The shared payload was stored at .github/setup.js. The article describes it as a single-line, 4,643,745-byte obfuscated JavaScript file containing the credential harvester. All four configuration paths pointed to this file.

Disabled repositories

After the malicious commit, GitHub disabled 73 repositories across four Microsoft GitHub organizations. The article says the disabled repositories returned HTTP 403 with a Terms of Service reason when checked through the GitHub API. It also says the list was first reported by OpenSource Malware and independently verified by the article’s authors.

The block timestamps ranged from 16:00:50 to 16:02:35 UTC on June 5. The article describes two waves separated by a 56-second gap, concluding that the enforcement was automated rather than manual.

Wave 1

The first wave occurred from 16:00:50 to 16:01:28 UTC and disabled 39 repositories in 38 seconds. Repositories in this wave included Azure Functions tooling, WebJobs components, Durable Task projects, Azure-Samples projects, and MicrosoftDocs/windows-driver-docs.

Wave 2

The second wave occurred from 16:02:24 to 16:02:35 UTC and disabled 34 repositories in 11 seconds. Repositories in this wave included Azure/azure-functions-host, Azure/durabletask, Azure/azure-functions-durable-extension, language workers, extension projects, connector SDKs, and several Azure-Samples repositories.

Organizations affected

The article groups the 73 disabled repositories across four organizations:

  • Azure: 49 repositories.
  • microsoft: 10 repositories.
  • Azure-Samples: 13 repositories.
  • MicrosoftDocs: 1 repository.

The affected Azure repositories included Azure Functions runtime, worker, SDK, extension, deployment action, connector, Durable Task, and AI-related repositories. The affected microsoft repositories included Durable Task SDKs and backends, DurableFunctionsMonitor, Microsoft-Performance-Tools-Apple, and secure-azureai-agent. The affected Azure-Samples repositories included AI, OpenAI, connector, container apps, and Azure Functions samples. The affected MicrosoftDocs repository was windows-driver-docs.

CI/CD impact from Azure/functions-action

The article identifies Azure/functions-action as the most immediately damaging repository to be disabled. Azure/functions-action is described as the official GitHub Action used to deploy Azure Functions. When the repository was disabled, workflows referencing Azure/functions-action@v1 stopped resolving.

The article says a Microsoft Learn Q&A thread opened within hours, with more than 20 developers reporting broken CI/CD pipelines. Microsoft initially described the issue as a GitHub policy violation. Twelve minutes later, a revised response characterized it as an “internal management issue” under investigation and recommended alternative deployment methods such as Azure CLI, Azure DevOps Pipelines, VS Code deployment, Zip Deploy, or Azure Pipelines.

The article uses this incident to discuss the mutable-tag problem. It states that workflows referencing @v1 depend on whatever GitHub serves for that tag, and when the repository disappears, the tag evaporates and dependent pipelines fail. It recommends pinning to a specific commit SHA when the action is restored.

Relationship to the broader campaign

The article links the June 5 repository injection to the May 19 PyPI attack through the same compromised contributor account. It says the contributor’s personal fork of Azure/azure-functions-durable-extension was also blocked during the same sweep, which the authors say confirmed the account’s involvement.

The article lays out three possibilities for how the account was used again:

  • The account’s credentials may not have been fully rotated after May 19, leaving the attacker with a working GitHub token.
  • The contributor may have been re-compromised through the worm’s propagation loop after opening an infected repository in an AI coding tool.
  • A different contributor’s token may have been used with commit author metadata spoofed through the Git Data API.

The article connects the incident to the broader Miasma worm campaign, which it says has infected more than 113 GitHub repositories across dozens of accounts. It also states that the May 19 payload connected to the TeamPCP threat group through the secondary C2 domain t.m-kosche[.]com, described as known TeamPCP infrastructure.

The article further states that TeamPCP has previously targeted TanStack, Mistral AI, the @antv ecosystem, the @redhat-cloud-services npm scope, LiteLLM, Telnyx, and Checkmarx.

Recommended actions in the article

The article recommends that anyone who cloned affected repositories after June 2 and opened them in VS Code, Claude Code, Cursor, or Gemini CLI treat the system as compromised. It advises rotating all credentials accessible from the system, including GitHub tokens, npm tokens, AWS keys, Azure service principals, GCP service accounts, SSH keys, Kubernetes secrets, Docker configs, environment variables, and shell history contents.

It also recommends auditing repositories for unexpected commits containing:

  • .gemini/
  • .cursor/
  • .vscode/tasks.json
  • .github/setup.js

The article further recommends auditing npm and PyPI packages for unauthorized version publishes and checking network logs for connections to check.git-service[.]com and t.m-kosche[.]com.

For CI/CD users referencing Azure/functions-action@v1, the article recommends switching to alternative deployment methods while the repository remains disabled. It also recommends pinning to a specific commit SHA instead of a mutable tag after the action is restored.

For prevention, the article recommends inspecting cloned repositories for suspicious AI tool and editor configuration files before opening them, enabling branch protection rules requiring pull request reviews, using PyPI Trusted Publishing instead of long-lived API tokens, pinning GitHub Actions to commit SHAs, restricting outbound network access from CI/CD runners, and monitoring for PyPI or npm releases without matching GitHub tags or CI runs.

Authentication Failure and Allow-List Bypass

The message did not pass standard authentication checks. The article states that SPF returned none, DKIM returned none, DMARC returned none, and CompAuth returned none with reason=405. By the article’s assessment, the message should have been rejected or quarantined by authentication standards.

Instead, the message was delivered with SCL=-1. The reason given is that the email transited a Votiro CDR relay at 44[.]206[.]213[.]130. That relay was on the recipient organization’s allow-list, causing the gateway to trust the relay’s judgment and suppress its own authentication verdict.

The article characterizes this as an allow-list converting total authentication failure into unconditional delivery. It recommends auditing CDR relay allow-lists and evaluating whether messages passing through those relays should still undergo authentication checks rather than inheriting blanket trust.

Payload and Delivery Elements

The email contained both link-based and attachment-based elements. The primary call to action was labeled “Review Document” and routed through us[.]list-manage[.]com, a Mailchimp redirect domain. The article identifies this as part of the credential-harvesting flow.

The message also carried two attachments:

  • invite.ics, a 580-byte calendar file.
  • document.png, a 339KB file served with a JPEG content type despite having a PNG extension.

The article explains that the calendar file could create persistence because a calendar entry may remain after the email is deleted, retaining attacker-controlled content, links, and reminders. It also describes the document.png MIME mismatch as a minor evasion technique, because some attachment scanners may make filtering decisions based on declared content type rather than file signature.
