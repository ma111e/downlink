Overview

Two banking trojan campaigns documented by WatchGuard and ESET are actively targeting financial users in Latin America and Europe. 1. Grandoreiro Banking Malware Campaign

1.1 Background and Persistence

  • Grandoreiro has been active since 2016 and is designed to steal credentials from thousands of financial institutions across 45 countries and territories.
  • Despite arrests and infrastructure dismantling attempts by Brazilian authorities in early 2024, the malware has not only survived but expanded its targeting footprint.
  • The campaign was observed by WatchGuard and also aligns with an earlier Kaspersky report from October 2024.

1.2 Technical Implementation: DLL Side-Loading and WebRTC Communication

  • The campaign abuses DLL side-loading within legitimate software to execute malicious Dynamic Link Libraries (DLLs) developed in Delphi 11.
  • Two sets of DLLs are used to achieve stealthy peer-to-peer command-and-control:
    • mingwm10.dll and libwebp.dll: Incorporate the sgcWebSockets library and utilize the Session Traversal Utilities for NAT (STUN) protocol over WebRTC for communication.
    • libffi-6.dll and libpng15.dll: Leverage the Interactive Connectivity Establishment (ICE) protocol instead of STUN to achieve the same goal.
  • By packaging C2 traffic within WebRTC, the malware exploits the noisy, high-volume nature of web conferencing data that is commonly trusted by organizations, making monitoring and detection extremely difficult.

1.3 Distribution and Anti-Analysis Techniques

  • Phishing emails deliver links to a ZIP archive hosted on Mediafire.
  • Inside the ZIP, an obfuscated Visual Basic Script launches an executable that displays a fake Adobe Reader update prompt.
  • When the user clicks the prompt, the malware runs a series of anti-analysis checks (including CAPTCHA) to evade sandboxes and security researchers, and only then deploys the final banking payload.
  • This method underscores Grandoreiro’s continuous adaptation, blending benign-looking lures with technical obstacles to automated analysis.

1.4 Targeted Financial Institutions

    • Abanca

  • The malware is believed to be the successor to the CraxsRAT, CypherRAT, and SpySolr families.

Recommended Practices

Adopt Passphrases Over Complex Passwords

  • Prioritize length over complexity; multiple-word passphrases are easier to remember and much harder to crack.
  • NIST recommends allowing passwords up to 64 characters; raising the minimum length (e.g., ≥15 characters) strengthens security and reduces user error.

Block Weak and Compromised Passwords

  • Even with longer passwords, users still choose weak or common options, making them vulnerable to password spraying attacks.
  • Organizations should actively block weak password creation using custom banned-word lists (tailored dictionaries that reflect the environment) and by checking passwords against a database of over 5.4 billion known breached credentials.
  • Specops Password Policy enables these capabilities, stopping compromised passwords from being used in AD and allowing issues to be addressed proactively.
  • Preventing weak passwords at creation is more effective than remediation after compromise.

Rethink Password Expirations

  • Frequent mandatory expiration causes users to make minimal, predictable changes.
  • Move away from forced expiry unless there is evidence of compromise. When users create long, robust passwords and breached-credential detection is in place, expiration can be safely extended or eliminated.
  • Length-based aging ties expiration to password length, incentivizing stronger credentials with the reward of longer validity periods and reducing user churn.

Use a Password Manager

  • Reuse is a major day-to-day problem; even employees with good AD passwords repeat them across other systems.
  • Enterprise-approved password managers generate and store unique, long passwords for every account, removing the memory burden.
  • They also support better control over shared and privileged credentials, complementing passphrase-friendly AD policies.

Implement Self-Service Password Resets

  • Password resets are a top cause of helpdesk tickets. Strict policies increase the frequency of lockouts, filling support queues.
  • Secure self-service reset, verified through MFA or other methods, allows users to regain access quickly without raising tickets.
  • Faster recovery reduces downtime, limits risky workarounds, and improves user experience, making strict policies feel less disruptive.

Customizable Notifications

  • Users should not be surprised by sudden lockouts or last-minute expiry warnings; unanticipated disruption drives support calls.
  • Clear, timely, and informative notifications help users stay compliant and reduce friction.

Provide Dynamic Feedback at Password Creation

  • Vague error messages like “password does not meet requirements” are unhelpful.
  • Real-time, specific feedback—strength meters, banned-password checks, clear requirement prompts—makes it easy for users to understand and meet policies immediately, improving password quality.

Conclusion

The article advocates a strategy that combines passphrase-focused length policies, breached-password blocking, incentive-based expiration (length-based aging), password managers, self-service resets with MFA, and clear user communication.
