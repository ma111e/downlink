Researchers diffed vulnerable and patched versions and found that the relevant change was in mics-core within ConfigServiceController.java. The vulnerable implementation accepted a user-supplied message parameter and passed it into backend configuration handling logic.

The response returned the Linux command output inside a JSON response, confirming pre-authenticated OS command execution. Overview

Ivanti published an advisory for two vulnerabilities affecting Ivanti Sentry: CVE-2026-10520 and CVE-2026-10523. The article focuses primarily on CVE-2026-10520, which Ivanti described as an OS command injection vulnerability in Ivanti Sentry before versions R10.5.2, R10.6.2, and R10.7.1. According to the advisory text quoted in the article, CVE-2026-10520 allows a remote unauthenticated user to achieve root-level remote code execution.

The advisory also covers CVE-2026-10523, an authentication bypass vulnerability in Ivanti Sentry before versions R10.5.2, R10.6.2, and R10.7.1. The article states that this issue allows a remote unauthenticated attacker to create arbitrary administrative accounts and obtain full administrative access. Credit for CVE-2026-10523 is given to Bryan Lam, while CVE-2026-10520 is credited to an unknown party.

Product context

The article describes Ivanti Sentry, formerly MobileIron Sentry, as an in-line gateway that manages, encrypts, and secures traffic between mobile devices and backend enterprise systems. It commonly sits between corporate mobile fleets and resources such as Microsoft Exchange, controlling ActiveSync email traffic and application data. Sentry works alongside Ivanti Endpoint Manager Mobile and enforces device-level access decisions so that only compliant, registered devices can reach internal services.

The researchers note that the application is exposed under the /mics context path, while Ivanti EPMM is exposed under /mifs. They identify the visible Sentry application login path as /mics/login.jsp. Their analysis compares Ivanti/MobileIron Sentry 10.5.1, described as vulnerable, against Ivanti/MobileIron Sentry 10.5.2, described as different.

Diffing and vulnerable endpoint identification

The researchers examined mics.war and found that only one JAR changed between the vulnerable and patched versions:

  • mics-core-10.5.1-R10.5.1.jar, described as vulnerable.
  • mics-core-10.5.2-R10.5.2.jar, described as different.

The change was located in mics-core/com/mi/middleware/rest/controller/ConfigServiceController.java. The controller had a parent mapping of /api/v2/sentry/mics-config, and the changed method used a @PostMapping of /handleMessage. Combining these mappings with the /mics application context produced the target path /mics/api/v2/sentry/mics-config/handleMessage.

Vulnerable controller behavior

In the vulnerable implementation, the handleMessage controller method accepted a message string. If the value was not null or empty, it passed that user-supplied value directly to this.configService.handleMessage(message). The method returned a success response if the result was non-null and did not contain an error, otherwise it returned a bad request response.

This meant that the endpoint did not treat the message parameter as a passive string. Instead, it passed the value into the backend MICS configuration-processing system. The article follows that execution flow into ConfigServiceHandler.java.

Backend command parsing

Inside ConfigServiceHandler.handleMessage(String msg), the user-controlled message was parsed with a StringTokenizer. The first token became command, the second became module, and the third became xpath. Remaining tokens were assembled into a value string for commands including set, delete, get, add, modify, import, export, test, copy, execute, query, and migratepasswordhash.

The parsed values were then passed into another handleMessage method with the signature handleMessage(String module, String xpath, String command, String value). That method created a ConfigRequestProcessor, optionally set a module handler, and switched on the parsed command value. The EXECUTE case routed execution to reqProcessor.handleExecute(xpath, value).

Execution path and reflection

The article traces the EXECUTE path into ConfigRequestProcessor.handleExecute(). That method validated the XPath, read the XML configuration file, set handler configuration, selected XML objects for the supplied XPath and value, and then called CommonUtilities.executeNativeCommand(this.handler.getNativeModule(), tmpObjType, "Execute"). The method returned a success response containing the native command response.

The analysis then follows execution into CommonUtilities.executeNativeCommand(). That method derived method and class information from the XML object, retrieved methods such as getExecutemethod, getClassname, and getBeanclass, and invoked a module method through ReflectionUtilities.excuteModuleMethod(). The article characterizes this as a path where the parsed XML object identifies which native method should handle the request and invokes that method via reflection.

Patch clue and proof of reproduction

The patched version of the controller stopped using the user-supplied message value. Instead, it hardcoded a command string:

execute system /configuration/system/commandexec <commandexec>\n<index>1</index>\n<reqandres>/bin/cat /sys/devices/virtual/dmi/id/product_name</reqandres>\n</commandexec>

The article treats this hardcoded commandexec XML block and reqandres field as a strong indicator of the expected command structure. Using that structure, the researchers sent a POST request to /mics/api/v2/sentry/mics-config/handleMessage on a vulnerable instance. They replaced the benign /bin/cat command with uname -a in the reqandres field.

The response returned HTTP 200 and included the Linux uname -a output inside the JSON data field. The article states that this reproduced CVE-2026-10520 as a pre-authenticated OS command injection in Ivanti Sentry.

Patch and additional defensive changes

The article shows the patched implementation of ConfigServiceController, where handleMessage(String message) still receives a message parameter but does not use it. Instead, it always calls configService.handleMessage() with the hardcoded /bin/cat /sys/devices/virtual/dmi/id/product_name command. The article notes that the underlying functionality was not removed; rather, attacker control over the input passed into the command string was removed.

The researchers also found additional defensive changes outside the Java code. The bundled Apache configuration now contains regex-based rules that block access to the vulnerable endpoint entirely. According to the article, unauthenticated requests are redirected to the login page with a 302 response.

Detection artifact

The article states that the researchers are providing a Detection Artefact Generator to help teams determine whether their environment is vulnerable. The linked GitHub repository is described as watchTowr-vs-Ivanti-Sentry-RCE-CVE-2026-10520-CVE-2026-10523. The article presents it as a way for teams to check exposure to the Ivanti Sentry issues discussed.

  • The patch indicates that the vulnerable endpoint was not removed; instead, Ivanti eliminated user control over the command string and added Apache-level blocking for unauthenticated access to the endpoint.
  • A single changed JAR, mics-core-10.5.2-R10.5.2.jar, and a single changed controller method were sufficient for the researchers to trace the vulnerable code path.
  • The patched hardcoded command revealed the exact commandexec XML structure needed to reproduce the vulnerability on a vulnerable instance.
  • The endpoint treated the message parameter as an internal MICS configuration command rather than ordinary user input, allowing access to system-level configuration actions through the EXECUTE path.

Overview

Security researcher Nightmare Eclipse released a new Microsoft Defender zero-day exploit called RoguePlanet. The release occurred just hours after Microsoft fixed two previously disclosed flaws during the June 2026 Patch Tuesday updates. The article reports that the new vulnerability affects fully patched Windows 10 and Windows 11 systems.

The exploit is described as capable of spawning a Windows command prompt with SYSTEM privileges. Nightmare Eclipse published the proof-of-concept exploit on a self-hosted Git repository after saying that GitHub and GitLab repositories hosting their exploits had previously been removed by Microsoft.

Core technical issue

RoguePlanet is described by Nightmare Eclipse as a Microsoft Defender race condition vulnerability. The researcher said the exploit is not uniformly reliable across systems, writing that it achieved a 100% success rate on some machines while struggling on others.

When exploitation succeeds, the result is a command prompt running with SYSTEM privileges. The article says the flaw was reportedly tested against:

  • Windows 11 Official builds
  • Windows 11 Canary builds
  • Windows 10 systems with the June 2026 security updates installed

Third-party reproduction

Cybersecurity firm ThreatLocker told BleepingComputer that it successfully reproduced the flaw in testing. ThreatLocker said it confirmed that the exploit worked against fully patched Windows 11 systems with KB5094126 installed. The company also shared a video demonstrating the exploit.

Danny Jenkins, CEO of ThreatLocker, told BleepingComputer that the company’s initial analysis confirmed the RoguePlanet exploit is viable and performs as described. He also said that organizations using application allowlisting can prevent the exploit from executing, which he described as an effective layer of protection against the attack.

Original remote code execution scenario

Nightmare Eclipse said RoguePlanet was originally developed as a remote code execution vulnerability. According to the researcher, the original issue exploited Microsoft Defender’s handling of files hosted on remote SMB shares.

The researcher said the original exploitation path required an attacker to coerce a victim into opening a .vhd(x) file on a remote SMB server. Successful exploitation reportedly caused Defender to overwrite its own files, resulting in remote code execution.

Nightmare Eclipse also described another possible scenario involving remote code execution by coercing a victim into opening an SMB share if symlink evaluation settings were enabled. The article does not state that this scenario was completed or demonstrated in the current public release.

Reported Microsoft hardening

Nightmare Eclipse claimed Microsoft silently hardened Defender in mid-May by patching the mpengine!SysIO* API. The researcher said this blocked junction attacks.

The researcher wrote that rewriting RoguePlanet to make it functional again was difficult and that they could not complete the other scenarios. As a result, the article says it remains unclear whether RoguePlanet is limited to local privilege escalation or whether there is still a way to turn it into remote code execution.

Disclosure dispute context

The release is part of an ongoing dispute between Nightmare Eclipse and Microsoft over vulnerability disclosure and bug bounty practices. The article says the researcher has publicly released multiple Windows zero-days over the past several months.

The previously released flaws named in the article include:

  • BlueHammer
  • RedSun
  • GreenPlasma
  • YellowKey

Some of those zero-days targeted Microsoft Defender, while others targeted BitLocker and Windows components. Microsoft fixed the GreenPlasma and YellowKey flaws as part of the June 2026 Patch Tuesday updates.

Repository removals and Microsoft response

Nightmare Eclipse claimed Microsoft repeatedly targeted and removed previous exploit repositories hosted on GitHub and GitLab. The researcher said this prompted the creation of a self-hosted code platform at projectnightcrawler.dev.

The article also notes that Microsoft previously responded to the disclosures with warnings that it would work with law enforcement when people engage in “malicious activity causing real harm to our customers.” BleepingComputer said this led many in the cybersecurity community to think Microsoft was threatening the researcher.

At the time of the article, BleepingComputer had contacted Microsoft about the new RoguePlanet zero-day. The article states that BleepingComputer would update the story if it received a statement from Microsoft.

01 High-Impact Vulnerability Research and Exploit Releases

WatchTowr Labs published technical analysis of CVE-2026-10520, an OS command injection vulnerability in Ivanti Sentry versions before R10.5.2, R10.6.2, and R10.7.1. According to the analysis, the vulnerable unauthenticated endpoint is /mics/api/v2/sentry/mics-config/handleMessage, where a user-controlled message parameter is passed into backend configuration handling logic. The issue can allow a remote unauthenticated attacker to achieve root-level remote code execution.

The researchers described how the message value is parsed as a MICS configuration command into fields including command, module, XPath, and XML body. They found that the EXECUTE path could reach executeNativeCommand through reflection-based backend handling, and reproduced the vulnerability by sending a POST request containing a commandexec XML block with uname -a in the reqandres field. The HTTP response returned command output, demonstrating practical exploitation of the flaw.

The patched Ivanti implementation reportedly hardcodes the command passed to handleMessage and adds Apache configuration rules that block access to the vulnerable endpoint, redirecting unauthenticated requests to the login page. The analysis is notable because it provides endpoint-level and execution-path detail for a vulnerability that sits on an exposed management/security product boundary, where unauthenticated reachability materially increases operational risk.

BleepingComputer reported on RoguePlanet, a Microsoft Defender zero-day exploit released by researcher Nightmare Eclipse. The exploit is described as a race condition that can spawn a command prompt with SYSTEM privileges on fully patched Windows 10 and Windows 11 devices, though its reliability varies by system.
