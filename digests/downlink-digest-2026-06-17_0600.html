    Overview

    Aikido reported detecting a large-scale npm supply chain attack on June 17 targeting the entire @mastra npm scope. The article describes Mastra as a popular open-source AI agent framework. According to the article, an attacker republished 141 packages between 01:15 and 02:00 UTC and silently added a malicious dependency to every one of them.

    The affected packages included @mastra/core, which the article says has 918K weekly npm downloads. Other named affected packages included mastra and create-mastra. The article provides a long list of affected package versions in its Indicators of Compromise section.

    Malicious Dependency Injection

    The single new dependency added to the affected packages was easy-day-js. The article describes easy-day-js as a malicious clone of the popular date library dayjs. It was published by a separate attacker-controlled account a day before the @mastra scope takeover.

    A key detail in the article is the two-stage publication pattern for easy-day-js:

    • easy-day-js@1.11.21 was described as completely clean and a straight copy of dayjs with no install hooks.
    • easy-day-js@1.11.22 was published the next day, around the time the @mastra packages were republished, and added the malicious behavior.

    The compromised packages depended on ^1.11.21. The article emphasizes that npm’s caret resolution automatically pulled in the malicious 1.11.22 version as the latest version. It also notes that auditing the pinned version would reveal nothing suspicious.

    Postinstall Execution

    Version 1.11.22 of easy-day-js added a postinstall hook that ran setup.cjs. The article describes this as an obfuscated script that executed automatically at install time without user interaction.

    The stripped-down logic shown in the article performed the following actions:

    • Fetched a second-stage payload from https://23[.]254[.]164[.]92:8000/update/49890878.
    • Wrote the payload to a randomly named .js file in the operating system temporary directory.
    • Spawned the payload using the Node.js executable as a fully detached background process.
    • Passed 23[.]254[.]164[.]123:443 to the spawned payload as a second command-and-control host.
    • Used ignored standard input/output, windowsHide: true, and detached: true to run invisibly across platforms.
    • Deleted its own script file with fs.rmSync(__filename, { force: true }).

    The article highlights the self-deletion as an attempt to remove forensic evidence of the postinstall hook.

    Second-Stage Behavior

    The second stage is described as a long-lived background process. According to the article, it collected system information and targeted over 160 browser-based crypto wallet extensions. Named examples included MetaMask, Keplr, and Coinbase.

    The article states that the second stage established persistence on multiple operating systems:

    • macOS
    • Windows
    • Linux

    It also says the malware disguised itself as node-related tools on these platforms. All of the second-stage activity phoned home to 23[.]254[.]164[.]123:443.

    Similarity to the Axios Compromise

    The article compares this attack to an axios compromise that Aikido said it covered in March 2026. It states that the playbook was nearly identical in several respects.

    Shared elements described in the article included:

    • The attacker avoided changing the target package’s own code.
    • The attacker injected a malicious dependency instead.
    • The payload relied on npm’s postinstall hook to execute automatically during installation.
    • A clean decoy version was published before the malicious one.
    • The droppers self-deleted after execution.

    In the axios compromise, the article says plain-crypto-js played the role that easy-day-js played in the Mastra incident. The article also says both attacks used infrastructure with a similar pattern: the axios dropper called back to a Hostwinds VPS on port 8000, while the easy-day-js dropper called 23[.]254[.]164[.]92:8000 on Hostwinds infrastructure.

    • 23[.]254[.]164[.]123:443 — second-stage callback host passed to the spawned RAT.

    Overview

    Socket reported a coordinated npm supply chain campaign involving compromised @mastra/* packages published under the Mastra namespace. The campaign occurred on 2026-06-17, when a single human npm account, ehindero, published malicious versions of 141 Mastra packages in a tight time window between roughly 01:15 and 02:36 UTC.

    The affected packages included @mastra/core, which the article says receives more than 918,000 weekly npm downloads. Socket states that any workstation, CI runner, or build environment that installed the affected versions should be treated as potentially compromised because the payload executed during installation.

    Delivery Mechanism

    The compromised Mastra packages did not contain modified application code. According to Socket, the published @mastra/* code was byte-for-byte identical to the last legitimate build, aside from a patch version bump, routine manifest re-normalization, and one functional manifest change.

    That change was the addition of a dependency:

    "easy-day-js": "^1.11.21"

    Socket identifies easy-day-js as a typosquat of the popular dayjs library. It was published on 2026-06-16 by a separate npm account, sergey2016. Version 1.11.21 was a clean copy of dayjs without an install hook, while version 1.11.22 added a malicious postinstall script:

    "scripts": { "postinstall": "node setup.cjs --no-warnings" }

    Because the compromised Mastra packages depended on easy-day-js, installing any affected @mastra/* package could pull easy-day-js@1.11.22 and run setup.cjs automatically during npm install.

    Stage 1 Loader

    The first stage, setup.cjs, was obfuscated with obfuscator.io techniques including a string array, a custom Base64 decoder, and array rotation. Once deobfuscated, Socket described it as a compact downloader.

    The loader performed several actions:

    • Set NODE_TLS_REJECT_UNAUTHORIZED=0, disabling TLS certificate validation.
    • Contacted https://23.254.164[.]92:8000/update/49890878.
    • Wrote .pkg_history and .pkg_logs files in the temporary directory.
    • Downloaded a second-stage payload.
    • Wrote the payload to a randomized .js file in the temporary directory.
    • Launched it as a detached background process.
    • Passed 23[.]254[.]164[.]123:443 as the second-stage exfiltration target.
    • Deleted itself to limit forensic traces.

    The detached execution meant the payload could continue running after the npm install process exited.

    Stage 2 Implant

    Socket recovered and analyzed the second-stage payload, referred to as protocal.cjs. The article describes it as a roughly 41 KB cross-platform Node.js tasking client rather than a one-time stealer.

    The payload installed persistence across three operating systems:

    • Windows: created a NvmProtocal Run-key value under HKCU\...\CurrentVersion\Run, pointing to a hidden PowerShell launch mechanism, and used C:\ProgramData\NodePackages\ for dropped files.
    • macOS: created a LaunchAgent named com.nvm.protocal.plist under ~/Library/LaunchAgents/, running ~/Library/NodePackages/protocal.cjs at login.
    • Linux: created a systemd user unit named nvmconf.service, with the payload stored at ~/.config/systemd/nvmconf/protocal.cjs and configuration under ~/.config/NodePackages/config.json.

    The article notes that these persistence names are deliberately themed around Node and NVM tooling, including protocal.cjs, NodePackages, NvmProtocal, com.nvm.protocal, and nvmconf.service.

    Command-and-Control and Tasking

    The second stage sends a Start beacon and then enters a repeated Check polling loop named Cycle. Tasks returned by the server are dispatched to built-in runners, including a Node runner and a Shell runner. The implant also honors config update and exit commands.

    Socket states that this design allows the operator to push and execute arbitrary follow-on code on the host at any time. The article also notes that, although the recovered core does not itself read saved passwords or cookies, credential and secret theft can still occur through later modules delivered over this tasking channel.

    The second stage exfiltrates data to the command-and-control host passed in by the loader using the bot path /49890878.

    Built-In Collection Capabilities

    Socket identified several built-in collection functions in the recovered second-stage sample.

    Cryptocurrency Wallet Inventory

    The payload contains a hardcoded list of 166 cryptocurrency wallet browser-extension IDs. The article names MetaMask, Phantom, Coinbase Wallet, Binance Wallet, TronLink, and others as examples.

    The implant lists each browser profile’s Local Extension Settings directory, matches installed extensions against the wallet list, and exfiltrates matched extension IDs and profile paths as extInfo in the Start beacon. Socket notes that the recovered sample inventories wallet extensions but does not copy or read the wallet extensions’ stored LevelDB data.

    Browser History

    The recovered payload targets browser history from Chrome, Edge, and Brave. It copies each profile’s History database to a temporary browser-hist-… directory and reads it using Node’s built-in node:sqlite module.

    Host Reconnaissance

    The implant also gathers host reconnaissance data, including hostname, architecture, platform, user ID, installed applications, and running processes.

    Notable Technical Details

    Socket highlighted several notable details in the malware and campaign:

    • The second stage uses custom ICAP-style HTTPS POST tasking and exfiltration payloads.
    • The code includes strings such as reqmod, PrimaryUrl, SecondaryUrl, sub_net_resolve, and sub_net_splithostport.
    • Hostname resolution uses node:dns, including resolve4 and isIPv4.
    • Traffic carries a hardcoded spoofed User-Agent string: mozilla/4.0 (compatible; msie 8.0; windows nt 5.1; trident/4.0).
    • Both the loader and second stage set NODE_TLS_REJECT_UNAUTHORIZED=0, disabling TLS certificate validation.
    • An identical loader sample named a.js was first seen on public sandboxes on 2026-05-29, about 19 days before the Mastra publish.

    Detection and Response

    Socket says it flagged the malicious easy-day-js package within six minutes after publication. Because the malicious behavior was in a dependency install hook, Socket says users were protected automatically, with installs of the compromised packages flagged and blocked before the malicious hook could execute.

    The article recommends treating any affected system as potentially compromised if it installed one of the impacted @mastra/* versions. It specifically states that deleting node_modules or uninstalling the npm package is not sufficient because the second-stage payload installs login persistence outside the project directory.

    Recommended Remediation

    Socket recommends that affected organizations and users:

    • Search repositories, lockfiles, CI logs, build artifacts, developer machines, package-manager metadata, package caches, SBOMs, and artifact manifests for affected Mastra packages and easy-day-js.
    • Use npm ls easy-day-js as a fast first check on projects or runners.
    • Isolate affected developer workstations and preserve logs before remediation.
    • Suspend affected CI/CD workflow runs and review whether releases, container images, npm packages, or deployment artifacts were produced after installation.
    • Remove affected versions and remove easy-day-js entirely.
    • Replace affected @mastra/* packages with a known-clean prior version after verifying metadata and release provenance.

    • Rebuild from clean environments rather than reusing existing runners, workspaces, or node_modules.

01 Developer Supply Chain Compromise: Mastra npm Packages and easy-day-js

Multiple reports from Aikido, Socket, and The Hacker News describe a coordinated npm supply chain attack affecting more than 140 packages in the @mastra namespace. The malicious publications occurred on June 17, when a hijacked npm account belonging to a legitimate former Mastra contributor mass-published compromised versions across the scope within a short window. The affected packages reportedly did not embed malware directly, but instead added a dependency on easy-day-js, a malicious clone or typosquat of dayjs.

The easy-day-js package used a postinstall hook to execute an obfuscated setup script during installation. That script downloaded a second-stage payload from attacker-controlled infrastructure, including 23[.]254[.]164[.]92:8000, wrote it to a randomly named JavaScript file in the operating system temporary directory, spawned it as a detached background process, and deleted itself. Socket reported that the loader also disabled TLS certificate validation before retrieving the payload.

The recovered second-stage malware was described as a cross-platform Node.js tasking client with persistence mechanisms for Windows, macOS, and Linux. Its collection logic targeted host reconnaissance data, browser history from Chrome, Edge, and Brave, and more than 160 cryptocurrency wallet browser extensions. Researchers noted similarities to an earlier axios compromise, including the use of a malicious dependency, a clean decoy version preceding the malicious release, automatic postinstall execution, and self-deleting droppers.

02 AI Tooling Abuse: Malicious JetBrains Plugins and Chatbot-Monitoring Chrome Extensions

Aikido Security reported a coordinated malware campaign on the JetBrains Marketplace involving at least 15 malicious plugins masquerading as AI coding assistants. The plugins reportedly functioned as advertised while secretly exfiltrating AI provider API keys entered by users, including keys for OpenAI, SiliconFlow, and DeepSeek. The stolen keys were sent over plaintext HTTP to 39.107.60[.]51.

Two of the reported JetBrains plugins, CodeGPT AI Assistant and DeepSeek AI Assist, were listed with more than 25,000 downloads each, although the authenticity of those counts was unclear. The campaign reflects continued attacker interest in developer environments where AI service credentials may be stored, entered, or reused across workflows.

Separately, researcher Jean-Marie R. reported that two Chrome ad blocker extensions, Smart Adblocker and Adblock for Browser, captured users’ conversations with AI chatbots. The extensions reportedly provided genuine ad-blocking capability while operating an undisclosed telemetry channel that collected full AI conversation histories, model usage, and subscription tier metadata across services including ChatGPT, Claude, Gemini, Copilot, Perplexity, DeepSeek, Grok, and Meta AI.

03 Microsoft Defender RoguePlanet Zero-Day CVE-2026-50656

Microsoft confirmed it is working on a security update for a Microsoft Defender zero-day vulnerability publicly referred to as RoguePlanet and now tracked as CVE-2026-50656. The issue is described as an elevation-of-privilege flaw in the Microsoft Malware Protection Engine involving a race condition.

Researcher Nightmare Eclipse published a proof-of-concept exploit in a self-hosted Git repository and claimed the exploit affects fully patched Windows 10 and Windows 11 systems. According to the reporting, the exploit can spawn command prompts with SYSTEM privileges and may work regardless of whether Defender real-time protection is enabled. The disclosure is also tied to an ongoing dispute between the researcher and Microsoft regarding bug bounty and vulnerability disclosure practices.
