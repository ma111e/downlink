In plain words

The article explains a serious flaw in Progress Kemp LoadMaster, a product many companies use at the edge of their networks to direct traffic to their applications. If the product’s API is turned on, affected versions can let someone who has not logged in run commands on the device by sending a specially crafted request. This matters because these devices are often exposed near the internet-facing side of a company’s network, so a flaw in them can become a way for outsiders to get inside instead of keeping people out.

Summary
Progress Kemp LoadMaster versions with the API enabled are vulnerable to pre-authentication remote code execution through CVE-2026-8037, caused by an unterminated, uninitialized heap buffer in quote escaping for API credentials. Attackers can abuse heap spraying to turn the bug into command injection, so affected systems should be patched immediately.

watchTowr Labs analyzed CVE-2026-8037 in Progress Kemp LoadMaster, a load balancer and application delivery controller used at the edge of enterprise networks. Progress published a June 4 advisory describing it as a command injection remote code execution vulnerability exploitable by unauthenticated attackers when the API is enabled. The affected versions are Kemp LoadMaster GA v7.2.63.1 and older, and Kemp LoadMaster LTSF v7.2.54.17 and older.

The researchers compared vulnerable version 7.2.63.1 with 7.2.63.2 and found the relevant change in the escape_quotes() function. The vulnerable version used malloc() for an escaped string buffer and failed to add a null terminator, while the patched version used calloc() and explicitly wrote a terminating null byte. The article explains that this meant escaped input containing a single quote could become an unterminated, uninitialized heap string.

The vulnerable code is reached through the /accessv2 endpoint when validating apiuser and apipass values from a JSON request. The article shows how heap spraying with many JSON key/value pairs and an apiuser value containing four single quotes can make command construction read beyond the intended escaped buffer into attacker-controlled data. That data is then incorporated into a command executed through system(), allowing the researchers to demonstrate command execution such as reading /etc/passwd.

  • The patch did not add broader shell escaping; it fixed memory handling by zero-initializing the allocation and explicitly adding a null terminator, showing the root cause was not a simple missing quote filter.
  • The bug only appears on the code path where the input contains a single quote, because escape_quotes() returns the original pointer unchanged when no single quote is present.
  • The exploit relies on the quote-escaping logic itself to help exploitation, because four single quotes expand into sixteen bytes that can overwrite allocator metadata that would otherwise stop the out-of-bounds string read.
  • The vulnerable command construction uses system() after placing API credentials into a shell command, so a memory safety flaw in input escaping can become command execution rather than only a crash or information leak.
  • The article’s exploit approach depends on JSON parsing behavior that allows many attacker-controlled key/value pairs to be used as a heap-spraying primitive.
Key Points
  • 01Progress published a June 4 advisory for CVE-2026-8037, described as a command injection remote code execution vulnerability in Kemp LoadMaster that is exploitable by unauthenticated attackers when the API is enabled.
  • 02The affected versions listed in the article are Kemp LoadMaster GA v7.2.63.1 and older, and Kemp LoadMaster LTSF v7.2.54.17 and older.
  • 03watchTowr Labs compared version 7.2.63.1 with 7.2.63.2 and found the relevant patch changed the escape_quotes() function from using malloc() to calloc() and added a missing null terminator after the escaped output.
  • 04The vulnerable escape_quotes() function allocated an uninitialized heap buffer and failed to null-terminate the escaped string when the input contained a single quote.
  • 05The article explains that an attacker can use JSON key/value spraying and an apiuser value of four single quotes to make __sprintf_chk() read into a sprayed command injection payload that is then executed through system().

The Supreme Court ruled that police generally need a warrant before getting people’s cellphone location history from companies, including searches that ask for every phone in a certain area during a certain time. This affects phone users, law enforcement, courts, and companies that store location data such as Google, Apple, Uber, and Lyft. It matters because the ruling says people do not give up their privacy rights just by using a modern phone, and it may limit broad searches that can sweep in many people who are not suspected of a crime.

Summary
The Supreme Court ruled that police searches of cellphone location history, including geofence searches, are Fourth Amendment searches that generally require a warrant. The decision is a major digital privacy win and limits law enforcement’s ability to obtain location data from tech companies without judicial oversight.

The Supreme Court ruled that police use of cellphone location history data obtained from tech companies is a Fourth Amendment search and requires a warrant. The decision is described as a significant win for privacy and civil liberties advocates who have argued that geofence searches are unconstitutional or should be more heavily scrutinized and narrowed. Geofence searches occur when police ask tech companies to provide a list of every phone located in a particular area during a specific time frame.

The case centers on Okello Chatrie, a Virginia man charged with bank robbery in 2019 after police used Google location history records to identify him. Although police had a warrant in Chatrie’s case, a lower court ruled that a warrant was not needed at all, and Chatrie’s lawyers argued the search was unreasonable even with a warrant. The Supreme Court sent the case back to the U.S. Court of Appeals for the Fourth Circuit to decide whether the Chatrie warrant was reasonable, sufficiently narrowly tailored, and otherwise constitutional.

Justice Elena Kagan wrote that Google users regularly use Location History like a personal journal, similar to private materials such as emails, documents, photographs, or calendars. The opinion noted that location history can remain active even when a phone is not in use and can pinpoint a person within around twenty meters, record location about every two minutes, and capture elevation. The article says the ruling may affect law enforcement requests to other companies that track and store location data, including Uber, Lyft, and Apple.

  • The ruling does not end geofence searches outright; it leaves the Fourth Circuit to decide whether the specific warrant used against Okello Chatrie was sufficiently narrow and constitutional.
  • The court’s reasoning weakens the government’s reliance on the third-party doctrine by saying that using a modern cellphone does not mean a person gives up Fourth Amendment rights over data held by companies.
  • Although Google no longer stores location history, the decision still matters because other companies named in the article, including Uber, Lyft, and Apple, continue to track and store user location data.
  • The opinion emphasized the precision and persistence of location history, including that it can record location about every two minutes, pinpoint within about twenty meters, and capture elevation data showing what floor of a building someone is on.
  • Experts cited in the article said a contrary ruling could have encouraged broader police use of geofence searches and other reverse searches, such as keyword inquiries listing people who searched particular terms on Google.

Anyone who installed “Search for perplexity ai” could have had their searches, IP address, and browser details logged by the operator. It matters because popular AI names are being used to trick people into installing tools that collect private activity, even when the browser appears to work normally.

Summary
Microsoft found and reported a malicious Chrome extension impersonating Perplexity that hijacked the browser’s default search settings to log searches, address-bar typing, IP addresses, and browser details through an attacker-controlled domain before redirecting users to real results. Google removed the extension, and anyone who installed “Search for perplexity ai” should remove it and verify their search settings.

Microsoft found a malicious Chrome extension that posed as the AI search engine Perplexity and quietly logged what people searched for. The extension was called “Search for perplexity ai” and used the look-alike domain perplexity-ai[.]online to pass for the real service at perplexity.ai. Google removed it from the Chrome Web Store after Microsoft responsibly disclosed the issue.

Once installed, the extension set itself as the browser’s default search engine and routed searches through the attacker-controlled domain before redirecting users to real results from Perplexity, Google, or Bing. Microsoft said the attacker’s server logged the query along with browser headers, IP address, and user agent. The extension also sent live address-bar search suggestions to the same domain, meaning characters typed before pressing Enter were exposed.

Microsoft said it found no proof of password theft, but the extension had more access than a search box should need. It requested declarativeNetRequest permissions, included server-side code that logged every request, shipped disabled redirect rules for Google and Bing, and left room to run WebAssembly code later. Microsoft advised users who installed it to remove it and check their default search engine, while teams should allow only approved extensions and watch for changed search settings, unusual permissions, and traffic to unfamiliar domains.

  • The extension’s use of Chrome’s live search suggestion setting meant data could be collected before a user committed to a search, making partial or abandoned address-bar input exposed as well.
  • Microsoft said the server-side logging code showed the data collection was deliberate rather than an accidental result of redirecting searches.
  • Disabled redirect rules for Google and Bing showed the same collection setup could be expanded beyond the Perplexity-themed flow.
  • The extension’s unused ability to run WebAssembly stood out because Microsoft said a simple search tool has no reason to include that capability.
  • The incident shows that checking whether search results look normal is not enough, because the collection happened before users were redirected to legitimate search engines.

SpecterOps introduced Jailbreaker, a tool meant to help people test whether AI chat systems can be tricked into giving unsafe or unwanted answers. It is aimed at teams checking models such as customer-support assistants before or after changes, so they can repeat the same tests, compare results, and keep the evidence in one place. This matters because simply trying a few prompts by hand can miss problems or make them hard to prove later, while repeatable testing helps show whether a fix actually made the AI safer.

Summary
SpecterOps released Jailbreaker, an open-source platform for repeatable LLM jailbreak testing that lets operators configure models, run techniques, compare baselines, and review evidence in one workflow. Its main value is turning ad hoc prompt testing into structured experiments that can be rerun and compared after mitigations.

SpecterOps introduced Jailbreaker, a defensive LLM jailbreak testing tool created as part of its GhostWorks initiative. The article says the tool is meant to make jailbreak, prompt-injection, and agent-behavior testing repeatable instead of relying on copied prompts, screenshots, spreadsheets, and one-off scripts. Jailbreaker provides a UI for configuring targets, running techniques, tracking experiments, reviewing evidence, and comparing results across models and prompts.

The article emphasizes that manual LLM testing often becomes difficult to manage because operators must track the target, system prompt, model response, technique, baseline behavior, and evidence. Jailbreaker addresses this by treating jailbreak testing as an evaluation workflow where users can run a baseline, execute one technique, a family of techniques, or the full registry, and inspect prompts, responses, traces, judge output, and scoring details. It separates model roles into Target, Attacker, and Judge, with saved LLM profiles and a Vault for reusable profile credentials.

Jailbreaker includes a technique registry covering direct and indirect prompt injection, roleplay and instruction override, encoding and multilingual obfuscation, system prompt extraction, iterative techniques such as PAIR, TAP, Crescendo, AutoDAN, and GPTFuzz, context poisoning, and customer-service boundary tests. Its experiment workflow lets users upload prompt datasets, select technique families, run matrix-style evaluations, monitor and retry cells, export CSV or JSON, and review aggregate metrics by technique and prompt category. The article says the tool is available on GitHub and can be run from a fresh clone with Docker Compose using a frontend, backend, and PostgreSQL database.

  • Running a baseline alongside each jailbreak technique is critical because an alarming response has a different meaning if the model would have produced the same response without the jailbreak.
  • Jailbreaker is positioned less as an attack-automation framework and more as an evidence and comparison system, emphasizing repeatability, saved context, experiment history, and review over one-off prompt execution.
  • The article highlights that meaningful LLM safety testing requires tracking the full chain of context, including the unsafe intent, transformed prompt, model configuration, response, scoring, traces, and judge output.
  • The experiment workflow is especially actionable for mitigation validation because the same dataset and technique matrix can be rerun before and after changes to a system prompt, retrieval policy, or tool-use guardrail.
  • Separating target, attacker, and judge model roles lets operators reuse saved model profiles without rebuilding configurations when iterative techniques or automated scoring require multiple models.
