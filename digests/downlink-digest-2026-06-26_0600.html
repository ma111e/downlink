Google Details Turla's New STOCKSTAY Backdoor Used in Ukraine Espionage Attacks

In plain words

Google reported that Turla, a Russian state-backed hacking group, has been using a newly identified spying tool called STOCKSTAY against government and military organizations in Ukraine and groups connected to European foreign policy. The tool can collect information, capture the screen, move files, run programs, and change settings on infected Windows computers. This matters because it shows an ongoing espionage effort tied to Russia’s interests, with Ukraine and some European targets facing attempts to steal sensitive information and maintain access inside their networks.

Summary
Google says Russian state-backed Turla is using a newly documented .NET backdoor called STOCKSTAY for espionage against Ukrainian government and military targets and entities tied to European foreign policy. The malware overlaps with Turla’s Kazuar toolkit and has been delivered through phishing, malicious RDP files, MSI installers, and WinRAR exploit chains.

Google Threat Intelligence Group attributed a previously undocumented .NET backdoor called STOCKSTAY to the Russian state-sponsored threat actor Turla. The tool has been deployed against government and military organizations in Ukraine, as well as entities with an interest in Italian foreign policy. Google described STOCKSTAY as a Windows backdoor that is under continued development and shares significant code and functional overlaps with Turla’s Kazuar implant.

STOCKSTAY is a multi-component backdoor that communicates with its command-and-control infrastructure through a secure WebSocket connection and uses Windows inter-process messages between its components. Its modules include a downloader, a proxy-aware tunneler, a main backdoor for information gathering, and an orchestrator that controls configuration and command execution. Supported actions include file upload and download, directory enumeration, screen capture, process execution, Windows Registry operations, system information collection, and ZIP extraction.

Google said Turla has distributed STOCKSTAY through multiple methods, including phishing emails with malicious RDP files, MSI installers, RAR archives containing HTA scripts, and RAR archives exploiting CVE-2025-8088 in WinRAR. In some cases, payloads were hosted on GitHub or compromised WordPress sites. Google also noted architectural similarities between STOCKSTAY and Kazuar, raising the possibility that both tools may have been developed and maintained in part by the same developer or team.

  • STOCKSTAY has been used both for initial access into previously unprofiled environments and later in operations against specific hosts, indicating Turla deploys the same tool at different stages depending on what it already knows about the target network.
  • Google found a publicly accessible GitHub repository containing a Python implementation of the victim-facing STOCKSTAY WebSocket server controller, but the server’s inability to decrypt inbound messages limits visibility for platform operators and helps obscure Turla’s infrastructure.
  • The malware was originally designed to resemble a stock market data viewer before being adapted to impersonate other harmless programs such as PDF viewers and calculator utilities.
  • STOCKSTAY’s component separation mirrors Turla’s Kazuar architecture closely enough that Google assessed the tools may have been developed and maintained in part by the same developer or team.
  • Recent STOCKSTAY delivery chains reused multiple intrusion methods, including malicious RDP files, MSI installers, HTA scripts in RAR archives, compromised WordPress hosting, GitHub hosting, and exploitation of CVE-2025-8088 in WinRAR.
Key Points
  • 01Google Threat Intelligence Group attributed a previously undocumented .NET backdoor called STOCKSTAY to the Russian state-sponsored threat actor Turla.
  • 02STOCKSTAY has been deployed against government and military organizations in Ukraine and entities with an interest in Italian foreign policy.
  • 03The malware consists of multiple components, including STOCKSTAY.MARKETMAKER, STOCKSTAY.STOCKBROKER, STOCKSTAY.STOCKTRADER, and STOCKSTAY.STOCKMARKET, which communicate via Windows inter-process messages and a secure WebSocket connection.
  • 04STOCKSTAY.STOCKTRADER supports commands for file deletion and upload, directory enumeration, screen capture, process execution, Windows Registry operations, system information gathering, and ZIP extraction.
  • 05Google said STOCKSTAY shares significant code, functional, and architectural overlaps with Turla’s Kazuar backdoor, raising the possibility that both tools were developed and maintained in part by the same developer or team.

In plain words

Russian authorities used Cellebrite’s phone-unlocking tools to get data from the seized iPhone of jailed opposition activist Andrey Pivovarov in June 2021, after Cellebrite had said it stopped selling to Russia and Belarus. Investigators searched his messages and contacts for opposition groups and people connected to them, showing how one confiscated phone can expose a wider network of activists. The case matters because old tools already in police or intelligence offices can keep working even after a company cuts off sales, leaving people at risk when their devices are taken.

Summary
Citizen Lab found that Russian authorities used Cellebrite UFED tools to extract data from jailed opposition activist Andrey Pivovarov’s seized iPhone in June 2021, months after Cellebrite said it had stopped sales to Russia. The case shows that sales cutoffs can be ineffective when older offline forensic hardware remains usable by government agencies.

Russian authorities used Cellebrite’s UFED forensic tools to break into the iPhone of detained opposition activist Andrey Pivovarov in June 2021. This happened three months after Cellebrite said it would stop selling its tools and services to Russia and Belarus. Citizen Lab’s finding is based on traces found on the phone and an official Russian government report that names Cellebrite tools.

Pivovarov ran Open Russia, an opposition group the Kremlin had labeled “undesirable,” and his iPhone 12 and MacBook were confiscated after he was pulled off a flight at St. Petersburg airport on May 31, 2021. He did not consent to a search and did not provide his passwords. The Russian forensic report documented extraction of data from apps including WhatsApp, Telegram, and Viber, and searches for Open Russia Civic Movement and named opposition figures.

The article says the timing highlights a gap in Cellebrite’s sales cutoff because existing UFED hardware could keep working offline after support and updates ended. Cellebrite told Citizen Lab and Access Now that any use of its legacy hardware in Russia after March 2021 was entirely unauthorized. The MacBook extraction failed because of encryption, while the phone evidence showed how a seized device could expose political contacts and potential future targets.

  • Cellebrite’s March 2021 cutoff mainly stopped future sales and updates, but it did not prevent already-deployed offline UFED hardware in Russia from continuing to work on seized devices.
  • The case is unusually well-supported because it combines forensic traces found on the iPhone with an official Russian government report that explicitly names Cellebrite tools.
  • The failed MacBook extraction shows that device encryption and not providing passwords can still block access in at least some custody-room searches.
  • The article highlights a practical risk for activists: data taken from one seized phone can reveal contacts and names that may become useful for later targeting campaigns.
  • Cellebrite’s move toward subscription licenses that stop working after expiration is presented as a response to the weakness of older tools that could keep operating without vendor support.

Packagist is now protected by Aikido Intel and other updates to the PHP registry
aikido.dev · 26 Jun 2026 07:52 UTC openai-codex / gpt-5.5

In plain words

Packagist and Composer, the main place and tool many PHP developers use to get software packages, now have stronger protections that can stop known harmful package versions from being installed by default in Composer 2.10 and newer. This matters because attackers sometimes take over maintainer accounts or change trusted releases to include harmful code, and developers could accidentally download it before it is removed. PHP developers are directly affected, especially those using older Composer versions who need to update, while the broader stakes are safer open-source software systems that block whole types of attacks instead of reacting one bad package at a time.

Summary
Packagist and Composer are adding default supply-chain protections for PHP, including Aikido Intel-powered blocking of known malicious package versions in Composer 2.10 and mandatory stable version immutability to stop re-tagging attacks. PHP developers should update Composer to benefit from these protections.

Packagist.org and Composer, the registry and package manager behind the PHP world, have added preventative security improvements aimed at reducing malicious package installs. Aikido’s Intel feed now powers malware blocking in Composer downloads and is enabled by default for Composer 2.10 and higher. The article presents this as part of a broader shift in open-source registries toward changes that stop whole categories of attacks instead of reacting to individual bad packages.

When Aikido flags a malicious package version, the flag is included in the package metadata Composer downloads. Composer 2.10’s dependency policy framework removes flagged versions from the resolution pool, so commands such as composer update, composer require, and composer create-project will not select them, while composer install can fail if a locked version is later flagged. Users can manually override the block if they still want to download a blocked package.

Packagist has also introduced stable version immutability, which prevents a published stable version from being rewritten and is intended to end re-tagging attacks like the one described against Laravel-Lang. Private Packagist can require a current Composer client and can refuse to serve flagged version files to any client, including older Composer versions. The article says Packagist is planning additional improvements such as minimum release age, organizational ownership for mandatory MFA, and staged releases, while other ecosystems including npm, PyPI, crates.io, and RubyGems are also moving toward safer package registry practices.

  • The most important malware-blocking check is on composer install because it can stop a malicious version that was added to a lockfile before Aikido flagged it.
  • Packagist made stable version immutability mandatory, while the article says GitHub’s equivalent protection is opt-in, making Packagist’s approach stronger by default.
  • Older Composer versions remain a practical weak point because CI images and AI agents may still run versions from years ago.
  • The Aikido Intel feed is CC-BY licensed, which means Packagist’s protection is not tied to an expensive private contract and could allow other data providers to participate.
Key Points
  • 01Aikido’s Intel feed now powers malware blocking in Composer downloads and is enabled by default for Composer 2.10 and higher.
  • 02Composer uses flagged package metadata to remove malicious package versions from dependency resolution for commands such as composer update, composer require, and composer create-project, and composer install fails if a locked version is later flagged.
  • 03Packagist introduced stable version immutability so a published stable version can no longer be rewritten, preventing re-tagging attacks like the one described against Laravel-Lang.
  • 04Private Packagist can require a current Composer client to connect and can refuse to serve flagged version files to any client, including older Composer versions.
  • 05Packagist plans additional security improvements including minimum release age, organizational ownership for mandatory MFA, and staged releases.
