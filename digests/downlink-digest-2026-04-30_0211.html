The affected packages include @cap-js/sqlite, @cap-js/postgres, @cap-js/db-service, and mbt, all of which have since been deprecated. Attack Mechanism & Payload Execution

The malicious code was injected via a preinstall script that runs automatically during package installation. This script downloads the Bun JavaScript runtime from GitHub to bypass standard execution environments. The runtime then executes a heavily obfuscated execution.js payload. The payload operates as a comprehensive information stealer targeting both developer machines and CI/CD pipelines.

It extracts npm and GitHub authentication tokens, SSH keys, developer credentials, and cloud credentials for AWS, Azure, and Google Cloud. It also targets Kubernetes configurations, CI/CD pipeline secrets, and environment variables. On CI runners, the malware executes an embedded Python script to perform memory scraping. The script reads /proc//maps and /proc//mem files for the Runner.Worker process.

It extracts every secret matching the JSON pattern "key":{ "value": "...", "isSecret":true} directly from memory. This technique successfully bypasses all log masking applied by the CI platform. The memory scanning logic is structurally identical to tools used in previous Bitwarden and Checkmarx incidents.

Data Exfiltration & Dead-Drop Tactics

Collected data is encrypted and uploaded to public GitHub repositories created under the victim’s account. These repositories are labeled with the description “A Mini Shai-Hulud has Appeared,” echoing strings from prior supply-chain campaigns. The malware also implements a dynamic dead-drop mechanism using GitHub commit searches. It scans commit messages for the string “OhNoWhatsGoingOnWithGitHub:” to retrieve tokens.

The base64 portion is decoded into GitHub tokens, which are then validated against repository access permissions. This approach eliminates the need for hardcoded command-and-control infrastructure. The dead-drop strategy allows attackers to dynamically distribute and rotate access tokens across compromised environments.

Self-Propagation & Attribution Context

The deployed payload contains self-propagation code designed to spread the infection further. Using newly stolen npm or GitHub credentials, the malware modifies other packages and repositories. It injects the same malicious code into these targets to expand its footprint. Researchers have linked this campaign to the TeamPCP threat actors with medium confidence.

The group previously utilized similar code and tactics in supply-chain attacks against Trivy, Checkmarx, and Bitwarden. This pattern of memory scraping, dead-drop token retrieval, and self-propagation establishes a clear operational signature. The attack demonstrates how supply-chain compromises can rapidly escalate into widespread credential theft across enterprise CI/CD ecosystems.

The exact method by which the threat actors compromised SAP’s npm publishing process remains unconfirmed. Security Engineer Adnan Khan reported that an NPM token may have been exposed through a misconfigured CircleCI job. BleepingComputer contacted SAP for details regarding the compromise but did not receive a reply at the time of publication. The Quick Page/Post Redirect plugin, a basic WordPress utility with over 70,000 installations, was found to contain a dormant backdoor added five years ago. Austin Ginder, founder of the WordPress hosting provider Anchor, uncovered the malware after twelve sites on his network triggered security alerts.

Official versions 5.2.1 and 5.2.2, released between 2020 and 2021, contained a hidden self-update mechanism that pointed to a third-party domain. This mechanism allowed attackers to push arbitrary code directly to user sites, bypassing WordPress.org’s official distribution controls. By February 2021, the malicious self-updater was removed from the official plugin directory, but the damage had already been done.

In March 2021, sites running the compromised versions silently received a tampered 5.2.3 build from the external server. This tampered build introduced a passive backdoor that triggered exclusively for logged-out users to avoid detection by site administrators. The malicious code hooked into the WordPress content filter to fetch external data, which was used to run cloaked parasite SEO spam campaigns.

While the backdoor itself is currently dormant because the malicious subdomain no longer resolves, the underlying update vulnerability remains active across all affected installations. WordPress.org has temporarily pulled the plugin from its directory pending a security review. The investigation has not determined whether the plugin author intentionally added the backdoor or if their account was compromised by a third party.

Researchers warn that the 70,000 affected sites still maintain an update check pointing to the compromised external server. The recommended mitigation is to immediately uninstall the plugin and replace it with a clean copy of version 5.2.4 once it is restored to the official directory. Additionally, researchers have urged the party controlling the backdoor to publish a static update manifest that would force all compromised sites to automatically upgrade to the clean version.

The backdoor was originally injected into the plugin code approximately five years prior to the discovery.
