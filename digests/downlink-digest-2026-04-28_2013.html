Cybersecurity researchers have disclosed a critical vulnerability affecting GitHub.com and GitHub Enterprise Server, identified as CVE-2026-3854 with a CVSS score of 8.7. The flaw enables an authenticated user with repository push access to achieve remote code execution on the backend infrastructure using a single git push command. Google-owned cloud security firm Wiz discovered and reported the issue on March 4, 2026, prompting GitHub to validate and deploy a fix to GitHub.com within two hours. The vulnerability also received patches in GitHub Enterprise Server versions 3.14.25, 3.15.20, 3.16.16, 3.17.13, 3.18.8, 3.19.4, and 3.20.0 or later. At its core, the issue occurs because user-supplied git push option values are not adequately sanitized before being placed into the internal X-Stat header. Since the internal metadata format relies on a semicolon as a delimiter that can also appear in user input, attackers can inject additional metadata fields through crafted push option values. The exploit chain requires chaining three specific injections: injecting a non-production rails_env value to bypass the sandbox, injecting a custom_hooks_dir value to redirect the hook directory, and injecting a crafted repo_pre_receive_hooks entry to trigger path traversal and execute arbitrary commands as the git user. Once unsandboxed code execution is achieved, attackers gain full control over the GHES instance, including filesystem read/write access and visibility into internal service configuration. On GitHub.com, an enterprise mode flag normally defaults to false, which keeps the custom hooks path inactive, but because this flag is also passed in the X-Stat header, it can be injected using the same mechanism to enable code execution. Due to GitHub’s multi-tenant architecture and shared backend infrastructure, obtaining code execution on GitHub.com enables cross-tenant exposure, potentially allowing an attacker to read millions of repositories on the shared storage node regardless of organization or user boundaries. There is currently no evidence that the vulnerability has been exploited in a malicious context prior to disclosure. Approximately 88% of instances were vulnerable at the time of public disclosure, and the remote code execution chain targets shared storage nodes. GitHub’s Chief Information Security Officer emphasized that the flaw demonstrates how assumptions about data in shared internal protocols become critical attack surfaces when multiple services pass data across different languages. Teams building multi-service architectures are encouraged to audit how user-controlled input flows through internal protocols, especially where security-critical configuration is derived from shared data formats. Users are advised to apply the update immediately for optimal protection.

  • The flaw impacts GitHub.com, GitHub Enterprise Cloud, GitHub Enterprise Cloud with Data Residency, GitHub Enterprise Cloud with Enterprise Managed Users, and GitHub Enterprise Server.
  • It allows an authenticated user with push access to a repository to achieve remote code execution (RCE) on the backend instance using a single git push command.
  • Google-owned cloud security firm Wiz discovered and reported the issue on March 4, 2026.

Technical Mechanism

  • The vulnerability is a command injection flaw rooted in inadequate sanitization of user-supplied git push option values.
  • These unsanitized values are incorporated into the internal X-Stat header during push operations.
  • The internal metadata format uses a semicolon as a delimiter character, which can also legitimately appear in user input.
  • This overlap allows attackers to inject additional metadata fields through crafted push option values, breaking the expected header structure.

Exploit Chain

  • Achieving RCE requires chaining three specific injections:
    • Inject a non-production rails_env value to bypass the sandbox environment.
    • Inject a custom_hooks_dir value to redirect the hook execution directory.
    • Inject a crafted repo_pre_receive_hooks entry that triggers path traversal to execute arbitrary commands as the git user.
  • Once the sandbox is bypassed and hooks are redirected, the attacker gains unsandboxed code execution as the git user.
  • This level of access provides full control over the GHES instance, including filesystem read/write access and visibility into internal service configuration.

Impact & Scope

  • Approximately 88% of instances were vulnerable at the time of public disclosure.
  • The RCE chain targets shared storage nodes, amplifying the potential blast radius.
  • On GitHub.com, an enterprise mode flag normally defaults to false, which keeps the custom hooks path inactive. However, because this flag is also passed in the X-Stat header, it is equally injectable, enabling code execution on GitHub.com as well.
  • GitHub’s multi-tenant architecture and shared backend infrastructure mean that obtaining code execution on GitHub.com enables cross-tenant exposure.
  • This cross-tenant exposure effectively allows an attacker to read millions of repositories on the shared storage node, irrespective of the organization or user.
  • There is no evidence that the issue was ever exploited in a malicious context prior to disclosure.

Remediation & Response

  • GitHub validated the report and deployed a fix to GitHub.com within two hours of the March 4, 2026 disclosure.
  • Patches are available for GitHub Enterprise Server versions 3.14.25, 3.15.20, 3.16.16, 3.17.13, 3.18.8, 3.19.4, 3.20.0, or later.
  • Users are strongly advised to apply the update immediately for optimal protection.
