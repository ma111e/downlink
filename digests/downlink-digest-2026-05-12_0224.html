Executive Summary and Threat Landscape

Active Directory Certificate Services (AD CS) functions as the backbone of enterprise public key infrastructure, managing digital certificates that authenticate users, devices, and services across corporate networks. Despite its critical role, AD CS is frequently undermined by insecure default configurations, overly permissive enrollment rights, and complex template management that security teams hesitate to modify. These structural weaknesses have established AD CS as a primary vector for privilege escalation and unauthorized identity impersonation in modern environments. Unlike traditional vulnerability exploitation, AD CS attacks rarely depend on zero-day exploits or malware execution. Instead, adversaries systematically misuse native certificate issuance mechanisms to impersonate privileged accounts and establish long-term persistence.

Attack Lifecycle and Operational Phases

The exploitation of AD CS follows a structured five-phase lifecycle that transforms a single compromised account into enterprise-wide dominance. The process begins with initial access through phishing or credential theft, followed by comprehensive discovery of certificate authorities, template permissions, and account keys. During the exploitation phase, attackers submit certificate requests against misconfigured templates to impersonate high-privileged accounts. These certificates are then utilized for Kerberos authentication, enabling privilege escalation and lateral movement across the network. The final phase focuses on persistence, where attackers leverage shadow credentials and key trust mechanisms to maintain access that survives password resets and certificate revocations.

Core Exploitation Techniques

Certificate template misuse represents the most consistently observed escalation method, particularly through the ESC1 technique. ESC1 exploits templates configured with low-privileged enrollment rights, disabled manager approval, and the ENROLLEE_SUPPLIES_SUBJECT flag that allows requesters to define certificate subjects. This configuration enables attackers to mint authentication certificates for privileged accounts they should not control. Shadow credential abuse operates through the msDS-KeyCredentialLink attribute, which stores cryptographic keys for Windows Hello for Business and smartcard authentication. Attackers manipulate this attribute to register their own public keys against target accounts, creating passwordless authentication paths that bypass traditional credential rotation defenses. Key techniques include:

  • ESC1 template exploitation for privilege escalation
  • Shadow credential registration for persistent access
  • PKINIT authentication for Kerberos ticket requests
  • LDAP enumeration for template and attribute discovery

Adversarial Tooling and Ecosystem

A rapidly expanding toolkit of open-source utilities has democratized AD CS exploitation across the threat landscape. Certify and Certipy serve as primary enumeration and exploitation frameworks for identifying vulnerable templates and executing ESC-style attacks. PKINITtools extends this capability into Kerberos authentication by leveraging certificates to request Ticket Granting Tickets. Whisker and pyWhisker specialize in shadow credential manipulation, directly modifying the msDS-KeyCredentialLink attribute to establish persistent access. The widespread availability of these tools has significantly lowered the technical expertise required to execute sophisticated AD CS attacks.

Detection Strategies and Telemetry Correlation

Effective defense against AD CS abuse requires behavioral analytics that correlate multiple event types across authentication and directory services. Security teams must monitor specific Windows Event IDs, including 4886 and 4887 for certificate requests and issuance, 4898 for template loading, and 5136 for directory object modifications. Correlating these events with Kerberos authentication requests (4768⁄4769) reveals attempts to leverage stolen or shadow credentials for lateral movement. LDAP reconnaissance activity serves as a critical early warning indicator, with attackers executing high-volume queries targeting pKICertificateTemplate, PKI-Enrollment-Service, and msDS-KeyCredentialLink attributes. Monitoring for these specific LDAP queries, combined with tracking unusual enrollment patterns and SAN usage, enables defenders to identify adversarial activity before privilege escalation occurs.

Overview

GhostLock is a proof-of-concept security tool created by Kim Dvash of Israel Aerospace Industries. It demonstrates how attackers can abuse the Windows CreateFileW API to block file access on local systems and SMB network shares. The technique relies on manipulating the dwShareMode parameter to enforce exclusive file locks.

Technical Mechanism

The attack operates by calling the CreateFileW function with the dwShareMode parameter set to zero. This configuration grants the initiating process exclusive access to the file. Windows subsequently returns a STATUS_SHARING_VIOLATION error to any other process attempting to open the locked file. The researcher provided a code example showing how to open a finance.xlsx file in exclusive mode using this method.

Execution and Impact

The GhostLock tool automates this process by recursively opening a large number of files across SMB shares. It requires only standard domain user privileges to function, eliminating the need for elevated access. Attackers can scale the disruption by launching the tool from multiple compromised devices simultaneously. The tool is designed to continuously reacquire file handles as terminated processes are replaced, maintaining the lock. Once the SMB session ends, the processes are killed, or the system reboots, Windows automatically closes the handles and restores normal access. The researcher classifies the impact as disruption-based rather than destructive, drawing a parallel to operational downtime windows rather than ransomware data loss.

Detection Challenges

Traditional security products and behavioral detection systems typically focus on mass file writes or encryption operations. GhostLock evades these controls by generating large volumes of legitimate file open requests. The attack lacks the typical telemetry signatures found in Windows event logs, EDR data, or network flow records. The only reliable observable for detection is the per-session open-file count with ShareAccess equal to zero at the file server layer. This metric resides within storage platform management interfaces rather than standard endpoint or network monitoring tools.

Mitigation and Resources

Defenders can use the researcher’s whitepaper to implement detection templates. The document includes SIEM queries and NDR detection rules designed to identify the abnormal file locking patterns. The tool and technical documentation are publicly available on GitHub and Zenodo. Security teams should monitor file server layer metrics to identify unauthorized exclusive file locks.
