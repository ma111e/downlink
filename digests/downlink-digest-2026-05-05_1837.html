Vulnerability Overview

The Apache Software Foundation has released critical security updates for the HTTP Server, addressing a severe flaw in HTTP/2 protocol handling. Tracked as CVE-2026-23918, the vulnerability carries a CVSS score of 8.8 and is officially described as a double-free condition with potential remote code execution implications. The issue affects Apache HTTP Server 2.4.66 and has been patched in version 2.4.67. Bartlomiej Dmitruk and Stanislaw Strzalkowski are credited with discovering and reporting the flaw to the foundation.

Technical Mechanism

The vulnerability resides in the mod_http2 module, specifically within the stream cleanup path of the h2_mplx.c file. It triggers when a client sends an HTTP/2 HEADERS frame immediately followed by a RST_STREAM frame containing a non-zero error code. This sequence occurs before the multiplexer has registered the stream, causing two nghttp2 callbacks to fire sequentially. Both the on_frame_recv_cb and on_stream_close_cb callbacks ultimately invoke the h2_mplx_c1_client_rst function. This function incorrectly pushes the same h2_stream pointer onto the spurge cleanup array twice. When the c1_purge_streams function later iterates through the array, it calls h2_stream_destroy and apr_pool_destroy on each entry. The second call attempts to free memory that has already been deallocated, resulting in a double-free error.

Exploitation Paths

The denial-of-service impact is trivial to achieve and requires only a single TCP connection and two HTTP/2 frames. This attack works on any default deployment utilizing mod_http2 with a multi-threaded MPM. Attackers can sustain the DoS by continuously sending the malicious frame pattern, causing Apache to repeatedly respawn crashed workers and drop all subsequent requests. Remote code execution becomes possible when the server uses the Apache Portable Runtime mmap allocator, which is the default configuration on Debian-derived systems and official Apache Docker images. Researchers successfully developed a working proof of concept on x86_64 architectures. The exploit chain places a fake h2_stream struct at a freed virtual address through mmap reuse. It then points the pool cleanup function to system() and uses the Apache scoreboard memory as a stable container for the fake structures and command strings. The scoreboard remains at a fixed address for the server’s lifetime, bypassing ASLR protections and making the exploitation practical. While practical exploitation requires an info leak for system() and scoreboard offsets, the heap spray is probabilistic but highly effective in lab conditions.

Impact and Mitigation

The MPM prefork module is completely unaffected by this vulnerability. However, the attack surface remains large because mod_http2 ships in default builds and HTTP/2 is widely enabled in production environments. Security teams are strongly advised to apply the latest fixes immediately to mitigate both the denial-of-service and potential remote code execution risks.

Incident Overview

A 23-year-old university student was arrested for halting four high-speed trains on April 5. The suspect, referred to by his surname Lin, used software-defined radio equipment to transmit a high-priority General Alarm signal. This action triggered emergency braking procedures that delayed trains for 48 minutes. The Taiwan high-speed railway network serves 81.8 million passengers annually along a 350-kilometer coastal route. Trains on this state-funded line can reach speeds of up to 300 kilometers per hour.

Technical Execution and Vulnerabilities

The attack targeted the TETRA communication system, which had been in operation for 19 years without parameter rotation. Lin intercepted and decoded the radio parameters using software-defined radio equipment purchased online. He then programmed these parameters into handheld radios to impersonate legitimate network beacons. A 21-year-old accomplice provided critical THSR parameters that enabled the unauthorized cloning process. Because the authentication parameters remained static, the attacker successfully bypassed seven verification layers. The system failed to detect the spoofed signal until THSR logs flagged a transmission from an unassigned beacon.

Investigation and Seizure

THSR investigators initially checked the network logs and noted the signal originated from an unassigned radio beacon. After confirming the physical device was not missing, they concluded the scenario pointed to unauthorized cloning. Police were immediately alerted and combined CCTV footage with TETRA network logs to trace the source. This investigation led officers to the suspect’s residence, where they seized the following equipment:

  • 11 handheld radios
  • An SDR device
  • A laptop

Lin was formally arrested on April 28 following the equipment seizure.

Legal and Political Fallout

The suspect is currently out on NT$100,000 bail while facing charges under Article 184 of the Criminal Law. This statute carries a maximum penalty of 10 years of imprisonment. His legal counsel claimed the transmission of the emergency signal on April 5 was accidental. Authorities have dismissed this claim as unconvincing. The incident has sparked criticism from Taiwanese politicians, who have publicly blamed responsible oversight bodies for systemic negligence.
