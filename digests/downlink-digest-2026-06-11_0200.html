Overview

The article argues that MCP and APIs are complementary interfaces rather than competing technologies. Both expose the same underlying capabilities, but they are built for different consumers and different operational patterns. APIs serve developers and scripts, while MCP serves AI agents, analysts, and natural-language-driven investigations.

The author’s main position is that security teams should run both. APIs remain the right choice for deterministic automation, while MCP is positioned as the right choice for Agentic SOC workflows and human analysts who want to ask questions instead of writing code.

Core distinction between MCP and APIs

The article defines an API as a contract for code. A user sends a structured request and receives a structured response, making it predictable and deterministic. This makes APIs useful when the workflow is known in advance and needs to behave consistently.

MCP is defined as a contract for AI agents. It exposes platform capabilities as tools that a large language model can discover, reason about, and call in response to natural language. According to the article, MCP is what allows an AI assistant to reach into security data, run a hunt, pivot on an indicator of compromise, or otherwise use the platform’s capabilities rather than merely chat about them.

The article emphasizes that the plumbing can be the same, but the consumer is different. APIs are for developers and scripts, while MCP is for agentic workflow tools, AI assistants, and analysts working through natural language.

When the article says to use an API

The article recommends using APIs for deterministic, scripted automation that a developer controls line by line. APIs are best when the user knows exactly what needs to be called, what data should come back, and how the system should behave.

Examples in the article include:

  • Wiring a SIEM alert into a ticketing system.
  • Pulling alert data into a daily report.
  • Building custom enrichment that runs the same way every time.

The API path is described as powerful but slower to implement. It requires reading documentation, setting up authentication, mapping fields, handling pagination, catching errors, and troubleshooting HTTP issues. The article says even a “quick” API integration can take a few days of work.

When the article says to use MCP

The article recommends MCP for Agentic SOC workflows and for humans in the loop who want to work in natural language. MCP is presented as useful when the analyst or agent is thinking in questions rather than endpoints.

The article lists several places where this pattern appears:

  • Purpose-built Agentic SOC platforms such as Torq and Tines, which are described as moving beyond static playbooks into LLM-driven investigations.
  • General workflow builders such as n8n, which can let security teams build their own agentic flows on top of MCP.
  • Human analysts using AI assistants such as Claude, ChatGPT, or other MCP-compatible clients.

An example question in the article is: “has this hash been seen anywhere in the last 90 days, and was it touched by a privileged account?” The point is that the user does not need to read API documentation or manually construct endpoint calls; they ask a question and expect an answer.

Time-to-value argument

A major claim in the article is that MCP can shorten time-to-first-answer dramatically. API integrations are described as taking days because of documentation, authentication, data mapping, pagination, and error handling. MCP, by contrast, is described as requiring the user to connect the server once and then ask a question.

The article says this speed comes from offloading workflow decisions to the LLM. The LLM figures out which tool to call, chains results, and formats the output. The user describes the desired outcome rather than pre-building a workflow.

The article gives an example where a question such as “investigate this alert and tell me if it looks like lateral movement” might lead to different tool-call sequences depending on what the data reveals. One investigation might involve five tool calls, while another might involve three. The workflow is assembled per question rather than designed in advance.

Agentic SOC and playbook direction

The article connects MCP to the concept of the Agentic SOC. It describes a direction where an LLM selects a relevant playbook based on a specific alert type and then uses that playbook to guide which MCP tools to use. This keeps MCP’s adaptability while adding more structure so the agent is not starting from scratch every time.

The article presents this as a future direction for agentic investigation. The workflow remains intent-driven, but the agent works from a pattern that fits the case in front of it.

Skills and operational requirements

The article contrasts the skills needed for each interface. To use an API well, the user needs familiarity with HTTP, authentication flows, Swagger or OpenAPI, GraphQL, and an automation platform. The user must also be comfortable reading documentation and troubleshooting HTTP errors.

To use MCP, the article says the user needs the endpoint to connect it to the environment. After that, the key skill is writing a clear question and judging whether the answer makes sense. The article describes this as shifting the bottleneck from engineering capacity to analytical judgment.

Security caveat around MCP

The article includes a warning that MCP convenience comes with security risk. An MCP server gives an agent credentialed reach into the security stack. As a result, MCP tokens and scopes become part of the attack surface.

The article states that Mitiga has observed MCP tokens being stolen in the wild. It advises teams to connect fast but treat MCP access like privileged access. This caveat is presented as an important operational consideration when adopting MCP.

Mitiga’s position

Mitiga says it ships both API and MCP side-by-side. The API is positioned for scripted, deterministic automation, while MCP is positioned for analysts and Agentic SOC platforms that drive live investigations through natural language.

The article’s conclusion is that the future of security tooling is multi-protocol by default. Teams should stop comparing MCP and APIs as if one replaces the other, and instead choose the right interface per use case, persona, and workflow.

This digest centers on the operational and security implications of Model Context Protocol (MCP) adoption in security environments. The reporting frames MCP not as a replacement for APIs, but as a parallel access model designed for AI agents and analyst-driven investigation workflows, with new implications for credential exposure and access control.

01 MCP and API Roles in Security Automation

Mitiga’s analysis distinguishes APIs and MCP by their intended consumers. APIs are described as contracts for code, best suited to deterministic automation where developers know which endpoints to call and require predictable behavior. MCP, by contrast, is presented as a contract for AI agents that need to discover, select, and chain tools during live investigative tasks.

The article positions APIs as better aligned with scripted workflows, integrations, and repeatable automation. MCP is framed as more useful for analysts, AI assistants, and Agentic SOC platforms operating through natural language prompts, where the agent determines which tools to use and how to combine outputs during an investigation.

02 Agentic SOC Workflows and Emerging Attack Surface

A key reported benefit of MCP is reduced time-to-first-answer in security investigations. The article states that MCP can allow an LLM-driven system to choose tools, chain results, and format outputs without requiring a pre-built workflow, potentially compressing early investigative steps into minutes.

The same access model also expands the security surface of AI-enabled SOC operations. Mitiga notes that MCP servers provide agents with credentialed access into security environments, making MCP tokens, scopes, and server permissions part of the effective attack surface. In this framing, MCP adoption introduces identity and authorization considerations similar to API security, but applied to autonomous or semi-autonomous agent activity.

The dominant theme is the coexistence of deterministic automation and agentic investigation models. APIs remain central for predictable machine-to-machine execution, while MCP is presented as an emerging interface layer for AI-assisted security operations that can accelerate analysis while concentrating privileged access paths around agent tooling.
