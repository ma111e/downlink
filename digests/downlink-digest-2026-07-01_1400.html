In plain words

Hackers tried more than 81 million times to sign in to Microsoft 365 accounts over two weeks, using username and password pairs exposed in earlier breaches. Huntress said 78 accounts at 64 organizations were compromised, often because extra login checks were not set up to cover the sign-in method the hackers used. This matters because even companies that thought they had added stronger account protection could still be exposed if those protections did not apply everywhere.

Summary
A large password-spraying campaign hit Microsoft 365 tenants with over 81 million login attempts, compromising 78 accounts across 64 organizations. Attackers abused Azure CLI and the ROPC OAuth flow to bypass MFA where Conditional Access policies were misconfigured or incomplete.

An aggressive password-spraying campaign targeting Microsoft 365 environments generated more than 81 million login attempts over a two-week period. Huntress observed the activity against its customers between June 12 and 26 and confirmed that 78 Microsoft accounts were compromised across 64 organizations. The attacker used still-valid username and password combinations that had been exposed in past breaches.

The threat actor attempted to authenticate through Microsoft’s Azure command-line interface and then used the ROPC OAuth mechanism when a valid username and password pair was found. Huntress said many impacted organizations had multi-factor authentication configured through Conditional Access policies, but those policies did not cover the specific flow the attackers used. ROPC sends the password directly to the token endpoint and does not provide an interactive MFA prompt.

Huntress highlighted several policy weaknesses, including MFA applied only to specific applications, enforced only for selected user groups, required only from untrusted locations, or left in report-only mode. In some impacted cases, there was no MFA policy at all. Huntress also reported a more than 155-fold increase in password-spraying attacks and said the activity originated from an IPv6 range owned by LSHIY LLC, which had not responded to its abuse report at the time of publication.

  • MFA coverage can fail if Conditional Access policies are scoped only to certain apps, groups, or locations rather than applying to All Cloud Apps.
  • ROPC is especially risky in this campaign because it sends passwords directly to the token endpoint without an interactive MFA prompt.
  • Policies set to report-only mode did not protect organizations because they were not actually enforced.
  • Traffic appearing to come from trusted locations allowed some attackers to avoid MFA requirements when policies only required MFA from untrusted locations.
Key Points

  • 02The threat actor attempted authentication through Microsoft's Azure command-line interface using still-valid username and password combinations exposed in past breaches.
  • 03Huntress observed the campaign between June 12 and 26 and confirmed that 78 Microsoft accounts were compromised across 64 organizations.

  • 05Huntress reported that the activity originated from an IPv6 range owned by LSHIY LLC and said it had not received a response after reporting the findings through the company's abuse portal.

In plain words

Researchers found two serious flaws in Cursor, an AI tool used by developers to write code, that could let hidden instructions in something the tool reads cause commands to run on the developer’s computer. Anyone using a Cursor version before 3.0 is affected, and the company has already released a fix. This matters because Cursor says more than half of the Fortune 500 use the tool, and a successful attack could give control over a developer’s machine and any cloud or online workspaces the editor is signed into.

Summary
Two critical Cursor flaws dubbed DuneSlide could let a hidden prompt injection escape the editor’s sandbox and run arbitrary commands on a developer’s machine without user approval. The bugs affect all versions before Cursor 3.0, so users should update immediately.

Cato AI Labs found two critical flaws in Cursor, an AI code editor, that could let a single ordinary-looking prompt escape the editor’s sandbox and run commands on a developer’s computer. The flaws are named DuneSlide and tracked as CVE-2026-50548 and CVE-2026-50549, both rated 9.8 out of 10 under CVSS 3.x. Every Cursor version before 3.0 is affected, and Cursor 3.0, released April 2, includes fixes.

The attack works through prompt injection, where hidden instructions are planted in something the Cursor agent reads, such as a connected service through the Model Context Protocol or a page returned by a web search. The article says the attack needs no click or approval from the user, making it “zero-click.” Both flaws use the same basic approach: get the agent to write a file it should not be allowed to write, then use that write to turn off the sandbox.

CVE-2026-50548 abuses Cursor’s handling of the working_directory parameter for terminal commands, while CVE-2026-50549 abuses a fallback in Cursor’s symlink safety check. In both cases, the attacker can target files such as the macOS Cursor sandbox helper, after which later commands run without the sandbox. The article says there is no sign the flaws have been used in real attacks, and it presents DuneSlide as part of a broader pattern of Cursor bugs where poisoned prompts led to code execution.

  • The attack path is especially actionable because it does not require the victim to approve a prompt or click anything; the malicious instructions can be hidden in material the agent reads on the user’s behalf.
  • The sandbox meant to protect users could be disabled by a single unauthorized file write, showing that its security depended heavily on preventing writes to specific helper or startup files.
  • Cursor initially rejected the reports because it said its threat model did not cover misuse of MCP servers, but later reopened the reports and shipped fixes after Cato escalated.
  • DuneSlide follows several earlier Cursor issues where poisoned prompts led to code execution, indicating that the new sandbox in the 2.x line was a response to prior flaws but did not fully solve the problem.
  • There is no sign the flaws were used in real attacks, so the article presents the issue as research and coordinated patching rather than an active campaign.

  • 02Both vulnerabilities affect every Cursor version before 3.0, and Cursor patched them in version 3.0 released on April 2.

The attack begins with a phishing PDF disguised as a corrupted file and uses checks to make sure visitors are in Spain or Portugal before delivering malware. The delivery chain hides the payload inside an image that looks like a PDF icon but contains a ZIP file, then deletes the image, ZIP, and script after running. Ousaban adds a Windows startup registry entry named Financeiro and uses a command-server setup that includes a decoy Pastebin link while the real server changes daily.   • The Pastebin-linked command server address embedded in Ousaban is a decoy, so defenders should not treat it as the malware’s actual control endpoint.
  • The campaign combines older Brazilian banking-trojan behavior with newer delivery protections, including country filtering, hidden payloads, and throwaway daily infrastructure.
