Overview

A flaw in the Google Cloud Vertex AI SDK for Python allowed an attacker to hijack a victim’s machine learning model upload and execute code inside Google’s serving infrastructure. The article says the attacker did not need access to the victim’s Google Cloud project, credentials, phishing, or an existing foothold. According to Palo Alto Networks Unit 42, the attacker needed only a Google Cloud project of their own and the victim’s project ID, which is often publicly visible.

Unit 42 discovered and reported the bug through Google’s bug bounty program and referred to the technique as “Pickle in the Middle.” Google said it saw no exploitation in the wild. The article states that Google has patched the issue and advises SDK users to update to version 1.148.0 or later.

Root Cause

The flaw was in the Python SDK’s logic for selecting a temporary Cloud Storage bucket during model uploads.

If the user did not explicitly set a bucket, the SDK generated a predictable default bucket name using the project ID and region, with an example format like:

  • project-vertex-staging-region

The article says the SDK verified only whether the bucket existed, but did not verify whether the bucket was owned by the victim. Since Google Cloud Storage bucket names are globally unique, an attacker could create the expected bucket name in their own project before the victim did.

That meant the victim’s SDK could upload the model files into an attacker-controlled bucket without realizing it.

Attack Chain

The article describes the attack flow as follows:

  1. The attacker learns the victim’s project ID.
  2. The attacker predicts the default staging bucket name based on the project ID and region.
  3. The attacker creates that bucket first in their own Google Cloud project.
  4. The victim uploads a model through the Vertex AI SDK without setting staging_bucket.
  5. The SDK finds the attacker’s bucket and uploads the model there.
  6. The attacker replaces the uploaded model with a malicious one before Vertex AI reads it.
  7. Vertex AI later loads the swapped model, triggering code execution inside the serving container.

The article notes that many Python machine learning models are serialized using pickle or joblib, both of which can execute code when deserialized. That made model substitution particularly dangerous in this workflow.

Timing and Proof of Concept

Unit 42 said the exploit depended on timing. In its measurements, there were about 2.5 seconds between the victim’s upload and Vertex AI reading the file.

In the proof of concept described in the article:

  • The attacker used a Cloud Function triggered after the upload.
  • That function replaced the model in 1.4 seconds.
  • This was fast enough to win the race before Vertex AI loaded the file.

The article presents this timing detail as an important practical component of the attack’s success.

Impact Described by Unit 42

According to the article, the malicious payload stole an OAuth token from the serving container’s metadata server and sent it to the attacker.

In Unit 42’s test environment, that token was not limited to the single compromised deployment. The article says it could access:

  • Other model artifacts in the same Google-managed tenant project
  • A full TensorFlow model with trained weights
  • BigQuery metadata
  • Access lists
  • Tenant logs
  • GKE cluster names
  • Internal container image paths

The article frames this as a significant cross-tenant exposure path because code execution in the serving container led to access beyond the original victim model deployment.

Conditions Required for the Attack

The article says the technique worked only under specific conditions:

  • The victim’s default staging bucket in that region did not already exist
  • The victim left the staging_bucket parameter unset

It adds that the first condition is common for a new Vertex AI project in a region. The second condition depends on the developer relying on the SDK’s default behavior instead of explicitly choosing a bucket they control.

Versions, Disclosure, and Fix Timeline

Unit 42 reported the flaw through Google’s Vulnerability Reward Program on March 5, 2026. The researchers tested versions 1.139.0 and 1.140.0, which were the latest available at the time, and found both vulnerable.

The article describes Google’s remediation in two stages:

Initial Fix

Google released v1.144.0 on March 31.

This version:

  • Added a random uuid4 to the bucket name

Completed Fix

Google released v1.148.0 on April 15.

This version:

  • Added bucket ownership verification in Model.upload()
  • Blocked the bucket-squatting scenario described in the report

The article notes that, at the time of publication, neither Unit 42 nor Google’s Vertex AI security bulletins listed a CVE for this issue.

Guidance to Users

The article recommends two concrete actions:

  • Update to 1.148.0 or later so the ownership check is active
  • Set an explicit staging_bucket pointing to a Cloud Storage location the user controls

It also emphasizes that the vulnerable logic lives in the client SDK, not just in a backend service. Because of that, users should verify the google-cloud-aiplatform version anywhere it runs, including:

  • Notebooks
  • CI jobs
  • Training pipelines
  • Production services

Broader Context

The article says this is the second predictable-bucket-name flaw to surface in Vertex AI this year.

It references a separate issue, CVE-2026-2473, which Google patched in February. That bug affected Vertex AI Experiments and also involved bucket squatting that enabled:

  • Cross-tenant code execution
  • Model theft
  • Model poisoning

The article also mentions Unit 42’s earlier work on Vertex AI’s default service-agent permissions, which traced a related path from a deployed AI agent into customer and tenant data.

  • Extract browser artifacts including cookies, browsing history, saved credentials, preferences, and local-state encryption keys
  • Traverse directories and select files using configurable rules
  • Read and exfiltrate file contents
  • Capture screenshots and display information
  • Execute shell commands or external processes and collect output
  • Transfer data back to command-and-control infrastructure
  • Use native Windows APIs for process interaction, memory operations, DPAPI access, Restart Manager behavior, and advanced file access

A second BabaDeda-related chain drops a ZIP archive that uses DLL side-loading to launch DanaBot and SectopRAT (also called ArechClient). The article emphasizes the use of a staged loader component named Storage Crypter, which reads payload material from external storage-like files such as List.Control.dat.

Morphisec said this design keeps the visible application package looking legitimate while hiding the malicious payload in external containers that are decoded only shortly before execution. According to the article, that approach minimizes forensic visibility, complicates automated analysis, and reduces opportunities for traditional security tools to detect the activity before it runs.

Lorem Ipsum Loader Campaign

Delivery shift to compromised WordPress sites

BlueVoyant reported an active ClickFix campaign that uses at least five compromised WordPress sites as the starting point to deliver Lorem Ipsum Loader and a backdoor of the same ecosystem. The hacked sites span sectors including:

  • Architecture
  • Legal services
  • Construction technology

The article contrasts this with earlier opportunistic campaigns that used trojanized Microsoft Teams installers distributed via fake download portals promoted through SEO poisoning and malvertising. It says the loader is believed to have been active in the wild since February 2026.

Why the delivery model changed

The article links this change in initial access to Microsoft’s disruption of Fox Tempest (also known as Forging Marauder). Fox Tempest is described as a threat actor that advertised a malware-signing-as-a-service operation, helping malware operators deliver payloads using fraudulently signed Microsoft Trusted Signing certificates.

BlueVoyant said the loss of certificate supply made the previously used signed-installer model unworkable, forcing the operators to move to a delivery mechanism that removes the need for code signing. The article presents this as a concrete example of how threat actors can quickly adapt after defensive disruption.

Attribution and operator profile

The article states that the Lorem Ipsum ecosystem has been attributed with high confidence to Vanilla Tempest, also known as Rapid Brigantine, Vice Society, and Vice Spider. It describes the actor as financially motivated and says the group is known for deploying ransomware families including:

  • Rhysida
  • BlackCat
  • Zeppelin
  • Quantum Locker

Technical execution chain

The Lorem Ipsum delivery sequence uses fake Microsoft Edge security update lures in a ClickFix format. Victims are induced to run a malicious command that downloads:

  • A ZIP archive
  • An outdated Node.js release from 2017 (version 7.10.1)

The article says the old Node.js version is used to execute JavaScript-based payloads from the archive while lowering the chance of detection.

The JavaScript acts as a dropper that deploys more malware, including a batch script that establishes persistence and launches a DLL side-loading chain. That chain executes a malicious DLL such as:

  • mscoree.dll
  • msvcp140.dll

That DLL then decodes the embedded Lorem Ipsum Loader payload.

Backdoor retrieval and follow-on activity

BlueVoyant said Lorem Ipsum Loader is designed to fetch the next-stage Lorem Ipsum Backdoor from command-and-control infrastructure whose location is obtained from attacker-controlled profiles on social networking platforms. The backdoor then provides the capability to run further payloads delivered by the C2 server.

The article says the chain ultimately hands off to Rapid Brigantine’s established post-exploitation tooling and then to its documented ransomware deployments, primarily Rhysida.

Potemkin, RMMProject, and EtherRAT Campaign

Initial infection chain

The third ClickFix campaign discussed in the article involves an MSI package that drops a previously undocumented loader named Potemkin using an HTML Application (HTA) payload. Huntress said Potemkin serves as a conduit for:

  • EtherRAT
  • RMMProject

RMMProject is described as a Lua-scriptable DLL containing modules for:

  • Remote screen control
  • Browser credential theft
  • Bypassing Chromium’s App-Bound Encryption (ABE) protections

RMMProject capabilities

The article says RMMProject includes a task dispatcher that can:

  • Run a file or process
  • Take screenshots
  • Steal browser autofill data
  • Execute arbitrary Lua scripts
  • Terminate browser processes
  • Download and run an additional module from a URL at runtime

Potemkin technical details

Huntress described Potemkin as a custom x64 loader that:

  • Uses a domain generation algorithm (DGA) to locate C2 infrastructure
  • Reflectively loads follow-on modules in memory

The article includes several implementation details:

  • A built-in 1,000-word dictionary is used for DGA-driven C2 discovery
  • A unique UUID identifying the victim is written to %LOCALAPPDATA%\hyper-v.ver
  • The loader supports task polling, DLL retrieval, and DLL execution
  • A custom byte cipher protects both C2 traffic and the DGA dictionary

Post-compromise activity

After access was established, the article says the unidentified threat actor conducted hands-on-keyboard operations that included:

  • Configuring Microsoft Defender exclusions
  • Deploying Chisel reverse SOCKS tunnels
  • Conducting additional reconnaissance
  • Setting up a Cloudflare tunnel for persistent access
  • Moving laterally using WMIExec and SMBExec
  • Reaching the domain controller
  • Propagating EtherRAT across more than 11 hosts

This section makes clear that the Potemkin campaign was not limited to simple malware execution on a single endpoint, but extended into broader network compromise and propagation.

Why ClickFix Remains Effective

The article concludes that ClickFix is still highly effective because it exploits predictable user behavior. Huntress is quoted as saying that people naturally follow instructions that look authoritative, such as “press Win+R, paste this, hit Enter,” especially when those instructions resemble routine troubleshooting steps.

The article says the social engineering does not need to be highly sophisticated to succeed. In this framing, the effectiveness comes from making the action appear normal and urgent rather than from technical novelty.

Broader ClickFix Activity Mentioned

The article also notes that ClickFix has been used recently against both Windows and macOS users. It cites examples including:

  • Phexia Stealer, a macOS infostealer
  • HellsUchecker, a backdoor delivered through EtherHiding
  • Fake Claude MSI installers that run PowerShell payloads

These examples are used to reinforce the point that ClickFix is an enduring and adaptable delivery mechanism across operating systems and lure themes.

Defensive Response Mentioned

The article highlights a response from Apple related to the broader risk of malicious command pasting. It says Apple introduced a new security pop-up in macOS Tahoe 26.4 that warns users who attempt to paste commands into Terminal from websites, chat agents, messaging apps, or email.

Apple is quoted as saying that scammers use those channels to trick people into pasting malicious commands that can damage the Mac or compromise privacy, and that the alert is intended to help users avoid running unexpected commands.

Significance

Several broader implications are explicit in the article:

  • Loader ecosystems are becoming more modular, splitting delivery, storage, execution, and payload deployment into separate components.
  • Threat actors adapt quickly when older delivery paths are disrupted, as shown by the shift from signed installers to ClickFix lures in the Lorem Ipsum ecosystem.
  • ClickFix remains durable because it relies on user compliance with simple, plausible instructions rather than requiring exploit chains.
  • Post-compromise impact can be extensive, particularly in the Potemkin case where the intrusion progressed to lateral movement and domain-level access.

EXEC Executive Overview

This digest highlights three converging themes: cloud and AI supply-chain weakness, continued success of user-driven malware delivery, and abuse of trusted consumer platforms for large-scale distribution. Across the reporting, attackers repeatedly relied on legitimacy by design—predictable cloud resource naming, familiar update prompts, and established content ecosystems—to move malicious payloads closer to victims.

A notable pattern is the blending of low-friction initial access with flexible post-exploitation tooling. Whether through a hijacked machine learning upload path, social engineering via ClickFix prompts, or malicious Steam Workshop wallpapers, the common thread is adversary use of expected workflows to conceal code execution and credential theft.

01 Cloud & AI Supply-Chain Exposure

A flaw in the Google Cloud Vertex AI SDK for Python showed how predictable infrastructure conventions can create a supply-chain style compromise path for machine learning workflows. According to Unit 42, an attacker needed only a Google Cloud project and the victim’s project ID to pre-create the default staging bucket name that the SDK would use, because the bucket name was derived predictably from project ID and region and the SDK verified only bucket existence rather than ownership.

In the demonstrated attack path, a victim’s model upload could be redirected into an attacker-controlled bucket, then swapped for a malicious pickle or joblib artifact before Vertex AI loaded it. Unit 42 reported an approximately 2.5-second gap between upload and model read, with its proof of concept replacing the file in 1.4 seconds via a Cloud Function, creating a realistic code-execution window during model loading. Google reportedly addressed the issue in stages by introducing random uuid4 bucket naming in version 1.144.0 and ownership checks in version 1.148.0.

02 Social Engineering Loaders and User-Executed Malware Delivery

ClickFix activity continues to expand as a practical malware distribution model, with multiple campaigns now tied to distinct loader families. Reporting cited BabaDeda Loader, Lorem Ipsum Loader, and Potemkin as the latest tooling associated with fake verification or update prompts that trick users into pasting and executing attacker-supplied commands.

Morphisec observed an April 2026 chain targeting education and financial organizations, where BabaDeda Loader was used to deliver information stealers and remote access trojans using hidden PowerShell, in-memory shellcode, DLL side-loading, and externally hosted payloads. Separate reporting from BlueVoyant linked compromised WordPress sites hosting ClickFix lures to Lorem Ipsum Loader and attributed the activity with high confidence to Vanilla Tempest, while Huntress documented an MSI-driven chain in which an HTA payload dropped Potemkin and ultimately deployed EtherRAT and RMMProject.

The campaigns underscore how ClickFix has matured from a simple social engineering trick into a repeatable malware-delivery ecosystem. The use of multiple loaders and varied hosting infrastructure suggests a broad operational model that can support different payload types, operators, and victim sectors while preserving the same core user-execution mechanic.

03 Trusted Platforms Abused for Malware Distribution

Kaspersky reported that threat actors abused Steam Workshop through the Wallpaper Engine application to distribute malicious wallpaper packages, extending malware delivery into a mainstream gaming ecosystem. The abuse centered on Wallpaper Engine’s support for application wallpapers, which are executable Windows applications capable of running as animated desktop backgrounds.

Researchers found dozens of malicious wallpapers with download counts in the thousands or tens of thousands, indicating meaningful reach and suggesting the activity has been ongoing since at least late 2025. In one analyzed sample masquerading as the game NTRaholic, the wallpaper functioned as expected while also installing the DarkKomet backdoor and a malicious AggregatorHost.dll that searched for Steam accounts and stole credentials.

Steam reportedly removed the malicious wallpaper applications identified by Kaspersky, but the case illustrates how user-generated content pipelines can become malware channels when executable formats are embedded in normal platform functionality. For defenders, this places a consumer software distribution vector alongside enterprise cloud and web-based lures in the same broader pattern of adversaries hiding malicious execution behind trusted interfaces.
