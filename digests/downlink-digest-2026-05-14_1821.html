Vulnerability Overview and Discovery

An 18-year-old security flaw in the NGINX open-source web server has been identified as CVE-2026-42945. The vulnerability received a critical severity rating of 9.2 based on the latest Common Vulnerability Scoring System metrics. Researchers at the AI-native security firm DepthFirst AI discovered the issue using an autonomous scanning system. The flaw resides in the ngx_http_rewrite_module and impacts NGINX versions ranging from 0.6.27 through 1.30.0.

Technical Mechanism and Exploitation

The vulnerability is triggered when NGINX configurations employ both the rewrite and set directives. This configuration pattern is frequently utilized in API gateways and reverse proxy setups. The root cause involves inconsistent state handling within NGINX’s internal script engine, which processes rewrites across two distinct passes. The first pass calculates the required memory allocation, while the second pass copies the actual data. An is_args flag incorrectly remains set after a rewrite containing a question mark. This causes the engine to calculate buffer sizes using unescaped URI lengths while subsequently writing larger escaped data containing characters like plus signs and ampersands. The resulting heap buffer overflow can corrupt adjacent NGINX memory pool structures and overwrite cleanup handler pointers.

Researchers demonstrated unauthenticated code execution by crafting HTTP requests that spray fake structures into memory via POST bodies. This technique forces NGINX to execute system commands during the memory pool cleanup phase. However, the published proof-of-concept required Address Space Layout Randomization to be disabled. ASLR is enabled by default but can be turned off to improve performance in embedded systems or virtual machines. NGINX’s multi-process architecture inadvertently aids attackers because crashed worker processes spawn new instances with identical memory layouts. This design allows for reliable heap manipulation and repeated exploitation attempts without memory layout shifts.

Additional Vulnerabilities and Impact Scope

During the same six-hour scanning session, researchers uncovered three additional memory corruption issues. The secondary flaws include:

  • CVE-2026-42946: Excessive memory allocation in SCGI/UWSGI modules that can crash workers through allocations approaching one terabyte.
  • CVE-2026-40701: A use-after-free flaw in asynchronous OCSP DNS resolution handling.
  • CVE-2026-42934: An off-by-one UTF-8 parsing bug that causes out-of-bounds reads. All three secondary flaws received a medium severity rating. NGINX powers approximately one-third of the top-ranked websites globally and is widely deployed by cloud providers, financial institutions, and Kubernetes clusters. F5, which owns and maintains the platform, issued a security advisory detailing the affected builds. The impact spans NGINX Open Source versions 0.6.27 through 1.30.0, NGINX Plus R32 through R36, Instance Manager 2.16.0 through 2.21.1, and various WAF and DoS modules. The vulnerabilities were discovered on April 18, 2026, and reported to the vendor four days later.

Remediation and Real-World Exploitability

Patches have been released in NGINX Open Source 1.31.0 and 1.30.1, alongside NGINX Plus R36 P4 and R32 P6. Organizations unable to upgrade immediately should replace unnamed PCRE capture groups in vulnerable rewrite rules with named captures. This configuration change eliminates the primary exploitation prerequisite for the heap buffer overflow. Independent analysts from AlmaLinux and security researcher Kevin Beaumont have pushed back on claims of reliable real-world remote code execution. They note that the proof-of-concept relies on highly specific conditions and deliberately disabled ASLR. Beaumont emphasized that the exploit was tested against a vulnerable setup and does not guarantee reliable code execution on hardened systems.

AlmaLinux confirmed that crashing NGINX worker processes via crafted requests is trivial and highly reliable. However, they cautioned that transforming the heap overflow into dependable remote code execution on ASLR-enabled systems is not straightforward. Despite the difficulty of achieving reliable RCE, the straightforward denial-of-service potential remains a significant threat. Security operators are advised to treat the issue as urgent and apply patches or configuration mitigations immediately.
