Vulnerabilities Identified

The six vulnerabilities listed in the article are:

  • CVE-2026-44289, with a CVSS score of 7.5, described as denial of service through unbounded protobuf recursion.
  • CVE-2026-44290, with a CVSS score of 7.5, described as process-wide denial of service when loading schemas with unsafe option paths.
  • CVE-2026-44291, with a CVSS score of 8.1, described as a code generation gadget after prototype pollution.
  • CVE-2026-44292, with a CVSS score of 5.3, described as prototype injection in generated message constructors.
  • CVE-2026-44294, with a CVSS score of 5.3, described as denial of service from crafted field names in generated code.
  • CVE-2026-44295, with a CVSS score of 8.7, described as code injection in pbjs static output from crafted schema names.

Root Cause Described by Cyera

Cyera said the vulnerabilities stem from protobuf.js handling schema and metadata as trusted by default. The article says this validation oversight could influence application behavior and lead to code execution.

Morag noted that exploitation generally requires specific conditions. However, the article quotes him as saying those conditions are increasingly common in data and AI ecosystems that exchange data, schemas, and configuration files across services, repositories, cloud platforms, and third-party integrations.

Technical Detail on CVE-2026-44291

The article identifies CVE-2026-44291 as the most severe of the disclosed issues. It says the flaw can result in code execution when a Node.js application accepts attacker-controlled input.

Security researcher Vladimir Tokarev explained that the attacker-controlled input reaches a prototype pollution gadget. Later, the same process uses protobuf.js to encode or decode a message. Because protobuf.js resolves type names through plain property lookups, a polluted Object.prototype can make an attacker-controlled string appear to be a valid protobuf primitive.

Tokarev said protobuf.js then inserts that string into a generated encoder or decoder function and compiles it with Function(). According to the article, this gives the attacker arbitrary JavaScript execution inside the Node.js process.

The article describes a scenario in which a malicious protobuf schema could be introduced into CI/CD workflows. In that case, CVE-2026-44295 could poison CI/CD workflows and leak build secrets.

The article also gives an example involving Node.js services such as WhatsApp bots built using Baileys, a WhatsApp Web API automation TypeScript library. It says those services could be crashed through a specially crafted message by exploiting CVE-2026-44292.

Overview

SpecterOps presents a technical walkthrough of using GEPA, or Genetic-Pareto optimization, to make LLM prompt engineering for security agents measurable. The article is tied to the company’s GhostWorks initiative for AI security and focuses on a practical proof-of-concept rather than a product release. The central problem is that prompt changes for agents are often judged by subjective impressions rather than by repeatable measurements.

The author applies GEPA to a simple web CTF agent and attempts to improve the agent’s performance by optimizing its AGENTS.md prompt. The goal is to find and submit flags in CTF web challenges using the agent’s available tools. The article emphasizes that the optimization is performed through prompt refinement, not model training.

What GEPA Is Used For

GEPA is described as an optimization framework positioned by its authors as an alternative to reinforcement learning techniques such as GRPO. The article says GEPA was primarily created for prompt refinement, although it can be applied to other optimization tasks. The framework takes an existing prompt and uses an LLM to produce refined versions that are evaluated against measurable goals.

The author highlights three practical advantages of GEPA:

  • It can improve LLM efficiency on closed-source and remote-hosted models because it only modifies prompts.
  • It requires relatively little compute because it relies on inference and evaluation rather than training.
  • GEPA already has a Python framework, GEPA-ai, which makes implementation straightforward.

The article also warns that GEPA runs can consume many tokens during optimization, even though the framework includes controls to reduce token usage.

CTF Agent Scenario

The proof-of-concept centers on a web CTF agent specialized in solving web challenges. The article assumes a simple ReAct-style agent with several common tools:

  • Shell execution.
  • Reading and writing files.
  • Making web requests.
  • A submit_flag tool that validates whether a submitted flag is correct.

The starting AGENTS.md prompt is intentionally basic. It tells the agent that it is an expert CTF player specializing in web CTF challenges, must find and submit the flag, should explain its reasoning before and after commands, cannot ask for help, and should provide a final writeup after solving the challenge. The flag format is described as usually being csictf{…}.

The challenges used in the proof-of-concept come from the CSI CTF challenge repository. Each challenge is hosted in a Docker container, which makes the lab environment easy to deploy.

Defining What “Good” Means

The article argues that GEPA requires explicit measurement criteria and a clear optimization goal. For the CTF agent, the goal is: find the flag in the CTF challenge and submit it using the submit_flag tool.

The author evaluates each run using three questions:

  • Whether the challenge was solved.
  • How many agent turns were needed.
  • How long the agent took.

The reward function returns 0.0 if the challenge is not solved. If the challenge is solved, it calculates a duration score and a turns score, then combines them with 25% weight on duration and 75% weight on turns. This makes turn efficiency more important than elapsed time in the proof-of-concept.

The article gives examples showing that a failed run scores zero regardless of time or turns, while solved runs receive higher scores when they use fewer turns and less time. The author cautions that reward functions should be designed carefully because GEPA uses the score heavily to steer prompt optimization.

Actionable Side Information

A core part of the GEPA process is actionable side information, abbreviated as ASI in the article. The paper calls this natural language reflection, while the GEPA-ai optimize_anything framework calls it actionable side information.

In the CTF example, ASI includes useful textual information from each lab run, such as:

  • Agent reasoning output.
  • Tools used by the agent.
  • Tool parameters.
  • Tool return values.
  • Errors encountered during tool usage.

The article explains that ASI lets the optimizer go beyond a numerical score. It can identify what worked, what failed, and what should be adjusted in future prompt candidates.

For example, a run might show that the agent quickly identified SQL injection and extracted the flag, but took too many turns to get SQL syntax correct and misidentified MySQL as Postgres. Feeding these traces into the optimizer can encourage useful behavior while adding prompt guidance for verifying SQL syntax and database type.

Batch Evaluation and Candidate Filtering

The article explains that GEPA uses minibatches to manage evaluation cost. In the example, a pool of 10 CTF challenges is split into 7 training challenges and 3 validation challenges. If the minibatch size is set to 3, GEPA selects 3 of the 7 training challenges for a round of execution.

Each selected challenge is run, ideally in parallel. After the runs finish, the score and ASI for each run are combined and passed to the optimizer LLM. The optimizer receives the optimization goal, the current prompt, and the evaluation results, then produces a complete replacement prompt.

The new candidate prompt is then evaluated on the same set of minibatch challenges. If its scores decrease or stay the same, the candidate is discarded. If it improves the minibatch score, it is also evaluated against the validation set.

The article says this filtering helps avoid wasting tokens on prompts that do not show measurable improvement.

Pareto Frontier Selection

The article describes the Pareto frontier as a way to keep strong candidates for multi-objective problems. It first illustrates the idea with hypothetical C2 implants graded on stealth and stability, then applies the concept to CTF prompt performance across different vulnerability types.

In the CTF setting, different prompts may perform better on different challenge categories, such as:

  • SQL injection.
  • Deserialization.
  • Remote file include.

Rather than choosing only one prompt as universally best, GEPA keeps candidates that are strongest across different parts of the score matrix. Prompts are selected stochastically, weighted by the number of challenge scores where a prompt performs best. This means prompts that perform well across more challenges are more likely to be selected, while specialist prompts that excel on fewer challenges can still remain in the process.

The article describes two optimization mechanisms:

  • Reflective Mutation, which uses a selected parent prompt and ASI.
  • System-Aware Merge, which combines two Pareto candidates that excel on different challenges.

The walkthrough focuses on Reflective Mutation because it is enabled by default in the optimize_anything framework.

Implementation Details

The article includes Python code for a GEPATrainer class using the GEPA-ai optimize_anything API. The code defines a list of 10 CTF challenge targets with difficulty labels, splits them into training and validation sets, configures GEPA, and evaluates candidate prompts.

Important configuration details include:

  • max_metric_calls, set in the class constructor and demonstrated with 250 metric calls.
  • Parallel evaluation with configurable workers.
  • Evaluation caching.
  • A reflection minibatch size of 3.
  • An EpochShuffledBatchSampler using a random seed.

The _evaluate method runs a CTF challenge with the candidate prompt, collects runtime statistics and context, calculates the score, and returns both the score and ASI. The returned ASI includes whether the challenge was solved, run statistics, and context.

The article states that GPT-5.5-Cyber is used for both evaluation and optimization to avoid guardrails impeding progress.

Results

After 250 metric calls, the GEPA run produced a matrix of prompt candidates and scores across three validation challenges. The article says the latest GEPA-ai release includes a tree view that helps visualize the optimization process.

GEPA selected prompt 11 as the strongest candidate. The resulting optimized prompt is much more detailed than the initial prompt and includes specific instructions for web CTF methodology, including:

  • Initial reconnaissance with curl.
  • Inspecting headers, redirects, cookies, comments, forms, hidden inputs, scripts, and assets.
  • Searching fetched responses for flag-like strings.
  • Handling cookies, headers, tokens, and JWTs.
  • Testing file and path behavior for local file inclusion and path traversal.
  • Handling encodings such as URL encoding, base64, hex, ROT13, HTML entities, JSON escapes, XOR, Caesar shifts, and reversed strings.
  • Fetching explicitly referenced external resources such as GitHub files when the challenge points to them.
  • Performing focused route discovery.
  • Using compact Bash and Python snippets for low-volume checks.

The prompt also includes challenge-specific heuristics learned from prior failures, including instructions for file-library-style challenges and CCC/JWT/GitHub-style challenges.

Measured Impact

The article compares the baseline first run with the refined prompt selected by GEPA. The baseline scores were:

  • Challenge 1: 0.9172955506046614.
  • Challenge 2: 0.8092837869127592.
  • Challenge 3: 0.8670885186394055.

The optimized prompt’s scores were:

  • Challenge 1: 0.9121042403578759.
  • Challenge 2: 0.8430549861987432.
  • Challenge 3: 0.9175024194518726.

The author notes that Challenge 1 decreased by about 0.005, or 0.5%, while Challenge 2 improved by about 0.04, or 4%, and Challenge 3 improved by about 0.05, or 5%.

Limitations and Lessons

The article emphasizes that the proof-of-concept used a very small training and validation set. The optimized prompt includes highly specific instructions tied to particular lab challenges, which the author identifies as overfitting. One example is guidance related to /getFile?file=a.cpp returning code that runs cat flag.txt, which is clearly tied to a specific challenge pattern.

The author states that using more labs would give GEPA more opportunity to generalize and reduce this type of overfitting. The article also notes that the reward function used in the proof-of-concept is naive because it gives high scores for simply solving the challenge and heavily penalizes unsolved runs.

The final lesson is that GEPA does not eliminate standard machine learning concerns. Dataset selection, lab variety, and mature reward functions remain key to achieving useful improvements.

  • GEPA can discard candidate prompts before validation if they do not improve scores on the same minibatch, which helps avoid spending additional tokens on prompts that show no measurable gain.
  • The quality of actionable side information is critical because the optimizer can use both high-scoring and low-scoring traces to preserve useful behaviors while explicitly avoiding observed failure patterns.
  • Pareto-based selection keeps specialist prompts in the optimization process even when they are weaker overall, allowing lessons from prompts that perform well on specific challenge types to influence later candidates.
  • The proof-of-concept showed that prompt optimization can introduce challenge-specific overfitting, such as adding instructions tailored to a particular file-library challenge, when the training and validation set is too small.
  • Reward-function design directly steers GEPA’s improvements, so naive scoring that heavily rewards solved challenges and turn efficiency can shape the optimized prompt toward those measured behaviors rather than broader generalization.
Key Points
  • The article describes GEPA as an optimization framework that refines prompts using LLM inference rather than reinforcement learning, making it applicable to closed-source and remote-hosted models with relatively little compute.
  • The author demonstrates GEPA by optimizing an AGENTS.md prompt for a web CTF agent that uses tools such as shell execution, file access, web requests, and a submit_flag tool.
  • The evaluation function scores only solved challenges and weights turn efficiency more heavily than duration, using a formula with 75% weight on turns and 25% on time.
  • GEPA uses actionable side information from agent runs, including reasoning output, tool usage, return values, and errors, to help the optimizer identify success patterns and failure modes.
  • In the proof-of-concept run, the optimized prompt slightly reduced the score on one challenge but improved two others, while also showing signs of overfitting due to the small training and validation set.

01 Proto6 Vulnerabilities in protobuf.js and Node.js Supply Chain Exposure

Researchers at Cyera disclosed six vulnerabilities collectively named Proto6 in protobuf.js, a widely used JavaScript implementation for Protocol Buffers. The issues can expose affected Node.js applications to remote code execution and denial-of-service conditions, particularly where applications deserialize Protobuf data or generate code from schemas.

The flaws center on protobuf.js treating schema and metadata as trusted by default. According to the reporting, this behavior can allow attacker-controlled schema-related input to influence application behavior and, in certain paths, reach code execution conditions. The most severe issue highlighted is CVE-2026-44291, which can allow arbitrary JavaScript execution when attacker-controlled input reaches a prototype pollution gadget before protobuf.js encodes or decodes a message.

The affected exposure surface extends beyond direct protobuf.js use. The article identifies Google Cloud client libraries, messaging frameworks such as Baileys, Node.js services handling Protobuf data, and CI/CD pipelines that generate code from schemas as potentially impacted environments. Fixes are reported in protobufjs 7.5.6 and 8.0.2, along with protobufjs-cli 1.2.1 and 2.0.2.
