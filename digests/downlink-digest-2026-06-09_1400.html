    During Windows setup, users can now choose a custom name for the user folder on the Device Name page. If this step is skipped, Windows uses the default folder name and continues setup as usual.

CVE-2026-45586: Windows Collaborative Translation Framework elevation of privilege

CVE-2026-45586 affects the Windows Collaborative Translation Framework, also identified in the article as CTFMON. Microsoft says the vulnerability involves improper link resolution before file access, also described as “link following.”

The flaw allows an authorized attacker to elevate privileges locally. The article states that the vulnerability grants SYSTEM privileges. Microsoft credited the issue to an anonymous researcher but did not share details about how it was disclosed.

CVE-2026-49160: HTTP.sys denial of service and HTTP/2 Bomb

CVE-2026-49160 is an HTTP.sys denial-of-service vulnerability. The article says it is the publicly disclosed “HTTP/2 Bomb” issue disclosed by researchers at the offensive security firm Calif.

Microsoft describes the issue as uncontrolled resource consumption in HTTP/2 that allows an unauthorized attacker to deny service over a network. The article explains that HTTP/2 Bomb abuses how HTTP/2 compresses and manages web traffic headers. Attackers can send very small amounts of data that force servers to allocate disproportionately large amounts of memory.

The article further states that attackers can keep memory tied up by manipulating flow-control settings. This can prevent the server from freeing resources and may cause performance issues or outages.

Microsoft introduced a new MaxHeadersCount registry setting in response. This setting limits the number of headers accepted in HTTP/2 and HTTP/3 requests by the HTTP server. Microsoft also published a support bulletin about controlling the maximum number of HTTP/2 and HTTP/3 request headers in Windows clients and servers.

The flaw was attributed to Quang Luong and Codex of Calif.io.

CVE-2026-50507: Windows BitLocker security feature bypass

CVE-2026-50507 is a Windows BitLocker security feature bypass vulnerability. Microsoft says the issue involves a protection mechanism failure that allows an unauthorized attacker to bypass a security feature through a physical attack.

Microsoft attributed the vulnerability to an anonymous researcher. However, BleepingComputer reports that it learned this patch fixes the YellowKey vulnerability publicly disclosed the previous month by a cybersecurity researcher named Nightmare Eclipse.

The article says YellowKey could be exploited by placing specially crafted files on a USB drive or EFI partition and booting into the Windows Recovery Environment, or WinRE. Holding down the CTRL key in that environment triggered a command shell with unrestricted access to encrypted BitLocker-protected drives.

The flaw primarily affects systems using TPM-only BitLocker protection on Windows 11 and Windows Server 2022⁄2025 devices. Microsoft had previously shared temporary mitigations, including enabling TPM+PIN authentication instead of relying only on TPM protection.

The article also notes that Nightmare Eclipse has released several Windows zero-day vulnerabilities, including BlueHammer, MiniPlasma, RedSun, and UnDefend. These releases were described as a protest of Microsoft’s handling of its bug bounty and vulnerability disclosure programs.

The malware does not generally encrypt or encode strings. Instead, it constructs strings at runtime byte-by-byte on the stack.

The article highlights that this becomes more difficult to analyze when combined with MBA and CFF. For example, a C2 domain such as cwrtwright[.]com may be constructed through multiple functions and stitched together, while the flattened state machine fragments the string-building logic. ThreatLabz states that this defeats tools such as FLOSS that look for consecutive characters in memory.

API resolution and indirect system calls

MLTBackdoor resolves Win32 APIs, system calls, and Beacon Object File symbols at runtime using DJB2 hashing. ThreatLabz observed three hashing patterns:

  • Normal WinAPI lookups, such as djb2("WinHttpConnect").
  • Lowercase API name lookups, such as djb2("enumwindows").
  • Beacon-prefixed lookups, such as djb2("BeaconNtCreateFile").

The malware also uses a Hell’s Gate-style indirect system call technique. At startup, it walks ntdll exports, matches them against a list of 31 Nt-function hashes, and builds a runtime table containing hashes, system service numbers, and syscall gadget addresses. When it needs to call a Windows API function, it calls its own wrapper, looks up the hash, retrieves the system service number and gadget address, and jumps to the corresponding ntdll system call address.

The article lists the Nt APIs covered by the malware’s resolved hashes, including NtAllocateVirtualMemory, NtProtectVirtualMemory, NtFreeVirtualMemory, NtWriteVirtualMemory, NtCreateThreadEx, NtQueryInformationProcess, NtQuerySystemInformation, NtOpenProcess, NtClose, NtOpenFile, NtCreateFile, NtReadFile, NtWriteFile, and several registry-related calls.

Anti-analysis behavior

MLTBackdoor includes multiple anti-analysis techniques for detecting debuggers and sandboxed environments. The article explicitly notes that detection does not halt execution. Instead, the malware aggregates the results of 10 checks into a bitmask and sends that bitmask as part of its initial request.

The documented checks include:

  • Hypervisor checks that look for known virtualization vendor IDs such as VMwareVMware, VBoxVBoxVBox, XenVMMXenVMM, and KVMKVMKVM.
  • An additional hypervisor-related check that treats some other hypervisor vendor IDs differently and accounts for Windows 10⁄11 hosts with Virtualization-Based Security enabled.
  • A timing check using repeated RDTSC, CPUID, and RDTSC loops.
  • A debugger check using NtQueryInformationProcess with the ProcessDebugPort information class.
  • A process-name check that hashes running process names with SHA256 and compares them to a hardcoded list.
  • A window-title check using strings such as x64dbg, x32dbg, ollydbg, windbg, idapro, process monitor, process explorer, wireshark, fiddler, dnspy, and cff explorer.
  • A sandbox driver check for names such as vbox, vmci, vmhgfs, virtio, vioscsi, and xenbus.
  • A RAM check for systems below 2 GB.
  • A CPU-count check for systems with one processor.
  • An uptime check for systems running less than five minutes.

ThreatLabz also includes an appendix of cracked SHA256 hashes corresponding to process names such as x64dbg.exe, x32dbg.exe, ollydbg.exe, windbg.exe, ida.exe, ida64.exe, procmon.exe, procmon64.exe, procexp.exe, procexp64.exe, wireshark.exe, fiddler.exe, pestudio.exe, dnspy.exe, vmtoolsd.exe, vboxservice.exe, df5serv.exe, cffexplorer.exe, autoruns.exe, and tcpview.exe.

Built-in capabilities

MLTBackdoor includes a small set of built-in commands focused on filesystem operations. The commands described in the article are:

  • download, which retrieves a file from the victim’s machine.
  • upload, which drops a file on the victim’s machine.
  • ls, which lists files in a directory.
  • delete, which deletes a file or folder.
  • rename, which renames or moves a file or folder.
  • mkdir, which creates a new folder.

The article states that the BOF loader is what significantly increases MLTBackdoor’s capabilities beyond this built-in command set.

Beacon Object File loader

MLTBackdoor can load Beacon Object Files, which the article describes as Microsoft Common Object File Format compiled files containing sections, a symbol table, and relocations. The malware can map and execute BOFs inside its own process and then free them.

ThreatLabz describes the BOF dispatcher as performing the following steps:

  • Creating one memory block per section.
  • Walking the COFF symbol table.
  • Applying relocations per section.
  • Changing section permissions to read and execute only.
  • Handling crashes and returning them as results.
  • Locating the entry point in the symbol table.
  • Removing and freeing allocated memory.

The BOF loader is compatible with Cobalt Strike beacons that rely on a small subset of DJB2-hashed imports. The article lists supported imports such as BeaconDataParse, BeaconDataInt, BeaconDataShort, BeaconDataLength, BeaconDataExtract, BeaconPrintf, and BeaconOutput.

A notable difference described by ThreatLabz is that the loader also includes 19 additional Beacon-prefixed cases that route calls to MLTBackdoor’s own indirect system call wrappers. These include wrappers for calls such as BeaconNtAllocateVirtualMemory, BeaconNtProtectVirtualMemory, BeaconNtFreeVirtualMemory, BeaconNtClose, BeaconNtCreateFile, BeaconNtOpenFile, BeaconNtReadFile, BeaconNtWriteFile, BeaconNtQueryInformationFile, BeaconNtSetInformationFile, BeaconNtQueryDirectoryFile, BeaconNtQuerySystemInformation, BeaconNtOpenProcessToken, BeaconNtQueryInformationToken, BeaconNtCreateKey, BeaconNtOpenKey, BeaconNtSetValueKey, BeaconNtDeleteValueKey, and BeaconNtDeleteKey.

Network communication

MLTBackdoor uses a custom encrypted binary protocol over TLS on port 443. The fixed HTTP path is /api/v1/telemetry, and the User-Agent is Microsoft-Delivery-Optimization/10.1, which the article says is used to masquerade as legitimate traffic.

The malware performs an Elliptic-Curve Diffie-Hellman key exchange using NIST curve P-256. It generates a new key pair per session and performs ECDH with a P-256 public key shared by the C2 server. The resulting shared secret is concatenated with both the session public key and the C2 server public key, then hashed with SHA256 to derive a shared secret used as an AES-256-GCM session key.

After the key exchange, subsequent messages are encrypted with the AES session key and a random 12-byte nonce. The article describes the packet structure as containing a header, encrypted payload, and a 16-byte AES-GCM tag.

DGA and command-and-control resilience

Some MLTBackdoor samples use hardcoded stack-built C2 domains together with a DGA. Other samples rely only on hardcoded domains or only on the DGA.

The article describes the DGA as a deterministic date-based algorithm that generates a new domain per day. ThreatLabz states that the DGA is designed to maintain control of infected systems if hardcoded C2 domains are unreachable.

ThreatLabz notes that the domain generated for April 29, 2026, hrs2y15sungu[.]com, was used not only for C2 communication but also for the distribution campaign described in the infection-chain section.

Protocol naming and message types

ThreatLabz named the malware MLTBackdoor based on the first four magic bytes of the network communication protocol’s header. The magic value is 0x014D4C54, represented as \x01MLT.

The packet header includes:

  • A 32-bit magic value.
  • A 32-bit session ID.
  • A 32-bit message type.
  • A 32-bit payload length.
  • A 12-byte nonce.
  • A 4-byte unknown field.

The session ID consists of four random bytes generated through BCryptGenRandom and is regenerated on each handshake.

The article lists supported message types for C2 communication, including check-in messages, BOF tasks, sleep commands, process exit, command execution results, ECDH key exchange, file download and upload operations, directory listing, delete, rename, mkdir, and BOF stdout.

Indicators and detections

The article states that Zscaler’s multilayered cloud security platform detects indicators related to MLTBackdoor. The listed threat name is Win64.Backdoor.MLTBackdoor.

ThreatLabz provides SHA256 indicators for several files, including:

  • A stage one loader.
  • An archive containing the stage one loader and encrypted MLTBackdoor.
  • MLTBackdoor samples with domains and DGA.
  • MLTBackdoor samples using DGA only.
  • An MLTBackdoor sample using domains only.
  • An MLTBackdoor update sideload archive.

The article also lists domains associated with the campaign and C2 infrastructure:

  • hrs2y15sungu[.]com, described as a DGA domain also used in the distribution campaign.
  • carrolc[.]com, described as an MLTBackdoor C2.
  • cwrtwright[.]com, described as an MLTBackdoor C2.
  • thomphon[.]com, described as an MLTBackdoor C2.
  • powwowski[.]com/payloads/update.zip, described as an MLTBackdoor update URL.

Conclusion from the article

ThreatLabz concludes that MLTBackdoor is a relatively new but formidable post-exploitation malware framework. The article emphasizes its filesystem access capabilities, BOF loader extensibility, CFF and MBA obfuscation, anti-analysis behavior, custom encrypted binary network protocol, and DGA-based backup communications. It also states that these features complicate reverse engineering, evade sandbox and analysis environments, and hinder takedown attempts by researchers and law enforcement.

  • MLTBackdoor does not stop running when it detects analysis environments; instead, it aggregates 10 anti-analysis results into a bitmask and sends them during the initial C2 key exchange.
  • The DGA domain hrs2y15sungu[.]com was used both for C2 communication and for malware distribution on April 29, 2026, showing that the backup domain mechanism can also support the infection chain itself.
  • MLTBackdoor’s BOF loader extends beyond standard Cobalt Strike-style imports by exposing 19 Beacon-prefixed indirect system call wrappers, allowing loaded BOFs to use the malware’s own syscall evasion layer.
  • The malware’s network protocol can be identified by the magic value 0x014D4C54, or \x01MLT, which appears in every client-side and server-side packet header.
  • MLTBackdoor’s stack-built strings are fragmented across flattened state-machine logic, making automated string recovery tools such as FLOSS unreliable according to the article.

Overview

This article is a Windows Kerberos internals deep dive focused on user-to-user authentication, also called U2U. The author explains how U2U differs from traditional Kerberos and why it became more visible after Active Directory Certificate Services, UnPAC-the-Hash, and Shadow Credentials attacks. The article states that it focuses on how U2U works under the hood rather than providing attack execution steps.

Standard Kerberos Baseline

The article begins by reviewing the normal Kerberos flow. A client sends an AS-REQ to the KDC with a timestamp encrypted using a key derived from the client’s password, while the username is sent in plaintext so the KDC can look up the matching key in Active Directory. If verification succeeds, the KDC issues a TGT encrypted with the KRBTGT key and returns an AS-REP containing an EncASRepPart encrypted with the client’s key.

The TGT contains the user’s metadata in the PAC and includes a session key. A second copy of the session key is delivered to the client in the encrypted EncASRepPart, and the client caches both the TGT and session key in LSASS. When the client needs a service ticket, it sends a TGS-REQ containing the TGT, the target SPN, and an authenticator encrypted with the TGT session key.

The KDC decrypts the TGT using the KRBTGT key, extracts the session key, verifies the authenticator, and returns a service ticket. In standard Kerberos, that service ticket is encrypted with the target service account’s long-term key. The client then sends an AP-REQ to the service, and the service decrypts the ticket with its long-term key and uses the ticket session key to validate the authenticator.

Why U2U Exists

The article explains that Windows services commonly run as built-in local accounts or under service accounts with registered SPNs. Regular Active Directory user accounts usually do not have SPNs. If a service is running under an account with no SPN, the KDC cannot look up the account through the requested SPN and may return KDC_ERR_S_PRINCIPAL_UNKNOWN.

U2U was designed to support peer-to-peer cases where a user needs to authenticate to a service hosted by another user. Instead of encrypting the service ticket with a service account’s long-term key, U2U encrypts the service ticket with the session key from the target service user’s TGT. This allows Kerberos authentication to work even when the target service is not represented by a conventional SPN-backed service account key.

U2U Ticket Flow

The U2U process described in the article begins when the client sends a KERB-TGT-REQUEST directly to the target service. The service responds with a KERB-TGT-REPLY containing its own TGT. The client then sends a modified TGS-REQ to the KDC with that TGT included in the additional-tickets field.

The TGS-REQ also sets the ENC-TKT-IN-SKEY flag. This flag tells the KDC to encrypt the resulting service ticket using the session key from the additional ticket rather than a long-term service key. The KDC decrypts the additional TGT with the KRBTGT key, extracts its session key, and uses that session key to encrypt the new U2U service ticket.

Why Sharing a TGT Is Not Enough to Impersonate the Account

The article emphasizes that a TGT alone is not sufficient to use Kerberos. The TGT is encrypted with the KRBTGT account’s long-term key, so the session key inside it is inaccessible without that key. The article notes that if an attacker could crack the KRBTGT key, they could forge TGTs for any account in the domain, which is the Golden Ticket attack, but that risk exists independently of U2U.

A TGT must be paired with its session key to be useful. The session key is required to encrypt the authenticator in later TGS requests. The article uses this to explain why Rubeus exports tickets as KRB-CRED structures, because those structures contain the ticket along with the session key, flags, expiration time, and other metadata needed to use it.

Session Keys Versus Long-Term Keys

The article describes session keys as randomly generated by the KDC and delivered to the client inside the AS-REP. They are not stored by the KDC, and their validity is tied to the TGT lifetime, which defaults to 10 hours. When the TGT expires, a new TGT with a different session key is issued.

The author argues that this limits the usefulness of U2U tickets. Since session keys are randomly generated and not password-derived, there is no dictionary or brute-force path against them as there is with password-derived keys. The article states that capturing a U2U ticket off the wire would leave at most the TGT lifetime to crack a random key, which it describes as not a realistic attack surface.

Draft U2U Scenarios and Windows Behavior

The article references the draft-swift-win2k-krb-user2user draft, which describes three scenarios for initiating U2U. The first is when the client already knows it needs U2U and proactively requests the target service’s TGT before sending a U2U TGS-REQ. The article states that this is the path observed in Windows, including in RDP with Network Level Authentication.

The second draft scenario is KDC-enforced U2U through account policy. In that model, the KDC would return KDC_ERR_MUST_USE_USER2USER if a client requested a standard service ticket for an account that requires U2U. The author states that Windows has no such user attribute, userAccountControl flag, or Group Policy setting, and that investigation of kerberos.dll found that the error does not exist.

The article says attempts to make the KDC return KDC_ERR_MUST_USE_USER2USER failed. Requests using a UPN in the SPN field and an empty SPN returned KDC_ERR_S_PRINCIPAL_UNKNOWN instead. The author also notes that even if the KDC returned such an error, it could not provide the target TGT because the KDC does not retain TGTs after issuance.

The third draft scenario is server-enforced U2U after a conventional AP-REQ. The article explains that this path is not generally reached in a standard Windows environment because the KDC detects the missing SPN during the TGS exchange. However, the article says the server-side error path can apply if a service is hardcoded to require U2U and explicitly returns KRB_AP_ERR_USER_TO_USER_REQUIRED when it receives a conventional AP-REQ.

RDP With Network Level Authentication

The article identifies RDP with Network Level Authentication as the most common legitimate place to observe U2U in the wild. NLA requires authentication before the server renders a login screen, which prevents a full desktop session from being started for an unauthenticated connection. The article explains that NLA relies on CredSSP to forward the user’s credentials to the server in encrypted form over a protected channel.

In the author’s lab, after the initial RDP handshake and TLS tunnel were established, the client requested a standard Kerberos service ticket for TERMSRV/hostname.domain.local. The KDC returned a valid service ticket. The article notes that this ticket was not used or cached in the logon session.

The client then contacted the RDP server directly to request its TGT, beginning the U2U exchange. The author states that there were no errors from the KDC or RDP server telling the client to use U2U. Instead, CredSSP initiated U2U because it called InitializeSecurityContext with the ISC_REQ_USE_SESSION_KEY flag, which requires a new session key for the security context.

The KERB-TGT-REQUEST and KERB-TGT-REPLY were wrapped inside SPNEGO tokens, which were carried in CredSSP TSRequest structures over the established TLS channel. The article emphasizes that this TGT exchange does not appear as raw Kerberos traffic on TCP port 88. The client receives the RDP server’s TGT inside the encrypted CredSSP negotiation before any credentials are forwarded.

With the RDP server’s TGT, the client sends a TGS-REQ to the KDC that includes the server’s TGT in additional-tickets and sets ENC-TKT-IN-SKEY. The article notes that the sname can still reference a valid SPN such as TERMSRV/castelblack, but the KDC ignores it once ENC-TKT-IN-SKEY is set. The resulting ticket is issued for the computer account and encrypted with the session key from that account’s TGT.

The TGS-REP reflects the same sname supplied in the request. The author notes that this differs from the expired draft, which says a U2U exchange should reflect the UPN of the target service. Further testing with Rubeus confirmed that whatever target value is supplied is reflected back in the sname of the issued ticket.

Finally, the client sends an AP-REQ to the RDP server inside CredSSP. The AP-REQ sets USE-SESSION-KEY to true, telling the receiver which keying material to use to decrypt the ticket. The RDP server then obtains the session key from LSASS and decrypts the AP-REQ to authenticate the user.

Why RDP With NLA Uses U2U

The author states that the need for U2U in RDP with NLA is debatable because the client can already request a service ticket for the TERMSRV SPN. The article suggests that the legitimate reason is the use of non-cacheable service tickets and a fresh session key. CredSSP calls InitializeSecurityContext with ISC_REQ_USE_SESSION_KEY, and Windows performs a U2U exchange to satisfy that requirement.

The article says the new session key is then used to deliver the client’s plaintext credentials. It explains that reusing an existing cached session key would create a risk if an attacker extracted the key and was also intercepting the traffic, because the attacker could decrypt the CredSSP exchange. The article also reiterates that NLA requires authentication before any RDP session is spawned and states that this prevents pre-auth RDP vulnerabilities like BlueKeep.

PKINIT and PAC_CREDENTIAL_INFO

The article then connects U2U to UnPAC-the-Hash. It explains that when a user logs in through PKINIT, the user’s NT hash is included in the PAC inside the TGT. This is done to support NTLM-based authentication for legacy services that do not support Kerberos.

In PKINIT, the user authenticates with a certificate rather than a password, so the client host does not have the user’s NT hash to cache locally for the msv1_0 authentication package. The article states that Microsoft included the NT hash in the PAC only if the TGT is obtained through certificate authentication. The PAC is encrypted inside the TGT or service tickets, and PAC_CREDENTIAL_INFO is separately encrypted with what the article calls the AS reply key.

The AS reply key is derived from the Diffie-Hellman shared secret and the client and server nonces. In PKINIT, this key encrypts the EncASRepPart of the AS-REP, whereas traditional Kerberos uses the user’s encryption key for that purpose. After the AS exchange, LSA derives the AS reply key, decrypts EncASRepPart, and retrieves the TGT session key.

At that point, LSA has the TGT, TGT session key, and AS reply key, but it cannot read the PAC inside the TGT because the TGT is encrypted with KRBTGT. To access the PAC, LSA requests a service ticket for the local computer account using an SPN such as HOST/FQDN. The KDC copies the PAC into that service ticket and encrypts the ticket with the computer account’s long-term key.

LSA can decrypt that service ticket using the computer account key, access the PAC, and use the AS reply key to decrypt PAC_CREDENTIAL_INFO. The recovered NT hash is then cached in the logon session for NTLM authentication. The article states that the AS reply key exists in LSASS memory only long enough for LSA to complete this service ticket decryption process.

UnPAC-the-Hash Connection

The article explains that the same primitives used by U2U make UnPAC-the-Hash possible. To perform the attack as described, a certificate issued by the domain Certificate Authority and its corresponding private key are required. The article states that this is typically obtained through ADCS ESC attacks, Shadow Credentials attacks, or by dumping a TGT obtained via PKINIT from a workstation.

With the certificate and private key, the operator initiates a PKINIT exchange against the KDC and generates an ephemeral Diffie-Hellman key pair under their control. The KDC returns an AS-REP, from which the AS reply key is derived using the same process Windows LSA uses. This provides a valid TGT and its associated session key.

The next step described is requesting a U2U service ticket for the same account. Because the request is U2U, the KDC encrypts the service ticket using the TGT session key instead of a long-term service account key. Since the requester already has that TGT session key, the service ticket can be decrypted.

Once decrypted, the embedded PAC becomes accessible. PAC_CREDENTIAL_INFO contains the user’s NT hash and is encrypted with the AS reply key derived during PKINIT. The article explains that using that AS reply key allows the NT hash to be recovered, after which it can be used for Pass-the-Hash or to request Kerberos tickets directly if RC4 encryption is enabled.
