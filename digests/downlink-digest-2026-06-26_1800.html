In plain words

The U.S. cyber agency CISA told federal agencies to fix a serious Cisco Unified Communications Manager Server weakness by June 28 because attackers are already using it. CISA also set the same deadline for a serious flaw in PTC Windchill and FlexPLM, software used by industries like manufacturing, engineering, retail, footwear, apparel, and consumer products. These problems matter because they could let outsiders reach affected systems remotely, including writing files or running commands, so agencies and organizations need to update or stop using the affected products quickly.

Summary
CISA ordered U.S. federal agencies to urgently patch an actively exploited critical Cisco Unified Communications Manager SSRF flaw, CVE-2026-20230, by June 28. The agency also set the same deadline for a critical PTC Windchill and FlexPLM remote code execution flaw, CVE-2026-12569.

CISA is giving U.S. federal agencies until Sunday, June 28, to patch CVE-2026-20230, a vulnerability in Cisco Unified Communications Manager Server that is being actively exploited. The flaw is a server-side request forgery issue and has been added to CISA’s Known Exploited Vulnerabilities catalog. Under Binding Operational Directive 26-04, the remediation is considered urgent.

Cisco rated CVE-2026-20230 as critical and released a patch on June 3. The company warned that the flaw could be exploited remotely and without authentication through specially crafted HTTP requests. At the time, Cisco said proof-of-concept exploit code existed but that it had found no evidence of active exploitation. Last weekend, Defused observed the vulnerability being exploited to write arbitrary text files to affected endpoints.

CISA also added CVE-2026-12569 to the Known Exploited Vulnerabilities catalog and set the same June 28 deadline for federal agencies to patch it. The flaw affects PTC Windchill and FlexPLM, product lifecycle management systems used in industries including manufacturing, engineering, retail, footwear, apparel, and consumer products. CVE-2026-12569 is a critical remote code execution vulnerability involving deserialization of untrusted data, and PTC said it affects all versions up to 11.0 and multiple versions of the 11.1, 11.2, 12.0, 12.1, and 13.0 release branches.

  • CISA’s June 28 deadline is unusually urgent, giving agencies only until Sunday to remediate or stop using the affected products under BOD 26-04.
  • Cisco’s flaw moved from having only proof-of-concept exploit code and no known exploitation on June 3 to observed in-the-wild attacks within the same month.
  • The observed Cisco exploitation involved writing arbitrary text files to affected endpoints, making the issue more concrete than a theoretical remote flaw.
  • The PTC issue has broad operational relevance because Windchill and FlexPLM are used across manufacturing, engineering, retail, footwear, apparel, and consumer products industries.

Summary
The FBI and CISA warn that Russian intelligence-linked hackers are phishing high-value targets into revealing Signal Backup Recovery Keys, letting attackers restore message archives and potentially take over accounts. Signal users should never share recovery keys, codes, or PINs in chats and should regenerate the key immediately if it may have been exposed.

The FBI and CISA updated their March warning about Russian intelligence-linked phishing of Signal accounts, saying the operators are now trying to get targets to hand over their Signal Backup Recovery Key. If attackers obtain the key, they can restore the account’s backup, read private and group message history, and take over the account. The advisory says the old key can still be used even if a new account is created on the same phone number.

The updated advisory identifies the activity as UNC5792 and UNC4221 and ties it to multiple Russian Intelligence Services groups, including FSB officers embedded with the FSB Border Guards and others working for Russian military services. The campaign targets high-intelligence-value individuals such as current and former U.S. and international government officials, military personnel, political figures, journalists, and officials in Ukraine. The March notice said the broader campaign had already compromised thousands of accounts worldwide.

The phishing messages pose as Signal support and walk targets through turning on backups, opening the Recovery Key, and pasting it into the chat. The agencies say this does not break Signal’s encryption or the app itself, but compromises individual accounts through social engineering and a legitimate feature. Recommended steps include treating in-app messages from “Signal support” as hostile, never sharing recovery keys, codes, or PINs in chat, checking linked devices, and generating a new key if the old one may have been shared.

  • A compromised Signal Backup Recovery Key can remain useful to attackers even if the victim creates a new account on the same phone number, so regenerating the key is necessary to block future backup downloads.
  • Generating a new Signal Backup Recovery Key invalidates the old one going forward, but it cannot undo any message history that attackers may have already restored.
  • The phishing has shifted from asking for one-time codes or PINs to targeting the backup key, which can expose an entire saved message archive rather than only enabling account access at one moment.
  • Messages claiming to be from “Signal support” inside the app are specifically called out as hostile because real support does not ask users in chat for codes, PINs, or recovery keys.

Summary
Kaspersky reports a broad, likely Chinese-speaking campaign called StrikeShark that uses a new SharkLoader malware to deploy Cobalt Strike after exploiting public-facing server vulnerabilities or using malicious installers. Targets include diplomatic, government, and software development organizations across multiple countries, suggesting possible espionage despite no confirmed data exfiltration yet.

Kaspersky has identified a newly discovered cyberattack campaign called StrikeShark that delivers a previously undocumented malware family named SharkLoader. SharkLoader acts as a loader for deploying Cobalt Strike Beacon on compromised hosts. The campaign has targeted a diplomatic organization in Indonesia, government organizations in Taiwan, software development companies across multiple countries, and entities in Hong Kong, Lebanon, Syria, Colombia, North Macedonia, Nepal, and Serbia.

The campaign has no direct links to a known threat actor, but Kaspersky believes it is likely the work of a Chinese-speaking threat actor based on the use of open-source post-compromise tools commonly used by Chinese-speaking developers. Initial access has involved exploiting known vulnerabilities in public-facing systems, including flaws in Exchange Server, Openfire, GeoServer, Apache Shiro, Hikvision products, Microsoft SharePoint, Zimbra Collaboration Suite, F5 BIG-IP, Fortinet FortiOS, React Server Components, and Cisco IOS XE Web UI. The attackers are assessed as likely using publicly available proof-of-concept exploits from GitHub or other open-source platforms.

After gaining access, the operators deploy web shells and use a DLL side-loading chain involving SystemSettings.exe to deliver SharkLoader, or they use custom droppers disguised as legitimate installers such as Google Update and Cisco AnyConnect. SharkLoader uses Perfect DLL Hijacking, decrypts and loads additional components, and ultimately runs Cobalt Strike Beacon after installing API hooks and resuming a suspended thread. The activity also includes reconnaissance, Active Directory enumeration, credential theft targeting LSASS and the NTDS database file, and deployment of tools such as FScan, Searchall, and Pillager, while Kaspersky said there is no clear evidence of active data exfiltration so far.

  • StrikeShark appears to rely on opportunistic exploitation of many known public-facing application flaws, indicating that long-known vulnerabilities such as ProxyLogon and newer flaws such as CVE-2024-36401 are both relevant to this campaign.
  • The campaign uses two distinct SharkLoader delivery approaches: server-side web shells with DLL side-loading and custom droppers disguised as legitimate software installers, so detection efforts cannot focus on only one infection path.
  • SharkLoader uses Perfect DLL Hijacking and API hooks around VirtualAlloc and Sleep to execute Cobalt Strike while attempting to evade memory scanning techniques that look for executable memory regions.
  • SharkLoader does not include its own persistence mechanism, but the operators compensate by using Registry Run keys and scheduled tasks to launch SystemSettings.exe during user logon or even when no user is logged in.
  • Although Kaspersky has not observed active data exfiltration, the combination of government and software-development targets with post-compromise reconnaissance and credential theft makes the campaign significant even before confirmed theft is seen.

The article describes an experiment where SpecterOps used OpenAI’s Codex to solve a Windows challenge program that hides the correct 25-digit code, comparing a normal analysis approach with one that could replay and search what the program actually did while running. The replay-based approach found the real code and flag faster, while the normal approach reached a believable but wrong answer. This matters because people who study suspicious or confusing programs often do not have the original source code, and being able to check what really happened can reduce wrong conclusions in malware analysis and vulnerability research.

Summary
SpecterOps shows that giving Codex access to Time Travel Debugging traces via TTDObjectsPy helped it solve the FLARE-ON FlareAuthenticator reverse-engineering challenge faster and correctly by grounding analysis in real execution history instead of static decompiler guesses. The key takeaway is that queryable runtime traces can reduce speculative reasoning and false confidence when LLMs analyze obfuscated binaries.

SpecterOps describes an experiment using Codex with Time Travel Debugging traces through TTDObjectsPy to solve the FLARE-ON 12 FlareAuthenticator challenge. The target was a Windows Qt GUI program where a user enters a 25-digit code and receives either a rejection or a flag. The article compares two local Codex sessions using the same goal: one without any TTD tooling and one with TTDObjectsPy enabled.

The session without TTD made progress by identifying the Qt GUI structure, keypad mapping, relevant slots, and success and failure message-box paths. However, it relied on indirect evidence, forced execution down a success-looking path, inferred the flag from a partial visual clue, and produced an incorrect flag after 1:51:28.706 and 69,501,208 total logged tokens. The article presents this as an example of how static analysis and instrumentation can still leave an LLM with plausible but wrong conclusions.

The TTDObjectsPy session used recorded execution history to ask what code ran on keypad and OK clicks, what state was compared, and how keypad input changed that state. It found that the OK handler compared a 64-bit state field at window+0x78 against 0x0bc42d5779fec401, extracted a 25-by-10 digit contribution table, solved the constraint with OR-Tools CP-SAT, and recovered the correct passcode and flag. The article concludes that queryable execution history can help keep LLM-assisted reverse engineering grounded in runtime facts, while noting that TTD has operational overhead such as disk usage, capture failures, GUI automation races, and trace cleanup.

  • The TTD-enabled session avoided fully untangling the heavily flattened validation code by turning the problem into targeted runtime questions about which state was compared and how keypad presses changed it.
  • The failed non-TTD run shows that an LLM can do extensive useful reverse-engineering work and still arrive at a plausible but incorrect result when its evidence depends on forced paths, screenshots, and inference.
  • The decisive runtime artifact was a single 64-bit state field at window+0x78 compared against 0x0bc42d5779fec401, which let Codex focus on extracting a digit contribution table instead of reasoning about the entire obfuscated submit path.
  • TTDObjectsPy made execution history useful as a searchable data source for Codex, enabling queries about register values, memory changes, function arguments, and the instruction responsible for writing a value.
  • The article highlights that TTD has practical overhead, including disk usage, capture failures, GUI automation races, and trace cleanup, so its value is strongest when important behavior depends on runtime state.
