Overview

CISA added CVE-2026-42271, a high-severity vulnerability affecting BerriAI LiteLLM, to its Known Exploited Vulnerabilities catalog after evidence of active exploitation. The flaw is a command injection issue with a CVSS score of 8.7. According to the article, it can allow any authenticated user to run arbitrary commands on the host.

The vulnerability affects the LiteLLM Python package in versions greater than or equal to 1.74.2 and less than 1.83.7. BerriAI addressed the issue in version 1.83.7. Users are advised to update LiteLLM to version 1.83.7 or later.

Technical Details of CVE-2026-42271

The flaw is tied to two LiteLLM endpoints used to preview an MCP server before saving it:

  • POST /mcp-rest/test/connection
  • POST /mcp-rest/test/tools/list

BerriAI said these endpoints accepted a full server configuration in the request body. That configuration included the command, args, and env fields used by the stdio transport.

When called with a stdio configuration, the endpoints attempted to connect and spawned the supplied command as a subprocess on the proxy host. The command ran with the privileges of the proxy process. The article says the endpoints were secured only by requiring a valid proxy API key.

Because of that authorization model, any authenticated user could execute arbitrary commands on a susceptible system. The article specifically notes that this included privileged internal-user keys.

Patch and Authorization Change

BerriAI released patches in LiteLLM version 1.83.7. As part of the fix, both MCP test endpoints now require the PROXY_ADMIN role. The article says this makes the test endpoints consistent with the save endpoint.

This change narrows access to the affected functionality. The article does not state that the endpoints were removed; it states that their required role was changed.

Chained Unauthenticated Remote Code Execution

Horizon3.ai said it chained CVE-2026-42271 with CVE-2026-48710 to achieve unauthenticated remote code execution against vulnerable LiteLLM deployments. CVE-2026-48710 is described as a “BadHost” host header validation bypass vulnerability affecting Starlette, a lightweight ASGI framework.

According to Horizon3.ai, CVE-2026-48710 can be used to bypass the authentication mechanism entirely in LiteLLM deployments whose dependency tree includes Starlette versions 1.0.0 or earlier. Horizon3.ai said this transforms the LiteLLM vulnerability into unauthenticated remote code execution with no credentials required.

The article states that Horizon3.ai assigned the chained vulnerability a combined CVSS score of 10.0, making it critical in nature.

Potential Impact

Successful exploitation of the chained vulnerabilities could allow attackers to run arbitrary commands on the LiteLLM host. The article also says attackers could access model provider credentials, siphon API keys and secrets stored by the proxy, move laterally into connected AI infrastructure, and compromise downstream systems integrated with the gateway.

The article does not provide details on confirmed compromises. It also does not state whether the observed in-the-wild exploitation is using the Starlette chain.

Known Unknowns

Recommended Actions

The article advises users to update LiteLLM to version 1.83.7 or later. It also advises updating Starlette to version 1.0.1 or later.

If immediate patching is not possible, the article recommends the following mitigations:

  • Block POST /mcp-rest/test/connection and POST /mcp-rest/test/tools/list at the reverse proxy or API gateway.

01 Exploited Remote Access and Internet-Facing Services

CISA ordered U.S. Federal Civilian Executive Branch agencies to secure affected Check Point devices against CVE-2026-50751 by June 11 after the flaw was exploited as a zero-day. The vulnerability allows unauthenticated remote attackers to bypass authentication and establish a remote access VPN connection on targeted Check Point Mobile Access/SSL VPNs, Remote Access VPNs, or Spark firewalls.

The exposure is limited to specific legacy or weakened configurations, including deployments using the deprecated IKEv1 key exchange protocol, gateways that do not require a machine certificate, and legacy Remote Access clients. Check Point reported exploitation beginning on May 7, with activity surging over the weekend and affecting a few dozen targeted organizations globally. The company also linked at least one post-compromise incident involving CVE-2026-50751 to a Qilin ransomware affiliate, elevating the issue from access abuse to ransomware-enabled intrusion risk.

CISA also added CVE-2026-42271 in BerriAI LiteLLM to its Known Exploited Vulnerabilities catalog based on evidence of active exploitation. The flaw is a high-severity command injection issue affecting LiteLLM Python package versions 1.74.2 through versions earlier than 1.83.7, allowing any authenticated user to execute arbitrary commands on the host.

Horizon3.ai reported that CVE-2026-42271 can be chained with CVE-2026-48710, a Starlette host header validation bypass, to achieve unauthenticated remote code execution against vulnerable LiteLLM deployments. BerriAI addressed the issue in LiteLLM 1.83.7 by requiring the PROXY_ADMIN role for two MCP server preview test endpoints, while the reported chain highlights how AI infrastructure components can become externally exploitable when combined with framework-level weaknesses.

02 Chrome V8 Zero-Day Exploitation

Google released emergency Chrome updates for CVE-2026-11645, a high-severity zero-day vulnerability in the V8 JavaScript engine that the company said has been exploited in the wild. The patched Stable Desktop versions are 149.0.7827.102 for Windows and Linux and 149.0.7827.103 for Mac.

CVE-2026-11645 is described as an out-of-bounds read and write weakness exploitable through crafted HTML pages. Successful exploitation can allow remote attackers to execute arbitrary code inside Chrome’s sandbox, access data beyond the intended memory buffer through heap corruption, expose sensitive information, or trigger browser crashes. Google said access to bug details and related links may remain restricted until a majority of users have received the fix, consistent with its handling of actively exploited browser vulnerabilities.
