Azure CLI Password Spray Hits at Least 78 Microsoft Accounts in 81M+ Attempts
thehackernews.com · 01 Jul 2026 05:46 UTC openai-codex / gpt-5.5
TAGSas32167chinaconditional-access-policieshuntresslshiy-llcmicrosoftmicrosoft-azure-climicrosoft-entra-idoauth-2-0password-sprayingresource-owner-password-credentialsunited-states

In plain words

Researchers say someone tried to sign in to Microsoft accounts more than 81 million times through Azure’s command-line tool and succeeded in taking over at least 78 accounts at 64 organizations. The attackers used old leaked username-and-password pairs, and some organizations were affected even though they had extra sign-in protections because those protections were not set up to cover this kind of login. This matters because it shows that simply having added security checks is not enough if they are not applied broadly and correctly.

Summary
Huntress says attackers made over 81 million Azure CLI login attempts and compromised at least 78 Microsoft accounts by using old breached credentials and the deprecated ROPC OAuth flow. The key takeaway is that MFA and Conditional Access can be bypassed if not applied to all users, cloud apps, and client app types, including Azure CLI logins.

Cybersecurity researchers at Huntress warned of a massive, ongoing, automated password spray attack aimed at Microsoft’s Azure command-line interface. Between June 12 and June 26, the threat actor made more than 81 million login attempts and compromised at least 78 Microsoft accounts across 64 organizations. Huntress said the targeting appeared to be based on password prevalence in compromised password combo lists, rather than a specific business type or industry.

The activity largely originated from the IPv6 range 2a0a:d683::/32, which is controlled by internet infrastructure provider LSHIY LLC. Some IP addresses resolved to the United States, while a few resolved to China. Huntress said the campaign was part of a larger wave of credential spray attacks across several ASNs, with credential spray volume across its customer base increasing by more than 155 times.

A notable technical detail is that the attackers used the deprecated Resource Owner Password Credentials OAuth flow, known as ROPC, to bypass some Conditional Access Policy protections. Microsoft recommends against using ROPC and says it is incompatible with multi-factor authentication. Huntress said organizations should require MFA for all users, all cloud apps, and all client app types, restrict Azure CLI for non-admin users, and prioritize response based on credential validity.

  • The campaign shows that Conditional Access policies can leave a major gap when MFA is scoped only to certain apps, user groups, or locations instead of covering All Users, All Cloud Apps, and All Client App types.
  • The use of the deprecated ROPC flow was especially significant because it can avoid the authorization endpoint where some Conditional Access policies are enforced, allowing attackers to get in even where MFA existed.
  • The attacks appear to have succeeded by using old breached username-and-password combinations that had never been rotated, making credential age and validity a practical response priority.
  • The spike on June 22, when 30 identities across 23 businesses were impacted, indicates the campaign’s success rate was not steady and could escalate sharply within a single day.
  • Eight affected businesses had no MFA policy at all, but Huntress emphasized that the broader lesson is not that MFA fails, but that incomplete MFA coverage creates exploitable openings.
Key Points

  • 02The campaign compromised at least 78 Microsoft accounts across 64 organizations, with targeting based on password prevalence in compromised password combo lists rather than business type or industry.
  • 03The activity originated largely from the IPv6 range 2a0a:d683::/32 controlled by LSHIY LLC, with some IP addresses resolving to the United States and a few to China.
  • 04The attackers leveraged the deprecated Resource Owner Password Credentials OAuth flow to bypass some poorly configured Conditional Access policies, including cases where MFA was not applied to all cloud apps, all users, or all client app types.
  • 05Huntress advised organizations to require MFA for all users, all cloud apps, and all client app types, restrict the Azure CLI application for non-admin users, and prioritize response by credential validity.

Researcher Analyzes 3,000 Live ClickFix Payloads, Exposing API-Driven Malware Delivery
thehackernews.com · 01 Jul 2026 05:32 UTC openai-codex / gpt-5.5

In plain words

A researcher studied about 3,000 live ClickFix tricks, where fake “prove you’re human” pages convince people to paste and run a harmful command themselves. He found that criminals now use back-end servers to give each visitor a freshly disguised version, and some pages hide the bad code in a downloaded file so it is harder to spot. This matters because ordinary computer users, Windows and macOS users, cryptocurrency workers, and organizations targeted by criminal or state-backed groups can all be affected, and the method is spreading quickly because it is cheap to change when defenders catch up.

Summary
ClickFix campaigns have evolved into API-driven malware delivery platforms that generate unique, obfuscated payloads for each visitor, making detection harder. Researchers also found a newer Downloads-folder method designed to bypass clipboard and script-scanning defenses, so defenders should focus on suspicious process chains rather than clipboard contents alone.

Security researcher Bert-Jan Pals analyzed roughly 3,000 payloads from live ClickFix campaigns and found that the technique has evolved into an API-driven malware delivery setup. ClickFix uses fake CAPTCHA or error pages to trick people into pasting and running commands that execute malware themselves. Pals found backend servers that check access tokens, log callers, and return freshly scrambled commands for each request.

In one test, a server returned 100 different payloads for 100 requests, using rotating combinations of Base64, AES, TripleDES, Rijndael, and Deflate. After the wrapping was removed, the payloads unpacked to the same script, which ran in memory through a PowerShell runspace. The platform also serves lures in 25 languages, matches commands to the visitor’s operating system, and includes macOS versions alongside Windows versions.

The research also found a newer Downloads-folder method that quietly downloads a file and copies only a short command that moves, unpacks, and runs it, which is designed to avoid clipboard-focused detection and slip past Windows script scanning. The article says defenders should focus on process chains such as explorer.exe or WindowsTerminal.exe launching powershell.exe, cmd.exe, or msiexec.exe and making network connections soon afterward. Pals also listed comicstar[.]lat, babybon[.]cfd, and merkantalolol[.]asia as payload servers observed during the research, while noting that a connection to one of them does not prove infection.

  • ClickFix has moved beyond static lure pages into a commercialized, on-demand delivery model, making it cheap for attackers to keep changing payload disguises when defenders catch up.
  • Clipboard monitoring alone is no longer a reliable detection strategy because newer ClickFix pages can copy only a harmless-looking orchestrator command while keeping the malicious code in a downloaded file.
  • Windows Terminal use is notable for investigations because, unlike the older Windows+R Run-box method, it does not leave evidence in the RunMRU registry key that investigators normally check.
  • A connection to one of the listed payload servers indicates that a command was most likely placed in a user's clipboard, but the article warns it does not by itself prove infection.
  • ClickFix is no longer limited to ordinary cybercriminal campaigns, because Proofpoint tied Russia-, Iran-, and North Korea-backed groups including APT28, MuddyWater, and Kimsuky to its use.
Key Points

  • 02One ClickFix server returned 100 different payload wrappers for 100 requests, using rotating combinations of Base64, AES, TripleDES, Rijndael, and Deflate while unpacking to the same script at the time of the research.
  • 03Newer ClickFix pages can quietly download a file to the Downloads folder and copy a short command that moves, unpacks, and runs the file, a method designed to avoid clipboard-focused detection and slip past Windows script scanning.
  • 04ClickFix lures are served in 25 languages, can match commands to a visitor’s operating system, and include macOS versions alongside Windows versions.
  • 05The article says defenders should focus on process chains such as explorer.exe or WindowsTerminal.exe launching powershell.exe, cmd.exe, or msiexec.exe and making network connections soon afterward.

In plain words

Citrix released fixes for six problems in its NetScaler ADC and NetScaler Gateway products, which are used by customers to manage access to services. If left unfixed in certain setups, these problems could let someone read files they should not see or make the device stop working properly. Citrix says there is no evidence these issues have been used in real attacks, but these products have been targeted in the past for ransomware, so customers are urged to install the updates and make one recommended settings change for the HTTP/2 issue.

Summary
Citrix released patches for six NetScaler ADC and Gateway vulnerabilities that could allow arbitrary file reads, memory overreads, or denial-of-service under certain configurations. Customers should upgrade to fixed versions immediately and, for CVE-2026-13474, ensure the HTTP/2 small-window timeout setting is configured as required.

Citrix released security updates for six vulnerabilities in NetScaler ADC and NetScaler Gateway that could be exploited to read arbitrary files or cause denial-of-service conditions. The flaws affect several configurations, including SAML IDP, Gateway or AAA virtual server, Oracle load balancing, DNS proxy or recursive resolver deployments, TCP Timestamp, and HTTP/2 profiles. Citrix said there is no evidence that the issues have been exploited in the wild.

The fixed versions include NetScaler ADC and NetScaler Gateway 14.1-72.61 and later, 13.1-63.18 and later, NetScaler ADC 14.1-FIPS 14.1-72.61 FIPS and later, and NetScaler ADC 13.1-FIPS and 13.1-NDcPP 13.1.37.272 and later. For CVE-2026-13474, Citrix also advised customers to update the Http2SmallWndTimeout setting, because appliances not using HTTP Strict Profiles have a default value of 0 and upgrading alone will not completely address the vulnerability. Citrix credited Michael Tucker from the XOR team at JPMorgan Chase, Aliz Hammond of watchTowr, and Maxim Suhanov for reporting the vulnerabilities.

watchTowr Labs said CVE-2026-8451 was discovered while attempting to reproduce CVE-2026-3055, an earlier insufficient input validation flaw. The company said the issue comes from how NetScaler parses SAML authentication requests and can result in out-of-bounds memory reads when malformed SAML requests are sent. watchTowr noted that the newer issue leaks less data than CVE-2026-3055, but still raised concern about a broader pattern of fragile memory management in Citrix NetScaler appliances.

  • For CVE-2026-13474, upgrading alone may not fully fix the issue on appliances that are not using HTTP Strict Profiles, because customers must also manually set Http2SmallWndTimeout to 30 seconds.
  • watchTowr found CVE-2026-8451 while trying to reproduce CVE-2026-3055, and said the new SAML parsing issue shares the same root cause as the earlier flaw.
  • watchTowr reported that CVE-2026-8451 leaks less data than CVE-2026-3055, but varying the request length still allowed researchers to consistently extract a few bytes from the server.
  • The listed vulnerabilities are tied to specific NetScaler configurations, such as SAML IDP, Gateway or AAA virtual server, Oracle load balancing, DNS proxy or recursive resolver, TCP Timestamp, and HTTP/2 profiles.
  • The article emphasizes urgency because Citrix appliances have repeatedly been targeted in recent years, including by threat actors deploying ransomware.

And another one. GitHub ships break-glass credential revocation
aikido.dev · 01 Jul 2026 04:06 UTC openai-codex / gpt-5.5
TAGSaqua-securityazure-functionscheckmarxdurabletaskforce-pushgithubgithub-actionsgithub-enterprisemiasma-wormmicrosoftnpmpull-request-targetpypitrivytrivy-action

In plain words

GitHub added a new emergency option for large organizations to quickly cut off stolen access keys across an entire organization, instead of hunting them down one by one during a crisis. The article connects this to recent cases involving Microsoft, Aqua Security’s Trivy project, and Checkmarx, where leftover stolen access let attackers come back and spread harmful code through software projects and build pipelines. This matters because many teams rely on these projects and tools to release software, so one missed access key can lead to wider disruption and stolen secrets beyond the first victim.

Summary
GitHub’s new enterprise break-glass credential revocation lets owners quickly cut off compromised PATs, SSH keys, and OAuth tokens across an organization, addressing a gap that enabled recent repeat compromises like Trivy, Checkmarx, and Microsoft durabletask. The key takeaway is to assign the required permission and understand the feature before an incident, because fast bulk revocation can stop credential reuse even if it temporarily breaks CI/CD.

GitHub released self-service credential revocation for Enterprise, giving organization owners a break-glass way to cut off compromised credentials across an organization in one action. Enterprise owners with the “Manage enterprise credentials” permission can bulk-revoke or delete credentials for all users in an organization, or target a specific account. The feature covers SSO authorizations for Personal Access Tokens, SSH keys, and OAuth tokens, and actions create audit log entries and send email notifications to affected users.

The article argues this matters because recent incidents showed how slow or incomplete credential cleanup can let attackers return. Microsoft’s durabletask package and related GitHub repository were compromised through the same account after earlier malicious PyPI package versions had been pulled, and GitHub disabled 73 repositories across four Microsoft GitHub organizations. Aqua Security’s Trivy compromise also returned after incomplete rotation, and stolen credentials from affected pipelines were later used to poison Checkmarx’s GitHub Actions.

The article says true atomic credential rotation is unrealistic at large scale, but instantly killing every credential in scope is the one action that can be made atomic during an incident. This can break CI/CD, but the article frames that tradeoff as worthwhile when the priority is getting an attacker out of infrastructure. It recommends assigning the “Manage enterprise credentials” permission before an incident, reviewing Settings > Credentials, and pinning GitHub Actions to full commit SHAs rather than version tags.

  • The article frames break-glass revocation as more realistic than true atomic credential rotation, because killing every credential in scope at once is executable under pressure even if it breaks CI/CD temporarily.
  • The Trivy-to-Checkmarx chain shows that an incomplete cleanup after one compromise can turn a single surviving credential into a multi-organization supply chain incident.
  • The durabletask case highlights that pulling malicious packages quickly is not enough if the same compromised account can later be used to modify the related GitHub repository.
  • The credential revocation API’s usefulness was limited in compromised-account scenarios because it required already having the token string, while the new feature can act across credentials tied to an account or organization.
  • Pinning GitHub Actions to full commit SHAs directly addresses the force-pushed tag technique used in the Trivy compromise, where version tags were moved to malicious commits.
Key Points
  • 01GitHub released self-service credential revocation for Enterprise, allowing organization owners with the "Manage enterprise credentials" permission to bulk-revoke or delete credentials for all users in an organization or target a specific account.
  • 02The feature covers SSO authorizations for Personal Access Tokens, SSH keys, and OAuth tokens, creates audit log entries, and sends email notifications to affected users.
  • 03The article says recent compromises involving Microsoft durabletask, Aqua Security’s Trivy, and Checkmarx showed that slow or incomplete credential rotation can allow attackers to reuse surviving credentials.
  • 04GitHub Actions tokens are out of scope for this revocation feature because they are minted per job and expire when the job ends.
  • 05The article recommends assigning the "Manage enterprise credentials" permission before an incident, reviewing Settings > Credentials, and pinning GitHub Actions to full commit SHAs rather than version tags.

It is not an active exploitation alert or critical patch, but the article grounds its importance in concrete recent compromises involving durabletask, Trivy, and Checkmarx.
