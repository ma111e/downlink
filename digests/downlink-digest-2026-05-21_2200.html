    Background: GitHub Actions Attack Surface

    GitHub Actions workflows execute arbitrary shell commands with full access to all secrets injected into the CI environment. When a repository grants id-token: write permission, workflows can mint short-lived OIDC tokens to authenticate directly to cloud providers (AWS, GCP, Azure) without static credentials. This makes a CI runner a high-value target: a single workflow execution can yield cloud access keys, SSH private keys, Kubernetes configs, Docker registry credentials, and all repository secrets simultaneously.

    Branch protection rules are the primary gate preventing unauthorized workflow injection. Repositories that do not require mandatory pull request reviews before merging to the default branch allow any account with write access—or, under certain weak configurations, accepted pull requests from public forks—to land workflow changes directly on the main branch. Megalodon systematically exploited this gap at scale.

    The campaign is a textbook example of direct Poisoned Pipeline Execution (d-PPE), a CI/CD attack where an adversary with write access injects malicious code directly into workflow definition files. Unlike indirect PPE (i-PPE), which requires tricking a maintainer into merging a malicious pull request from a fork, d-PPE exploits direct push access to the default branch, completely bypassing pull request review gates.

    Key enablers identified in the article:

    • Workflow YAML files receive less security scrutiny than application code.
    • Bot-like commit messages (e.g., “ci: add build optimization step”) blend into high-velocity repositories without raising suspicion.
    • CI runners have wide-open outbound internet access by default, with no firewall to bypass.
    • Ephemeral runner environments leave minimal forensic traces after job completion, making post-incident host forensics ineffective.

MiniUpdate Malware Family

AppDomainManager Hijacking & Native EDR Evasion

MiniUpdate introduced a technique that manipulates a .NET application’s .config file to hijack the initialization process. The configuration includes:

  • <etwEnable enabled="false"/> – disables Event Tracing for Windows, the primary telemetry source for .NET EDR monitoring.
  • <bypassTrustedAppStrongNames enabled="true"/> – skips strong‑name signature validation, allowing unsigned payloads to load silently.
  • <publisherPolicy apply="no"/> – prevents Microsoft publisher redirects to patched assembly versions.
  • <requiredRuntime safemode="true" imageVersion="v4.0.30319"/> – forces a predictable runtime environment to reduce crash‑related alerts.
  • <probing privatePath="."/ – forces local sideloading of a custom AppDomainManager type.

This means the malicious InitInstall.dll runs before the host application’s Main() method, in a context where the .NET runtime itself has been told to not report telemetry or validate code signatures.

Execution Flow & Persistence

  1. setup.exe loads setup.exe.config, which triggers InitInstall.dll.
  2. InitInstall.dll decrypts configuration strings using a two‑step reverse‑then‑ROT13 cipher, then launches a fake installer spinner UI thread.
  3. It stages the payload by copying and renaming files (e.g., setup.exe → update.exe, UpdateChecker.dll → the MiniUpdate payload) into AppData\[videoconferencing app]\bin\update.
  4. A scheduled task is created (daily at 09:30 local time) and immediately triggered to run update.exe.
  5. update.exe with its own .config disables ETW again and loads Updater.dll. This loader verifies the process is update.exe and its parent is svchost.exe (anti‑sandbox).
  6. Finally, UpdateChecker.dll is loaded and the CheckForUpdates export is called, starting the RAT.

Core RAT Capabilities

The MiniUpdate RAT communicates via HTTP GET to /agent/poll with a Chrome‑like User‑Agent and cycles through three C2 domains using Base64‑encoded binary commands. Across variants (March: 16 opcodes; April: 18 opcodes), capabilities include:

  • Arbitrary shell command execution (cmd.exe /c).
  • In‑memory DLL loading and export invocation.
  • Process enumeration and termination.
  • File upload, including chunked transfers for large files (added in April variants).
  • UAC elevation request.
  • Logon‑triggered scheduled task creation (WindowsSecurityUpdate) for persistence, with self‑removal and reinstallation functions.

Campaign 1: Rising Tension (February 2026)

In the weeks leading up to the conflict, Nimbus Manticore initiated a phishing campaign targeting employees in the software and aviation sectors, primarily in Saudi Arabia and Australia. AppDomain hijacking abuses the .NET runtime to load a malicious DLL through a trojanized XML .config file that specifies an attacker-controlled AppDomainManager class. When the victim launched Setup.exe, the configuration forced the loading of uevmonitor.dll. Campaign 2: During Operation Epic Fury

First-Stage Execution

When the user executed Setup.exe, the configuration file triggered the loading of InitInstall.dll via AppDomain hijacking. The loader demonstrated light obfuscation (ROT13 and reversed strings) but contained readable function names. Its execution began by displaying a fake installation progress window to mimic a legitimate software installation while simultaneously launching the genuine Zoom installer (Zoom_cm.exe). This arrangement made the victim believe a normal Zoom installation was underway.

The malware then entered a loop that lasted roughly one minute, continuously monitoring the system for the creation of a scheduled task with the format ZoomUpdateTaskUser-<current user SID>. This task is normally created by the legitimate Zoom installer during setup. When detected, the loader hijacked the task, modifying it to execute the second-stage component instead. By leveraging an existing Zoom task rather than creating a new one, the malware attempted to blend into legitimate system activity and evade detection.

Second-Stage Deployment and MiniFast Loading

The next-stage components were copied to C:\Users\<USER>\AppData\Local\Zoom\bin\update. This directory contained the same trusted Microsoft-signed binary, now renamed Update.exe, a configuration file (UpdateConfig.xml) for another AppDomain hijack, the second-stage loader Updater.dll, and the final backdoor UpdateChecker.dll. When the hijacked scheduled task ran Update.exe, the .NET runtime loaded Updater.dll.

The second-stage loader performed anti-analysis checks: it only continued execution if the hosting process name was update.exe and the parent process was svchost.exe. This ensured that the DLL was executed through the scheduled task persistence mechanism rather than directly by a user or analyst. After validation, the loader dynamically loaded UpdateChecker.dll and called its exported function CheckForUpdates, the entry point for the MiniFast backdoor.

MiniFast Backdoor Analysis

MiniFast is a 64-bit Windows DLL exposing a single export (CheckForUpdates). It verifies the same process chain (process update.exe, parent svchost.exe) before beginning host reconnaissance. The backdoor collects the username, hostname, and domain information, then registers with its command-and-control (C2) infrastructure using a structured HTTP API. To blend into normal traffic, it impersonates Chrome with the user-agent string Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/146.0.0.0 Safari/537.36.

Communication follows a REST-like pattern with JSON-formatted data:

  • POST /rg – initial handshake providing a clientId (machine name and user)
  • POST /agent/init – victim registration with token, PC name, user, domain, and elevation status
  • GET /agent/poll?token= – task retrieval
  • POST /agent/result – command result upload
  • PUT /upload/ – file exfiltration
  • GET /files/ – file download from C2

Task retrieval polls return Base64-encoded structures containing opcodes and arguments. The supported command set includes:

  • 0x02 list directory
  • 0x03 move/rename
  • 0x04 execute shell command via cmd.exe /c
  • 0x05 enumerate processes
  • 0x06 delete file/directory
  • 0x07 download file
  • 0x08 upload file
  • 0x09 enumerate drives
  • 0x0A kill process by PID
  • 0x0B load DLL and call exported function
  • 0x0C create directory
  • 0x0D create ZIP archive
  • 0xB0 request UAC elevation via runas
  • 0xB1 install persistence (scheduled task WindowsSecurityUpdate)
  • 0xF0 set poll interval
  • 0xF1 idle command acknowledge
  • 0xF2 set jitter

Results are serialized, Base64-encoded, and sent back to the C2 with the task identifier, status code, and output.

2. The Offensive Value of Kernel Drivers

  • Kernel driver vulnerabilities can offer local privilege escalation or, in BYOVD attacks, the ability to tamper with EDR and other security products.

3. Device Object Creation and the Central Role of AddDevice

3.1 Driver Load Without Hardware

  • Some non-PnP drivers create device objects unconditionally in DriverEntry (e.g., the Multidev_WDM demo driver). These can be deployed with a simple sc.exe create and sc start.
  • Most device drivers, however, only fully initialize when the Plug and Play (PnP) manager invokes their AddDevice callback upon detecting a matching device.

3.2 The AddDevice Callback

  • All PnP-compatible drivers must supply an AddDevice routine. It is responsible for creating functional device objects (FDOs) and filter device objects, and for initializing device extensions that hold driver-specific state.
  • AddDevice is not called from DriverEntry; it is invoked by the PnP manager when a new device node is discovered and the driver is selected to control it or filter its stack.
  • The function also attaches the new FDO to the device stack (with IoAttachDeviceToDeviceStack), making the stack ready for IRP dispatch. If AddDevice is never called, the driver’s dispatch routines will never receive IRPs through a PnP device stack, and many vulnerable code paths—guarded by checks on device extension fields—remain unreachable.

3.3 PnP Minor IRP Handling

  • After AddDevice, further initialization occurs in IRP_MJ_PNP dispatch routines (for WDM) or PnP/power event callbacks (for WDF KMDF). These handle start, stop, and device validation steps.
  • The article notes that drivers may perform configuration file checks, hardware probing, and device interface registration in these callbacks, adding more layers of hardware dependency.

  • Pure filter drivers pass through IRPs but neither deny nor accept IRP_MJ_CREATE; they forward it down. On a stack where the PDO rejects CREATE, no handle can be opened.
  • The article presents a class‑agnostic deployment method: add the filter’s service name to the ‘UpperFilters’ registry value of a device class like Disk Drive (GUID {4d36e967‑e325‑11ce‑bfc1‑08002be10318}), then mount a virtual disk. The new disk creates a full device stack (disk.sys, partmgr, etc.) that includes an FDO capable of accepting CREATE, with the filter now sitting on top.
  • This was demonstrated with a gaming mouse filter driver (GMLXDFltr) successfully loaded on a disk stack, receiving IRP_MJ_CREATE when the handle to PhysicalDrive was opened.
  • The technique works regardless of the original hardware class of the filter driver, because the UpperFilters setting is global per class.

4.4 Forced Driver Replacement Without INF Files

  • The article shows that INF file digital signatures are not a kernel‑mode security boundary. An attacker can bypass the entire INF mechanism by:
    1. Deploying the driver with sc.exe.
    2. Creating a software device node (or selecting an existing one).
    3. Directly writing the required registry keys to bind the driver’s service name to the device node under HKLM\SYSTEM\CurrentControlSet\Enum and the class key under HKLM\SYSTEM\CurrentControlSet\Control\Class.
    4. Restarting the device (disable/enable) to trigger AddDevice.
  • This was demonstrated with AwinicSmartKAmps.sys, which was successfully bound to a spoofed hardware ID and produced a functional device stack, even though the driver was shown as “not digitally signed” in Device Manager—a warning that refers only to the absent INF/CAT file, not to the kernel driver binary itself.

5. Active Hardware Probing and Its Limitations

The article distinguishes three categories of hardware interaction:

  • Neutral use: fixed-address MMIO like Local APIC, present on all systems, no gating.
  • Vulnerable use: user‑controllable physical addresses, unconditionally reachable, no gate.
  • Hardware gating: code that reads a chip ID register via MMIO from a PCI BAR and only proceeds if a specific value is returned.

For hardware‑gated code that performs active probing (PCI config, remote ACPI methods, I2C/SPI GPIO, etc.), the article acknowledges that bypassing these checks purely from user mode is generally not possible without additional kernel components. However, it notes that the majority of driver code does not perform such probing, and that the techniques described still expose a large amount of otherwise unreachable code.

6. Implications for BYOVD and Defense

  • The work demonstrates that hardware gating is often an illusory barrier: many vulnerabilities previously dismissed as “requires hardware” can be exploited by an admin with only user‑mode capabilities.
  • As Microsoft continues to reduce the pool of easily exploitable cross‑signed drivers, attackers may shift focus to these hardware‑gated drivers, making it critical for defenders to understand the forensic footprint of the demonstrated techniques.
  • Forensic indicators include the presence of software‑emulated device nodes (especially with unusual hardware IDs or driver bindings), unexpected UpperFilters and LowerFilters values in device class registry keys, and Device Manager entries showing drivers installed without a digital signature (no INF/CAT link).

7. Conclusion

The article provides a practical methodology for security researchers and defenders to assess whether a kernel driver vulnerability that appears hardware‑dependent is actually reachable. By manipulating the PnP subsystem with software‑emulated devices and filter class restacking, an attacker can trigger driver initialization and access vulnerable code paths without the original hardware. These findings expand the BYOVD threat landscape and underscore the need to monitor for anomalous device configurations that may indicate attacker preparation.

  • A bare software-emulated device node created via SetupAPI or Software Device API is not user-accessible because PnpManager's dispatch table returns STATUS_INVALID_DEVICE_REQUEST for IRP_MJ_CREATE; at least one driver in the stack must handle CREATE for a handle to be opened.
  • Filter drivers can be run on entirely unrelated hardware classes—such as a gaming mouse filter placed on a disk stack—by adding their service name to the UpperFilters entry of a device class like Disk Drive and then mounting a virtual disk, dramatically expanding the reachable attack surface.
  • The INF file digital signature requirement is not a security boundary; an attacker can completely bypass it by directly writing the necessary registry structures under HKLM\SYSTEM\CurrentControlSet\Enum and Class keys to bind a driver to a device, needing only a standard sc.exe service creation.
  • Creating custom device stacks without the original upper filter drivers that normally block user-mode IRP_MJ_CREATE can expose vulnerabilities in drivers that were never intended for user interaction, turning otherwise unreachable bugs into viable BYOVD targets.
Key Points
  • Many Windows kernel driver vulnerabilities are gated by hardware presence because the driver only creates its device objects and initializes internal state when AddDevice is called by the PnP manager upon detecting the hardware.
  • Using software-emulated device nodes with spoofed hardware IDs created through SetupAPI or Software Device API can trigger a driver's AddDevice routine, making device objects accessible without physical hardware.
  • Filter drivers that do not handle IRP_MJ_CREATE can be exposed by adding their service name to the UpperFilters registry key of a class like Disk Drive and then mounting a virtual disk to build an exploitable device stack.
  • INF file signature requirements are not a security boundary; attackers can force a driver onto a device by directly creating registry structures under Enum and Class keys after deploying the driver with sc.exe.

EXEC Executive Overview

This digest covers a broad array of threats and law enforcement actions reported within the coverage period, headlined by the Megalodon campaign that injected malicious CI/CD workflows across thousands of public repositories, and a significant escalation in Iranian state-sponsored operations following the regional conflict that began in late February 2026. 01 Megalodon: Large-Scale GitHub Supply Chain Attack and npm Poisoning

The Megalodon campaign pushed over 5,700 malicious commits to 5,561 public GitHub repositories within a six-hour window on May 18. Attackers exploited repositories lacking branch protection to inject counterfeit GitHub Actions workflows that exfiltrated cloud credentials, SSH keys, API tokens, and OIDC tokens via HTTPS POST to a remote C2 server. The adversaries used throwaway accounts with random eight-character usernames and forged commit identities such as build-bot and ci-bot to blend into legitimate development activity. Two payload variants were observed: SysDiag triggered on every push and pull request, while Optimize-Build activated only on workflow_dispatch to target CI/CD runners more covertly.

The campaign extended into the npm ecosystem; at least nine Tiledesk repositories were compromised, resulting in poisoned npm package versions that propagated the backdoor to downstream consumers. In response, npm invalidated all granular access tokens with write access that lacked two-factor authentication and urged migration to Trusted Publishing. The attack demonstrates the growing risk of direct poisoned pipeline execution attacks against open-source software supply chains.

02 Iranian APT Operations Escalate Amidst Regional Conflict

Two Iranian threat groups significantly intensified their activities following the start of the U.S. military campaign against Iran on February 28, 2026. Nimbus Manticore (IRGC-affiliated) resurfaced with newly adopted techniques, including the MiniFast backdoor developed using AI-assisted practices, SEO poisoning that directed victims to a fake SQL Developer download site, and a trojanized Zoom installer that hijacked the legitimate application’s scheduled task to execute follow-on malware. The campaign targeted the aviation and software sectors in the United States, Europe, and the Middle East, marking the group’s first confirmed targeting of US-based organizations.

Concurrently, Screening Serpens deployed six new remote access Trojan variants (grouped as MiniUpdate and MiniJunk V2) between February and April 2026. The group used AppDomainManager hijacking to manipulate .NET configuration files and disable runtime security mechanisms like Event Tracing for Windows. Their social engineering lures included fake job requisitions impersonating a global air carrier and spoofed video conferencing invitations, targeting technology sector professionals in the United States, Israel, the UAE, and two other Middle Eastern nations. Each victim received a dedicated set of Azure-hosted C2 domains for resilience.

03 Other State-Sponsored Espionage Activity

The Belarus-aligned Ghostwriter group (UAC-0057) targeted Ukrainian government entities with phishing emails themed around the Prometheus learning platform. The infection chain delivered a JavaScript downloader dubbed OYSTERFRESH that decoyed a document while writing the OYSTERBLUES payload to the Windows Registry. OYSTERBLUES collects system information and fetches further JavaScript to load Cobalt Strike. Separately, Cloud Atlas activity was documented deploying new tools including PowerCloud for exfiltrating administrator account data to Google Sheets, along with PowerShell backdoors for file theft and Kerberoasting-based credential theft. Attackers maintained persistence through reverse SSH, RevSocks, and Tor tunnels, and patched termsrv.dll to enable multiple RDP sessions. ROADtools, an open-source framework for enumerating Microsoft Entra ID and manipulating tokens via legitimate APIs, continues to be leveraged by nation-state actors including Midnight Blizzard and Curious Serpens for discovery and defense evasion in cloud environments.

04 Law Enforcement Actions Disrupt DDoS Infrastructure and Fraud Schemes

Dutch financial crime investigators seized 800 servers linked to Stark Industries, a hosting firm founded just before Russia’s invasion of Ukraine and later placed on EU sanctions lists. The operation was transferred to a front company, WorkTitans, which was tied to DDoS attacks by the pro-Russian hacktivist group NoName057(16). A company director and a connectivity firm head were arrested on charges of indirectly providing economic resources to sanctioned Russian and Belarusian entities.

In a multinational operation, the KimWolf DDoS botnet was dismantled and its alleged administrator, 23-year-old Canadian Jacob Butler, arrested. KimWolf and its AISURU variant infected nearly two million IoT devices, launched over 25,000 attacks including against DoD IP addresses, and achieved record DDoS volumes peaking at 31.4 terabits per second. Law enforcement also seized 45 DDoS-for-hire platforms. Additionally, two former US executives pleaded guilty to misprision of a felony for knowingly concealing a global tech support fraud scheme that cost Americans an estimated $2.1 billion last year.

06 Research: Exploiting Vulnerable Drivers Without Hardware

New research demonstrates that attackers can trigger kernel driver vulnerabilities gated by hardware presence by using software-emulated device nodes with spoofed hardware IDs via SetupAPI or the Software Device API. This technique triggers the driver’s AddDevice routine and creates accessible device objects without physical hardware, broadening the attack surface for Bring Your Own Vulnerable Driver (BYOVD) attacks.

The Supreme Court is hearing Chatrie, the first digital privacy case since 2018’s Carpenter decision, to decide whether geofence warrants constitute unconstitutional general warrants under the Fourth Amendment. The case’s outcome could broadly affect reverse search techniques, including reverse keyword and AI-based searches that scan entire user bases for evidence of crimes.
