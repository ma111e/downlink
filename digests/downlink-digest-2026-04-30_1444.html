Campaign Overview and Distribution Architecture

Atos Threat Research Center documented a sophisticated threat campaign in March 2026 that targets enterprise administrators, DevOps engineers, and security analysts. The operation leverages search engine order poisoning across Bing, Yahoo, DuckDuckGo, and Yandex to manipulate search results for niche IT terminology. The delivery mechanism relies on a dual-stage GitHub distribution architecture designed to evade platform takedowns and maintain operational longevity. The first stage consists of a clean, SEO-optimized facade repository that appears benign to both users and automated scanners. This storefront contains a professional README file that redirects victims to a secondary GitHub repository hosting the malicious payload. This separation allows threat actors to rapidly rotate distribution accounts while preserving their search engine rankings through the untouched facade. Between early December 2025 and April 2026, the campaign deployed 44 distinct facade repositories impersonating various administrative and diagnostic utilities.

Tool Impersonation and Victim Profiling

The campaign strategically impersonates the administrative stack to automate victim profiling and ensure high-value targets. Malicious MSI installers are distributed under the names of essential IT tools including PsExec, AzCopy, Sysmon, LAPS, Kusto Explorer, ProcDump, and BgInfo. This targeting approach ensures that infections primarily land on workstations belonging to personnel with elevated system and network permissions. The psychological component leverages an irony lure by disguising payloads as utilities that defenders routinely use for investigation. Compromise of these accounts provides immediate pathways for lateral movement and sustained access to enterprise environments.

Malware Architecture and Execution Stages

The delivered payload is identified as EtherRAT, a Node.js-based remote access trojan that employs a layered, fileless-style execution model. The installation process begins with Stage zero, an obfuscated Windows batch script that downloads the Node.js runtime and prepares the execution environment. Stage one functions as an in-memory loader that decrypts subsequent payloads using AES-256-CBC encryption and executes them via module._compile(). Stage two establishes persistence through a registry Run key and disguises the process tree by wrapping the payload in conhost.exe with the –headless argument. The final stage deploys the core RAT, which generates a unique bot ID and computes a system-specific working directory. All strings, including server addresses and API endpoints, are encrypted to hinder static analysis. The malware continuously re-encrypts its own source code on disk after each execution cycle.

Decentralized Command and Control Infrastructure

A defining technical characteristic of the campaign is its blockchain-based dead drop resolving mechanism for command and control resolution. The malware avoids hardcoded domains or IP addresses by querying nine public Ethereum RPC endpoints in parallel. It retrieves the active C2 address from a specific Ethereum smart contract and uses majority voting to ensure reliable resolution even if some gateways experience downtime. A background timer executes this blockchain lookup every five minutes, allowing attackers to dynamically rotate infrastructure by submitting a single on-chain transaction. This decentralized model provides extreme resilience against traditional disruption techniques such as domain seizure or IP blocklisting. The smart contract’s transaction history reveals repeated state-changing calls that directly correspond to C2 address updates.
