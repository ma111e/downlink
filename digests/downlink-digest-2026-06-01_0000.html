Because the nonce was exposed publicly, Wordfence said the check was ineffective as an access control mechanism. This meant unauthenticated attackers could invoke the wpgmp_temp_access_support() handler.

    To observe what the malware did during installation, StepSecurity says it installed @redhat-cloud-services/notifications-client@6.1.4 in a controlled GitHub Actions workflow with Harden-Runner enabled in audit mode.     Execution Chain

    Stage 1: Preinstall Hook

    Each malicious package reportedly includes a preinstall script in package.json:

    "scripts": {
  "preinstall": "node index.js"
}

    The article emphasizes that this causes the malicious code to execute automatically when npm install processes the package, before application code runs.

    Stage 2: ROT-21 Outer Obfuscation

    StepSecurity says the top-level structure of index.js wraps the payload in a try { eval(...) } catch(e) {} block. Inside that wrapper, the content is ROT-21 encoded and reconstructed from a large numeric array using String.fromCharCode.

    The article notes that the silent catch block prevents errors from surfacing to the developer console. Decoding this first layer revealed a 1.27 MB JavaScript file containing two AES-128-GCM encrypted payload blobs.

    Stage 3: AES-128-GCM Decryption

    The decoded Stage 2 script contains two encrypted blobs:

    • _b: Bun runtime downloader
    • _p: Main implant

    The article provides the hardcoded decryption material for both blobs:

    • _b
      • Key: 4c26cf9791bce1bfd4b84eba80ce2754
      • IV: 1241582f04234f7192feacba
    • _p
      • Key: ec514c074caf0ffdce6c66a0e95753d8
      • IV: 1251c3b85365f9b56a956c10

    According to the article, _b decrypts into a small async JavaScript module that downloads bun-v1.3.13 from the official GitHub releases CDN, extracts it into a randomly named directory under /tmp/, and registers the Bun path in globalThis.getBunPath. The _p payload is then written to a randomly named JavaScript file in /tmp/ and executed via the downloaded Bun binary.

    Stage 4: obfuscator.io and B5 Cipher

    The main implant, _p, is described as 634 KB after decryption and obfuscated with obfuscator.io. The article says it uses a non-standard base64 alphabet with lowercase letters preceding uppercase letters, and a string table of 2,219 encoded entries that is rotated by 284 push/shift cycles before lookups occur.

    The most sensitive strings, including C2 domain names, the User-Agent string, and API paths, are additionally encrypted with a custom cipher designated B5. The article describes B5 as follows:

    • Master key derivation via PBKDF2 with:
      • 200,000 iterations
      • 32-byte output
      • SHA-256
    • Per-message key derived as SHA-256(master_key ∥ iv)
    • Decryption involving 3 rounds of inverse Fisher-Yates byte-substitution with XOR feedback chaining
    • Ciphertexts stored as standard base64 within the obfuscator.io string table, which itself is custom-base64 encoded

Containerization principles and why they matter

  • Cgroups
  • Capabilities
  • Seccomp

This architecture is central to the article’s threat model. Because containers rely on the host kernel and runtime components, successful compromise inside a container can become relevant to the host system itself, particularly when vulnerabilities or misconfigurations weaken isolation.

Main attack vectors identified

  • Exploitation of host system and container runtime vulnerabilities

It emphasizes that these vectors may appear independently or as components of a larger attack chain. The article also states that attackers rarely stop at a single compromised container; instead, they usually pursue access to Kubernetes clusters, secrets management systems, or other critical infrastructure components.

Host and runtime vulnerability exploitation

  • Secrets theft

Vulnerability examples

CVE-2019-5736

The article calls CVE-2019-5736 one of the most prominent and illustrative container-related vulnerabilities. It affected the runC runtime and allowed an attacker with root access inside a container to execute arbitrary code on the host with root privileges. The flaw stemmed from runC’s improper handling of the file descriptor for its own executable through /proc/self/exe, which enabled an attacker to overwrite the runC binary when the runtime briefly executed in the container’s context.

CVE-2022-0492

CVE-2022-0492 is described as a critical Linux kernel vulnerability that allows container escape and arbitrary command execution on the host. According to the article, the flaw involved improper privilege validation in the cgroups release_agent mechanism. The article says this posed particular risk to container infrastructures because an attacker with code execution inside a container could use it to break isolation and take control of the host.

CVE-2024-21626

The article presents CVE-2024-21626 as a critical runC flaw that allowed access to the host file system from within a container and, in some scenarios, complete escape. The root cause is described as improper handling of file descriptors and the process working directory during container startup or command execution through mechanisms such as docker exec.

Malicious actions inside a compromised container

A significant point made in the article is that attackers often do not need a sophisticated escape to achieve meaningful objectives. In many cases, the container itself contains valuable information and access material.

Examples listed in the article include:

  • User credentials
  • Service credentials
  • API keys
  • Access tokens
  • SSH keys
  • Environment variables containing secrets
  • Kubernetes ServiceAccount tokens
  • Configuration files
  • Application service data or databases

The article says these assets are often exposed due to operational practices or configuration mistakes, such as baking secrets into images, passing them through environment variables, or mounting them directly into containers. In Kubernetes environments, automatically mounted ServiceAccount tokens are singled out as especially valuable because they provide a way to interact with the Kubernetes API.

The article describes several possible attacker outcomes from compromising just one container:

  • Gaining access to external services
  • Compromising cloud infrastructure
  • Stealing user data
  • Impersonating a trusted service
  • Establishing persistence in the environment

It concludes that even a fully isolated container can become a major foothold if it contains sensitive data or can access internal services.

Container escape

The article defines container escape as bypassing container isolation to directly interact with the host. It notes that escape can stem from vulnerability exploitation, misconfiguration, or insecure API usage.

Importantly, the article presents escape as the logical culmination of many attacks on container infrastructure, because attackers frequently want to move from the isolated environment to the host or broader cluster. It also states that, in practice, misconfigurations are a more common root cause of successful escapes than complex vulnerability exploitation.

Risky capabilities and configurations

The article devotes considerable attention to Linux capabilities as a mechanism that can greatly weaken isolation when misused.

Privileged containers

Running a container with the --privileged flag is described as one of the most dangerous configurations. In this mode, the container receives all Linux capabilities, direct access to host devices, and the ability to interact with kernel interfaces. The article says that such a container effectively stops being an isolated environment and can, in many cases, behave similarly to root on the host.

An example is given in which namespace and PID-sharing behavior allow an attacker to use nsenter to spawn a shell from the host’s init process, demonstrating execution outside the container.

CAP_SYS_ADMIN

CAP_SYS_ADMIN is described as one of the most dangerous Linux capabilities in container security. The article notes that, although capabilities were meant to break up superuser privileges into granular categories, CAP_SYS_ADMIN has become a catch-all for many sensitive kernel operations.

The article states that this capability can allow a container to:

  • Mount file systems
  • Interact with cgroups
  • Modify certain kernel parameters
  • Work with loop devices
  • Use namespace management features

When combined with settings such as hostPath, CAP_SYS_ADMIN can allow an attacker to mount host directories into the container and access critical files. The example given shows an attacker mounting the host disk inside the container and modifying the root user’s shell configuration to inject a malicious payload.

CAP_SYS_MODULE

CAP_SYS_MODULE is described as giving direct access to kernel module loading and unloading. The article says this is high-risk because it enables direct interaction with kernel space, where modules run with maximum privileges.

Modern containerized applications are said to rarely require this capability, and the article states that it is almost universally banned in modern infrastructures. It presents a container escape scenario in which an attacker loads a malicious kernel module that triggers a reverse shell. The article explains that such a module must match the target kernel version, so attackers typically perform reconnaissance and often compile the payload externally before transferring it into the container.

CAP_SYS_PTRACE

CAP_SYS_PTRACE is presented as a debugging-related capability that can severely weaken isolation when misconfigured. The article explains that it allows reading and modifying the memory of other processes, controlling their execution, injecting code, and extracting sensitive information from memory.

The article states that if the host PID namespace is shared with the container through hostPID: true, an attacker who compromises the container can use ptrace to attach to host processes. It describes an example using a public proof-of-concept to inject code and trigger a reverse shell in a host process.

CAP_NET_ADMIN

CAP_NET_ADMIN is described as providing broad network stack management privileges. While the article notes that these operations are formally limited to the container’s own network namespace, it says this capability is often combined with misconfigurations such as hostNetwork: true, which exposes host network resources.

According to the article, an attacker with CAP_NET_ADMIN can:

  • Modify interface configurations
  • Manipulate routing tables
  • Interact with traffic filtering mechanisms
  • Change network stack behavior
  • Alter iptables rules for traffic redirection

The article specifically mentions man-in-the-middle scenarios, internal traffic interception, and masking of malicious activity.

Orchestration and management API exploitation

The article describes abuse of container management and orchestration APIs as one of the most dangerous and common vectors. It emphasizes that these attacks can be straightforward because the APIs are inherently administrative interfaces.

The article identifies the following control points as especially sensitive:

  • Docker API
  • Kubernetes API
  • Kubelet API

It notes that these interfaces can create containers, modify configurations, access host file systems, and execute commands in running containers. As a result, misconfiguration can make them single points of failure for the environment.

Exposed Docker API

The article calls the exposed Docker API one of the most notorious examples. If the Docker daemon is reachable over TCP without TLS or authentication, an attacker can interact with the host system as if they were a local administrator.

Actions the article says become possible include:

  • Deploying attacker-controlled containers
  • Mounting the host’s entire root file system
  • Executing arbitrary commands inside containers

The article states that in practice, compromising an unauthenticated Docker API usually leads to full host takeover after only a few API requests.

Kubernetes API abuse

The Kubernetes API server is described as the central control point for an entire cluster. The article says that if an attacker obtains a privileged ServiceAccount token, exploits weak RBAC, or finds an exposed API server, they can carry out broad and destructive operations.

It gives an example workflow in which an attacker:

  1. Compromises a Kubernetes API token.
  2. Enumerates the token’s permissions, often by querying capabilities one by one.
  3. Discovers that the token grants very high privileges.
  4. Uses a curl POST request to create a privileged pod via the Kubernetes API.
  5. Uses that privileged pod to perform host escape techniques.

The provided JSON pod specification uses privileged: true to explicitly enable an escape path.

Mounted Docker socket

The article also highlights a scenario in which a Docker socket is mounted inside a container. It states that if such a container is compromised, the attacker effectively inherits the Docker daemon’s privileges and gains control over all containers on the host.

The described attack progression is:

  • Find a container with a mounted Docker socket.
  • Use the socket to interact directly with the Docker daemon.
  • Create a privileged container through the API.
  • Use an escape technique from that privileged container to compromise the host.

Supply chain attacks

The article describes supply chain compromise as an increasingly important vector because it allows attackers to manipulate components before they ever reach runtime. It says container ecosystems are tightly connected to many external components, making security dependent not just on applications but on the entire build and delivery chain.

Poisoned images

One common scenario discussed is contamination of container images. The article says organizations frequently use public images from Docker Hub and other sources without fully verifying their origin or content. Threat actors are said to publish contaminated images disguised as popular utilities or services, so that once launched, the attacker gains code execution inside the trusted environment.

CI/CD compromise

The article also says that CI/CD systems are among the most frequent targets because they typically operate with elevated privileges. If an attacker gains access to a CI/CD platform, they can covertly alter image build stages rather than changing application source code.

Examples of malicious build-stage modifications listed in the article include:

  • Downloading a third-party binary during the build
  • Adding a hidden script
  • Modifying container configuration
  • Implanting a remote management mechanism

A notable point in the article is that the resulting container may still appear legitimate because its visible core functionality remains unchanged.

Takeaways and significance

The article concludes that the main threat in container environments comes not just from the container itself, but from the implementation of the container infrastructure as a whole. It says containers are frequently used as the initial foothold, after which attackers attempt either to escalate to the host operating system or to seize control of the infrastructure through containerization and orchestration APIs.

The core weaknesses repeatedly highlighted are:

  • Weak configurations
  • Excessive capabilities
  • Isolation flaws
  • Overexposed administrative interfaces
  • Insecure software supply chains

The article also points to a visible shift toward CI/CD-focused attacks, where compromising a single component can enable broad infrastructure takeover.

Security implications described by the article

  • Host protection
  • Strict access control in the orchestrator
  • Minimization of container capabilities
  • Configuration auditing
  • Runtime protection
  • Activity monitoring
  • Validation of the software supply chain

01 Supply Chain Intrusions and Developer-Focused Malware

The most consequential supply-chain story was the compromise of more than 30 npm packages in Red Hat’s @redhat-cloud-services namespace. Researchers said attackers likely used a compromised Red Hat employee GitHub account to inject malicious code, add a GitHub Actions workflow, and publish backdoored packages through trusted publishing. The resulting malware, dubbed Miasma, was described as a credential and secret stealer targeting developer workstations and CI/CD environments, with overlap to Mini Shai-Hulud and reported downstream impact to hundreds of GitHub repositories.

Additional analysis showed the packages used a heavily obfuscated preinstall payload that decrypted implants at runtime, downloaded Bun, harvested secrets from GitHub Actions, cloud platforms, Kubernetes, Vault, SSH, Docker, GPG, and .env files, and in some cases read runner memory to recover masked CI secrets. Researchers also said the malware could enumerate writable GitHub repositories, commit malicious workflows, and establish persistence via developer tooling configuration changes.

A separate npm case involved codexui-android, which Aikido said exfiltrated OpenAI Codex authentication tokens from ~/.codex/auth.json to an attacker-controlled domain. The same behavior was reportedly observed in Android apps packaging the npm component, reinforcing the broader trend of package-level compromise targeting developer credentials and machine-to-cloud trust chains.

02 Web Malware Delivery and WordPress Ecosystem Abuse

Multiple reports underscored the continued use of compromised websites as malware distribution infrastructure. Silent Push said the DriveSurge actor hijacked thousands of legitimate sites and used the zTDS traffic distribution system to profile visitors before serving either ClickFix lures or FakeUpdate prompts. WordPress remained a major abuse surface. GoDaddy identified malware on roughly 1,980 WordPress sites that hid command-and-control data inside Steam Community profile comments using invisible Unicode characters, then pulled JavaScript from hello-mywordl.info and deployed a PHP backdoor. In a separate WordPress threat, attackers were actively exploiting CVE-2026-8732 in WP Maps Pro to create rogue administrator accounts; Wordfence said it blocked thousands of exploitation attempts in a 24-hour period.
