The two critical flaws impact versions 2.20.1 and older and can be chained to achieve remote code execution. The first issue, CVE-2026-3965, involves a misconfigured rewrite rule that maps requests to /open/* directly to /api/*, unintentionally exposing protected administrative endpoints. The second flaw, CVE-2026-4047, arises because the authentication check treats paths as case-sensitive while the router matches them case-insensitively. This discrepancy allows attackers to use variations like /aPi/... to bypass security controls entirely.

Snyk researchers attribute both vulnerabilities to a fundamental mismatch between the security middleware’s assumptions and the Express.js framework’s actual behavior. Attackers exploit the flaws to modify the config.sh file and inject shell commands that download a miner to /ql/data/db/.fullgc. The malicious binary is hosted at file.551911.xyz and includes variants for Linux x86_64, ARM64, and macOS architectures. Infections have been confirmed across various server configurations, including setups protected behind Nginx and SSL. Hackers are exploiting two authentication bypass vulnerabilities in the Qinglong open-source task scheduler to deploy cryptominers on developers’ servers. Qinglong is a self-hosted time management platform that is highly popular among Chinese developers, featuring over 19,000 GitHub stars and more than 3,200 forks.

Vulnerability Breakdown

The security issues impact Qinglong versions 2.20.1 and older. When chained together, these flaws enable remote code execution. The specific vulnerabilities are documented as follows:

  • CVE-2026-3965: A misconfigured rewrite rule maps /open/* requests to /api/*, which unintentionally exposes protected administrative endpoints through an unauthenticated path.
  • CVE-2026-4047: The authentication check treats URL paths as case-sensitive, while the router matches them case-insensitively. This allows attackers to bypass authentication by sending requests with altered casing, such as /aPi/....

Exploitation & Malware Behavior

Snyk researchers report that attackers have targeted publicly exposed Qinglong panels since February 7 to deploy cryptominers. The initial compromise was identified by Qinglong users who reported a rogue hidden process named .fullgc utilizing between 85% and 100% of their CPU power. The process name deliberately mimics “Full GC,” an innocuous but resource-intensive operation, to evade detection. Once authenticated, attackers exploit the flaws to modify Qinglong’s config.sh file and inject shell commands. These commands download a miner to /ql/data/db/.fullgc and execute it in the background. The remote resource at file.551911.xyz hosts multiple binary variants, including builds for Linux x86_64, ARM64, and macOS.

Response & Mitigation Timeline

The attack activity continued with multiple confirmed infections across various server setups, including configurations running behind Nginx and SSL. Qinglong maintainers only responded to the situation on March 1. The maintainer acknowledged the vulnerability and urged users to install the latest update. However, the initial mitigation focused on pull request #2924, which concentrated on blocking command injection patterns. Snyk researchers noted that this approach was insufficient to address the underlying authentication bypass. The effective fix was ultimately delivered in pull request #2941, which corrected the authentication bypass directly in the middleware.

Technical Root Cause

Both vulnerabilities stem from a fundamental mismatch between the security middleware’s authorization assumptions and the Express.js framework’s routing behavior. Snyk researchers explain that the authentication layer assumed certain URL patterns would always be handled in a specific manner, while Express.js processed them differently. This architectural disconnect allows unauthenticated requests to reach protected endpoints despite the presence of middleware security controls.
