Overview

Splunk has released security updates to address CVE-2026-20253, a critical vulnerability in Splunk Enterprise. The flaw is rated 9.8 on the CVSS scoring system. According to the article, the issue could be exploited to perform unauthenticated file operations and even remote code execution.

Splunk said the vulnerability affects Splunk Enterprise versions below 10.2.4 and 10.0.7. The company described the vulnerable component as a PostgreSQL sidecar service endpoint that lacks authentication controls. Because of this missing authentication, any network-reachable user could invoke file operations without credentials.

Affected and Fixed Versions

The article lists the following affected and fixed Splunk Enterprise versions:

  • Splunk Enterprise 10.0.0 to 10.0.6 is fixed in 10.0.7.
  • Splunk Enterprise 10.2.0 to 10.2.3 is fixed in 10.2.4.
  • Splunk Enterprise 10.4 is not affected.

Splunk, which is part of Cisco, said Splunk Cloud is not impacted. The reason given is that Postgres sidecars are not used in Splunk Cloud.

Root Cause and Initial Impact

Splunk said an unauthenticated user could create or truncate arbitrary files through a PostgreSQL sidecar service endpoint. The vulnerability exists because that endpoint lacks authentication controls. This allows a network-reachable user to perform file operations without credentials.

The article frames the issue as significant because it can go beyond arbitrary file operations. watchTowr Labs reported that the weakness could be exploited to achieve pre-authenticated remote code execution on susceptible systems.

watchTowr Labs Technical Analysis

watchTowr Labs released additional technical details on CVE-2026-20253. The analysis focused on the /v1/postgres/recovery/backup and /v1/postgres/recovery/restore endpoints. According to the article, these endpoints can be abused as part of an attack chain leading to pre-authenticated remote code execution.

The described attack chain works as follows:

  • An attacker connects Splunk to an attacker-controlled database and dumps its contents into an arbitrary file using the /backup endpoint.
  • The attacker loads the dump of the attacker-controlled database into the local PostgreSQL instance using the /restore endpoint.
  • The restore operation includes a passfile argument specifying the path to /opt/splunk/var/packages/data/postgres/.pgpass.
  • That .pgpass file contains the password for the postgres_admin user.
  • SQL queries defined in the database dump are executed by Splunk’s PostgreSQL instance.

The article says this process allows attacker-controlled SQL to be restored into the local PostgreSQL instance. watchTowr Labs researchers Piotr Bazydlo and Yordan Ganchev said that once they could restore attacker-controlled SQL, they created a database dump template that provided a controlled file write.

File Write to Remote Code Execution

The article describes how an attacker could use PostgreSQL functionality to write attacker-controlled content to the file system. Specifically, it mentions defining a new function that uses lo_export, a PostgreSQL function used to extract a BLOB from the database and save it as a file. The malicious function can execute during the restoration process.

With an arbitrary file write primitive on the Splunk file system, an attacker could escalate further to remote code execution. The article gives the example of overwriting a Python script that Splunk frequently executes. One cited example is /opt/splunk/etc/apps/splunk_secure_gateway/bin/ssg_enable_modular_input.py, which could be modified to include a malicious payload.

Full Exploitation Sequence Described

The article summarizes the exploitation sequence as follows:

  • Create a database and configure it so that a user can authenticate without a password.
  • Grant that user sufficient permissions to invoke functions such as lo_export.
  • Use the /backup endpoint to place a dump of the remote database onto the Splunk file system.
  • Use the /restore endpoint to load the malicious database dump.
  • Trigger execution of the malicious function during the restore process.
  • Write an attacker-controlled Python script to the Splunk file system.

Reported Exploitation Status and Response

The article states that there is no evidence the flaw has been exploited in the wild. However, it also notes that the availability of exploit specifics could be enough to drive opportunistic attempts. The article concludes that users should move quickly to apply the fixes to stay protected.

Splunk’s response was to release fixed versions for the affected Splunk Enterprise branches. Users of the impacted 10.0.x and 10.2.x versions are directed to upgrade to 10.0.7 or 10.2.4, respectively.

  • The vulnerable PostgreSQL sidecar endpoint lacks authentication entirely, so any network-reachable user can invoke file operations without credentials.
  • The exploit path does not require direct PostgreSQL credentials at first because the restore flow can be made to use Splunk’s local .pgpass file containing the postgres_admin password.
  • A controlled file write can become remote code execution by overwriting a Python script that Splunk frequently executes, such as /opt/splunk/etc/apps/splunk_secure_gateway/bin/ssg_enable_modular_input.py.
  • Although Splunk reported no evidence of exploitation in the wild, watchTowr Labs has published enough technical detail about the /backup and /restore endpoint chain to make rapid patching especially urgent.

The email was disguised as a Medicare Transaction Facilitator (MTF) Independent Provider Attestation request and targeted an employee at a healthcare outcomes management company.

The phishing email used healthcare regulatory compliance language to create urgency and legitimacy. Its subject referenced a “Medicare Transaction Facilitator (MTF) Independent Provider Attestation,” and the body included a call-to-action button with the same Medicare attestation wording.

The call-to-action linked to:

The email was sent through Salesforce Email-as-a-Service infrastructure. The sending hostname and IP listed in the article were:

  • 18[.]117[.]31[.]220

X-SFDC headers confirmed an active Salesforce organization. The article notes that the EmailCategory was quickActionEmail, indicating that the message was triggered from within the CRM interface itself.

Authentication Results

The message passed all major email authentication checks described in the article:

  • SPF passed.
  • DKIM passed.
  • DMARC passed.
  • Composite authentication returned compauth=pass with reason code 100.

IRONSCALES reported a credential-harvesting campaign that used a compromised or malicious Salesforce CRM organization to send an email disguised as a Medicare Transaction Facilitator attestation request. The message passed SPF, DKIM, and DMARC with compauth=pass because Salesforce was an authorized sender for the outcomes.com domain. The email also used identical From and To addresses, aryan@outcomes.com, creating a self-send pattern through Salesforce.

The campaign’s call-to-action linked to nexaroco.com/providerrelations/, which served an Office 365 credential-harvesting page. Themis initially flagged the message at 50% confidence, and additional phishing signals from the IRONSCALES community confirmed the classification. The activity illustrates how attackers can leverage trusted CRM infrastructure and valid domain authentication to reduce the effectiveness of controls that rely heavily on sender legitimacy.

The reported lure also reflects a broader trend in phishing operations: attackers are increasingly aligning messages with business workflows that recipients may view as routine or compliance-driven. In this case, a healthcare- and Medicare-themed attestation request was paired with authenticated Salesforce delivery, increasing the appearance of legitimacy despite the credential theft destination.

03 AI Model Access Disruption Under U.S. Export Controls

Anthropic suspended worldwide access to its Fable 5 and Mythos 5 AI models after receiving a U.S. government export-control directive ordering it to block access by foreign nationals. According to Anthropic, the directive applies to foreign nationals both inside and outside the United States, including the company’s own foreign-national employees. The company said this forced it to disable the models for all customers.

Fable 5 had started rolling out on June 9 as a free limited-time offering for Pro, Max, and Enterprise customers through June 22 before being taken offline. Anthropic said new sessions would fall back to a user’s default model or Claude Opus 4.8, existing Fable 5 sessions would terminate with an error, and Platform requests to Fable 5 would fail.

Anthropic disputed the rationale for the directive, stating that it appeared to stem from a narrow reported jailbreak involving a prompt to read a specific codebase and fix software flaws. The incident is notable for security teams tracking AI governance, model availability risk, and the intersection of software vulnerability analysis capabilities with export-control policy.
