Credential Exposure and Account Types

According to SOCRadar, generic admin accounts and built-in Fortinet system accounts represented the majority of compromised credentials. Generic admin accounts accounted for 35% of the compromised credentials, while built-in Fortinet system accounts accounted for 28.3%. It said this gave the attacker a reliable target list before brute force was needed. SOCRadar also noted that organization-specific accounts were significant because they showed the attacker had compromised accounts created by organizations themselves, possibly sourced from prior breaches where passwords were never changed.

Affected Sectors and Geography

The article identifies telecom, government, and education as the top three impacted sectors. The most exposures were located in India, the U.S., Mexico, Colombia, and Thailand. Hudson Rock said the scale of the breach touched nearly every sector of the global economy and that the threat actors had built a verified database of working credentials for some of the largest enterprises.

The incident was described as affecting thousands of firewalls and VPN gateways across 194 countries. Reported Attack Workflow

The article says the threat actor mass-scanned the internet for Fortinet remote login endpoints. After identifying endpoints, the attacker used a bespoke tool to spray known login and password combinations against them. The goal was to break into exposed devices using credentials that were already known or leaked.

The attack was described as fully automated and built around a self-sustaining, two-step approach:

  • The threat actor attempts a curated list of leaked Fortinet passwords against devices across the internet.
  • Once access is obtained, the attacker passively monitors network traffic going through the devices to collect additional credentials, which are then used to compromise more appliances.

The article says the credentials used in the campaign are legitimate and valid. It also says the attackers verify each credential before adding it to a database of confirmed, working logins.

Techniques Described by NCSC

The U.K. NCSC described FortiBleed as a global campaign targeting internet-facing Fortinet firewalls and VPN gateways. It said the methods included brute-force, dictionary attack, and credential stuffing. The article frames these methods as part of the broader effort to access Fortinet appliances through exposed login surfaces.

FortiOS Password Hashing Detail

The article says the threat actors may have exploited older credential hashing mechanisms and historical storage practices for credentials in FortiGate configuration files. Arctic Wolf said Fortinet introduced PBKDF2-based password hashing for administrator credentials in FortiOS 7.2.11, 7.4.8, and 7.6.1. This replaced a legacy SHA-256-based storage mechanism.

However, Arctic Wolf said that when organizations upgrade from earlier versions, existing administrator passwords remain stored as SHA-256 hashes until the corresponding administrator successfully logs in after the upgrade. As a result, many organizations may still be storing administrator credentials using older SHA-256 with Salt hashing mechanisms. CISA’s recommendations include ensuring PBKDF2 is used to store administrator credentials and removing weaker legacy hashes.

A Fortinet spokesperson told The Hacker News that the data involved is likely a resharing of data from previous incidents, as well as brute-forcing of credentials. Fortinet said it was not related to any current incident or advisory. The company urged organizations to follow best practices, including regularly rotating security credentials and enabling multi-factor authentication.

CISA Recommendations

CISA outlined several defensive steps for Fortinet customers. These include terminating all active SSL VPN and administrative sessions and resetting all Fortinet VPN and administrative passwords, especially on internet-facing systems. CISA also recommended enforcing strong password policies.

CISA further advised organizations to ensure PBKDF2 is used to store administrator credentials and to remove weaker legacy hashes. It recommended reviewing firewall, VPN, authentication, and domain controller logs for suspicious actions, including unauthorized configuration changes. CISA also urged enabling phishing-resistant MFA on all external gateways and administrative interfaces, reducing the attack surface, and locking down management.

Overview

Microsoft researchers detailed an exploit chain named AutoJack that can turn an AI browsing agent into a delivery vehicle for remote code execution on the host system. The article says the attack works when an agent is steered to load an attacker-controlled web page. After that page is loaded, its JavaScript can reach a privileged local service on the same machine and spawn a process on the host.

The article emphasizes that the attack requires no credentials, no sign-in screen, and no further user interaction once the agent loads the page. It says an attacker only needs to get the agent to open the malicious page, which could happen through a planted link, a URL field, or prompt injection.

Affected software and packaging details

The flaw is in AutoGen Studio, the open-source prototyping interface for Microsoft Research’s AutoGen multi-agent framework. The article makes a distinction between the stable PyPI package and pre-release builds.

According to the article:

  • A plain pip install autogenstudio installs the current stable release, 0.4.2.2.
  • Microsoft inspected 0.4.2.2 and found that it has no Model Context Protocol, or MCP, route.
  • Microsoft’s statement that the vulnerable MCP WebSocket surface “was never included in a PyPI release” is described as accurate for the stable build.
  • The vulnerable handler did ship to PyPI in two pre-release builds: 0.4.3.dev1 and 0.4.3.dev2.
  • The Hacker News says it downloaded and inspected both pre-release builds and found the MCP WebSocket route present.
  • The article says neither of the two pre-release builds had been yanked.

The article also notes that pip does not install pre-releases unless the user passes --pre or pins a pre-release version. As a result, a plain install was not exposed, but anyone who installed one of the two pre-release builds was exposed.

Technical chain

AutoJack chains three weaknesses in the MCP WebSocket.

Localhost trust check

The first weakness is that the socket trusted localhost. The article says this check was meant to block a normal browser pointed at a malicious site. However, a browsing agent running on the same machine is also localhost, so content it loads can inherit that localhost identity and pass the check.

Authentication bypass on MCP paths

The second weakness is that authentication middleware skipped MCP paths. The article says this was done on the assumption that the handler would verify tokens itself. The handler did not verify tokens, so the socket accepted unauthenticated connections regardless of the configured authentication mode.

Command execution from request parameter

The third weakness is that the endpoint took a command directly from a request parameter and ran it. The article says there was no allowlist restricting which executable could be launched.

Together, these conditions meant that a page on the open internet, when rendered by a local agent, could run an attacker-chosen command under the account running AutoGen Studio.

Demonstrated impact

Microsoft described the work as research and reported no exploitation in the wild. The proof of concept used a “Web Content Summarizer” agent. When the agent was fed an attacker URL, it launched calc.exe on the developer’s desktop through the AutoGen Studio process.

The article does not identify any active campaign or named threat actor exploiting AutoJack.

Response and fixes

Microsoft reported the behavior to the Microsoft Security Response Center. The maintainers hardened the main branch in commit b047730, associated with PR #7362.

The article says the fixed handler includes several changes:

  • It no longer reads the command from the URL.
  • Parameters are stored server-side behind a one-time session ID.
  • Unknown session IDs are refused.
  • MCP routes now run through the normal authentication path.

However, the article says this hardening had not landed in a PyPI release at the time of publication. It states that affected users of the pre-release builds should pull from GitHub main at or after commit b047730.

Mitigation guidance in the article

The article says users of the stable 0.4.2.2 release installed through a plain pip install autogenstudio are not affected because that release has no MCP route.

For users who installed a pre-release build, the article says there was no patched PyPI build available and that the real fix is to pull from GitHub main at or after commit b047730.

Until a release is available, the article recommends separating the components required for the attack. Specifically, it says not to run AutoGen Studio on the same machine as a browsing or code-execution agent that touches untrusted content, because the chain only works when both share the same localhost. If they must run together, the article recommends isolating them in separate containers or virtual machines and running AutoGen Studio under a low-privilege account.

Broader pattern described

The article says the AutoGen Studio bugs are patched in source, but the broader pattern remains. It says Microsoft expects similar issues in other agent frameworks where a local service has too much power, localhost is treated as a security boundary, and an agent opens untrusted pages.

The article compares this pattern to other referenced research and coverage, including ChatGPhish, where ChatGPT page summaries became a phishing vector, and Microsoft’s Semantic Kernel remote code execution research tracked as CVE-2026-26030 and CVE-2026-25592.

The article concludes that another localhost check is not enough. It says control planes should be authenticated, process execution should be kept behind an allowlist, and agents should have an identity that is not the developer’s own session.
