Summary
GitHub’s upcoming npm v12 will block dependency install scripts, Git-based dependencies, and remote URL dependencies from running or resolving automatically unless explicitly approved. The change is meant to reduce npm supply-chain attacks that abuse npm install for automatic code execution.

GitHub announced that npm v12, expected next month, will introduce security-focused changes to reduce supply-chain attacks that abuse behaviors triggered by the npm install command. The command is widely used to download dependencies and run install-related package scripts after cloning projects, pulling updates, or during CI/CD builds. GitHub’s main change is to stop trusting automatic code execution and non-registry dependency sources by default.

Starting in npm v12, npm install will not run dependency preinstall, install, or postinstall scripts unless they are explicitly approved. The same approval requirement applies to native module builds triggered through node-gyp and prepare scripts from Git, local file, and linked dependencies. GitHub will also require explicit permission before fetching dependencies from Git repositories or resolving remote URL dependencies such as HTTPS tarballs.

The article says these defaults could have disrupted techniques used in recent npm supply-chain attacks, including malicious install-script campaigns, data-stealing packages, and Git dependency abuse documented in Shai-Hulud attacks. Projects that rely on these behaviors for legitimate workflows will need to explicitly opt in before upgrading to npm v12. GitHub recommends upgrading to npm 11.16.0 or newer first, because it displays warnings for actions that will break under version 12.

Shadow AI in the Software Supply Chain

The article describes Shadow AI as the new Shadow IT. It says AI is already inside the software supply chain and did not wait for formal organizational approval. Developers are described as adopting AI tools to improve productivity, even when those tools are not officially governed.

Examples given in the article include:

  • Developers pulling code suggestions through personal accounts.
  • Developers using unvetted third-party plugins.
  • Developers embedding AI-generated code blocks directly into production environments.

The article argues that each day organizations delay governance, the unmanaged AI footprint grows. It also says that simply blocking or blacklisting AI tools does not solve the problem. Instead, blocking tools may push developer usage underground.

Limits of AI Agent Self-Governance

The article identifies a major misconception: the idea that AI agents can be trusted to self-police compliance and validation. It states that AI agents cannot be relied on for this role. The reason given is that AI agents operate non-deterministically.

The article says AI agents do not understand:

  • Word-of-mouth policies.
  • Unwritten developer best practices.
  • The contents of a shared wiki.

If compliance boundaries exist only in documentation or verbal agreements, the article says automated agents will bypass them entirely. This is presented as a pipeline governance risk, especially as automated coding tools become more common.

Dependency and Package Risks

The article gives examples of how AI coding agents can introduce supply chain problems. When trying to resolve a bug, an AI agent may hallucinate package names. It may also silently pull vulnerable open-source dependencies from public repositories.

The public repositories named in the article are npm and PyPI. The article states that flawed or malicious components may not be discovered until after they have already blocked builds. In that scenario, any velocity gain provided by the agent may be wiped out.

Recommended Response

The article argues that the solution is not to slow AI adoption or restrict what agents can do. Instead, it calls for a shift in how governance is applied. The recommended approach is to build smarter guardrails at the software supply chain layer.

The article describes platform-level governance as the viable path for managing AI adoption safely within corporate security and governance practices. It says organizations that are succeeding are not necessarily working harder, but are applying smarter guardrails. The article also notes that most organizations are still in early stages.

EXEC Executive Overview

This digest window was dominated by active exploitation, developer ecosystem compromise, and policy movement around faster remediation of high-risk vulnerabilities. Reporting highlighted attacks against Langflow and Oracle PeopleSoft, continued evolution of self-propagating supply-chain malware, and defensive changes planned for npm in response to recent package ecosystem abuse.

01 Active Exploitation of Langflow and Oracle PeopleSoft

Attackers are actively exploiting CVE-2026-5027, a high-severity path traversal flaw in Langflow that allows arbitrary file writes on exposed servers. The issue affects the POST /api/v2/files endpoint, where the filename parameter in multipart form data is not properly sanitized. Tenable disclosed the vulnerability on March 27, 2026, after reporting it to Langflow more than two months earlier without receiving a response; Snyk reported fixes in langflow-base 0.8.3 and Langflow 1.9.0, with 1.10.0 listed as the latest release.

Separately, ShinyHunters claimed responsibility for ongoing data theft attacks against Oracle PeopleSoft cloud and on-premises customer instances. The group claimed to have stolen data from 300 PeopleSoft instances across more than 100 organizations, with education reportedly making up most of the affected sector. The actor described its exploitation path as a “gadget chain” involving old and zero-day vulnerabilities, with impact dependent on instance configuration.

Researcher Michael R identified exposed online directories containing apparent attack tooling related to the PeopleSoft activity, including MeshCentral agents, a defacement and credential spray script, and staging materials. The two cases underscore continued attacker focus on externally reachable enterprise platforms where file write, deserialization, or configuration-dependent exploit chains can quickly become data theft or persistence opportunities.

02 Software Supply-Chain Abuse and Developer Credential Theft

The Miasma credential-stealing framework was briefly exposed through GitHub repositories named “Miasma-Open-Source-Release” across numerous compromised developer accounts. SafeDep assessed that Miasma appears to be an evolution of the earlier Shai-Hulud worm, sharing many features, techniques, and code. The framework steals build environment and cloud credentials, compromises legitimate repositories and packages, and publishes trojanized versions to reach downstream developers.

A notable feature of Miasma is its use of GitHub itself for command-and-control-like functions, reducing dependence on separate infrastructure. The leaked code also revealed a dead-man switch capable of deleting files in a user’s home and Documents folders if a stolen GitHub token used for exfiltration is revoked. This behavior combines credential theft, repository abuse, and destructive pressure against victims attempting to cut off attacker access.

GitHub announced npm v12 security changes intended to disrupt similar supply-chain tactics. In npm v12, npm install will not run dependency preinstall, install, or postinstall scripts unless explicitly approved, and it will no longer fetch dependencies from Git repositories or remote URLs such as HTTPS tarballs unless permitted. GitHub said the defaults could have disrupted techniques seen in malicious install scripts, data-stealing npm packages, and Git dependency abuse documented in Shai-Hulud attacks.

03 AI Development, Shadow AI, and Developer Device Exposure

JFrog’s discussion of IDC’s 2026 Software Engineering Challenges Report framed Shadow AI as a growing unmanaged enterprise risk, replacing Shadow IT as developers use personal accounts, unvetted plugins, and AI-generated code in production workflows. The report, based on a survey of 1,000 enterprise and mid-sized organizations, described a widening gap between AI adoption goals and governance readiness. The article argued that AI agents cannot be assumed to self-police compliance because their behavior is non-deterministic and may bypass policies that exist only as documentation or informal agreements.

Aikido described a parallel shift in where code is written, citing PostHog’s observation that roughly as many pull requests are being merged through Slack as through the code editor. The article said developers increasingly write code through AI agents, chat interfaces, and MCP servers that may be outside security team visibility. It connected this dispersion to recent supply-chain incidents such as TeamPCP activity, the Axios maintainer account hijacking, and the Glassworm attack, which emphasized compromise of developer devices and accounts rather than application vulnerabilities.

Developer endpoints were characterized as high-value control points because they often hold cloud credentials, SSH keys, npm publish tokens, Kubernetes configs, source code access, and production access. The reporting also noted limitations in traditional MDM and EDR visibility for malicious packages, IDE extensions, browser plugins, and postinstall hooks before credential theft occurs.

04 Federal Vulnerability Remediation Policy Tightens

CISA issued a new binding operational directive requiring U.S. federal civilian agencies to patch certain high-risk vulnerabilities within 72 hours. The accelerated timeline applies when vulnerabilities meet three of four criteria: public internet exposure, presence in the Known Exploited Vulnerabilities catalog, automated exploitability, and the degree of adversary control over the vulnerable system.

Agencies have 180 days to adopt the new time frame. The directive also requires agencies to assess when and how a vulnerable system may have been compromised before patching, reflecting CISA’s position that applying a patch does not generally evict an existing threat actor. CISA also urged state, tribal, and local governments, along with critical infrastructure owners and operators, to adopt similar practices.
