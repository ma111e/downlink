In plain words

Splunk is a tool that many organizations use to watch their computer systems for signs of break-ins. A critical security flaw in Splunk Enterprise allows anyone on the internet to take control of a Splunk server without a password, just by sending a specially crafted web request. Because Splunk acts as a watchdog, a compromised server lets attackers delete or change the very logs that would catch them, steal passwords and other secrets stored in those logs, and then use the server to break into other parts of the network. The US government ordered federal agencies to fix the flaw immediately because hackers are already exploiting it in the real world.

Summary
A critical unauthenticated remote code execution vulnerability (CVE-2026-20253) in Splunk Enterprise allows attackers to exploit a PostgreSQL sidecar service recovery endpoint via Splunk Web to achieve arbitrary file write and code execution; active exploitation has been confirmed, and federal agencies must patch immediately.

On June 10, 2026, Splunk disclosed CVE-2026-20253, a critical unauthenticated remote code execution vulnerability in Splunk Enterprise scoring 9.8 on the CVSS scale. The flaw arises from missing authentication on a PostgreSQL sidecar service recovery endpoint that is reachable through the Splunk Web interface, which acts as a reverse proxy without enforcing access controls. Versions 10.2.x prior to 10.2.4 and 10.0.x prior to 10.0.7 are affected, while version 9.4 and earlier are not vulnerable. By June 18, CISA added the vulnerability to its Known Exploited Vulnerabilities catalog and mandated federal agencies remediate by June 21 under Binding Operational Directive 26-04, after Splunk confirmed active exploitation in the wild.

The attack chain begins with an unauthenticated HTTP request to the exposed recovery endpoint, which accepts an empty Authorization header. An attacker leverages path traversal in the backupFile parameter to create or truncate files at arbitrary filesystem locations, then injects connection string parameters to redirect the database connection to an attacker-controlled server. This allows dumping attacker-supplied content to disk, turning a file creation primitive into an arbitrary file write. By pointing the connection at the local .pgpass file containing plaintext credentials, the attacker authenticates as the privileged postgres_admin user without knowing the password, and ultimately overwrites a scheduled Python script to achieve code execution under the Splunk service account.

After compromise, an attacker can tamper with or delete security telemetry, harvest credentials and API keys from indexed data, disable logging, and pivot laterally into the internal network from the trusted Splunk host. Organizations are urged to upgrade to fixed versions, or temporarily disable the PostgreSQL sidecar service if upgrading is not immediately possible, though this workaround breaks Edge Processor, OpAmp, and SPL2 data pipelines. Placing the Splunk Web interface behind a zero trust access layer is recommended to reduce exposure, especially in AWS deployments where the vulnerable sidecar may be enabled by default.

  • The vulnerability hinges on the proxy passing requests to a localhost-only sidecar endpoint without any authentication enforcement, and the sidecar itself accepts an empty Authorization header (Basic Og==), completely bypassing access control without any credential guessing.
  • Attackers can supply a connection string in the database parameter (e.g., hostaddr=attacker-db.com) to replace the target host, transforming a file-truncation bug into a full arbitrary file write by dumping attacker-controlled data from an external PostgreSQL server.
  • Splunk stores plaintext credentials in a .pgpass file, and an attacker injects a passfile parameter in the connection string to force the system to read that file, authenticating as the privileged user without knowing the password and enabling remote code execution.
  • The temporary workaround of disabling the PostgreSQL sidecar service breaks Edge Processor, OpAmp, and SPL2 data pipelines, but leaves core search and indexing unaffected, forcing organizations to choose between remediation and operational functionality.
  • CISA's emergency directive mandated federal agencies to patch within three days of the KEV listing, an unusually rapid deadline that signals both confirmed active exploitation and the extreme risk a compromised Splunk instance poses for log tampering and lateral movement.

In plain words

The developers of Vite, a widely used website-building tool, decided to remove a new feature that would have sped up work for React users because it made the tool’s download file 5 megabytes larger. Since Vite is downloaded 12 million times a week by all kinds of projects—not just React ones—that extra size would waste time and bandwidth for a huge number of people who would never use the feature. This shows how teams that build shared tools must balance helpful additions against the hidden costs of making software heavier for everyone.

Summary
Rolldown and Vite maintainers pulled a Rust-based React Compiler integration after determining the 5MB binary size increase was unjustifiable for all users, opting instead to explore a smaller opt-in implementation.

Rolldown and Vite maintainers removed a Rust-based React Compiler integration after it increased the Rolldown binary from 28.7MB to 33.8MB — a 17% jump. Boshen, an Oxc maintainer, said the cost could not be justified for a feature that would ship to all Vite users by default. The integration had been in discussion for over a year and arrived through an Oxc plugin that raised the napi transform addon from 3.51 MiB to 8.66 MiB.

The decision sparked pushback from both sides. Some developers argued React-specific compiler support should live in an opt-in vite-plugin-react instead of burdening every install. Evan You stressed that Vite is framework-agnostic and downloaded 12 million times a week, so a 5 MB increase paid by non-React users was unacceptable. The arrival of Rspack 2.1 with Rust React Compiler support — 7–13x faster than Babel — made the performance versus size tradeoff more pressing.

Technical work is now focused on reducing the binary cost. Boshen removed a JSON-serialization round-trip that converted a Rust AST to a Babel AST and back, cutting 759 KiB and improving transform speed by 27%. A new napi-rs proposal for compressed addons could shrink a 25.74 MB native addon to 8.15 MB with no runtime penalty. Boshen is also experimenting with a smaller implementation that yields a 2x performance gain with only a 1.4 MB increase, while the team explores raw AST transfer to eliminate plugin overhead.

  • The Rust React Compiler integration incurred a performance penalty from an unnecessary JSON serialization round-trip that converted the Rust AST to a Babel AST and back; removing it cut 759 KiB and sped up transformation by about 27%, showing how interop overhead can undercut native performance gains.
  • A proposed napi-rs compression flag could shrink native addon binaries by nearly 70% with no runtime penalty, potentially resolving the binary size objection by making large addons small on disk until first use.
  • The direct integration is up to 50% faster than a plugin because plugins introduce data passing and AST cloning overhead, meaning that an opt-in plugin model might reintroduce the very build-time costs the native integration was meant to eliminate.
