In plain words

A GitHub tool called codfish/semantic-release-action was secretly changed so that projects using certain version labels would run an attacker’s hidden code during their automated release process. Any open-source or company project that used those affected labels after June 24, 2026 at 15:39:06 UTC may have exposed GitHub access tokens, personal access tokens, and other secrets. The stolen access could let the attacker copy private material, alter other repositories, and spread the same hidden code through developer tools and software packages, putting downstream users at risk too.

Summary
The codfish/semantic-release-action GitHub Action was compromised via force-moved version tags, causing affected workflows to run an obfuscated Bun-based payload that steals GitHub tokens, uses GitHub as command-and-control, and attempts supply-chain and developer-machine propagation. Any workflows that used the affected tags after June 24, 2026 15:39:06 UTC should be treated as potentially compromised and have secrets rotated.

On June 24, 2026 at 15:39:06 UTC, an attacker force-pushed a malicious commit to codfish/semantic-release-action and redirected several version tags to that commit. Any GitHub Actions workflow that used one of the affected tags after that time executed the attacker’s payload inside the runner. The compromised action affects workflows using tags including v5, v4, v3, v2, and several specific version tags listed in the article.

The attacker changed the action from a Docker-based runner into a composite action that installed Bun through oven-sh/setup-bun and ran an obfuscated index.js payload. Both injected steps used if: always(), ensuring the payload ran even if earlier steps failed or were skipped. The payload steals GitHub OIDC tokens, searches for Personal Access Tokens matching known GitHub token patterns, encrypts collected data, and attempts to propagate a backdoor into other accessible repositories.

The malware uses public GitHub commit messages as command-and-control through dead-drop identifiers named RevokeAndItGoesKaboom and TheBeautifulSandsOfTime. It also targets AI coding assistant configurations, SSH hosts, and package publishing credentials for npm, PyPI, and RubyGems. StepSecurity reported that Harden-Runner observed GitHub API traffic used for exfiltration and added the compromised action to its Compromised Actions Policy to block affected workflow runs for customers with that policy enabled.

  • The attacker changed the action from a Docker-based action to a composite action so the malicious steps could install Bun and run the payload directly in the workflow, with both added steps guarded by if: always() to execute even after failures.
  • The payload’s use of Bun was deliberate because the article states Bun lacks the --require hook interception used by most Node.js security tooling.
  • Because the command-and-control and exfiltration traffic used GitHub’s public API, runtime monitoring saw outbound traffic primarily to api.github.com, making domain-based blocking less effective.
  • The campaign reused fingerprints from prior compromises of @antv/graphlib, echarts-for-react, and tanstack-react-router, including the same dead-drop identifiers, AES key, and Bun runtime requirement.
  • The malware attempted to spread from CI into developer environments by committing Claude Code and VS Code configuration files that automatically execute the payload when a repository is opened.
Key Points

  • 02Workflows that ran against the affected tags after the compromise executed a Bun-based, obfuscated JavaScript payload inside the GitHub Actions runner.
  • 03The payload steals GitHub OIDC tokens, harvests Personal Access Tokens matching known GitHub token patterns, encrypts the collected material with AES-128-GCM, and attempts to propagate a backdoor into other repositories accessible with the stolen credentials.
