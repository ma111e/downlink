The exploited flaw is CVE-2026-35273, a remote code execution vulnerability in Oracle PeopleSoft Enterprise PeopleTools. It is rated 9.8 out of 10 and requires no login or user interaction. The article states that an attacker only needs network access over HTTP to take over the server.

The vulnerability sits in the Updates Environment Management component, which is the piece behind the Environment Management Hub, also referred to as PSEMHUB. Oracle lists PeopleTools 8.61 and 8.62 as affected and says earlier unsupported versions are probably vulnerable too. Oracle credits researchers from TrendAI Zero Day Initiative and TrendAI Research for the report.

The key exposure described in the article is PeopleSoft with the Environment Management Hub reachable from outside. The article says organizations in that position should immediately lock those endpoints down.

Mandiant CTO Charles Carmakal confirmed that the bug is being exploited in the wild. Oracle’s advisory points to a patch availability document behind a support login, and the article says it is unclear whether a full fix is broadly available. For now, the article says the guidance centers on mitigation.

Operational details became public because the attackers left their own gear exposed. Oracle’s guidance is to disable the Environment Management Hub service on multi-server setups. On single-server setups, Oracle recommends removing the PSEMHUB application outright. If organizations cannot do either, the article says they should block external access at the perimeter to:

  • /PSEMHUB/hub in particular
  • /PSIGW/HttpListeningConnector

Mandiant warns that WAF body-inspection rules alone are not enough because they can be bypassed. The article also states that restricting these endpoints does not break normal user sessions. The article lists several indicators and behaviors organizations should investigate for signs of compromise. The article also calls out recently changed .xml files under the web document root’s envmetadata/data/environment path, which can be abused for XMLDecoder persistence that triggers on the next restart.

Another sign is outbound SMB traffic on port 445 from PeopleSoft hosts to external destinations. The article says the exploit chain may use that traffic to capture machine-account NetNTLM hashes.

LangGraph is an open-source framework for building stateful, multi-agent AI systems and is described in the article as an extension of LangChain with over 50 million monthly downloads according to PyPI stats. The first vulnerability, CVE-2025-67644, is a SQL injection in the SQLite checkpointer. It occurs when the filter parameter passed into the checkpointer’s list() function is processed by _metadata_predicate. CVE-2025-67644: SQLite checkpointer SQL injection

CVE-2025-67644 is a SQL injection in the SQLite checkpointer. The vulnerable flow begins with the list() function, which accepts a filter parameter used to query checkpoints based on metadata. This filter is passed to an internal function called _metadata_predicate, which constructs SQL predicates.

The article highlights the vulnerable line:

json_extract(CAST(metadata AS TEXT), '$.{query_key}') {operator}

The problem is that query_key comes from the filter dictionary and is interpolated into the SQL expression. An attacker-controlled key containing a quote character can escape the JSON path string and inject SQL code.

From SQL injection to arbitrary deserialization

The SQL query used by list() selects checkpoint records from the checkpoints table, including the type, checkpoint, and metadata columns. After results are retrieved, LangGraph processes each row and calls self.serde.loads_typed((type, checkpoint)). This means that checkpoint data returned by the SQL query is deserialized.

The article explains that an attacker can use the SQL injection to add a UNION SELECT that returns a fake checkpoint row. In that injected row, the attacker can supply values for fields such as thread_id, checkpoint_ns, checkpoint_id, type, checkpoint, and metadata. If the fake row specifies a serialized payload in the checkpoint column, LangGraph will deserialize that attacker-controlled data while processing the query results.

CVE-2026-28277: Unsafe msgpack deserialization

CVE-2026-28277 is an unsafe msgpack deserialization issue in LangGraph’s serialization logic. The article reviews the loads_typed() function, which handles multiple serialization types. It supports null, bytes, bytearray, json, msgpack, and, if enabled, pickle.

The article notes that pickle fallback is disabled by default and that JSON deserialization with an object hook does not lead to code execution in this case. The relevant path is msgpack, where the code calls ormsgpack.unpackb() with a custom extension hook. That hook is used by LangGraph to support custom Python objects.

For msgpack extension data using EXT_CONSTRUCTOR_SINGLE_ARG, the hook unpacks a tuple and executes:

getattr(importlib.import_module(tup[0]), tup[1])(tup[2])

The article states that if an attacker controls the serialized data, they control the extension code and the data bytes. By supplying a tuple such as os, system, and a command string like echo PWN > /tmp/pwned.txt, the deserialization path imports os, retrieves system, and calls it with the attacker-controlled command.

  • The attacker sends a malicious filter parameter that exploits the SQL injection vulnerability.

  • The application processes the query results and deserializes the malicious checkpoint.
