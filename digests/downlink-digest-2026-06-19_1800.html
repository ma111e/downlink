Hackers exploit info disclosure bug in Gravity SMTP WordPress plugin

Summary
Hackers are actively exploiting an unauthenticated information disclosure flaw in the Gravity SMTP WordPress plugin that can expose API keys, email service credentials, and detailed site configuration data. Site owners should update to Gravity SMTP 2.1.5 immediately and check logs for requests to /wp-json/gravitysmtp/v1/tests/mock-data.

Threat actors are actively exploiting CVE-2026-4020, an unauthenticated information disclosure vulnerability in the Gravity SMTP WordPress plugin, which is active on 100,000 sites. The flaw affects Gravity SMTP versions 2.1.4 and older and was fixed in version 2.1.5, released on March 17. Defiant warned that hackers are exploiting the issue, and Wordfence said its firewall has blocked more than 17 million attempts against protected customers.

The vulnerability comes from an exposed REST API endpoint whose permission_callback always returns true, allowing unauthenticated GET requests to receive a JSON “System Report” generated by the plugin. The exposed data may include API keys, secrets, OAuth tokens, third-party email service credentials, WordPress configuration details, server and PHP environment information, and database configuration details. Wordfence warned that the exposed credentials could let attackers abuse connected email services, while the system report can reduce the effort needed to plan further attacks.

Wordfence reported that exploitation activity spiked on June 7, when 4 million requests were blocked, with similar activity continuing for several days afterward. A key indicator of compromise is requests to /wp-json/gravitysmtp/v1/tests/mock-data in web server access logs, particularly with the ?page=gravitysmtp-settings query parameter. The article also notes a separate Wordfence advisory for CVE-2026-8713, a critical unauthenticated arbitrary file-deletion flaw in the Avada Builder WordPress plugin fixed in version 3.15.4.

  • Although CVE-2026-4020 is rated medium severity, the article emphasizes that unauthenticated access to live third-party email credentials can enable abuse of connected email services and impersonation of the victim.
  • The exposed Gravity SMTP system report can reduce attackers’ reconnaissance effort by revealing the site’s plugins, themes, software versions, server environment, and database details.
  • The most actionable log indicator is not just the REST endpoint path but requests to /wp-json/gravitysmtp/v1/tests/mock-data that include the ?page=gravitysmtp-settings query parameter.
  • Wordfence reported a concentrated exploitation surge on June 7 with 4 million blocked requests, indicating mass scanning or automated exploitation rather than isolated probing.
Key Points

  • CVE-2026-4020 affects Gravity SMTP versions 2.1.4 and older and was fixed in version 2.1.5 released on March 17.
  • The vulnerability exposes a REST API endpoint that can return a JSON system report containing API keys, OAuth tokens, email service credentials, WordPress configuration details, server information, and database details.

Summary
JetBrains removed 15 malicious Marketplace plugins that posed as AI developer tools but stole users’ AI API keys and sent them to a live command-and-control server in China. Developers who installed any affected plugins should uninstall them, rotate exposed keys, and audit AI usage and billing for abuse.

On June 16, 2026, JetBrains received security reports about a coordinated supply chain attack involving 15 malicious third-party plugins on the JetBrains Marketplace. The plugins were published under seven vendor accounts and posed as legitimate AI-powered developer tools for code review, unit test generation, Git commit message writing, and chat features. JetBrains removed all 15 plugins, permanently banned the seven publisher accounts, and triggered a remote kill-switch to disable the extensions inside IDEs upon relaunch.

The campaign ran for approximately eight months, from late October 2025 through June 2026, and accumulated roughly 70,000 total installations. The two most recent plugins, CodeGPT AI Assistant and DeepSeek AI Assist, were published on June 9 and June 10, 2026 and accounted for more than 53,000 downloads. JetBrains confirmed that no internal systems, source code, or corporate infrastructure were compromised.

The plugins captured AI provider API keys when developers entered them into plugin settings and clicked Apply, then exfiltrated the keys over unencrypted HTTP to a hardcoded command-and-control server at 39.107.60.51. The code checked for keys starting with sk- and 51 characters long, used deduplication to avoid sending duplicate keys, and posted JSON payloads with a static X-Api-Key header. StepSecurity independently confirmed on June 19, 2026 that the C2 server was still live and responding to API requests, and the article advises affected users to uninstall the plugins, revoke and rotate exposed keys, block the C2 server, and audit API usage and billing.

  • The plugins used a deduplication check before exfiltrating keys, which reduced repeated network transmissions and likely made the malicious activity less noisy in monitoring data.
  • The stolen API keys were sent over plaintext HTTP, making the exfiltration detectable through network inspection despite the plugins’ attempt to suppress TLS certificate warnings inside the JVM.
  • The campaign’s two newest plugins accounted for more than 53,000 of the roughly 70,000 total installations, concentrating much of the exposure in the final days before discovery.
  • The donation wall allegedly returned working API keys from the C2 server to paying users, creating a fraud cycle in which stolen credentials could be used to fund other users’ AI activity.
  • A live C2 server after Marketplace removal means rotating exposed API keys remains urgent because removing or disabling the plugins alone does not prevent continued abuse of already stolen keys.
Key Points
  • JetBrains received security reports on June 16, 2026 identifying 15 malicious third-party plugins on the JetBrains Marketplace that masqueraded as legitimate AI-powered developer tools.
  • The plugins stole AI provider API keys entered into their settings and exfiltrated them over unencrypted HTTP to the hardcoded command-and-control server at 39.107.60.51.
  • The campaign ran from late October 2025 through June 2026 and accumulated roughly 70,000 total installations across plugins published under seven vendor accounts.
  • JetBrains removed all 15 plugins from the Marketplace, permanently banned the seven publisher accounts, and triggered a remote kill-switch that disables the extensions inside IDEs upon relaunch.
  • StepSecurity independently confirmed on June 19, 2026 that the command-and-control server at 39.107.60.51 was still live and actively responding to API requests.

The Gentlemen ransomware group is centrally providing affiliates with a standardized EDR-killer toolkit, GentleKiller, that uses BYOVD techniques to disable hundreds of security processes before encryption. The Gentlemen ransomware-as-a-service operation is actively developing and maintaining endpoint detection and response killers that it provides to affiliates before deploying its encryptor. ESET said this toolset centers on a framework called GentleKiller and also incorporates third-party or leaked tools such as HexKiller, ThrottleBlood, and HavocKiller. The group has been active since March 2025 and has claimed 504 victims, with most located in Southeast Asia, South America, and Western Europe.

ESET described The Gentlemen as one of the most technically agile RaaS groups because it can quickly operationalize newly disclosed proof-of-concept exploits related to bring your own vulnerable driver techniques, sometimes within days of public release. GentleKiller has eight variants, each mimicking a different legitimate product and abusing a different vulnerable or malicious driver. The framework targets 400 processes associated with 48 distinct security programs from multiple vendors.

The tools use a shared defense-evasion layer that impersonates security vendors with fake version information, copied legitimate certificates, and icons, while also using binary protection such as Enigma or Themida. ESET said the common structure across the variants suggests a shared development template designed for ease of deployment and operational flexibility for affiliates. The article also notes ESET’s detection of a Rust-based credential stealer called OxideHarvest and a CERT/CC advisory about vendor-signed UEFI applications vulnerable to Secure Boot bypass through a BYOVD attack.

  • The Gentlemen’s centralized EDR-killer suite materially lowers the technical barrier for affiliates, making the RaaS operation more attractive compared with groups that leave defense evasion to individual affiliates.
  • The group’s EDR killers use a shared defense-evasion layer with fake vendor metadata, copied legitimate certificates, and legitimate-looking icons, so defenders should not rely on file appearance or version information to assess trustworthiness.

  • CERT/CC’s related Secure Boot bypass advisory highlights that BYOVD risks extend beyond operating-system runtime defenses into the pre-boot phase when trusted vulnerable UEFI applications remain allowed.

Unpatchable 'usbliter8' Exploit Breaks Apple A12 and A13 SecureROM Boot Chain
thehackernews.com · 2026-06-19 18:37 openai-codex / gpt-5.5

Summary
Researchers released usbliter8, an unpatchable SecureROM exploit for Apple A12/A13 and related chips that enables privileged code execution via USB in DFU mode with physical access. Most users face low risk, but sensitive environments should treat affected devices as permanently vulnerable if an attacker can control USB/DFU access.

Security researchers at Paradigm Shift published a working exploit called usbliter8 that achieves arbitrary code execution inside the SecureROM of Apple A12 and A13 chips. Because SecureROM is burned into silicon at manufacture, the article says no software update can reach the flaw. The exploit is not remote and requires physical possession of the device, DFU mode, and a USB connection to a dedicated RP2350-based microcontroller board.

The public proof of concept supports A12, A13, S4, and S5 SoCs, with A12X and A12Z described as theoretically possible but not yet implemented. Affected device families include iPhone XS/XR and iPhone 11 models, the iPhone SE 2nd generation, several iPads, Apple Watch Series 4 and 5, the first-generation Apple Watch SE, and the HomePod mini. A11 is not affected, while A14 and later appear to be out of reach for this exploit path.

The root issue is described as a hardware flaw in the Synopsys DWC2 USB controller, where handling of smaller-than-standard Setup packets can create a repeatable buffer underflow. On A12 and A13, Apple’s USB DART configuration in SecureROM runs in bypass mode, allowing the underflowing DMA pointer to overwrite arbitrary SRAM. The article says the exploit can temporarily demote the SoC’s production mode or boot a raw, unsigned iBoot image, but it does not show a Secure Enclave compromise.

  • For high-security environments, affected A12, A13, S4, and S5 devices should be treated as a device-custody and hardware-retirement issue because the flaw cannot be fixed by firmware updates.
  • A14 and later devices are the preferred refresh target because the article says they appear to configure DART correctly and are out of reach for this exploit path.
  • The practical exposure depends heavily on physical USB control, making untrusted cables, hosts, and DFU-mode access the most important operational risks to manage.
  • The exploit does not demonstrate a Secure Enclave compromise, so its confirmed impact is control of the application processor’s SecureROM rather than a direct break of Apple’s separate Secure Enclave boundary.
Key Points
  • Paradigm Shift published a working exploit called usbliter8 that achieves arbitrary code execution inside the SecureROM of Apple A12 and A13 chips.
  • The exploit requires physical possession of the device, DFU mode, a USB connection to a dedicated RP2350-based microcontroller board, and completes in under two seconds before Apple’s signed boot chain loads.
  • The public proof of concept supports A12, A13, S4, and S5 SoCs, while A12X and A12Z support is described as theoretically possible but not implemented.
  • The root issue is a hardware flaw in the Synopsys DWC2 USB controller combined with Apple configuring the USB DART in bypass mode on affected devices, allowing an underflowing DMA pointer to overwrite arbitrary SRAM.
  • As of June 19, 2026, no CVE, CVSS score, Apple security advisory, CISA alert, or publicly reported in-the-wild exploitation had been issued.
