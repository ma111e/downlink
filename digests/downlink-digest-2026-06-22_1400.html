Cybersecurity researchers at Zafran Security disclosed details of four vulnerabilities in Dify, an open-source agentic workflow platform with more than 146,000 GitHub stars. The flaws, collectively named DifyTap, could allow attackers to stealthily read AI conversations from other customers’ applications. Zafran said two vulnerabilities were critical, two required no authentication, and three had cross-tenant impact on Dify’s multi-tenant cloud service.

The vulnerabilities could have enabled attackers to read private AI chats, create a covert exfiltration channel for every message and model response, traverse Dify’s internal Plugin Daemon API from unauthenticated requests, and trigger cross-tenant internal API calls. They also made it possible to preview documents uploaded by other tenants and leak files across users within a tenant by attaching another user’s file UUID. Separately, Zafran found that Dify’s file parsing stack used a PDFium version affected by CVE-2024-5846, a use-after-free bug that could potentially be exploited through a crafted PDF file.

The disclosed Dify flaws are tracked as CVE-2026-41947, CVE-2026-41948, CVE-2026-41949, and CVE-2026-41950. The article says missing tenant ownership checks could be abused to redirect messages and responses from victim applications to an attacker-controlled LLM trace provider, and notes that anyone can freely register for a Dify account. Following responsible disclosure, all vulnerabilities except CVE-2026-41948 were fixed in Dify version 1.14.2, with a fix for the remaining issue expected in the next release.

  • Because anyone can freely register for a Dify account, flaws that require an authenticated user may still be broadly exploitable against publicly accessible applications.
  • The trace configuration flaw is especially actionable because it could let an attacker create a persistent channel that silently forwards all messages and model responses from a victim application to an attacker-controlled tracing provider.
  • Upgrading to Dify 1.14.2 does not fully eliminate the reported risk because CVE-2026-41948 was not fixed in that release and is expected to be addressed later.
  • The findings highlight that container-image differences between deployments can create visibility gaps that traditional scanners may miss, according to Zafran Security.

Squidbleed (CVE-2026-47729) is a heap over-read in the Squid web proxy that can leak another user’s cleartext HTTP request to someone already allowed to send traffic through the same proxy. The leaked data can include credentials or session tokens carried in that request. Researchers at Calif.io disclosed the bug in June and said it traces back to a 1997 FTP-parsing change that remains active in Squid’s default configuration.

The issue affects shared proxy environments such as schools, offices, and public Wi-Fi where an attacker may be another authorized user of the same proxy. The leak only applies to traffic Squid can read, including cleartext HTTP and TLS-terminating setups where Squid decrypts and inspects traffic. Normal HTTPS over an opaque CONNECT tunnel is not exposed because Squid does not see inside it.

The bug is in Squid’s FTP directory-listing parser, where a whitespace-skipping loop can walk past the end of a buffer and copy leftover memory back to the attacker as a filename. Calif’s demo recovered an Authorization header from a victim sharing the same proxy, and proof-of-concept code is public, though no in-the-wild exploitation had been reported at the time of writing. The article recommends verifying the actual patch or distribution backport and says disabling FTP is the cleaner mitigation because it removes this attack surface.
