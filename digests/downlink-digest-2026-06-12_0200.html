Overview

watchTowr Labs analyzed CVE-2026-50751, a Check Point Remote Access VPN IKEv1 authentication bypass with a CVSS score of 9.3. The vulnerability affects Check Point Mobile Access/SSL VPN, Remote Access VPN, and Spark Firewall products, specifically in deprecated IKEv1 VPN code. Check Point released hotfixes on June 8, 2026 through sk185033.

The article states that the vulnerability had reportedly been exploited in the wild since May 7, 2026, roughly a month before patches were available. It also says Check Point reported “a few dozen targeted organizations,” and that at least one incident had been linked to a Qilin ransomware affiliate. The vulnerability was described as a logic flaw in how certificates are validated during the IKEv1 key exchange.

Affected Products and Preconditions

The article lists the affected Gaia versions as:

  • R80.20.X
  • R80.40
  • R81
  • R81.10
  • R81.10.X
  • R81.20
  • R82
  • R82.00.X
  • R82.10

watchTowr says exploitation depends on several conditions:

  • The target accepts legacy Remote Access clients.
  • IKEv1 is permitted rather than IKEv2-only.
  • Machine certificate authentication is not mandatory.

The article notes that these prerequisites may still leave many potential targets because legacy client support is common in enterprise environments.

Research Method

watchTowr compared R82.10 Jumbo Hotfix Take 19, which they describe as vulnerable, against the same build with the sk185033 hotfix applied. The researchers first examined vpnd, but found that vpnd did not bind UDP 500 or UDP 4500. They then determined that on R81.10 and later, Check Point redirects IKE negotiation to a newer daemon, iked, which listens internally on UDP 30500.

After diffing the correct binary, the changed code appeared in:

  • ikeMainMode.cc
  • ikeAggressive.cc
  • machine_cert_utils.cc

A key change was the removal of a trailing integer argument from process_cert_payloads. In the vulnerable version, that argument controlled whether machine certificate payloads were processed. In the patched version, the function instead consults the appliance’s own machine certificate policy from the negotiation state.

Root Cause

The vulnerable code allowed the decision to process certificate material and validate authentication to depend on flags that could be influenced by the client. In process_cert_payloads, the vulnerable implementation used a caller-supplied process_machine_certs flag to decide whether to process Check Point machine certificate payloads of type 0xF6. The patched version removed that argument and used a field populated from the gateway’s configured machine certificate policy.

The more important issue appeared in the caller process_auth_pl and in verify_peer_auth. In process_auth_pl, the code read a flags dword from state offset 0x4bc4 and tested bit 0x2. If this bit was not set, the code did not call seekPayload for the 0xF9 signature payload and did not call processSigPayload.

In verify_peer_auth, the same flag word controlled whether verifyMessagePhase1 ran. If bit 0x4 was set, an OR condition short-circuited and the function returned authentication success without calling verifyMessagePhase1. The article identifies verifyMessagePhase1 as the phase-1 signature check.

Client-Controlled Authentication Flags

watchTowr searched for writes to state offset 0x4bc4 and found two. One was initialization in FwIkeNewPhase1State, which zeroed the field for a new negotiation. The other was in processVendorIDPayload.

The Vendor ID involved was the 16-byte value:

  • 3c f1 87 b2 47 40 29 ea 46 ac 7f d0 ea f2 89 f5

The gateway logged this as VPNExtFeatures. According to the article, processVendorIDPayload required at least four bytes after the 16-byte magic value, read those four attacker-supplied bytes, byte-swapped them, and stored them directly into state + 0x4bc4. This meant the client could provide the flags used by the gateway to decide whether to perform authentication checks.

The article highlights an additional inconsistency: a peer that sent only the VPNExtFeatures Vendor ID was logged by the gateway as “not a Check Point peer,” while the gateway still honored the client-supplied flag that disabled authentication verification.

Proof of Concept Behavior

To prove the issue, watchTowr wrote a minimal IKEv1 Main Mode client using AES-256, SHA1, Diffie-Hellman group 2, and Check Point’s 0xFADD hybrid authentication method. They sent the VPNExtFeatures Vendor ID with the four trailing bytes set to 0x00000004, setting bit 0x4. They then completed Diffie-Hellman and sent a completely random HASH_I in encrypted message 5.

They tested identical packets against two gateways:

  • A vulnerable R82.10 Take 19 appliance.
  • A patched Take 19 appliance with sk185033.

The vulnerable gateway logged that the peer was not a Check Point peer, proceeded through MMCreate6, and saved the user in IkeSAFromState. The patched gateway logged an authentication failure with hybrid and sent an authentication failed notification. The article says the vulnerable gateway created a phase-1 SA based on a signature generated with os.urandom.

watchTowr also performed negative controls:

  • VPNExtFeatures with capability 0x00000004 skipped verifyMessagePhase1 and produced the bypass.
  • VPNExtFeatures with capability 0x00000000 caused verifyMessagePhase1 to run and resulted in rejection.
  • No Vendor ID caused verifyMessagePhase1 to run and resulted in rejection.

Authentication as a Provisioned User

The researchers then moved from proving acceptance of invalid phase-1 data to showing authentication as a provisioned Remote Access user. They switched from the hybrid method to RSA signature authentication, IKE authentication method 3, using the certificate path.

The credentials they presented were:

  • An ID payload containing a distinguished name in the form CN=,OU=users,O=.
  • A self-signed X.509 certificate minted by the researchers.
  • A signature payload containing 256 bytes of os.urandom.
  • The VPNExtFeatures Vendor ID with bit 0x4 set.

Because verifyMessagePhase1 was skipped, the gateway did not verify the signature and did not validate the certificate trust chain. The article states that the gateway only checked whether the subject distinguished name resolved to a provisioned Remote Access user.

According to watchTowr, an attacker needed two pieces of information:

  • A valid username.
  • The ICA organization string for the distinguished name.

The article says the ICA organization string is present in the gateway’s public TLS certificate and can be read from there. It also says a wrong username fails at user lookup while a correct username succeeds, making the same probe a username oracle.

In their demonstration, watchTowr’s tool authenticated as the user “watchtowr” with a forged certificate and invalid signature, decrypted the gateway’s final phase-1 message, and retrieved the gateway internal IP and VPN certificate. The gateway also logged that the user CN=watchtowr,OU=users,O=watchtowr was saved.

Exploitation Over UDP and TCP 443

The article states that the vulnerability was exploitable over UDP 500. It also says watchTowr was able to exploit it over TCP 443.

When UDP 500 and UDP 4500 are filtered, Check Point clients can fall back to Visitor Mode. Visitor Mode tunnels the same IKE exchange over a raw TCP stream on port 443 using a framing protocol called TCPT. watchTowr says they reversed the protocol enough to demonstrate that the same bypass worked over this path.

Authentication Modes

watchTowr reports that the vulnerability appeared exploitable across several Check Point certificate modes for SSL Network Extender user authentication. Their table states:

  • Certificate mode was bypassed over UDP 500 and TCP 443.
  • Certificate with enrollment mode was bypassed over UDP 500 and TCP 443.
  • Mixed mode was bypassed over UDP 500 and TCP 443.
  • Legacy username/password mode was not bypassable.

The article explains that Legacy username/password mode still required a real password through XAUTH, so the bypass did not remove that gate.

Response and Detection Artifact

Check Point released hotfixes through sk185033 on June 8, 2026. The patched code removed the client-controlled process_machine_certs function argument and instead based machine certificate processing on the gateway’s own configured policy. The patched gateway in watchTowr’s test rejected the malicious exchange with an authentication failure.

watchTowr also released a Detection Artefact Generator at the GitHub repository watchtowrlabs/watchTowr-vs-Check-Point-CVE-2026-50751. The article says the decision to release it was based on the attackers having already had the vulnerability for a month.

  • The vulnerable gateway could log a peer as “not a Check Point peer” while still honoring that peer’s client-supplied flag to skip authentication checks.
  • The bypass required only a valid username and the ICA organization string, and the article states the ICA organization value could be obtained from the gateway’s public TLS certificate.
  • Wrong username guesses failed at user lookup while valid usernames succeeded, making the same authentication probe usable as a username oracle.
  • Filtering UDP 500 or UDP 4500 was not sufficient mitigation in the researchers’ testing because the same IKEv1 exchange could be tunneled over TCP 443 using Check Point Visitor Mode.
  • The plain Legacy username/password mode was described as not bypassable because XAUTH still required a real password, while the certificate-based modes were bypassed.

Overview

Cyble Research & Intelligence Labs identified an active SEO poisoning campaign abusing abandoned cloud DNS zone delegations to serve Thai-language gambling content under trusted enterprise domains. The campaign compromised 163 organizations across more than 30 countries, including federal government agencies, national healthcare systems, financial institutions, critical infrastructure operators, and major universities. At the time of publication, Cyble stated that 161 organizations remained actively compromised.

The campaign’s core mechanism was the takeover of abandoned cloud DNS zones, especially Azure DNS zones. When organizations decommissioned cloud infrastructure but left NS delegations in their parent DNS zones, the actor identified those stale delegations, claimed the orphaned zones under a new Azure subscription, and used the victim’s domain authority to serve gambling SEO content. The pages were reachable with valid Let’s Encrypt wildcard TLS certificates and appeared under legitimate enterprise-owned domains.

Initial Discovery

Cyble first identified anomalous DNS resolution in the cardsforheroes.verizon.com subdomain environment during an ongoing vulnerability assessment. The Verizon environment contained more than 1,000 individually named subdomains serving Thai-language online gambling content under Verizon’s trusted domain authority. Each subdomain followed gambling brand keyword patterns with URL-encoded Thai characters, and all resolved to 51.79.199.51 at OVH SAS, AS16276.

A representative endpoint loaded a fully rendered Next.js gambling application with outbound affiliate redirect links. Cyble found that the endpoint shared structural similarities with 97 other enterprise subdomains across unrelated organizations. The article states that affiliate referral parameters on every redirect established the campaign’s monetization intent as gambling registration commission rather than defacement or malware delivery.

Vulnerability Class: Abandoned Cloud DNS Delegation

The article describes a DNS misconfiguration that occurs when enterprises delegate subdomains to Azure DNS zones for project environments and later delete the cloud resources without removing the NS delegation from the parent DNS zone. The parent zone continues pointing queries for that subdomain to Azure nameservers, even though the original Azure DNS zone no longer exists. Cyble observed cases where these delegations had remained in place for more than six years.

The exploitability depends on whether the DNS zone name can be claimed by a new subscriber. According to the article, Microsoft’s zone-creation model has historically allowed a subscriber to register a zone by name under their own subscription. If a zone was abandoned with a canceled subscription and the parent-zone NS delegation was not removed, a third party could recreate the zone and inherit delegated DNS authority for that subdomain.

DNS Compromise Mechanisms

Cyble identified four distinct mechanisms accounting for the 163 confirmed victim organizations.

Azure DNS Zone Takeover

More than 150 organizations were affected through Azure DNS zone takeover. The actor identified subdomains whose NS records still pointed to Azure DNS nameservers after the underlying subscription or zone had been canceled or abandoned. The actor then created a new Azure subscription, claimed the orphaned zone by name, added wildcard A records pointing all subdomains to a delivery IP, and used ACME HTTP-01 validation to obtain Let’s Encrypt wildcard certificates.

Cyble cites authoritative DNS evidence from Microsoft nameserver infrastructure showing the actor’s wildcard A records. Zone SOA serials of 1 on confirmed victims indicated that the zones were created from scratch by the actor rather than modified from an existing state. Certificate Transparency logs showed long dormancy gaps, including a pharmaceutical company pilot subdomain with a last legitimate certificate in October 2019 and an actor certificate on April 11, 2026.

DigitalOcean DNS Zone Takeover

Two organizations were affected through DigitalOcean DNS zone takeover: a U.S. luxury furniture retailer and an Indian university. Their compromised subdomains were delegated to DigitalOcean nameservers rather than Azure. Both had wildcard records resolving to 38.127.8.49.

The article states that DigitalOcean’s zone-claiming model carries the same structural vulnerability when abandoned DNS zones in canceled accounts become available for re-registration. Cyble concluded that the actor’s tooling targeted multiple cloud DNS providers.

Direct Wildcard Misconfiguration

Two organizations were affected through direct wildcard DNS misconfiguration: a U.S. energy company’s development subdomain and a mobile payments platform’s development subdomain. These subdomains resolved via wildcard to 139.99.82.106 despite not using cloud DNS zone delegation. Both used their own organizational nameservers, making cloud zone takeover mechanically impossible in those cases.

Cyble states that the wildcard records resolved from within the parent zone. The article says this indicated either an orphaned wildcard A record left in the organization’s DNS console when a project was decommissioned, or a direct DNS management access path exploited by the actor.

Per-Subdomain A Records

The Verizon environment represented a distinct mechanism. Instead of a single wildcard record, it contained more than 1,000 individually named A records pointing gambling-keyword subdomains to 51.79.199.51. Cyble states that this implied either an automated DNS API script with zone-write access or a compromised DNS management credential enabling bulk record creation.

Pivoting and Infrastructure Discovery

Cyble pivoted from the initial Verizon endpoint to a broader victim set using passive DNS data, page fingerprinting, and infrastructure indicators. The primary delivery IP, 51.79.199.51, revealed more than 90 enterprise subdomains serving identical content. These matched on the Next.js build ID QQOrXCFjoI6C9oF-4YVhl, the favicon path /img/ib99-hq.ico, and outbound affiliate redirects to the same three destination domains.

Passive DNS data against AS16276, OVH, surfaced two additional delivery IPs: 139.99.82.106 and 38.127.8.49. Both served the same gambling kit with the same build fingerprint across a partially overlapping but distinct victim set. Together, the three OVH delivery nodes accounted for the full 163-organization victim estate described in the article.

A JARM TLS fingerprinting pivot from the primary delivery node exposed 38.173.56.218 in Hong Kong under AS398478, PEG TECH INC. Unlike thousands of generic shared-hosting JARM matches, this host presented a ThinkPHP application framework error on port 443, a Chinese server management panel certificate identity on port 21, and MySQL 5.7.44-log on port 3306. Cyble described this as the entry point from the OVH delivery layer into the backend infrastructure.

Affiliate Monetization Architecture

The campaign monetized traffic through gambling affiliate redirects. Every destination redirect carried a ?rc= affiliate code, including ibiza99vip1, bigwinv1, link99, or seven77vip1. The article states that these codes attributed completed registrations to the actor and triggered commission payouts from the destination platform.

Cyble observed a server-side verification layer before redirect issuance. A POST request at a live endpoint returned a response including status “ok,” the message “TH - Referrer verified,” and data containing the referrer “link99.” The article states that the backend validated geographic origin server-side and did not redirect requests from outside Thailand.

The article describes link99 as a sub-affiliate publisher ID that sits above the ?rc= codes in the platform’s tracking hierarchy. Cyble states that the actor earned at both tiers: publisher-level credit under link99 for delivering Thai traffic and per-registration payout under the ?rc= code for each conversion. The two layers were described as independent, allowing campaign codes to rotate while link99 persisted as the actor’s platform identity.

Destination Platforms and Gambling Kit

Cyble confirmed four destination platforms: ibiza99.autos, big888.store, seven77.click, and stillsunday.pl. The article states that these domains were white-label deployments on a shared codebase, supported by identical CSS hashes and JavaScript resilience logic across all three except big888.store. Big888.store carried active Google Search Console verification, which Cyble said indicated that the platform operator ran its own independent SEO operation beyond this campaign.

The gambling kit served from compromised enterprise subdomains returned a fully rendered Next.js page when requested with Thai locale headers. The pages included lang=“th”, Thai-language gambling branding, valid Let’s Encrypt wildcard TLS certificates matching victim subdomains, Schema.org FAQ structured data, AMP alternate links, and static assets served from paths under the compromised domain. The article states that the minimum deposit advertised across variants was 1 Thai Baht, approximately $0.03 USD.

The kit fingerprint was consistent across the victim estate regardless of the victim domain or OVH delivery node. Cyble states that this confirmed centralized build and deployment by a single operator.

Backend Infrastructure

The three OVH delivery nodes handled content distribution, while the application layer was hosted in a separate backend fleet in Hong Kong. Cyble identified 103 backend nodes distributed across seven /24 subnets within 38.173.0.0/16, all under AS398478, PEG TECH INC. The fleet was discovered by pivoting on a Let’s Encrypt certificate issued to broker-xm.com that was deployed across the nodes.

Cyble listed multiple evidence points supporting single-operator control of the backend fleet:

  • An identical HTTP body MD5 hash, 7df3d7cf3358af3f470ac7229387ef94, across all 103 nodes on port 80.
  • A shared broker-xm.com Let’s Encrypt certificate across all 103 servers.
  • An identical Envoy proxy 503 error string on port 443 across all nodes.
  • A uniform MySQL version, 5.7.44-log, with binary replication logging enabled.
  • BT-Panel provisioning details showing admin@bt.cn and Dongguan, Guangdong, on FTP certificates.
  • A uniform JARM fingerprint across the fleet.
  • No PTR records on any of the 103 IPs.

Detection and Hunting Guidance

The article emphasizes that standard security controls produced little or no signal because the campaign used valid TLS certificates, clean IP reputation, trusted domain names, and no exploit delivery. Cyble identifies Certificate Transparency monitoring as the most reliable early indicator. Unexpected Let’s Encrypt wildcard certificates on corporate-owned subdomains, especially after a period of no certificate issuance, were presented as an anomaly that would have detected every Azure takeover victim at issuance time.

Cyble recommends auditing Azure DNS zone delegations by enumerating NS delegation records in parent DNS zones and verifying that corresponding Azure DNS zones exist in subscriptions the organization owns and manages. The article also recommends wildcard DNS testing by querying randomized nonsense hostnames under Azure-delegated subdomains. If such a hostname resolves unexpectedly, the zone should be treated as compromised until proven otherwise.

The article provides detection indicators including DNS answers resolving to 51.79.199.51, 139.99.82.106, or 38.127.8.49; URL parameters containing rc=ibiza99vip1, rc=bigwinv1, rc=seven77vip1, or rc=link99; HTTP requests to /img/ib99-hq.ico; and outbound connections to seven AS398478 backend subnets. It also lists internet scanning pivots such as the HTTP body MD5 7df3d7cf3358af3f470ac7229387ef94, TLS certificate CN broker-xm.com, and ASN AS398478.

Remediation Guidance

For Azure DNS zone takeover victims, Cyble recommends removing the NS delegation from the parent DNS zone immediately. The article states that this severs the attack chain regardless of what the actor maintains inside the Azure zone, because that zone is under the actor’s control and cannot be directly modified by the victim organization. Cyble also recommends reporting the abandoned zone to Microsoft MSRC with the zone name and evidence.

The article further recommends requesting revocation of any Let’s Encrypt certificates issued against compromised subdomains using serial numbers from crt.sh. Organizations are advised to audit remaining subdomains for environment-style entries such as dev, uat, staging, preprod, pilot, lab, test, and sandbox, especially where they carry NS delegations to cloud providers. For DigitalOcean takeover victims, Cyble recommends removing the NS delegation from the parent zone and verifying whether the DigitalOcean zone exists in an account the organization owns.

For direct wildcard misconfiguration victims, Cyble recommends removing the wildcard A record from the DNS management console. If the origin of the record cannot be identified, the article says the DNS management credential should be treated as compromised and rotated immediately.

Impact and Significance

Cyble states that the actor did not breach a perimeter or exploit an application vulnerability. Instead, the actor claimed abandoned DNS infrastructure and built a commercial affiliate operation on top of it. A single wildcard record beneath an abandoned Azure delegation could expose an unlimited subdomain namespace while inheriting the TLS-verified authority of the victim’s brand.

The campaign’s scale was described as the result of automated scanning applied to one vulnerability class rather than 163 separate attacks. Its use of OVH delivery nodes, Let’s Encrypt certificates, victim-owned domains, and organic search rankings allowed it to operate within legitimate infrastructure. The article concludes that abandoned NS delegations pointing to cloud providers remain candidates for future exploitation if organizations do not audit and remove them.

The reporting highlights how legacy protocol code and neglected DNS assets can both enable attackers to inherit legitimacy from trusted enterprise environments.

01 Check Point Remote Access VPN Authentication Bypass CVE-2026-50751

Check Point released hotfixes on June 8, 2026 for CVE-2026-50751, a critical authentication bypass rated CVSS 9.3 affecting Mobile Access/SSL VPN, Remote Access VPN, and Spark Firewall products. watchTowr Labs analyzed the issue and attributed it to deprecated IKEv1 VPN code, where attacker-controlled input in a VPNExtFeatures Vendor ID payload could influence authentication-related flags inside the gateway implementation.

According to the research, the vulnerable implementation could be made to skip certificate signature validation during IKE authentication. watchTowr demonstrated that a vulnerable gateway accepted a random HASH_I and created a phase-1 IKE security association, while a patched system rejected the same attempt with an authentication failure. The bypass reportedly worked across UDP 500 and TCP 443 Visitor Mode.

The article states that an attacker could authenticate as a provisioned Remote Access user by presenting a matching subject distinguished name, a self-signed X.509 certificate, and an invalid signature, because the gateway did not verify the signature or certificate trust chain. watchTowr reported successful exploitation across Certificate, Certificate with enrollment, and Mixed user authentication modes, making the issue relevant to multiple Remote Access deployment configurations.

02 Abandoned Cloud DNS Delegations Used for Enterprise-Domain SEO Poisoning

Cyble Research & Intelligence Labs reported an SEO poisoning campaign that served Thai-language gambling content from trusted enterprise domains by exploiting abandoned or misconfigured DNS infrastructure. The campaign affected 163 organizations across more than 30 countries, with the primary technique involving Azure DNS zone takeover through reclaimed nameserver delegations tied to deleted Azure DNS zones.

The campaign also used DigitalOcean DNS zone takeovers, direct wildcard DNS misconfigurations, and individually created A records in at least one Verizon subdomain environment. By inheriting authority over abandoned DNS delegations, the operators were able to publish gambling pages under otherwise legitimate organizational domains, increasing the apparent trust and search visibility of the content.

Cyble described a consistent technical footprint across the gambling kit, including valid Let’s Encrypt wildcard TLS certificates, a recurring Next.js build fingerprint, Thai-language SEO content, and affiliate redirect codes such as ibiza99vip1, bigwinv1, link99, and seven77vip1. The infrastructure was linked to three OVH delivery nodes and a 103-node backend fleet in Hong Kong under AS398478, with shared indicators including a broker-xm.com certificate and an identical HTTP body hash.
