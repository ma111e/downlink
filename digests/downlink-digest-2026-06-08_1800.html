Overview

On June 8, 2026, an attacker compromised the GitHub account of LeonOstrez, a co-founder and maintainer of Pythagora-io/gpt-pilot, and force-pushed a credential-stealing payload to the repository’s main branch. The article describes gpt-pilot as a popular open-source AI developer tool with more than 33,700 GitHub stars and 3,500 forks. The attacker was able to force-push directly because the repository had no branch protection rules configured on main.

The malicious payload was identified as a variant of the Shai-Hulud worm. It was stopped not by a dedicated security tool, but by Ruff, the Python formatter and linter used in the project’s CI pipeline. The attacker tried twice to get the malicious code past CI and failed both times because the injected code violated formatting and linting rules.

Attack Timeline

The malicious commit was titled “Revert ‘Implemented weekend discount’” and was authored with a backdated timestamp of August 24, 2025. The article states that it matched the metadata of a legitimate revert by Zvonimir Sabljic while adding malicious files and changes.

Key events described in the article include:

  • At 11:01:38 UTC on June 8, 2026, the compromised LeonOstrez account force-pushed a malicious chain over the clean commit chain.
  • At 11:02:07 UTC, CI failed because ruff format –check found a formatting violation in _hooks.py.
  • At 11:13:07 UTC, the attacker force-pushed again after fixing the formatting issue.
  • At 11:13:38 UTC, CI failed again because ruff check found E402 and I001 import-related violations in init.py.
  • Around 11:30 UTC, a community member reported the compromise in GitHub issue #1181.
  • Around 12:00 UTC, issue #1181 was deleted and later returned HTTP 410.

Repository Compromise Method

The attacker gained control of the LeonOstrez GitHub account and used it to force-push directly to the main branch. The GitHub API returned a 404 for the main branch protection endpoint, which the article uses to show that no branch protection rules were configured. Because of that, the attacker did not need pull request review, required passing checks before merge, or other approval.

The force push replaced the commit history. The article gives two push-event records: the first replaced clean HEAD 53154df1c66b with malicious head 90f59f5de681, and the second replaced 90f59f5de681 with a372904facd5 after the first CI failure.

Malicious Commit Strategy

The attacker used a commit that appeared to be an ordinary revert. The article compares two versions of the same-looking commit:

  • Clean commit 566fbb12, which only reverted frontend.py and spec_writer.py.
  • Malicious commit 065ee8eb, which made the same reverts but also added _hooks.py and _runtime.bin and modified init.py.

The malicious commit used identical-looking metadata, including message, author name, and timestamp. By backdating it to August 2025, the attacker made the commit appear months older than the actual June 2026 compromise.

Malware Components

The attacker injected three files or changes into core/telemetry/:

  • _hooks.py: a 4,022-byte Python loader that downloaded the Bun runtime and executed the JavaScript payload.
  • _runtime.bin: a 758,608-byte obfuscated JavaScript credential stealer, despite its .bin extension.
  • init.py: modified with nine added lines that spawned a daemon thread during telemetry module import.

The article emphasizes that the malware hid in the telemetry directory, a location that already contains network-related code and may attract less scrutiny. It also notes that _hooks.py used an underscore prefix suggesting internal code, while _runtime.bin disguised JavaScript as a binary-looking file.

Activation Chain

The injected code in core/telemetry/init.py imported threading as _th, defined a _setup_reporting function, imported run from core.telemetry._hooks, and started the function in a daemon thread. This meant that when a developer ran gpt-pilot and the telemetry module was imported, the malware could activate silently.

The Python loader was cross-platform and supported Linux, macOS, and Windows on multiple architectures. It used a .loader.lock lock file to prevent duplicate execution and suppressed output to /dev/null.

JavaScript Payload Details

The _runtime.bin payload was a 758KB single-line JavaScript file designed to run under Bun. The article says it used five layers of obfuscation:

  • A 1,266-element constant lookup table.
  • A 4,777-entry Base91-encoded string array rotated before use.
  • Three custom Base91 alphabets.
  • Control-flow flattening with generator functions and while/switch state machines.
  • Lazy property-access decoding with thousands of unique references.

The article also says the code renamed require() to rMq3gu() and used an @redacted namespace to hide package names.

Credential Targets and Capabilities

The payload targeted credentials from major cloud, developer, and CI/CD environments. The article specifically names AWS keys, npm tokens, GitHub secrets, Kubernetes service accounts, HashiCorp Vault tokens, and SSH keys. It also mentions token patterns for npm, GitHub tokens, and AWS access keys.

The payload could query cloud metadata APIs, read credential files, and monitor for newly created tokens during a session. It could also request signing certificates from Fulcio, create Rekor transparency log entries, and generate SLSA provenance attestations.

Command-and-Control and Exfiltration

The malware used GitHub commit messages as a command-and-control channel. It searched the GitHub commits API for the marker string “thebeautifulsnadsoftime” and extracted commands using a base64-oriented regular expression. The article states that commands were base64-encoded and executed through eval().

For exfiltration, the primary method was to create new GitHub repositories using stolen tokens and commit stolen credentials as files. The article says the malware impersonated claude@users.noreply.github.com as the commit author and created repositories with discussions, issues, and wiki disabled. A secondary method used DNS-resolved HTTP endpoints with AES-256-GCM encryption.

Persistence and Anti-Analysis

The malware planted persistence hooks in developer tools. In Claude Code, it could modify .claude/settings.json to run a malicious command on SessionStart. In VS Code, it could modify .vscode/tasks.json so that the payload re-executed when a folder was opened.

The article lists several anti-analysis techniques:

  • A Russian locale exemption that exited when the system locale included “ru.”
  • A singleton mutex to prevent multiple instances from running.
  • Daemonization through a detached child process with unref().
  • Token monitoring through installTokenMonitor.

How Ruff Blocked the Attack

The gpt-pilot repository used Ruff in CI with both linting and formatting checks. The first force-push attempt failed because ruff format –check found that _hooks.py did not match the project’s formatting rules. The article states that the malicious file was the only file that did not pass formatting.

The attacker then fixed the formatting issue and force-pushed again. The second attempt failed because ruff check found E402, “module level import not at top of file,” and I001, “import block is un-sorted or un-formatted,” in core/telemetry/init.py at line 399. All six CI jobs failed again, and the article says the attacker gave up.

Shai-Hulud Attribution and Related Campaign Context

The article identifies the attack as a direct instance of the Shai-Hulud malware family attributed to TeamPCP/UNC6780. It gives several matching indicators:

  • Identical Bun v1.3.13 runtime loader pattern.
  • The same _runtime file naming convention seen in PyTorch Lightning in April 2026.
  • Matching credential targets and exfiltration methods.
  • The same “thebeautifulsnadsoftime” C2 marker string.
  • Russian locale exemption.
  • Sigstore/Fulcio abuse for signing malicious packages.

The article also states that TeamPCP publicly released the Mini Shai-Hulud source code on May 12, 2026, which makes attribution uncertain. It says the payload’s sophistication suggests the original actors, but also notes that a copycat is possible.

Affected Users and Recovery Guidance

The article says users may be affected if they cloned or pulled gpt-pilot from GitHub after June 8, 2026 at 11:01 UTC and before the force push was reverted. It recommends checking for the malicious files core/telemetry/_hooks.py and core/telemetry/_runtime.bin, Bun runtime directories under /tmp named rt-*, a .loader.lock file, and persistence hooks in .claude/settings.json or .vscode/tasks.json.

The article recommends rotating credentials immediately, including AWS access keys, npm tokens, GitHub PATs, SSH keys, and secrets in environment variables or credential files. It also recommends auditing AWS CloudTrail, checking npm audit logs, inspecting GitHub repositories for unauthorized repositories, removing unexpected Claude Code or VS Code hooks, and killing suspicious Bun processes or processes with the __DAEMONIZED environment variable.

Response and Disclosure

The article says StepSecurity responsibly disclosed the compromise to the maintainers through GitHub issue #1182. It also says a community member reported the compromise through issue #1181 before that issue was deleted. The article states that only users with repository admin access can delete GitHub issues and recommends raising a GitHub support ticket to confirm the deletion through internal audit logs and suspend the compromised account.

The article notes that neither the legitimate nor malicious commits were GPG-signed. It says the malicious revert commit claimed to be authored by Zvonimir Sabljic, but because it was pushed through the LeonOstrez account, the identity was likely spoofed.

Lessons Highlighted by the Article

The article presents several lessons from the incident. First, it says code-quality tools can function as an underappreciated defense layer because malicious code injected outside normal development workflows may not match a project’s style. In this case, Ruff formatting and linting checks prevented the malicious code from passing CI.

Second, it states that branch protection is not optional. Recommended controls include required pull request reviews, required status checks, restrictions on force pushes to the default branch, and signed commits.

Third, it says force pushes to default branches are almost always suspicious in production repositories. The article also highlights that telemetry directories can be attractive hiding places for malicious code and that Bun can be used as an attack vector because it can execute JavaScript or TypeScript files with any extension, including .bin.

  • Code quality tooling acted as an effective security control because the injected malware did not conform to the repository’s established Ruff formatting and import-order rules.
  • The absence of branch protection allowed one compromised maintainer account to rewrite the default branch history directly, making required pull requests, status checks, and force-push restrictions especially actionable controls.
  • The attacker backdated the malicious commit and reused legitimate-looking revert metadata, showing that casual commit-history review can miss supply-chain tampering when commit timestamps and author fields are spoofed.
  • Telemetry code was used as a hiding place because it already appears network-related and is likely to receive less scrutiny during review.
  • The malware’s ability to create apparently valid SLSA Build Level 3 attestations through Sigstore means provenance signals alone may not prove a package is safe when publisher credentials are stolen.

01 Developer and SaaS Supply-Chain Compromise

Two reports describe new Shai-Hulud activity against open-source software ecosystems. In one case, attackers compromised the LeonOstrez GitHub account and force-pushed a credential-stealing payload to the Pythagora-io/gpt-pilot repository’s main branch, which lacked branch protection. The payload included a Python loader, an obfuscated JavaScript file named _runtime.bin, and import-triggered execution through core/telemetry/__init__.py; its collection targets included AWS keys, npm tokens, GitHub secrets, Kubernetes service accounts, Vault tokens, and SSH keys.

A separate Shai-Hulud campaign trojanized 19 science-focused PyPI packages across 37 malicious releases, including bioinformatics tools such as Dynamo, Spateo, CoolBox, U-FISH, and Napari-UFISH. Those packages used *-setup.pth execution and an obfuscated _index.js payload that downloaded the Bun runtime and searched for GitHub, npm, PyPI, RubyGems, JFrog, cloud, Kubernetes, Docker, SSH, configuration, and shell-history secrets. The GitHub incident was notable because CI tooling blocked execution paths twice: first through a ruff format failure and then through a ruff check import-order violation.

Cyera’s analysis of ShinyHunters frames a related but broader pattern: attackers abusing legitimate access through stolen credentials, session tokens, OAuth grants, SaaS integrations, and native export APIs. The report links this playbook to Snowflake-related intrusions, Vercel and Context.ai, Salesloft and Gainsight integration compromises, and the April 2026 Vimeo dataset leak, which reportedly exposed identity, Salesforce, product analytics, content metadata, marketing attribution, and API data layers.

02 Data Exposure Through Third Parties and Trusted Access

SoFi confirmed that customers of SoFi Securities Hong Kong were affected by unauthorized access to a third-party vendor database. The company said it discovered the incident on April 30, 2026, but had not yet determined which specific data categories were involved. SoFi declined to identify the vendor, disclose the number of affected customers, or say whether extortion was involved.

The SoFi disclosure aligns with the broader trend described in the ShinyHunters reporting: a single vendor, integration, or identity can provide access to data spread across business platforms. In both cases, the central issue is not necessarily direct compromise of a primary application, but the exposure created by trusted external systems and the data privileges granted to them.

03 Privilege Escalation, Legacy Infrastructure, and Container Escape Risk

CVE-2026-23111, a Linux kernel use-after-free in nf_tables, drew attention because public exploit details are now available. The flaw can allow an unprivileged local user to escalate to root and potentially escape containers on systems using nf_tables with unprivileged user namespaces enabled. Researchers demonstrated or reproduced exploitation across Debian Bookworm, Debian Trixie, Ubuntu 22.04 LTS, Ubuntu 24.04 LTS, and RHEL 10, while major Linux vendors are tracking or providing fixes.

A NetSPI penetration-testing case study showed how legacy and modern authentication infrastructure can combine into a full Active Directory compromise path. The tester found unauthenticated Apache NiFi servers and achieved root command execution, then discovered cleartext credentials on a Linux host. The environment also exposed Network Information Service password hashes without authentication, and a cracked privileged password led to access to a Duo Authentication Proxy host containing AD service credentials and a RADIUS shared secret.

The case study culminated in decrypted Duo proxy authentication traffic and relay of an MSSQL service account that belonged to Domain Admins. That relay enabled creation of a new domain admin account and DCSync, illustrating how Linux infrastructure, NIS, MFA proxy configuration, RADIUS secrets, and Windows domain privileges can collapse into a single escalation chain.
