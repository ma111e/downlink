Summary
The @mastra npm organization was compromised and 140+ packages were republished with a typosquatted dependency, easy-day-js, whose malicious postinstall script downloaded and ran a second-stage payload. Anyone who installed affected @mastra packages on June 17, 2026 should treat their environment and exposed secrets as compromised.

On June 17, 2026, an attacker compromised the @mastra npm organization and added easy-day-js as a dependency across more than 140 packages in the Mastra AI framework ecosystem. The affected packages had combined weekly downloads exceeding 1.1 million, and the article states that anyone who installed an @mastra package that day should treat the environment as compromised. The campaign was described as ongoing at publication time, with additional @mastra packages continuing to be compromised.

The malicious dependency, easy-day-js, was a typosquat of the popular dayjs date library. A clean bait version, easy-day-js@1.11.21, was first published on June 16, and the compromised Mastra packages referenced it with the caret range ^1.11.21, allowing npm installs to resolve later to the malicious 1.11.22 version. That malicious version added a setup.cjs file and a postinstall hook that executed an obfuscated dropper.

When installed, the dropper disabled TLS certificate verification, wrote beacon files into the temporary directory, downloaded a second-stage payload from https://23.254.164.92:8000/update/49890878, spawned it as a detached Node.js process with 23.254.164.123:443 as an argument, and deleted itself. StepSecurity reported that Harden Runner in block mode intercepted and blocked the outbound call to 23.254.164.92:8000 during a controlled GitHub Actions test, preventing the stage-two payload from downloading and stopping the rest of the chain.

On June 17, 2026, an attacker compromised the @mastra npm organization and republished more than 140 Mastra ecosystem packages with a new production dependency on easy-day-js. Mastra is described in the article as a TypeScript framework for AI agents, workflows, RAG pipelines, LLM integrations, memory, MCP servers, vector databases, and cloud deployment targets. Because Mastra packages are often installed in environments containing LLM API keys, cloud credentials, database tokens, CI/CD secrets, VCS tokens, and npm tokens, the article identifies the ecosystem as a high-value target.

The attack began with pre-positioning on June 16, 2026, when npm user sergey2016 published easy-day-js@1.11.21 as a clean copy of the legitimate dayjs library. The package mirrored dayjs versioning, metadata, homepage, repository URL, license, and keywords to appear credible. The compromised Mastra packages added easy-day-js as ^1.11.21, which meant npm would resolve fresh installs to the latest matching version once a newer release appeared.

At 01:01 UTC on June 17, sergey2016 published easy-day-js@1.11.22, which added setup.cjs and a postinstall script of node setup.cjs --no-warnings. Eleven minutes later, the attacker began publishing compromised @mastra packages, starting with @mastra/schema-compat@1.2.12, followed by core packages such as @mastra/core@1.42.1, @mastra/memory@1.20.4, @mastra/server@2.1.1, and the top-level mastra@1.13.1. The article states that the publishing cadence was consistent with an automated script enumerating and processing packages in batches.

The setup.cjs dropper was obfuscated with a custom-alphabet Base64 scheme, an array rotation and integrity check, and an XOR-encoded package marker. At install time, it set NODE_TLS_REJECT_UNAUTHORIZED=0, wrote .pkg_history and .pkg_logs into the temp directory, fetched a second-stage payload from https://23.254.164.92:8000/update/49890878, wrote it to a random 24-hex-character JavaScript filename, and spawned it detached with 23.254.164.123:443 as the command-and-control argument. The script then deleted itself with fs.rmSync(__filename, { force: true }).

StepSecurity tested the behavior in a controlled GitHub Actions workflow using Harden Runner in block mode. Harden Runner blocked the outbound connection to 23.254.164.92:8000, so the stage-two payload was not downloaded, no detached child process was spawned, and the stage-two command-and-control address was not contacted. The article also lists indicators of compromise, including easy-day-js@1.11.22, easy-day-js@1.11.21, npm maintainer sergey2016, setup.cjs, the two IP addresses, the stage-two URL, temporary files, and the postinstall script.

Overview

On June 17, 2026, an attacker compromised the @mastra npm organization and added the dependency easy-day-js across more than 140 packages in the Mastra AI framework ecosystem. The article describes easy-day-js as a typosquat of the popular dayjs date library. Packages with combined weekly downloads exceeding 1.1 million were exposed, and the article states that anyone who installed an @mastra package that day should treat the environment as compromised.

The article says the incident was ongoing at the time of publication. It also says additional @mastra packages were continuing to be compromised and that the issue was responsibly disclosed in the Mastra GitHub repository at https://github.com/mastra-ai/mastra/issues/18045.

Mastra Context and Significance

Mastra is described as a rapidly growing open-source TypeScript framework for building AI agents, multi-step workflows, and retrieval-augmented generation pipelines. It includes integrations for major LLM providers such as OpenAI, Anthropic, and Google, as well as persistent agent memory, Model Context Protocol servers, vector databases, and cloud deployment targets.

The article emphasizes that Mastra packages are commonly installed in sensitive development and infrastructure environments. Examples of credentials that may exist in those environments include:

  • OPENAI_API_KEY, ANTHROPIC_API_KEY, and GOOGLE_API_KEY
  • AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, and AZURE_TENANT_ID
  • Database connection strings and tokens
  • CI/CD secrets and VCS tokens such as GITHUB_TOKEN and NPM_TOKEN

Because of this, the article characterizes the Mastra ecosystem as a high-value target for supply-chain attackers.

Pre-positioning With a Clean Typosquat

The attack began on June 16, 2026 at 07:05 UTC, when npm user sergey2016 published easy-day-js@1.11.21. The article states that this version was clean and functioned as a copy of the legitimate dayjs library, with no malicious code. Its purpose was to look credible.

The package mirrored several attributes of dayjs, including:

  • 1.11.x version numbering
  • Author metadata associated with iamkun
  • Homepage URL
  • Repository URL
  • MIT license
  • Keywords such as dayjs, date, time, and moment

This clean version was the dependency inserted into the @mastra packages. The dependency was specified as "easy-day-js": "^1.11.21", which allowed npm to resolve the dependency to the latest matching version at install time.

Malicious Version Publication

At 01:01 UTC on June 17, 2026, sergey2016 published easy-day-js@1.11.22. The article says this version was identical to 1.11.21 except for the addition of a file named setup.cjs and a postinstall hook.

The postinstall hook was:

"postinstall": "node setup.cjs --no-warnings"

The article notes that --no-warnings suppresses Node.js runtime warnings that could alert the user. The added setup.cjs file was 4,572 bytes and contained an obfuscated dropper.

@mastra Organization Compromise and Mass Publishing

Eleven minutes after the malicious easy-day-js version was published, the attacker used compromised @mastra organization credentials to begin a mass publishing campaign. The campaign ran from 01:12 UTC to 02:39 UTC, during which more than 140 packages were republished with easy-day-js added as a production dependency.

The article lists this sequence of observed activity:

  • 01:01 UTC: easy-day-js@1.11.22 was published by sergey2016 with the malicious setup.cjs dropper.
  • 01:12 UTC: The first @mastra package was hit, @mastra/schema-compat@1.2.12.
  • 01:15–01:20 UTC: Core packages such as @mastra/core@1.42.1, @mastra/memory@1.20.4, @mastra/server@2.1.1, @mastra/loggers@1.1.3, @mastra/observability@1.14.2, and @mastra/deployer@1.42.1 were published.
  • 01:24 UTC: The top-level framework package mastra@1.13.1 was published.
  • 01:25–01:30 UTC: Infrastructure packages such as @mastra/pg@1.13.1, @mastra/mcp@1.10.1, @mastra/libsql@1.13.1, @mastra/evals@1.3.1, @mastra/rag@2.2.2, and @mastra/datadog@1.2.5 were published.
  • 01:34–02:09 UTC: Database adapters, observability integrations, voice providers, auth plugins, and deployer targets were published.
  • 02:19–02:39 UTC: More packages continued to be compromised, with @mastra/dane last observed at 02:39 UTC.

The article states that the publishing cadence was clearly not manual and was consistent with a script that enumerated the @mastra organization and processed packages in batches.

Install-Time Execution Chain

When a developer installed an affected package, npm resolved easy-day-js to version 1.11.22, because it was the latest patch version matching ^1.11.21. npm then automatically ran the malicious postinstall hook.

The article deobfuscates the dropper logic as performing the following actions:

  • Loading Node.js modules including child_process, crypto, fs, os, and path
  • Setting process.env.NODE_TLS_REJECT_UNAUTHORIZED = '0'
  • Defining the stage-two download URL as https://23.254.164.92:8000/update/49890878
  • Defining the stage-two C2 argument as 23.254.164.123:443
  • Writing the installation path to <tmpdir>/.pkg_history
  • Writing a package marker to <tmpdir>/.pkg_logs
  • Downloading the second-stage payload
  • Writing the payload to a randomly named JavaScript file in the temp directory
  • Spawning the payload as a detached Node.js process with ignored standard I/O and windowsHide: true
  • Calling .unref() so the process could continue after npm exited
  • Deleting setup.cjs with fs.rmSync(__filename, { force: true })

Evasion and Forensic-Relevant Behavior

The article highlights several behaviors that made the dropper effective at avoiding detection or complicating investigation.

First, the dropper disabled TLS certificate verification by setting NODE_TLS_REJECT_UNAUTHORIZED=0. The article says this allowed the dropper to reach the C2 server even if it used a self-signed certificate.

Second, the dropper wrote beacon information before fetching the payload. It wrote the package installation path to <tmpdir>/.pkg_history, which the article says gave the attacker a map of affected machines.

Third, the second-stage payload was spawned as a detached background process using detached: true, stdio: 'ignore', and .unref(). The article says this allowed it to keep running after npm exited, without visible output.

Fourth, the dropper deleted itself after running. The article identifies this as destruction of the main forensic artifact.

Stage-Two Payload

The stage-two payload was fetched at runtime from https://23.254.164.92:8000/update/49890878. The article says it was not embedded in any tarball, so static analysis of package contents could not reach it.

The stage-two process was executed with 23.254.164.123:443 passed as its first argument. The article describes this as almost certainly the command-and-control address. It also states that, given the kinds of credentials commonly present in Mastra environments, the most likely outcome of stage-two execution was environment variable harvesting and exfiltration to that C2.

Runtime Analysis With Harden Runner

StepSecurity tested the malware in a controlled GitHub Actions workflow by installing @mastra/core@1.42.1 with StepSecurity Harden Runner enabled in block mode. Harden Runner was configured to enforce an allowlist of permitted outbound endpoints at the step level.

When the easy-day-js@1.11.22 postinstall hook executed, Harden Runner intercepted and blocked the outbound call to 23.254.164.92:8000. The article states that because the stage-two payload could not be downloaded, the rest of the attack chain did not execute.

The article lists these results from the controlled run:

  • The stage-two payload download was blocked.
  • No payload file was written to the temp directory.
  • The detached child process was never spawned.
  • The stage-two C2 at 23.254.164.123:443 was never contacted.
  • Credential exfiltration was prevented because the chain stopped at the first network call.
  • The block was logged in Harden Runner network events.

The article also says 23.254.164.92 was added to the StepSecurity Harden Runner Global Block List.

Typosquat Characteristics

The article describes easy-day-js as deliberately chosen to impersonate dayjs. It copied metadata from the legitimate library closely enough that the article says routine review of package.json or npm audit would have no obvious signal that something was wrong.

The article identifies the reliable distinguishing features as:

  • The npm maintainer sergey2016
  • The email sergey2016@tutamail.com
  • The presence of setup.cjs in the tarball
  • The postinstall hook in easy-day-js@1.11.22

The legitimate dayjs package had no malicious setup.cjs file and no postinstall hook according to the comparison in the article.

Obfuscation Layers

The article says the setup.cjs dropper used three obfuscation layers.

First, string literals were stored using a custom-alphabet Base64 encoding scheme. The alphabet used lowercase letters first, then uppercase letters, then digits, so standard Base64 decoders would not produce meaningful output without reversing the alphabet mapping.

Second, the decoded 40-element string array had to be rotated 34 positions before an arithmetic checksum equaled 0x4c11d, or 311,581. The article says this design was intended to disrupt static analysis tools that evaluated the array before the rotation ran.

Third, the package name easy-day-js was stored as the byte sequence [0xe5, 0xe1, 0xf3, 0xf9, 0xad, 0xe4, 0xe1, 0xf9, 0xad, 0xea, 0xf3] and written to .pkg_logs as raw binary. The article says this prevented the plaintext string from appearing in files written to disk.

The article also notes that the C2 URL appeared in plaintext within the obfuscated source, which it describes as unusual and possibly indicating that the dropper was prepared under time pressure.

EXEC Executive Overview

The coverage window is dominated by a significant npm supply chain compromise affecting the Mastra AI framework ecosystem. Reporting from StepSecurity describes a coordinated package manipulation incident in which a typosquatted dependency was introduced across more than 140 packages under the @mastra npm organization, exposing downstream development and CI/CD environments to a postinstall-based payload chain.

The incident is notable for its scale, its targeting of an AI framework package namespace, and its use of a staged delivery mechanism designed to execute during package installation. The affected Mastra packages reportedly had combined weekly downloads exceeding 1.1 million, making this a high-impact software supply chain event for organizations consuming JavaScript and AI development dependencies.

01 npm Supply Chain Compromise Targeting Mastra AI Packages

StepSecurity reported that on June 17, 2026, an attacker compromised the @mastra npm organization and added a dependency named easy-day-js across more than 140 Mastra packages. The package name closely resembled the legitimate dayjs library, indicating a typosquatting approach intended to blend into JavaScript dependency trees and exploit developer familiarity with common date/time libraries.

The malicious package was staged across versions. Version 1.11.21 of easy-day-js was reportedly published as a clean bait package, while version 1.11.22 introduced an obfuscated postinstall dropper. This sequencing is consistent with supply chain tradecraft in which an attacker first establishes a benign-looking package history before pushing a malicious update after trust or dependency inclusion has been achieved.

02 Postinstall Dropper and Second-Stage Payload Delivery

The reported malicious behavior centered on npm’s postinstall lifecycle hook. According to StepSecurity, the obfuscated script attempted to download a second-stage payload from https://23.254.164.92:8000/update/49890878, spawn it as a detached Node.js process, and then delete setup.cjs after execution. This design indicates an effort to move beyond package-level code execution into a more persistent or independently running payload while reducing local forensic artifacts tied to the installer script.

The use of a direct IP address and nonstandard port for payload retrieval suggests attacker-controlled infrastructure rather than a commodity package registry-hosted payload. StepSecurity also reported that, during a controlled GitHub Actions test, its Harden Runner in block mode intercepted and blocked the outbound connection to 23.254.164.92:8000, preventing the stage-two payload from being retrieved in that environment.

03 Exposure Scope and Intelligence Significance

The reported scale of the compromised package set places this incident among high-concern npm ecosystem supply chain events, particularly because it affected an AI framework namespace with broad downstream usage. StepSecurity stated that the affected Mastra packages had more than 1.1 million combined weekly downloads, and assessed that anyone who installed an @mastra package on the day of the compromise should consider the environment compromised.

For threat hunters and detection engineers, the reporting provides concrete infrastructure and behavioral details: the package name easy-day-js, malicious version 1.11.22, postinstall execution, retrieval from 23.254.164.92:8000, and execution of a detached Node.js second stage. The incident reinforces ongoing adversary interest in developer ecosystems, CI/CD workflows, and trusted open-source dependency chains as distribution mechanisms for code execution at scale.
