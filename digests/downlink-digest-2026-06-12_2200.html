On uninstall, it used chrome.runtime.setUninstallURL with a google.com/url wrapper carrying ved and usg parameters, disguising the destination as a Google search-result click.

Because Manifest V3 service workers run in the extension’s own origin, the routine cannot delete website IndexedDB, cookies, localStorage, or sessions.

The extensions were tied to three brand domains: tabplugins.com, yowgames.com, and chromewallpaper.com, with chromewallpaper.com redirecting to owhit.com. The article says these domains were ad-monetized using Google ad products, and tabplugins.com also loaded an Advergic Prebid stack via avads.live. Overview

Socket’s Threat Research Team identified a family of Chrome Web Store new-tab “live wallpaper” extensions that the article characterizes as an adware-adjacent potentially unwanted program. The family consists of 152 unique extension IDs built from a shared codebase and distributed across 38 separate Chrome Web Store publisher accounts. The extensions were associated with three brand backends and had a combined total of approximately 105,000 reported installs.

The central issue is that the extensions claimed in Chrome Web Store privacy disclosures that they would not collect or use user data, while linked privacy policies admitted data collection and sharing. The article says the policies describe logging IP addresses, browser type, ISP, date and time stamps, referrers or exit pages, click counts, and other information. The policies also name Google AdSense, Google DoubleClick DART cookies, Google Analytics, and third-party advertising partners.

Traffic Attribution Laundering

A 54-listing subset using the newer tabplugins.com template performed the article’s most significant technical behavior: forged Google organic-search attribution. In the service worker js/bg.js, the extension defined an install URL on tabplugins.com containing utm_source=google&utm_medium=organic. On install, the worker opened that URL and also opened the extension’s own newtab.html.

The article explains that this caused extension-driven traffic to appear in analytics as if it came from Google organic search. It emphasizes that the visit was not the result of a person searching Google and clicking a normal result; it was the extension opening a tab on install. The article describes this as manufacturing a valuable “organic search” traffic signal.

The same subset also set an uninstall URL through chrome.runtime.setUninstallURL. The uninstall URL used a google.com/url redirect wrapper pointing to tabplugins.com/live-wallpaper/ and included ved and usg parameters. The article states that this disguised the uninstall ping as a Google search-result click and allowed the operator to detect uninstall events while laundering the traffic as Google activity.

IndexedDB Wipe Behavior

Every analyzed member of the family shipped an IndexedDB enumerate-and-delete routine in the service worker. On every service-worker start, the code called indexedDB.databases(), iterated through visible databases, called indexedDB.deleteDatabase(db.name), and logged Deleted IndexedDB database:. The article identifies this literal log string and routine as the family’s most reliable fingerprint.

The article carefully limits the impact of this behavior. Because a Manifest V3 background service worker runs in the extension’s own chrome-extension://<id> origin, the IndexedDB routine can only affect databases in that extension origin. It cannot touch websites’ IndexedDB, cookies, localStorage, or sessions.

In the analyzed build, the routine is described as inert because the extensions do not use IndexedDB for their own state. Instead, the article says they store saved shortcuts, background mode, custom wallpaper, and last-image index in localStorage. The article still treats the routine as notable because it is undisclosed, copied into the whole family, and has no legitimate purpose in a wallpaper app.

Other Extension Behaviors and Code Quality Issues

The article states that the new-tab search box calls chrome.search.query without overriding the search engine, so it uses the user’s existing default engine and does not hijack search. However, the search permission is described as the cleanest install-time tell for the family. The article also notes that saved shortcut domains are sent to Google through s2/favicons?domain= requests on every new tab.

Two lower-severity issues were described in the new-tab code and package layout. The shortcut renderer injects user-saved shortcut name and url values into the DOM through unescaped template strings, which the article calls self-XSS because a user would have to save their own malicious shortcut. The image-wallpaper mode references wallpapers/1.jpg through wallpapers/10.jpg, but the packages do not include a wallpapers/ directory, so image mode is broken.

The article also notes evidence of careless mass production. Three tabplugins.com extensions shipped a bg.js that did not parse because a closing quote appeared before the query string in the install URL. In those three cases, the syntax error prevented the install navigation, uninstall tracking, and IndexedDB wipe from registering, although the extensions still installed, overrode the new tab, and requested the search permission.

Scale and Distribution

Across the full dataset, Socket collected 152 unique extension IDs. It downloaded and SHA-256-verified the bg.js file for 141 of them, and all 141 resolved to live Chrome Web Store listings. The remaining 11 were already delisted from the Chrome Web Store, with the update endpoint returning HTTP 204.

The family was split across three brand backends:

  • tabplugins.com: 109 analyzed extensions, including the newer template that added forged Google organic attribution and a cloaked google.com/url uninstall redirect.
  • yowgames.com: 19 extensions using the same core without the forged Google attribution.
  • chromewallpaper.com: 13 extensions structurally identical to the yowgames.com variant.

The article says the same template was spread across 38 distinct Chrome Web Store publisher accounts, including accounts such as ZainAhamed1994, hirakiranpk, yowgames.com, tabplugins.com, and others listed in the IOC section. It describes this fragmentation across publisher identities as a takedown-resistance tactic because removing one account would leave the rest of the network live.

Infrastructure and Monetization

The article describes two distinct operator infrastructure clusters tied by a shared extension template rather than shared hosting. The yowgames.com, chromewallpaper.com, and owhit.com cluster used the same Cloudflare name-server pair, journey.ns.cloudflare.com and tim.ns.cloudflare.com, and was registered through Spaceship. chromewallpaper.com acted as a redirector and issued an HTTP 301 to owhit.com.

The tabplugins.com cluster used a separate Cloudflare name-server pair, fatima.ns.cloudflare.com and ned.ns.cloudflare.com, and was registered through Hostinger. The article says its origin was exposed on Hostinger IPs 147.79.120.202 and 92.112.198.22 rather than being fully proxied. All operator-controlled domains were registered behind WHOIS privacy.

The monetization described in the article was funnel-based rather than injection-based. The extensions drove traffic to ad-monetized brand pages through forced install tabs, in-page “More Extensions” and uninstall-guide links, and forged-organic attribution. The article says tabplugins.com loaded https://avads.live/s/av-tabplugins.js, an Advergic Prebid header-bidding bundle that included Google Ad Manager network codes 23301900962,23324153939, AppNexus/Xandr, PixFuture, SmileWanted, a full-screen interstitial ad slot, Google Analytics 4 property G-906NQ2GLXR, and FOU Analytics.

For the yowgames.com cluster, live retrieval was blocked by a Cloudflare bot wall from the researchers’ vantage, so archived copies were examined. The article says yowgames.com and owhit.com embedded Google AdSense and googlesyndication.com directly, using publisher IDs ca-pub-2685573472598175 and ca-pub-6596604135510481, respectively. Their Google Analytics 4 properties were listed as G-YJWVP0Q1KW and G-6V3WECV225.

Attribution Statements

The article characterizes the operation as financially motivated commercial adware and traffic-attribution fraud run by one or more small freelance teams. It explicitly states that the researchers cannot tie the operation to a specific country with confidence. It notes circumstantial signals that loosely point toward Turkey, including the owhit.com contact name “Saniye Yıldız,” the email yahyagazi06@gmail.com, and Turkish-reading publisher contact emails such as ferhatbadem831@gmail.com and keremsopar@gmail.com.

The article emphasizes that these are possibilities to investigate rather than attribution that can be stated with confidence. It says the listed contact may be a pseudonym and that no GitHub, freelance-marketplace, or social-media portfolio linked the publisher handles to one another or to a real identity. It also says no prior public reporting was found on the domains or handles.

Privacy Disclosure Conflict

The article identifies the clearest policy issue as a contradiction between Chrome Web Store privacy disclosures and the operators’ linked privacy policies. The Chrome Web Store Privacy practices tab for the listings said the developer would not collect or use user data, that data was not being sold to third parties, and that data was not being used or transferred for unrelated purposes. The article says identical “will not collect or use your data” disclosures appeared on all 141 live listings the researchers resolved.

The linked privacy policies stated the opposite. According to the article, they said log files collect data when visitors use the websites or extensions, including IP addresses, browser type, ISP, timestamps, referrers or exit pages, and possibly click counts. The policies also described tracking user movement on the website and extension and named Google AdSense, Google DoubleClick DART cookies, Google Analytics, and third-party advertising partners.

The article cites Chrome Web Store policy language requiring privacy fields to be up to date and accurate and warning that contradictions between dashboard disclosures, privacy policies, and item behavior may violate policy. It says the documented situation is a three-way mismatch between store disclosure, linked privacy policy, and observed phone-home behavior. The article states that the install and uninstall telemetry corroborates that the extensions phone home.

Impact

The article says the exposure is the family’s full install base, on the order of 100,000 users. The harm is described as privacy and measurement integrity rather than credential theft or remote code execution. It says affected users are enrolled in deceptive traffic measurement and undisclosed telemetry.

Specific impacts listed in the article include:

  • Users’ install and uninstall events are reported to the operator.
  • The operator’s privacy policy says this activity is associated with IP, ISP, referrer, and click data and shared with Google AdSense, DoubleClick, Analytics, and third-party ad partners.
  • Extension-driven visits are presented to advertisers and affiliates as Google organic search traffic through forged UTM tags and the google.com/url redirect.
  • Saved shortcut domains leak to Google through favicon requests on every new tab.
  • Every install ships an undisclosed IndexedDB wipe routine that runs on service-worker start.

Recommended Actions

For users, the article recommends removing any new-tab “live wallpaper” extension sourced from tabplugins.com, yowgames.com, or chromewallpaper.com. It also advises treating new-tab override extensions that request the search permission as something to scrutinize. After removal, users are advised to confirm that Chrome’s new tab and default search engine are restored to their preferences.

For developers, the article recommends ensuring that Chrome Web Store privacy disclosures match published privacy policies exactly. It warns against fabricating UTM attribution or reusing Google ved and usg redirect tokens to disguise traffic. It also says developers should not ship unexplained anti-forensic boilerplate such as indiscriminate IndexedDB wipe routines.

For security teams, the article recommends hunting for family fingerprints rather than individual extension IDs because the family is mass-produced and IDs rotate. Reliable signals listed include the literal string Deleted IndexedDB database:, an indexedDB.databases() enumerate-and-delete loop in bg.js, a setUninstallURL pointed at a google.com/url wrapper, and an onInstalled handler opening a tab with utm_source=google&utm_medium=organic. The article also recommends blocking tabplugins.com, yowgames.com, chromewallpaper.com, and owhit.com, and alerting on new-tab override extensions that request the search permission and originate from those domains.

This is a specific technical finding about a large Chrome Web Store extension network affecting roughly 100,000 users, with concrete IOCs, hunting guidance, and policy/privacy implications. It is not an emergency like active exploitation of critical enterprise software, but it is broadly relevant to security teams monitoring browser extensions and adware/PUP activity.

EXEC Executive Overview

This digest covers a browser-extension monetization cluster reported by Socket’s Threat Research Team involving 152 Chrome Web Store live wallpaper and new-tab extensions. The activity centers on deceptive data disclosures, traffic attribution manipulation, and ad-monetized redirect behavior rather than traditional credential theft, malware delivery, or page-level ad injection.

The reporting is notable for the scale of the shared extension ecosystem: approximately 105,000 reported installs across 38 publisher accounts. While the extensions presented themselves as benign customization tools, Socket found inconsistencies between Chrome Web Store data-use claims and linked privacy policy language describing collection of IP addresses, ISP information, click counts, referrers, and data sharing with advertising partners.

01 Chrome Extension Monetization and Traffic Manipulation

Socket identified 152 Chrome live wallpaper extensions built from a shared codebase and distributed through multiple publisher accounts. All resolved live listings reportedly claimed in the Chrome Web Store that they did not collect or use user data, while the associated privacy policies stated that telemetry such as IP address, ISP, click counts, and referrer data may be logged and shared with ad partners.

A subset of 54 listings tied to tabplugins.com allegedly forged Google organic-search attribution during installation. Socket reported that these extensions used a google.com/url uninstall redirect to make extension-driven traffic appear as though it originated from Google Search, obscuring the true source of visits and potentially improving the perceived legitimacy or value of the traffic.

The extensions monetized users primarily by forcing or funneling them toward ad-monetized brand pages, including tabplugins.com, yowgames.com, and chromewallpaper.com. The article emphasizes that the behavior did not involve injecting advertisements into pages users browsed; instead, the monetization model depended on controlling new-tab or extension-driven navigation flows.

02 Deceptive Disclosure and Dormant Code Artifacts

The campaign highlights a recurring browser-extension risk pattern: low-friction installation of consumer-facing customization utilities paired with opaque or contradictory privacy representations. In this case, the Chrome Web Store listings and linked privacy policies reportedly described user-data handling differently, creating a mismatch between public store assurances and external policy disclosures.

Socket also observed that every analyzed extension shipped a service-worker routine designed to enumerate and delete IndexedDB entries. The report notes that this routine appeared inert in the analyzed build because the extensions did not store their own state in IndexedDB, but its consistent presence across the shared codebase was still part of the technical footprint connecting the listings.

Overall, the reporting describes an ecosystem of Chrome extensions optimized for traffic acquisition, attribution laundering, and advertising monetization. For security teams tracking browser-extension abuse, the cluster illustrates how seemingly benign new-tab experiences can be used to shape user navigation, collect browsing-adjacent telemetry, and generate monetized traffic at scale.
